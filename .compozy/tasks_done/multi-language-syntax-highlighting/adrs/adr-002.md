# ADR-002: Default-on trustworthy code recognition

## Status

Accepted

## Date

2026-07-13

## Context

Developers use Kitten to read streamed agent output and hand-off summaries. They need to identify code quickly without risking changed source when a language cannot be enhanced. The approved idea defines a broad, practical V1 language set; clarification established that immediate recognition is the primary value, visibly labelled plaintext is the required fallback, the feature should be enabled by default once verified, and post-launch developer scanning clarity is the leading success signal.

## Decision

Ship trustworthy multi-language code recognition as a default-on Kitten capability once the documented V1 language set meets its quality bar. The experience will make code blocks visually distinct in live conversations and hand-off summaries, preserve their language label and source when no enhancement is available, and retain non-color code-block cues so readability does not depend on color perception alone.

## Alternatives Considered

### Alternative 1: Opt-in preview

- **Description**: Make the feature available only when users enable it manually.
- **Pros**: Limits initial exposure and allows gradual feedback collection.
- **Cons**: Hides a core reading improvement from users who would benefit and weakens learning from normal workflows.
- **Why rejected**: The feature is valuable as a reliable default, not as an advanced preference, once it meets the release standard.

### Alternative 2: Minimal language set

- **Description**: Limit the initial experience to the most common languages.
- **Pros**: Reduces the visible promise.
- **Cons**: Fails the approved core-plus-common-formats scope and makes mixed-language agent output inconsistent.
- **Why rejected**: The documented V1 set is bounded and better reflects real agent-assisted coding work.

### Alternative 3: Color-only best effort

- **Description**: Use visual coloring wherever possible without a strict fallback contract.
- **Pros**: May appear to improve more blocks quickly.
- **Cons**: Can mislead users, exclude limited-color terminals, and undermine trust in copied source.
- **Why rejected**: Code recognition must enhance, never alter or obscure, the source users read and copy.

## Consequences

### Positive

- All Kitten users receive a more scannable code-reading experience without configuration.
- The same reading expectation applies to live work and hand-off review.
- Explicit plaintext fallback preserves trust when coverage is incomplete.

### Negative

- The feature cannot be released merely because some languages look improved; the full documented user promise must be met.
- Niche language demand may require revisiting the V1 coverage after real usage feedback.

### Risks

- **Users over-rely on color** — retain labels, code-block boundaries, and readable source as independent cues.
- **A visibly inconsistent experience harms trust** — release only when the documented V1 experience is consistent for installed users.
- **Niche-language expectations outpace demand** — review feedback after release before expanding coverage further.

## Implementation Notes

Product execution must preserve code source and provide the same user-visible behavior across Kitten's Markdown-reading surfaces.
