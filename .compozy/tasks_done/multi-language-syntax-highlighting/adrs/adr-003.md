# ADR-003: Static parser manifest with pre-initialization registration

## Status

Accepted

## Date

2026-07-13

## Context

OpenTUI 0.4.3 creates one process-wide Tree-sitter client and registers its default parser set only when that client initializes. Its default Markdown injection map recognizes only a narrow set of fenced-code labels. Kitten must add Rust, Go, OCaml, ReScript, JSON, Bash, Python, Markdown, and diff-related file extensions without introducing per-surface mappings, runtime downloads, or behavior that changes source text.

The compiled Kitten binary also needs every parser asset embedded and resolvable before a code renderable creates the client. The selected product behavior requires plaintext fallback plus content-free diagnostics if a declared capability cannot load.

## Decision

Create one typed, static manifest in `src/ui/syntaxParsers.ts`. It owns canonical filetypes, aliases, statically imported parser/query assets, the Markdown injection override, fixtures, and content-free diagnostic metadata. The manifest registers its `FiletypeParserOptions` through OpenTUI's `addDefaultParsers()` before the shared client initializes.

Vendor the necessary version-pinned WASM and highlight-query assets under `src/ui/syntax-assets/`; this directory is necessary because OpenTUI does not export its installed parser assets as public package entry points. Use Bun `with { type: "file" }` static imports so source runs resolve local paths and `bun build --compile` embeds the same assets. Call the idempotent registration function during boot after worker configuration and from Markdown/diff render entry points as a direct-mount safety guard.

Override the `markdown` parser configuration with its existing inline mapping plus the approved fence-label map. Keep default parsers for their native filetypes. Unknown labels, absent parser assets, warnings, and parser failures preserve the original source as plaintext and emit only structured, content-free diagnostics.

## Alternatives Considered

### Alternative 1: Separate Markdown-only highlighter

- **Description**: Add a second tokenization/rendering engine beside OpenTUI Tree-sitter.
- **Pros**: May provide a larger prebuilt language catalog.
- **Cons**: Duplicates theme, streaming, selection, and compiled-asset responsibilities; Markdown and diff behavior can diverge.
- **Why rejected**: The existing OpenTUI parser pipeline already supports custom pre-initialization parser registration.

### Alternative 2: Patch OpenTUI's installed parser assets

- **Description**: Modify files under `node_modules/@opentui/core` or depend on undocumented internal asset paths.
- **Pros**: Appears to reuse the current default Markdown configuration.
- **Cons**: Is not reproducible across installs, upgrades, or compiled artifacts.
- **Why rejected**: Kitten must own a pinned, reviewable asset and alias contract.

### Alternative 3: Runtime download or user-installed grammars

- **Description**: Fetch missing parsers on demand or let users provide them.
- **Pros**: Avoids carrying assets in the release binary.
- **Cons**: Adds network, integrity, startup, and support risks; conflicts with the default-on trustworthy experience.
- **Why rejected**: V1 must be offline, deterministic, and release-verifiable.

## Consequences

### Positive

- One manifest keeps Markdown injections, diff extensions, asset packaging, and tests aligned.
- Static imports make source and standalone binaries use the same asset set.
- Parser failures degrade safely without exposing source content in diagnostics.

### Negative

- Kitten owns asset provenance, license review, upgrades, and ReScript grammar compatibility.
- The Markdown override must preserve the existing inline and fenced-content behavior.

### Risks

- **Registration happens after client initialization** — call registration before boot rendering and retain entry-point guards.
- **An asset is omitted from the binary** — require compiled-artifact coverage for every canonical capability.
- **A grammar lacks compatible highlights or license terms** — block that language from the supported manifest and retain plaintext fallback.
- **OpenTUI changes its parser contract** — pin the dependency and revalidate the manifest on each upgrade.

## Implementation Notes

Use OpenTUI's public `addDefaultParsers()` API and `FiletypeParserOptions`; do not import its unexported asset paths. Preserve `MARKDOWN_STREAMING`, existing Markdown normalization, and the existing diff `filetypeFor()` no-guess rule.
