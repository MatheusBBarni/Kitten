# ADR-001: Ambient per-agent headroom gauge over an on-demand `/usage` overlay

## Status

Accepted

## Date

2026-07-09

## Context

Kitten runs two AI coding agents (Claude Code + Codex) side by side over the Agent Client Protocol (ACP) and lets the user hand a live coding task between them.
The original request was a `/usage` slash command that opens an on-demand overlay showing each agent's context-window fill and token counts, primarily to help decide *when* to hand off.

Three facts reshaped the decision during the idea's council debate:

- ACP already emits a `usage_update` session notification (`used`/`size` context tokens), but Kitten currently discards it at `src/agent/acpTranslate.ts:57` (`return null`). The context-fill signal is one already-dropped event away.
- The feature's stated job is a *timing* signal - "is it time to hand off?" - yet an overlay you must summon with `/usage` only helps once the user already suspects the answer. A pull surface answers a question you are already asking; a timing cue has to be pushed.
- ACP `usage_update` is optional and only recently stabilized (2026-06-05). Real adapters lag: Zed's Codex ACP adapter emits no usage today, so one agent may report nothing. The design must degrade honestly rather than fabricate a number.

The advisory council (pragmatic-engineer, architect-advisor, product-mind, devils-advocate, the-thinker) converged: place the signal where the handoff decision is made, keep it honest, and defer everything that does not serve the primary job.

## Decision

V1 surfaces per-agent context **headroom** as an **always-on ambient signal**, not an on-demand overlay:

1. Translate `usage_update` into a first-class, absence-aware domain value derived **once** in the pure reducer/selector layer: `{ status: 'unknown' } | { status: 'known', used, size, percent }`. Selectors project a **primitive** (rounded percent or the discriminant) so the store's per-agent re-render isolation holds.
2. Render a headroom segment on each agent's chip in the existing `StatusStrip` (both agents, honestly side by side).
3. Show the **target** agent's headroom inside the existing `Ctrl+T` handoff preview, so custody and headroom sit together at the moment of decision.
4. Scope V1 to **context fill only**. Absent data renders as a dash or an explicitly-approximate raw token count - never a fabricated percentage.
5. **Instrument first:** log `usage_update` arrivals from both adapters before committing UI, since no code path has yet verified either adapter actually emits it.

The standalone `/usage` overlay, the per-turn token breakdown, cost, a slash-command parser, and any active handoff recommendation are deferred.

## Alternatives Considered

### Alternative 1: On-demand `/usage` overlay (the original request)

- **Description**: A `/usage` slash command opens a modal overlay listing both agents' context% and token breakdown; Esc dismisses.
- **Pros**: Matches the initial ask; room for detail; reuses the established help/approval overlay pattern.
- **Cons**: A timing signal you must summon is a lookup, not a signal - it arrives after the pain. Requires Kitten's first slash-command parser plus a new overlay slot and modal key-standdown machinery, all for the least-seen number. Does nothing to move the handoff earlier.
- **Why rejected**: Architecturally the most machinery for the surface least likely to change behavior; four of five advisors judged it mismatched to its own job, and the lone defender partially conceded.

### Alternative 2: Focused-agent headroom chip only

- **Description**: Show only the currently focused agent's headroom in its status chip; no side-by-side, no handoff-preview integration.
- **Pros**: Absolute-minimum honest signal; validates the ACP data path in a single leaf; ships fastest.
- **Cons**: Sacrifices the two-agent side-by-side comparison, which is Kitten's genuine differentiator.
- **Why rejected**: Kept as the guaranteed-shippable *core inside* the accepted design, not as the whole of V1 - the side-by-side view is what no single-agent tool can offer.

### Alternative 3: Active custody-advisor nudge

- **Description**: Kitten proactively suggests a handoff when the focused agent crosses a headroom threshold and the other agent has room.
- **Pros**: Turns a passive gauge into an active recommendation; unique; compounds as cost and task-fit become future inputs.
- **Cons**: Highest trust risk - a wrong nudge that drives a costly handoff erodes credibility, and the underlying percentage's denominator is not fully trustworthy yet.
- **Why rejected for V1**: Premature on unvalidated data; adopted as the explicit V2+ stretch direction.

## Consequences

### Positive

- The signal lives where the decision is made (status strip + handoff preview), so it can actually move the handoff earlier instead of merely confirming it.
- Cheapest implementation given the codebase: both targets are already-mounted, already-subscribed components; the context signal is one already-dropped ACP event.
- Honest degradation: absent Codex data reads as "unknown," never a fabricated value, consistent with Kitten's under-report-never-fabricate invariants.
- The derive-once domain value makes the deferred overlay, token breakdown, and cost purely additive leaves later.

### Negative

- Flips the user's original overlay-only framing; the `/usage` command is deferred rather than delivered in V1.
- An always-on gauge is unforgiving: if an adapter never emits usage, the strip shows a persistent "unknown," which is more visible than a hidden overlay would have been.

### Risks

- **Denominator trust**: the ACP-reported context `size` may not equal the effective limit under internal auto-compaction or reserved output headroom, so a raw percentage can mislead. Mitigation: frame as "headroom," consider a segmented bar (used vs reserved), tune thresholds to the handoff decision, and avoid false precision.
- **Emission unverified**: neither adapter is confirmed to emit `usage_update`. Mitigation: instrument-first spike before UI work; render honest "unknown" until confirmed.
- **Re-render leakage**: selecting the raw usage object would fail `Object.is` every tick. Mitigation: project a primitive from the selector so a chip re-renders only when its displayed value crosses.

## Implementation Notes

- Translate `usage_update` in `src/agent/acpTranslate.ts` into a new `DomainSessionEvent` kind instead of `null`.
- Add an absence-aware `usage` field to `SessionState` (`src/core/types.ts`) and a reducer case (the `assertNever` guard will force it); initialize as `{ status: 'unknown' }` in `createSessionState`.
- Add a memoized per-agent `selectSessionUsage(sessionId)` in `src/store/selectors.ts` that returns a primitive/discriminant.
- Render in `src/ui/StatusStrip.tsx` (`AgentStatusChip`) and `src/ui/HandoffPreview.tsx` (target agent).
- Extend the telemetry recorder (`src/telemetry`) to log usage arrivals and correlate with hand-off events.
