# ADR-005: Measure Prompt Recall Through Opt-In Content-Free Telemetry

## Status

Accepted

## Date

2026-07-12

## Context

The PRD measures recall adoption and edited resends, but prompt text is sensitive and Kitten telemetry is opt-in and content-free. The existing recorder writes only fixed event types, anonymous run references, session references, and coarse numeric fields.

## Decision

Extend the existing opt-in recorder with prompt-history eligibility, recall, clear, and edited-resend events. Events may include the anonymous session reference and owning agent session, but never prompt text, hashes, exact length, or history contents.

## Alternatives Considered

### Alternative 1: Store prompt text or hashes in telemetry

- **Description**: Emit text-derived data to associate recalls with resends.
- **Pros**: Richer analysis.
- **Cons**: Violates the product’s privacy boundary and risks sensitive-data retention.
- **Why rejected**: No product metric justifies exporting prompt-derived content.

### Alternative 2: Defer all instrumentation

- **Description**: Use only manual usability sessions.
- **Pros**: No recorder changes.
- **Cons**: Cannot measure production adoption or edited-resend behavior.
- **Why rejected**: The user selected local, opt-in counters for V1 validation.

## Consequences

### Positive

- Measures the PRD’s value hypothesis without recording prompt content.
- Reuses established opt-in gating, local JSONL output, and memory-sink tests.

### Negative

- Metrics show behavior, not the semantic quality of prompts or edits.

### Risks

- **Accidental content fields**: telemetry type and tests must reject text-bearing properties.
- **Misleading denominator**: emit eligibility once per session after its second composer submission.

## Implementation Notes

- The recorder maintains only per-run counters needed to decide when eligibility occurs.
