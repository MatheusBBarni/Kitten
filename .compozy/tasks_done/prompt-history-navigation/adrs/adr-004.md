# ADR-004: Gate Recall with OpenTUI Cursor-Movement Results

## Status

Accepted

## Date

2026-07-12

## Context

The prompt composer is a wrapping multiline `TextareaRenderable`. Up and Down must retain ordinary vertical cursor behavior whenever movement is possible, while the slash menu already consumes arrows when it is open. Coordinate heuristics are fragile around wrapping and terminal resize.

## Decision

In `PromptEditor`, keep slash-menu handling first. For an unmodified Up or Down in the plain composer, call the installed OpenTUI textarea’s corresponding cursor-movement method. If it reports movement, prevent the default and keep the editor movement. If it reports no movement at the boundary, prevent the default and request the next prompt-history state from `ControllerActions`. Replace the textarea contents only when that action returns a history result.

## Alternatives Considered

### Alternative 1: Infer boundaries from logical line counts and cursor offsets

- **Description**: Compare cursor offsets or logical line numbers in the component.
- **Pros**: Avoids calling editor movement methods directly.
- **Cons**: Can disagree with wrapped visual rows, selections, or future editor layout behavior.
- **Why rejected**: OpenTUI already exposes the authoritative movement result.

### Alternative 2: Always intercept Up and Down for history

- **Description**: Treat every arrow key as history navigation while the composer has focus.
- **Pros**: Simpler dispatch path.
- **Cons**: Breaks multiline editing and conflicts with the PRD’s boundary-aware behavior.
- **Why rejected**: It violates the primary interaction constraint.

## Consequences

### Positive

- Wrapped and multiline cursor behavior stays native to the installed editor.
- Recall activates only at an actual editor boundary.
- Menu precedence remains unchanged.

### Negative

- The component deliberately owns unmodified vertical navigation rather than relying on the textarea’s default dispatch.

### Risks

- **Editor API drift**: typecheck and focused interaction tests guard the current OpenTUI surface.
- **Accidental draft replacement**: no draft snapshot is retained; history replacement happens only after the editor reports no movement.

## Implementation Notes

- Use `setText` for a recalled result so the replacement is complete and does not retain a hidden pre-browse undo path.
