# ADR-001: Scope the provider-independent clarification bridge as a live-generation V1

## Status

Accepted

## Date

2026-07-14

## Context

Kitten already owns a protocol-free clarification model, an identity-guarded modal queue, lifecycle cleanup, and content-free telemetry. Native ACP elicitation remains correctly fail-closed because no production adapter recipe is verified. Issue #10 proposes a Kitten-owned local MCP `ask_user` bridge so eligible providers can request structured operator input without depending on native ACP support.

The new bridge creates a cross-process trust and lifecycle boundary. It must not accept a caller-supplied session identity, leak question or answer content, reorder user-configured MCP servers, misroute concurrent answers, or leave a provider call hanging. A process crash also cannot truthfully offer durable exactly-once delivery without a journal and recovery protocol.

## Decision

Ship a narrow, provider-independent V1: one Kitten-owned local `ask_user` contract injected alongside user-configured MCP servers for every eligible session, with Codex as the end-to-end proof path. Reuse the existing normalized clarification payload, modal queue, and coordinator; retain native ACP elicitation as a separate fail-closed fast path.

V1 supports stable question IDs; optional header/context; single-select, multi-select, custom answer, and skip; and explicit submitted, skipped, timed-out, and cancelled outcomes. The bridge owns an unguessable per-session capability, derives owner and generation server-side, applies schema, namespace, payload, choice, and queue bounds, and makes duplicate or late events inert.

The reliability guarantee is deliberately bounded: at most one accepted settlement per capability, request ID, and live session generation. Replacement, close, provider failure, bridge disconnect, and shutdown cancel or otherwise terminally settle the live request. A host-process crash is not durable replay; a future requirement for cross-crash delivery requires a separately owned journal, idempotency keys, and reconciliation.

## Alternatives Considered

### Alternative 1: Codex-only direct integration

- **Description**: Wire a Codex-specific question path directly to the existing dialog.
- **Pros**: Fastest visible proof and lower initial surface area.
- **Cons**: Creates provider-specific behavior, fails the selected provider-independent launch promise, and accumulates migration debt.
- **Why rejected**: The user selected one bridge contract for every eligible provider, with Codex as proof rather than as a product boundary.

### Alternative 2: Durable generalized interaction bus before the MCP bridge

- **Description**: Build persistence, replay, recovery, and generic tool hosting before exposing `ask_user`.
- **Pros**: Can eventually support cross-crash delivery and many interaction tools.
- **Cons**: Delays validation, greatly expands scope, and turns a focused operator unblock into platform work.
- **Why rejected**: Durable recovery is valuable but unsupported by present demand evidence; it is a V2+ architectural commitment.

### Alternative 3: Native ACP elicitation only

- **Description**: Wait for verified ACP adapter recipes rather than adding a bridge.
- **Pros**: No MCP runtime or new cross-process route.
- **Cons**: Leaves the production capability gap unresolved and excludes providers that do not implement elicitation.
- **Why rejected**: It contradicts the provider-independent goal while the verified ACP allowlist remains empty.

## Consequences

### Positive

- Gives operators one structured clarification experience across eligible providers.
- Reuses the existing UI, queueing, lifecycle, and privacy-safe telemetry investments.
- Preserves ACP and MCP as thin transport edges around a protocol-free core.
- Establishes testable ownership, privacy, and teardown invariants before broader rollout.

### Negative

- Requires a new local MCP runtime and a same-turn transport test harness.
- V1 cannot promise recovery or exactly-once delivery across a host-process crash.
- Security controls and lifecycle tests are non-negotiable launch work, not polish.

### Risks

- **Cross-session spoofing or answer leakage**: mitigate with bridge-owned capabilities, generation fencing, and fail-closed validation.
- **Stale or duplicate settlement**: mitigate with request namespacing, one terminal state, and late-event rejection.
- **Operator or provider blocking**: mitigate with bounded queues, timeout, cancellation, and teardown on all lifecycle exits.
- **Content exposure**: mitigate with untrusted rendering and telemetry restricted to fixed outcomes and coarse duration buckets.

## Implementation Notes

Technical design belongs in the later TechSpec. It must prove deterministic composition with user MCP servers, all terminal paths, and exact session ownership through fake-transport end-to-end tests.
