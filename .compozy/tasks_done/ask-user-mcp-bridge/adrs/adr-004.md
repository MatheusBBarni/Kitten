# ADR-004: Define a bounded multi-field contract with a Kitten-owned five-minute timeout

## Status

Accepted

## Date

2026-07-14

## Context

The PRD requires structured choices, custom answers, explicit skip, timeout, cancellation, and same-run continuation. Kitten’s current clarification model supports several fields in one dialog but has only `answered` and `cancelled` terminal outcomes. RepoPrompt CE shows a more expressive multi-question contract, including stable field IDs, optional title/context, and structured result serialization.

The bridge must give agents a predictable, bounded tool contract while keeping the operator experience focused and avoiding agent-controlled waiting behavior.

## Decision

Expose one `ask_user` form per MCP tool call. A form contains one to ten stable-ID fields, optional form title and context, optional per-field header and context, bounded options, single- or multi-select behavior, and optional custom responses. The bridge normalizes the form into Kitten’s protocol-free clarification payload before projecting it to the UI.

Extend the protocol-free outcome model to distinguish `submitted`, `skipped`, `timed_out`, and `cancelled`. Submitted answers preserve selected option IDs and custom text separately so both MCP response serialization and native ACP translation remain faithful. Skip applies to the whole form; cancellation remains a distinct lifecycle/UI outcome.

Kitten owns one fixed, configurable timeout with a default of five minutes. The MCP input schema has no timeout override. The timeout starts when Kitten accepts the form, applies while queued or active, and settles the exact request once. Form and field limits are fixed in code for V1: at most ten fields, twenty options per choice field, and 4 KiB per user-visible text value.

## Alternatives Considered

### Alternative 1: One question per tool call

- **Description**: Limit each call to one field.
- **Pros**: Smallest schema and simplest dialog behavior.
- **Cons**: Forces repeated interruptions for related decisions despite the existing multi-field UI.
- **Why rejected**: The selected V1 should reuse Kitten’s structured form capability while keeping a hard upper bound.

### Alternative 2: Agent-controlled timeout per call

- **Description**: Accept an optional timeout supplied by the MCP caller.
- **Pros**: Lets callers tune their own wait duration.
- **Cons**: Lets an untrusted agent extend operator blocking and makes product behavior unpredictable.
- **Why rejected**: The user selected a single Kitten-owned fixed timeout.

### Alternative 3: Flatten answers to strings only

- **Description**: Return only one text value per field.
- **Pros**: Minimal response shape.
- **Cons**: Loses selection versus custom-answer semantics and cannot faithfully represent multi-select answers.
- **Why rejected**: V1 requires structured answers and valid translation back to native ACP forms.

## Consequences

### Positive

- Gives agents one stable, expressive contract across eligible providers.
- Reuses one modal for related decisions and avoids agent-controlled long waits.
- Preserves enough answer structure for provider-neutral serialization.

### Negative

- Requires extending existing core payload, outcome, UI, telemetry, and adapter contracts.
- Fixed limits intentionally reject very large forms rather than silently truncating them.

### Risks

- **Oversized or malicious forms**: enforce all limits before queueing and return a structured tool error.
- **Timeout racing with submission**: use the coordinator’s exact request identity as the only settlement authority.
- **Inconsistent native and MCP semantics**: normalize both at the protocol-free payload and outcome boundary.

## Implementation Notes

The fixed timeout is configuration-owned, never a caller-provided MCP field. Update content-free telemetry only with the expanded terminal enum and coarse duration bucket.
