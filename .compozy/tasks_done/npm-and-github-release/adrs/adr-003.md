# ADR-003: One consolidated release workflow, driven by release-please outputs rather than a `release: published` trigger

## Status

Accepted

## Date

2026-07-09

## Context

ADR-001 assumed the release cut and the publish would stay in two workflows: release-please would create the tag and GitHub Release, and the existing `.github/workflows/release.yml` (`on: release: published`) would build and publish in reaction.

That assumption does not hold with GitHub's default token.
A Release created by a workflow using the built-in `GITHUB_TOKEN` does not trigger other workflows, by design, to prevent recursive workflow runs.
So a release-please Release would silently fail to fire `release.yml`, and nothing would build or publish.

There are two ways out: give release-please an elevated credential (a GitHub App token or a PAT) so its Release does trigger the second workflow, or fold the build and publish into the same workflow as release-please and gate them on its outputs.
The elevated-credential path reintroduces a stored long-lived secret, which cuts directly against ADR-001's goal of removing standing publish secrets via OIDC.

The existing `release.yml` build matrix already builds each target on its own native runner (OpenTUI's Zig FFI core prevents cross-compilation), and `release-please-action` exposes a `release_created` output plus the created tag.

## Decision

Replace the two-workflow design with one consolidated release workflow that absorbs today's `release.yml`.

1. **On push to `main`, run `release-please-action`.**
   It maintains the release PR and, when that PR merges, creates the tag and GitHub Release, emitting `release_created` and the tag name as outputs.
2. **Gate the build matrix on `release_created`.**
   A `build` job (`needs:` the release-please job, `if: release_created`) builds all four targets on their native runners, self-checks each, and generates the per-platform npm packages plus the `SHA256SUMS` manifest.
3. **Gate an atomic publish job on the full matrix.**
   The `publish` job `needs:` all four build jobs, so it cannot run on a partial matrix; it publishes the four platform packages first and the main shim last (exact-pinned), attaches the binaries + `SHA256SUMS` to the GitHub Release, and publishes to npm with `npm publish --provenance` under OIDC Trusted Publishing.
4. **No elevated credential.**
   Because build and publish live in the same run as release-please, nothing depends on a Release event crossing a workflow boundary, so no GitHub App token or PAT is introduced; the only permissions are `contents: write` (release assets) and `id-token: write` (OIDC provenance).

This refines ADR-001: its "release-please fires the `release: published` event `release.yml` already consumes" mechanism is superseded by "release-please's `release_created` output gates the build and publish jobs in the same workflow." ADR-001's substance (release-please as the cut, manifest as the single version source, atomic ordered token-less publish) is unchanged.

## Alternatives Considered

### Alternative 1: Two workflows with an elevated token (GitHub App / PAT)

- **Description**: Keep `release.yml` on `on: release: published`; give release-please a GitHub App token or PAT so its Release triggers the second workflow.
- **Pros**: Minimal change to the existing `release.yml`; clean separation of "cut" and "publish" workflows.
- **Cons**: Reintroduces a stored long-lived credential to manage and rotate - the exact standing-secret liability ADR-001's OIDC decision set out to remove; a broader-scoped token than the release needs.
- **Why rejected**: It trades the token elimination we just won for workflow tidiness.

### Alternative 2: `workflow_dispatch`-only manual publish

- **Description**: Drop event-driven publishing entirely; a human runs the publish workflow by hand after merging the release PR.
- **Pros**: Simplest wiring; an obvious escape hatch.
- **Cons**: Reintroduces a manual step, defeating the one-action-cut goal, and is easy to forget or run against the wrong tag.
- **Why rejected**: The whole point is to remove the manual cut; kept only as a guarded fallback (see Implementation Notes).

## Consequences

### Positive

- The release cut, build, and publish are one run from one merge, with no cross-workflow event dependency and no new long-lived secret.
- The atomic publish (job `needs:` all builds, platform packages first, shim last) is preserved in the consolidated shape.
- `release_created` cleanly no-ops the build/publish jobs on ordinary pushes, so the workflow is safe to run on every commit to `main`.

### Negative

- The existing `release.yml` is restructured rather than reused as-is, a larger diff than bolting onto the `release: published` trigger.
- The release-please job, build matrix, and publish job are coupled in one workflow file, which is longer and must be read as a whole.

### Risks

- **Ordinary pushes doing unwanted work** - mitigated by gating every build/publish job on `if: needs.release_please.outputs.release_created == 'true'`.
- **Partial publish** - mitigated (as in ADR-001) by the publish job depending on all four build jobs and publishing platform packages before the shim.
- **A stuck or misconfigured release-please job blocking all releases** - mitigated by a guarded, manually-triggered `workflow_dispatch` fallback that publishes an already-created tag without double-publishing.

## Implementation Notes

- Job graph: `release_please` -> (`build` matrix, `if: release_created`) -> (`publish`, `needs:` all builds).
- Keep a `workflow_dispatch` entry that can rebuild/publish a given tag, guarded so it cannot republish an existing version (npm 403) or duplicate a GitHub Release asset.
- Pin `release-please-action` and `setup-bun` to specific versions; the publish job runs `npm publish` (not `bun publish`) on an image meeting npm CLI >= 11.5.1 / Node >= 22.14.0.
