# ADR-001: V1 scope for the automated release train - release-please cut, npm-native binary via optionalDependencies, OIDC provenance

## Status

Accepted

## Date

2026-07-09

## Context

Kitten is an early-stage TUI (Bun, TypeScript, OpenTUI, React) that runs Claude Code and Codex side by side over the Agent Client Protocol.
It is distributed as per-platform standalone binaries produced by `bun build --compile` (darwin-arm64, darwin-x64, linux-x64, linux-arm64) and published to npm, under ADR-006 ("Distribution as a Compiled Standalone Binary").

A working release pipeline already exists, but it is purely *reactive*.
`.github/workflows/release.yml` triggers on `release: published`; it builds the four targets on native runners (OpenTUI's native Zig core loads per-platform via FFI, so a single machine cannot cross-compile), self-checks each artifact, uploads the binaries plus a `SHA256SUMS` manifest to the release, and runs `bun publish` behind a static `NPM_TOKEN` secret.
Nothing creates the tag or the GitHub Release; a human hand-cuts every release.
`package.json` ships `version: 0.0.0` and CI stamps the real version from whatever tag the human typed - an implicit, un-reviewed source of truth that would `403` on any republish.

The product owner set the intent: automate the maintainer's release *cut* to one deliberate action, driven by Conventional Commits, and make the release release-grade rather than just functional.
Two facts sharpen the scope.
First, the npm package today ships raw TypeScript (`files: [src, ...]`, `bin: src/index.ts`) with Bun-only APIs, so `npx kitten` and `npm i -g kitten` fail on any machine without Bun; the only Node-runnable artifact (the compiled binary) lives only on GitHub Releases.
The owner's explicit requirement is that `npm i` work, which means the npm package must ship the prebuilt native binary.
Second, `scripts/install.sh` and the README hardcode a placeholder repo slug `OWNER/kitten` (the real repo is `MatheusBBarni/Kitten`), so the advertised curl one-liner `404`s today.

A five-advisor council (pragmatic-engineer, architect-advisor, security-advocate, product-mind, devils-advocate) debated the V1 scope and the sharpest fork - how the npm package should deliver the binary - and converged after a rebuttal round.

## Decision

Ship V1 of the automated release train, stable channel only, as follows.

1. **Automate the cut with release-please (release-PR model).**
   release-please continuously maintains a release PR from the Conventional Commit history; merging it writes `CHANGELOG.md`, mints the version tag, and cuts the GitHub Release, which fires the `release: published` event the existing pipeline already consumes.
   The `.release-please-manifest.json` becomes the single authoritative version source, retiring the `0.0.0`-stamped-from-tag hack.
   Merging the release PR is the one human gate; branch protection keeps it the only path that can create a release tag.

2. **Ship the npm-native binary via the optionalDependencies platform-package pattern.**
   A tiny main `kitten` package plus four per-platform packages (`@kitten/darwin-arm64`, `@kitten/darwin-x64`, `@kitten/linux-x64`, `@kitten/linux-arm64`), each carrying the compiled binary bytes inside the tarball and gated by `os`/`cpu`, listed as `optionalDependencies` of the main package; a thin JS shim resolves and executes the host binary.
   No install scripts, and no download fallback in V1: if the matching platform package is missing, the shim fails loudly with a one-line message pointing at the GitHub Release for manual download.
   This is the live `@openai/codex@0.144.0` template for the same four targets (`hasInstallScripts: false`).

3. **Make the registry publish atomic, ordered, and token-less.**
   The publish job `needs:` all four native build jobs, so it can never run on a partial matrix.
   Publish the four platform packages first and the main shim last, with the shim's `optionalDependencies` exact-pinned (`=x.y.z`) to the release version; a partial failure leaves the previous version resolving cleanly.
   Move the registry push from `bun publish` to `npm publish` (Bun stays as the compiler), adopt npm OIDC Trusted Publishing with `--provenance`, add `id-token: write`, and delete the static `NPM_TOKEN`.

4. **Enforce clean commit input cheaply.**
   Squash-merge with "default to PR title for squash merge commits" plus PR-title linting via `amannn/action-semantic-pull-request`, so exactly one valid Conventional Commit lands on `main` per PR.
   No commitlint or husky local hooks.

5. **Keep the binary channel checksum-verified and honestly labeled.**
   Retain `SHA256SUMS` for the curl binaries; defer separate binary attestation (`actions/attest-build-provenance` + `gh attestation verify`) to a later slice, and do not advertise the binary channel as "attested" while it is checksum-only.

6. **Fix the placeholder repo slug** `OWNER/kitten` -> `MatheusBBarni/Kitten` in `install.sh` and the README (P0), with a CI check that the README's literal install commands resolve.

7. **Supersede ADR-006's channel framing.**
   npm becomes a co-primary, Bun-free distribution channel shipping the prebuilt binary, not the secondary "`bunx kitten` for people who have Bun" path ADR-006 records.

## Alternatives Considered

### Alternative 1: Thin JS launcher that downloads the binary from the GitHub Release on first run/postinstall

- **Description**: A single small npm package that fetches the correct binary from the GitHub Release at install or first run.
- **Pros**: Tiny npm footprint; one package; no lockstep versioning; reserves the `npm i -g kitten` contract by name.
- **Cons**: Fetch-and-exec of a native binary lives outside npm's integrity model (no SRI hash on the runtime download), so a MITM or a tampered release asset is arbitrary code execution `npm ci`/`npm audit` cannot see; breaks behind corporate proxies, offline, and in airgapped CI; the postinstall-download pattern is widely discouraged (napi-rs docs).
- **Why rejected**: It saves nothing at install time - npm installs only the one matching platform package, the same bytes the launcher would fetch - while keeping every risk. The Devil's Advocate conceded this in debate.

### Alternative 2: semantic-release straight-through publish (no release PR)

- **Description**: Analyze commits and publish on every qualifying merge to `main`, with no release PR and no human gate.
- **Pros**: Maximum automation; highest adoption of the three tools.
- **Cons**: Removes the reviewable human gate before a tag can trigger a publish, which the automation-poisoning threat model argues against; less legible for a solo maintainer who wants to see the version and changelog before they ship.
- **Why rejected**: release-please's release PR is the single reviewable act the owner asked for and the security posture wants.

### Alternative 3: Minimal cut only - defer provenance, enforcement, and keep `bun publish`

- **Description**: Fix the slug, adopt release-please, keep the static `NPM_TOKEN` and `bun publish`, defer OIDC and enforcement.
- **Pros**: Smallest surface; does not touch the one green publish step.
- **Cons**: Leaves a long-lived publish secret in CI - a standing, harvestable liability independent of user count - when removing it via OIDC is cheap and yields provenance for free.
- **Why rejected as a whole, adopted in part**: The binary-attestation machinery is deferred (as this alternative urges), but killing the token via OIDC is done now because the cost is low and the liability is real.

### Alternative 4: Keep npm as the `bunx`-only path and lead with the curl binary

- **Description**: Leave the npm package Bun-targeted; direct Node-only users to the curl installer.
- **Pros**: Simplest package; no multi-package machinery.
- **Cons**: Contradicts the owner's explicit "`npm i` should work"; JS-ecosystem evaluators reach for `npx`/`npm i -g` first (gemini-cli, OpenCode, Codex all offer it); a Bun-not-found error on the first command is an instant bounce.
- **Why rejected**: The product requirement is that `npx kitten` works without Bun.

## Consequences

### Positive

- One reviewable action (merge the release PR) cuts a release; the `0.0.0`/tag footgun is gone because the manifest is the single version source.
- `npx kitten` and `npm i -g kitten` work on a stock Node machine, matching JS-ecosystem muscle memory.
- The static publish secret is deleted; every npm publish carries provenance for free via OIDC.
- Both distribution channels are built from one set of artifacts at one version from one workflow run, so they cannot diverge.

### Negative

- The npm footprint grows: a `bun build --compile` binary embeds the Bun runtime, so each of four platform packages is tens of MB, duplicating bytes already on GitHub Releases.
- Five npm packages must publish atomically and in version lockstep every release.
- Swapping `bun publish` -> `npm publish` re-tools the single currently-green publish step.
- ADR-006's recorded channel decision is superseded.

### Risks

- **Partial multi-package publish** - mitigated by gating the publish job on all four build jobs, publishing platform packages first and the main shim last with exact version pins, so a failure leaves the previous version resolving.
- **OIDC/trusted-publishing bootstrap chicken-and-egg** (the packages must exist before a trusted publisher can be configured) - mitigated by a first publish with a scoped automation token, then locking each package to trusted publishing and going token-less.
- **Large npm packages / slow first `npx`** - mitigated by a progress indicator and a size note, and by steering daily users to the persistent global install; the "one command, zero commitment" promise survives latency but not failure.
- **Missing-platform-package resolution edge** (the known optionalDependencies global-install gap) - mitigated by a loud, actionable shim error pointing at the Release; a hash-verified download fallback is added only if it actually bites.
- **Version bootstrap off `0.0.0` with no existing tags** - mitigated by setting the first release-please version floor explicitly.
- **Binary channel remains checksum-only** - mitigated by honest labeling and a deferred, not denied, attestation slice.

## Implementation Notes

- Build each binary once on its native runner, then feed the identical bytes both to the GitHub Release upload (as today) and into its platform tarball; never rebuild per channel.
- Stamp the one release version into all five `package.json` files from the release; the platform packages are pure generated build outputs, never hand-versioned.
- Keep the package free of any install scripts (`hasInstallScripts: false`).
- Document `npx kitten` as the hero install, `npm i -g kitten` for daily use, and the checksummed curl one-liner as the "no Node" alternative.
- Each new target later (windows, linux-musl, more arches) is one build-matrix row plus one generated platform package plus one exact-pinned optional dependency - complexity grows linearly.
