# ADR-004: Version source of truth via a bundled `package.json` JSON import

## Status

Accepted

## Date

2026-07-09

## Context

kitten cannot report its own version.
The version string is hardcoded `"0.0.0"` in exactly two places: `package.json` (`version`) and `src/agent/agentConnection.ts:168` (the `clientInfo` kitten sends to agents over ACP).
There is no `--version`, and no runtime ever reads `package.json`.

The release feature adds `--version`/`--help` (PRD C5) and requires the real published version to reach both the binary's output and the ACP `clientInfo`.
release-please already bumps `package.json` `version` in the release PR, so `package.json` is the natural single source of truth.

The codebase constrains the mechanism.
`tsconfig.json` sets `resolveJsonModule: true` and the repo already imports JSON with import attributes (`import ... with { type: "json" }` in `test/tsconfig.test.ts`).
There is no `--define` usage anywhere, no `import.meta.env`, and `scripts/build.ts`'s `compileCommand` passes no define flags.
`bun build --compile` statically bundles imported modules, including JSON, into the standalone binary.

## Decision

Introduce one module, `src/version.ts`, that imports `package.json` as JSON and re-exports the version as a single constant, and have both consumers read it.

```ts
// src/version.ts
import pkg from "../package.json" with { type: "json" }

/** kitten's release version, the single source of truth (release-please bumps package.json). */
export const KITTEN_VERSION: string = pkg.version
```

- `src/index.ts` prints `KITTEN_VERSION` for `--version` (and includes it in `--help`).
- `src/agent/agentConnection.ts` sets `clientInfo: { name: "kitten", version: KITTEN_VERSION }`, replacing the hardcoded `"0.0.0"`.
- `bun build --compile` bundles `package.json` into the binary, so the compiled artifact reports the same version that was published, and `bun run` (dev) reads the same file - no divergence between the two run modes.

## Alternatives Considered

### Alternative 1: Compile-time `--define`

- **Description**: `scripts/build.ts` reads the version and passes `bun build --compile --define=KITTEN_VERSION='"x.y.z"'`, referencing an ambient `declare const KITTEN_VERSION`.
- **Pros**: No JSON in the bundle; the version is a pure compile-time literal.
- **Cons**: No `--define` precedent in the repo; `bun run` (dev) has no define, so a `typeof KITTEN_VERSION` guard/fallback is needed, splitting dev and build behavior; an ambient global sidesteps the module system the codebase uses everywhere.
- **Why rejected**: More moving parts and a dev/build split, for no real gain over a bundled import.

### Alternative 2: Generated version file

- **Description**: A build/release step writes `src/version.ts` from the version; a default is committed and regenerated in CI.
- **Pros**: Matches the repo's `Bun.write` habit; version is a plain constant.
- **Cons**: Reintroduces a committed version literal that drifts if a developer builds without regenerating, and adds a generation step to keep in sync.
- **Why rejected**: The bundled JSON import gets the same constant with zero generation step and zero drift.

## Consequences

### Positive

- `package.json` is the single version source; release-please already owns it, so `--version`, `--help`, and the ACP `clientInfo` all report the published version automatically.
- Identical behavior for `bun run` and the compiled binary.
- One tiny module; both existing `"0.0.0"` literals collapse into one import.

### Negative

- `package.json` (a few hundred bytes) is bundled into the binary; negligible size, but it does embed the dependency list.

### Risks

- **Type friction under `verbatimModuleSyntax`/strict** - mitigated by typing the export explicitly (`: string`) and relying on `resolveJsonModule`, already enabled.
- **A future non-bundled runtime** (e.g. shipping unbundled JS) reading a missing `package.json` - not applicable while distribution is the compiled binary; revisit only if that changes.

## Implementation Notes

- Add `--version` and `--help` as sibling predicates to `wantsSelfCheck` in `src/index.ts`, each writing to stdout and `process.exit(0)`; `--help` leads with examples and the install/upgrade commands.
- A unit test asserts `KITTEN_VERSION` equals `package.json`'s `version`, and `test/build.integration.test.ts` is extended to assert the compiled binary's `--version` output matches.
