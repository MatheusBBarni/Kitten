# ADR-001: Keep statusline colors item-local and declarative

## Status

Accepted

## Date

2026-07-20

## Context

GitHub issue #33 proposes optional foreground colors for fields in Kitten's existing single personal `/statusline` layout. Terminal prompt tools commonly expose foreground styling, but often through scripts, templates, palettes, or broader visual systems. Kitten already has a pure statusline normalizer and renderer, strict persisted configuration, a product-owned proposal flow, and an explicit preview-and-confirm persistence boundary.

The product goal is faster visual scanning for solo power users. The scope must retain Kitten's non-executable, fail-closed configuration model and avoid accidentally establishing a general theming platform. Fixed colors can become less legible after a theme or terminal change; V1 must make that trade-off explicit rather than silently guessing or transforming colors.

## Decision

Add one optional, item-local foreground `color` to each existing statusline field form. V1 accepts only a known CSS color name or an opaque six-digit `#RRGGBB` value, normalizes every accepted value at the pure-core boundary to canonical uppercase `#RRGGBB`, and persists and previews only that normalized representation.

An explicit color applies only to its field value. Separators remain `palette.muted`; a field without `color` retains the active theme's normal text color. The existing preview and explicit confirmation remain the sole write boundary. V1 does not infer terminal backgrounds, adjust colors for contrast, or alter a requested valid color across themes.

## Alternatives Considered

### Alternative 1: General palettes, backgrounds, templates, and profiles

- **Description**: Introduce a broader styling/theming system with palette references, separator or background colors, dynamic expressions, and profile scopes.
- **Pros**: Supports more visual customization and can centralize future style rules.
- **Cons**: Adds unsettled ownership, persistence, rendering, compatibility, security, accessibility, and support requirements.
- **Why rejected**: There is no evidence that several independently valuable styling dimensions need shared resolution semantics. It would turn a focused scanability improvement into a theming platform.

### Alternative 2: Defer all color customization

- **Description**: Keep the current theme-derived statusline until stronger demand is measured.
- **Pros**: Retains the smallest configuration surface and avoids fixed-color readability risks.
- **Cons**: Delays a low-cost, reversible improvement that fits existing validated seams and developer expectations for segment-level status information.
- **Why rejected**: A field-only, non-executable, previewed extension is sufficiently bounded to test the scanability hypothesis without committing to broader styling.

## Consequences

### Positive

- Faster visual scanning can be tested without scripts, templates, dynamic data, or new telemetry.
- One canonical value makes proposal parsing, config round-tripping, preview, and rendering deterministic.
- Existing uncolored layouts, theme behavior, separator treatment, and confirmation semantics remain intact.

### Negative

- User-selected fixed colors can be unreadable in a different terminal or theme.
- A new configuration option creates some documentation, test, and support responsibility.
- Requests for backgrounds, separator colors, palettes, and profiles must be deliberately declined in V1.

### Risks

- **Malformed or hostile input**: reject unknown names, non-opaque or non-six-digit hex, ANSI/control characters, `transparent`, and extra item keys at the existing strict boundary.
- **Unreadable colors after environment changes**: keep color optional, preserve theme defaults for uncolored fields, show the current preview, and provide the existing reset/recovery route; do not claim automatic contrast safety.
- **Scope creep into a theming system**: preserve this ADR's item-local, foreground-only contract and require evidence before adding another styling dimension.

## Implementation Notes

- Keep parsing and canonicalization in `src/core/statusline.ts`; do not import an OpenTUI color parser into the pure core.
- Carry an optional color through `StatuslineSegment`; render field text with the explicit color or `palette.text`, while rendering `separatorBefore` with `palette.muted` in both preview and active footer.
- Preserve the existing one-line budget, grapheme-width handling, `statuslineText()` behavior, strict config semantics, LLM disclosure, preview, confirmation, and watcher reload boundaries.
