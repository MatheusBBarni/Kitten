# ADR-003: Carry canonical colors through the pure statusline model

## Status

Accepted

## Date

2026-07-20

## Context

The existing statusline core owns the accepted layout shape, proposal parsing,
width-aware rendering, and the `StatuslineSegment` contract consumed by both
the active footer and preview. The current item union supports legacy simple
fields plus `ELLIPSIS_BRANCH`, and both UI surfaces flatten rendered segments
into one normal-theme string. Configuration loading, proposal parsing, and
confirmed persistence already converge on the same core normalizer.

Per-item foreground color must preserve these boundaries: legacy simple items
must continue to work; saved and agent-proposed layouts must have identical
validation; preview and active footer must show the same field colors; and no
UI dependency or terminal-control parser may enter the pure core.

## Decision

Extend the existing `StatuslineItem` union with a structured form for every
simple field and an optional `color` on `ELLIPSIS_BRANCH`. Keep legacy strings
valid. The pure statusline normalizer accepts only known CSS names and opaque
six-digit `#RRGGBB`, converts every accepted value to canonical uppercase
`#RRGGBB`, and rejects unknown names, shorthand or alpha forms, transparent
values, terminal controls, and unsupported item keys.

Carry the optional canonical color on `StatuslineSegment`. Add one small shared
UI presentation helper that renders field text using its explicit color or the
theme text color and always renders `separatorBefore` with the muted theme
color. The active footer and preview both consume that helper. Keep
`statuslineText()` color-agnostic for text and width behavior.

The configuration schema, writer, watcher reload, LLM proposal parser, preview
config diff, and confirmed active layout all reuse the same normalized model.
Invalid data fails closed at the existing boundary; only a canonical layout can
preview, save, load, or reload.

## Alternatives Considered

### Alternative 1: Parse colors in the UI with the terminal library

- **Description**: Store raw color strings and let each UI surface interpret
  them at render time.
- **Pros**: Avoids a maintained name table in the core.
- **Cons**: Makes saved values renderer-dependent and permits preview/footer
  drift or UI-specific acceptance behavior.
- **Why rejected**: The core is the established strict acceptance boundary and
  must remain independent of UI dependencies.

### Alternative 2: Store colors in a separate layout map

- **Description**: Keep field order and color preferences in separate objects.
- **Pros**: Leaves legacy item objects unchanged.
- **Cons**: Adds ordering and stale-key reconciliation, duplicates validation,
  and weakens item-local configuration semantics.
- **Why rejected**: Color belongs to the field it styles and should travel with
  it through proposal, persistence, rendering, and diff output.

### Alternative 3: Emit styled terminal strings from the core

- **Description**: Flatten colors into terminal control sequences before UI
  rendering.
- **Pros**: Gives consumers one pre-styled string.
- **Cons**: Violates the pure text contract, complicates width calculations,
  and bypasses OpenTUI presentation ownership.
- **Why rejected**: Core output must stay protocol- and terminal-control-free.

## Consequences

### Positive

- Every path shares one canonical, deterministic color representation.
- Legacy layouts remain compatible while structured items can express color.
- Preview and active footer use the same segment data and separator policy.
- Invalid persisted or agent-proposed input remains fail-closed.

### Negative

- The core owns a maintained CSS-name-to-RGB table.
- The affected statusline test matrix grows across core, config, proposal,
  preview, active-footer, resize, and reload behavior.

### Risks

- **Parser drift**: keep one normalizer as the sole acceptance boundary and
  assert canonical round trips through every entry point.
- **Preview/footer drift**: render both through one shared helper and cover
  their field and separator colors in mounted tests.
- **Regression of legacy layouts**: keep simple strings valid and retain all
  existing width, duplicate-kind, grapheme, and recovery assertions.

## Implementation Notes

- Modify existing statusline, config, proposal-flow, footer, and overlay
  modules; add at most one presentation helper in the existing `src/ui/`
  directory.
- Do not add a dependency, dynamic parser, telemetry event, profile model, or
  new persistence location.
- Require focused layer coverage followed by `bun run typecheck`, `bun test`,
  `bun run selfcheck`, and `bun run build`.
