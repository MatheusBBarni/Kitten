# ADR-003: Keep delegation state protocol-free and ephemeral in AppState

## Status

Accepted

## Date

2026-07-14

## Context

The PRD requires a developer-visible parent-child delegation model while preserving Kitten's current ownership boundaries. The core and store already own immutable protocol-free state, whereas the controller owns live agent runtimes, connection generations, interaction settlement, and teardown. V1 must not restore live delegation ownership after restart.

The delegation model needs stable parent-child identity, lifecycle state, terminal snapshots, and parent-close state that UI selectors can read without access to ACP objects or controller-private promises.

## Decision

Add a protocol-free `DelegationState` to `AppState`, reduced by pure core helpers and updated through narrow store actions. It records parent generations, child ownership, lifecycle state, terminal snapshots, and cleanup eligibility. The controller retains the only map of live child runtimes and is responsible for start, prompt dispatch, cancellation, generation fencing, and teardown.

Do not add delegation state to persisted run records in V1. On restart, existing sessions restore under their current best-effort rules but no restored session claims a live parent-child delegation relationship.

## Alternatives Considered

### Alternative 1: Controller-only registry

- **Description**: Keep ownership and lifecycle entirely in controller-local maps.
- **Pros**: Fewer store changes and direct access to runtime operations.
- **Cons**: UI cannot safely subscribe to authoritative delegation state; lifecycle and visibility become coupled to side effects.
- **Why rejected**: It violates the existing protocol-free state boundary and makes selector-driven UI completion unreliable.

### Alternative 2: Persist and restore the delegation graph

- **Description**: Add parent-child ownership and lifecycle data to the run-record schema and reconstruct it at startup.
- **Pros**: Preserves historical grouping across restart.
- **Cons**: Risks fabricating live ownership without reliable runtime recovery and expands V1 migration scope.
- **Why rejected**: The PRD explicitly prioritizes trustworthy live delegation over restart continuity.

## Consequences

### Positive

- UI and tests can observe a single immutable, provider-neutral delegation projection.
- The controller keeps exclusive responsibility for ACP-facing operations and teardown.
- Session restoration remains unchanged and cannot overstate ownership after restart.

### Negative

- Delegation grouping and terminal snapshots disappear when Kitten restarts.
- Store changes must remain synchronized with session creation and removal.

### Risks

- **Partial registration**: provide one atomic store operation that adds the child session, workspace metadata, and delegation child together.
- **Stale publication**: record and verify parent and child generation before every lifecycle transition.
- **State divergence**: derive child attention from existing session state instead of duplicating approval or clarification payloads.

## Implementation Notes

Add a small `src/core/orchestration.ts` reducer/selector module and `delegation` to `AppState`. Do not create a new package, persistence schema version, or parallel session runtime.
