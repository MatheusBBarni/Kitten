# ADR-001: Keep the Harness Contract Static, Deterministic, and Narrowly Extensible

## Status

Accepted

## Date

2026-07-14

## Context

Kitten needs a reviewable source of truth for host guidance supplied to fresh ACP sessions. The base contract must be provider-neutral, concise (at most 150 tokens), and owned by the protocol-free domain layer. Issue #18 must establish a future extension seam for capability-specific guidance, while #19 owns ACP lifecycle delivery and #20 owns capability confirmation and fragment selection.

The central trade-off is between a single immutable string, which minimizes the initial surface, and a broad composition platform, which would prematurely couple the contract to future runtime requirements. The contract also must not create a false authorization or prompt-injection defense boundary.

## Decision

Create one self-contained, ACP-free domain contract that owns reviewed base `v1` wording and renders it deterministically. The base is static, provider-neutral, content-free, and capped at 150 tokens. It states only universally true host behavior: repository instructions and user requests retain normal precedence; agents report and verify results honestly; Kitten runtime controls govern confirmation-gated actions; and agents do not assume unavailable tools or capabilities.

Render the contract as one explicit tagged text envelope. Tags are fixed text delimiters that make exact output and whitespace reviewable; they have no parsing, authorization, or security authority. Unsupported versions, malformed or duplicate extension identifiers, invalid delimiter/control content, and configured size or count limits fail closed. The renderer never silently substitutes the latest version.

Expose only a skeletal extension seam: a bounded, typed set of stable fragment identifiers and reviewed static fragment bodies, deterministically placed after the base. Issue #18 ships no optional fragments and accepts no user, repository, transcript, provider-output, environment, or configuration content. #20 may later own the allowlist and selection of confirmed capability fragments without changing the base renderer; #19 may later encode the already-rendered text for ACP transport.

## Alternatives Considered

### Alternative 1: Immutable base-only string with no extension seam

- **Description**: Render one fixed `v1` string and defer all composition concepts until #20.
- **Pros**: Smallest initial API and simplest golden fixture.
- **Cons**: Does not meet #18's required extension boundary; forces later redesign of the contract interface.
- **Why rejected**: The issue explicitly requires a reviewable seam for #20. A sealed, static-only seam preserves that boundary without building a composition platform.

### Alternative 2: Generic dynamic fragment platform

- **Description**: Accept arbitrary fragment text and runtime metadata, then compose broadly configurable guidance.
- **Pros**: Flexible for future features and providers.
- **Cons**: Creates an injection path, invites provider/lifecycle coupling, expands review burden, and risks presenting guidance as authorization.
- **Why rejected**: It violates the static content boundary and belongs, if ever needed, to later capability work with concrete requirements.

## Consequences

### Positive

- Maintainers review one canonical, versioned artifact with exact deterministic output.
- The core remains independent of ACP transport, lifecycle, providers, and user-visible transcript behavior.
- Future capability work has a narrow integration point without expanding V1 into a platform.
- Golden and semantic tests can distinguish contractual changes from harmless formatting changes.

### Negative

- The contract does not produce runtime behavior until #19 delivers it.
- Future fragment authors are constrained to reviewed static text and bounded identifiers.
- The 150-token cap requires ruthless wording discipline.

### Risks

- **False security posture**: reviewers may mistake tags or wording for enforcement. Mitigate by stating explicitly that Kitten runtime controls remain authoritative.
- **Premature extensibility**: a seam may grow into a generic prompt platform. Mitigate with no V1 optional fragments, stable-ID validation, static-only input, and strict bounds.
- **Prompt drift**: wording changes may silently alter behavior. Mitigate with a semantic version policy, exact base-only golden, and visible review diffs.

## Implementation Notes

- The likely ownership seam is a new pure core module with colocated tests, not the ACP adapter or shared session state.
- Exact limits for future extension count and rendered size must be explicit and tested before #20 enables any fragment; V1 renders the base alone.
- Tests use synthetic content only and enforce no ACP imports, no dynamic content sources, deterministic ordering, fixed whitespace, duplicate rejection, unsupported-version failure, and base-only exact output.
