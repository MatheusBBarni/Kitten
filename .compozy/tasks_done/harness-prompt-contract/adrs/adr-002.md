# ADR-002: Release the Harness Contract as a Reviewer-First Foundation

## Status

Accepted

## Date

2026-07-14

## Context

The harness contract has two audiences with different timing. Maintainers and reviewers need a canonical, trustworthy artifact now; developers benefit only after a later card delivers the contract into fresh ACP sessions. Bundling those outcomes would blur ownership, obscure review, and make it hard to tell whether a change affects contract meaning or delivery behavior.

Research also shows that developers expect concise, attributable instructions while retaining host-controlled permission boundaries. The clarification round selected an exact rendered diff plus a declared behavior-impact classification as the review standard, and an explicit unsupported state rather than silent version substitution.

## Decision

Release #18 as a reviewer-first foundation. Its product promise is one concise, versioned, provider-neutral host-guidance contract that maintainers can review independently. Every contract change must expose the exact rendered change and state whether it changes behavioral meaning. A requested unsupported version must remain visibly unsupported; it must never silently become the newest version or disappear.

The release must explicitly state that it creates guidance, not agent permissions, authorization, or guaranteed runtime outcomes. #19 owns the later developer-facing delivery outcome, and #20 owns future evidence-gated capability guidance.

## Alternatives Considered

### Alternative 1: Bundle contract creation with in-session delivery

- **Description**: Ship the contract only when it can immediately affect fresh agent sessions.
- **Pros**: Developers see a runtime effect in the same release.
- **Cons**: Couples the reviewable policy artifact to delivery, session state, and user-visible behavior; makes failures and ownership harder to explain.
- **Why rejected**: The contract is independently valuable as a durable, reviewable foundation and must remain separately verifiable before delivery work begins.

### Alternative 2: Silently use the newest version when a requested version is unavailable

- **Description**: Preserve continuity by automatically substituting the latest approved guidance.
- **Pros**: Fewer visible interruptions.
- **Cons**: Hides meaningful behavioral drift and weakens reviewers' ability to reason about what guidance was intended.
- **Why rejected**: Explicitness is more trustworthy than a hidden substitution for an API-like contract surface.

## Consequences

### Positive

- Maintainers get a clear review workflow and an auditable contract surface immediately.
- The product makes no premature promise of agent behavior before delivery exists.
- Version changes, unsupported states, and later phase boundaries remain legible to reviewers and future feature owners.

### Negative

- Developers see no new in-session behavior in this card alone.
- Release communication must explain why the foundation ships before its delivery companion.

### Risks

- **Foundation perceived as incomplete**: mitigate with release notes that state the current reviewer value and the planned #19 outcome.
- **Prompt text mistaken for security enforcement**: mitigate with explicit guidance-only language and host-control ownership.
- **Scope expansion into capability or lifecycle work**: mitigate with clear phase boundaries and non-goals.

## Implementation Notes

This decision governs product scope, review expectations, and release messaging. It deliberately does not decide delivery, provider behavior, or capability selection.
