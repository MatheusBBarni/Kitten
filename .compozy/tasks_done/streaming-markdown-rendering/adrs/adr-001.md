# ADR-001: V1 Scope - Elevate Markdown as Trustworthy Instrumentation for the Hand-off, Behind an Owned Seam

## Status

Accepted

## Date

2026-07-09

## Context

Kitten already renders Markdown through OpenTUI's `<markdown>` renderable, but only in the transcript (`src/ui/MessageView.tsx`), and `src/ui/theme.ts` registers only code-syntax scopes.
As a result, every heading, bold, italic, strikethrough, inline code, list marker, blockquote, and link currently renders in flat body-text color; only fenced code blocks are styled.
Every other prose surface (the hand-off summary preview, help, errors, approvals) is plain `<text>`, and the hand-off summary - the prose a user must trust before forwarding a task to the other agent - is an editable `<textarea>` that is never rendered as Markdown.

The team scoped a V1 to "elevate Markdown everywhere, target full CommonMark + GFM, and beat the competitor toad in blind head-to-head render preference."
A five-advisor council (pragmatic-engineer, architect-advisor, product-mind, devils-advocate, the-thinker) stress-tested that scope, priority, framing, and risk against three hard constraints.
First, OpenTUI is pre-1.0 (0.4.3) and sits on the critical path: it actively drops GFM task-list checkboxes, does not support footnotes or inline raster images, carries a finalization-blanking bug that Kitten already works around by pinning streaming permanently on, and is subject to open bug #807, in which Markdown syntax highlighting silently degrades to plain text inside `bun build --compile` binaries (which Kitten now ships).
Second, toad is built on Rich and Textual, is more than ten years mature, and treats Markdown quality as a stated pillar, so "beat toad on render preference" grades Kitten against the incumbent's most-fortified axis.
Third, Kitten's actual product wedge is the cross-agent hand-off, not rendering, and no measured line connects render preference to hand-off adoption.

## Decision

V1 elevates the Markdown reading surface as instrumentation for the hand-off, not as a standalone render beauty contest.

1. Extract a single owned `<Markdown>` component (the seam); OpenTUI lives entirely behind it, mirroring how `ToolCallDiffView` already isolates the shared `<diff>`.
2. Register the missing `markup.*` theme scopes so headings, emphasis, lists, blockquotes, and links render properly and stay theme-aware - the cheap, high-yield 80%.
3. Apply the shared component across prose surfaces, with the hand-off summary preview as the center of gravity: rendered in read mode, editable `<textarea>` in edit mode.
4. Fix or mitigate bug #807 so the compiled binary highlights code; this is table stakes and lives below the seam, so it needs its own fix (inline the tree-sitter worker or provide a main-thread fallback).
5. Allow only above-the-dependency extensions in V1: OSC 8 clickable links with terminal-capability detection and graceful `text (url)` fallback, display-only GFM checkbox glyphs, and robust table wrapping via OpenTUI's existing `columnFitter`/`wrapMode`.
6. Degrade gracefully on malformed Markdown (which LLMs routinely emit) and on unsupported elements (render footnotes as sensible text rather than dropping content).

Success is graded primarily on comprehension and forward-decision quality on the hand-off summary.
A "not visibly worse than toad" hygiene check on ambient surfaces, plus a pre-registered (non-curated) blind corpus comparison, serve as a floor and as lagging launch collateral - never as the headline objective.

## Alternatives Considered

### Alternative 1: Maximalist "beat toad, full CommonMark + GFM everywhere"

- **Description**: Own the rendering pipeline to fully match or exceed toad across all GFM, anchored on winning a blind preference test.
- **Pros**: Highest ceiling on ambient render quality; a clean external marketing claim.
- **Cons**: "Full GFM" is blocked by 0.4.3 engine limits; the differentiating tail requires renderer-internal surgery (soft-forking a pre-1.0 dependency); grades Kitten on toad's strongest axis via a riggable metric with no proven tie to the wedge.
- **Why rejected**: Concentrates cost and risk on the incumbent's home turf while starving the hand-off; the "full GFM" promise is not shippable on the current engine.

### Alternative 2: Cheap-80%-only (theme scopes + shared component, nothing else)

- **Description**: Register `markup.*` scopes and route surfaces through one component; ship in days; skip #807, OSC 8, and the preview instrumentation.
- **Pros**: Smallest, fastest, fully reversible; clears the "half-hearted rendering" bar versus the broken field.
- **Cons**: Leaves #807 shipping un-highlighted code in the compiled binary; leaves the highest-value surface (hand-off preview) untouched.
- **Why rejected as a whole**: Insufficient alone - but its core (seam + theme scopes) is absorbed into the decision.

### Alternative 3: Fork OpenTUI / own a native Markdown renderer

- **Description**: Replace the dependency with a Kitten-owned renderer to control quality end to end.
- **Pros**: Total control; removes all upstream blockers.
- **Cons**: Unbounded maintenance debt against a young ecosystem; aimed at rendering, not the wedge.
- **Why rejected**: Premature; revisit only if the comprehension metric proves render quality converts and OpenTUI 1.0 still blocks the needed features.

## Consequences

### Positive

- Closes the flat-rendering gap across every prose surface at low cost.
- Puts render quality in service of the wedge: a legible, trustworthy hand-off preview.
- Isolates the pre-1.0 OpenTUI dependency behind an owned, swappable seam.
- Removes the compiled-binary highlighting regression before it ships.
- Preserves a shareable external proof point without building for a benchmark.

### Negative

- Does not deliver "full GFM": task-list checkboxes are display-only, footnotes and inline images stay unsupported.
- Does not fix the streaming finalization bug; streaming stays pinned on.
- May not decisively out-render toad on ambient prose in V1.

### Risks

- Pre-1.0 OpenTUI churn on the critical path - mitigated by the owned seam and above-the-line-only extensions (no renderer-internal surgery in V1).
- Riggable preference metric - mitigated by comprehension-primary grading and a pre-registered, non-curated comparison corpus.
- Cosmetic-fake divergence for checkbox glyphs - mitigated by rendering them display-only and honestly, with no pretense of interactivity.
- Distraction from the hand-off wedge - mitigated by making the hand-off preview the center of gravity and capping ambient investment at the hygiene floor.

## Implementation Notes

Fix #807 below the seam (inline the tree-sitter worker via a Blob URL or a main-thread fallback) and add a golden-frame test run against the compiled artifact.
Keep `MARKDOWN_STREAMING` pinned until a finalize test proves settled multi-block content renders.
Extend the `ConversationView` streaming tests and add a `theme.ts` test that asserts the `markup.*` scopes are registered in both modes.
