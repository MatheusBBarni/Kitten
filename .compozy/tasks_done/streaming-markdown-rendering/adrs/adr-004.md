# ADR-004: #807 - Embed the Tree-sitter Worker and Assets in the Compiled Binary, Resolve via `workerPath` at Startup

## Status

Accepted

## Date

2026-07-09

## Context

OpenTUI parses syntax through a web worker (`parser.worker.js`) resolved from `import.meta.url`; there is no main-thread fallback, so if the worker cannot be constructed, highlighting silently fails while unstyled text still paints a valid frame.
`bun build --compile` does not embed the worker, because the worker specifier is computed into a variable rather than passed as a literal `new Worker(new URL(...))`, so Bun's bundler never traces it.
In the shipped binary the worker and its runtime assets are therefore missing, and every fenced code block and diff loses highlighting (OpenTUI issue #807).
Kitten has just shipped compiled-binary and npm packaging, and its build integration test only asserts that the binary boots, so it would not catch this.
OpenTUI exposes three seams to point at the worker: `TreeSitterClientOptions.workerPath`, the `OTUI_TREE_SITTER_WORKER_PATH` env var, and a `--define` global of the same name.

## Decision

Embed the worker plus its runtime assets (the `web-tree-sitter` `tree-sitter.wasm` and the per-language `.wasm`/`.scm` files) into the compiled binary.
On startup, extract them once, idempotently, to a stable per-user cache directory, and point OpenTUI at the extracted worker via the `workerPath` option (or the `OTUI_TREE_SITTER_WORKER_PATH` env) before the renderer and tree-sitter client are created.
This preserves the single-binary distribution used by `install.sh` and npm.
Centralize the build wiring in `scripts/build.ts` `compileCommand()`.
Add a golden-frame acceptance test: strengthen the self-check to render known Markdown and diff content and assert a highlighted (non-default foreground) span, and run that self-check against the compiled binary in `build.integration.test.ts`.

## Alternatives Considered

### Alternative 1: Ship assets beside the binary

- **Description**: Emit the worker and assets next to the binary in `scripts/build.ts` and set the `--define`/env worker path to them.
- **Pros**: Simpler build wiring; no runtime extraction.
- **Cons**: Breaks the single-file install and complicates distribution and the installer.
- **Why rejected**: The single-binary distribution is a deliberate product property (ADR-006 in the sibling task); a multi-file drop regresses it.

### Alternative 2: Rely on default resolution / do nothing

- **Description**: Ship as-is.
- **Pros**: No work.
- **Cons**: The shipped product silently loses its headline feature (highlighting).
- **Why rejected**: This is the bug being fixed.

### Alternative 3: Main-thread parse fallback

- **Description**: Parse synchronously on the main thread when the worker is unavailable.
- **Pros**: No asset embedding.
- **Cons**: OpenTUI 0.4.3 has no such path; `highlightOnce` still routes through the worker.
- **Why rejected**: Not available without forking the dependency.

## Consequences

### Positive

- The shipped binary highlights identically to a source run.
- The single-binary distribution is preserved.
- The regression is guarded by a color-asserting test rather than a boot check.

### Negative

- Net-new startup extraction and worker-path wiring, of which there is currently none app-side.
- A cache directory to create, locate, and version across releases.

### Risks

- The exact embedding recipe (which assets, and how they are embedded and located at runtime) needs empirical validation during the build task; the three seams are certain, the recipe is not.
- The fix depends on OpenTUI worker internals staying compatible across version bumps; revisit on every `@opentui` upgrade alongside the streaming pin.

## Implementation Notes

Set the worker path before the first renderer/tree-sitter client is constructed (`getTreeSitterClient()` currently builds the default client with no `workerPath`).
The self-check must assert a span color, not just that a frame paints, because highlighting failure is silent.
