# ADR-005: Provenance via the Structured Referenced-Files List, Not Inline Prose Links

## Status

Accepted

## Date

2026-07-09

## Context

The PRD's "provenance links" feature wants the hand-off reviewer to trace a claim in the summary to its source before forwarding.
The summary, however, is developer-editable prose that contains bare file paths as text, and there is no structured relation between a path mentioned in the summary and an entry in `bundle.files[]` - only heuristic substring matching.
OSC 8 hyperlink emission is native and automatic in OpenTUI given link metadata plus terminal support; Kitten cannot post-process it, and it requires URL-form links rather than bare paths.
An inline provenance overlay would therefore be fragile: paths-only, re-derived on every edit, and prone to false matches.
Meanwhile `bundle.files[]` already carries each referenced file as structured data (`path` plus a `read`/`edited` reason).

## Decision

Deliver provenance through the already-structured referenced-files list, making those rows the first-class, inspectable provenance surface in the review console, rather than linking paths inside the summary prose.
Where a file path is a valid target and the terminal supports it, the file rows may render as OSC 8 links (in `file://` form), degrading to plain readable path text otherwise.
The summary renders as Markdown prose without synthetic inline links.

## Alternatives Considered

### Alternative 1: Heuristic inline links in the summary

- **Description**: Substring-match `bundle.files[].path` inside the rendered summary and wrap matches as `file://` Markdown links so OSC 8 makes them clickable.
- **Pros**: Provenance is clickable in-prose, at the point of the claim.
- **Cons**: Fragile against edits, paths-only, false matches, re-derived on every keystroke, and with no structured identity behind the match.
- **Why rejected**: The fragility undermines the trust the console is meant to create.

### Alternative 2: Defer provenance entirely

- **Description**: Ship the rendered summary and redaction clarity only; no provenance in V1.
- **Pros**: Smaller.
- **Cons**: Provenance is a core console value, and the structured list already exists, so first-class provenance is cheap.
- **Why rejected**: Leaves an easy, high-value capability on the table.

## Consequences

### Positive

- First-class, stable provenance drawn from existing structured data, with no fragile matching.
- Behaves consistently whether or not the summary has been edited.

### Negative

- Provenance lives beside the prose (the files list) rather than inline in it, so tracing a specific in-prose claim is a glance to the list rather than a click on the word.
- Clickable file rows depend on terminal OSC 8 support and `file://` handling; the fallback is plain path text.

## Implementation Notes

The files list already renders as keep/drop rows in `HandoffPreview`; the provenance affordance extends those rows rather than adding a new panel.
