# ADR-003: Persist a structured statusline preference and share one pure renderer

## Status

Accepted

## Date

2026-07-13

## Context

The PRD requires a configurable single-line layout, immediate application after explicit confirmation, deterministic narrow-width behavior, and a legacy status strip until a developer saves a preference. The configuration must represent `ELLIPSIS_BRANCH` with its own bounded `maxChars`, so an array of strings alone cannot express every valid layout.

The status strip and preview must never independently interpret the same layout. Separate formatters would make the approved preview unreliable and create incompatible truncation behavior.

## Decision

Add a resolved `StatuslinePreference` to `AppConfig` and the reactive app preferences. Its initial value has no custom layout and therefore keeps the existing status strip unchanged. The user-config delta gains an optional `statusline` block that stores a disclosure acknowledgement plus, when saved, a printable separator and ordered line.

The line accepts simple field strings and a structured branch item: `{ "kind": "ELLIPSIS_BRANCH", "maxChars": 24 }`. A new pure core statusline module validates and normalizes this model, derives the selected-session values, and renders the same segments for both the overlay preview and `StatusStrip`. `maxChars` counts grapheme clusters. At a constrained width the renderer preserves declared order and omits trailing items; it does not truncate any other field internally. A custom layout with no fitting leading item produces no custom left-side item rather than overflowing.

Confirmation writes the complete statusline block atomically before committing the layout to the reactive store. External config reloads replace the reactive preference without writing it back. The writer will reject an existing symlink target before reading or replacing it.

## Alternatives Considered

### Alternative 1: String-only fields plus one global branch limit

- **Description**: Keep `line` as strings and place a single branch cap beside it.
- **Pros**: Smallest JSON shape.
- **Cons**: Cannot express item-local formatting and makes future bounded field options awkward.
- **Why rejected**: The selected structured branch item is clearer, validates locally, and keeps simple fields concise.

### Alternative 2: Separate preview and footer formatters

- **Description**: Let the overlay and status strip each convert layout data into text.
- **Pros**: Fewer immediate changes to the current footer.
- **Cons**: The preview can disagree with the saved result and width policy can drift.
- **Why rejected**: The PRD treats preview equality as a trust boundary.

### Alternative 3: Replace the legacy strip for every developer

- **Description**: Render the configurable field model even when no preference is saved.
- **Pros**: One rendering path from first launch.
- **Cons**: Changes existing information such as headroom and MCP state without developer consent.
- **Why rejected**: Preserve current behavior until a developer explicitly confirms a custom layout.

## Consequences

### Positive

- The saved layout, preview, and active footer share one deterministic interpretation.
- A structured branch item fulfills the required bounded ellipsis without a free-form template language.
- Existing developers retain their current strip until they opt in.

### Negative

- The schema and config-reload path gain one more first-class preference.
- Grapheme counting is deterministic but does not promise perfect terminal display-cell measurement for every Unicode glyph.
- A long leading non-branch field can leave no custom item at very narrow widths.

### Risks

- **Malformed or duplicate items** — reject invalid schemas and duplicate field kinds during normalization.
- **Config write race or unsafe target** — serialize explicit writes, retain the active layout on failure, and reject symlink targets.
- **Preview/footer drift** — test both surfaces through the same pure renderer and width budget.

## Implementation Notes

- The later implementation must not subscribe every preview edit to persistence; only acknowledgement and Confirm may write configuration.
- Existing `overflow: hidden` remains containment, not the defined omission policy.
