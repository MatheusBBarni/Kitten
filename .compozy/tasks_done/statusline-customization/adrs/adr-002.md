# ADR-002: Make the statusline flow immediate, disclosed, and conversational-first

## Status

Accepted

## Date

2026-07-13

## Context

The approved idea defines a safe, declarative personal statusline. Product clarification resolved four remaining experience decisions: a confirmed layout should take effect immediately; Kitten must obtain acknowledgement before the first request is sent to an LLM; MVP success is a high-quality first-use experience rather than ongoing telemetry; and presets should appear only when conversation is unavailable, cannot satisfy the request, or the developer opts out.

These choices determine whether `/statusline` feels like a trustworthy personal-control feature or an opaque configuration assistant. They also preserve the requested natural-language experience without making it mandatory when it is unavailable.

## Decision

MVP will make `/statusline` a conversational-first personal customization flow. Before the first request, Kitten shows a concise disclosure of what will be sent to the LLM and requires acknowledgement. The developer then receives a compact, single-line proposal and exact change preview. Only an explicit confirmation persists the layout, and confirmation updates the currently visible statusline immediately.

Three fixed layouts are a recovery path only: Kitten offers them after an unavailable or unsatisfied conversational request, or when the developer explicitly declines to use the LLM. MVP evaluation focuses on whether first-time developers can understand, review, and save a useful line quickly and confidently.

## Alternatives Considered

### Alternative 1: Apply saved layouts on the next launch

- **Description**: Save the choice but leave the active session unchanged.
- **Pros**: Avoids a visible mid-session change.
- **Cons**: Breaks the feedback loop between preview, confirmation, and outcome; makes the first-use experience harder to understand.
- **Why rejected**: Immediate visible confirmation is central to the product value and the user explicitly chose it.

### Alternative 2: Non-blocking or documentation-only data-use notice

- **Description**: Mention LLM processing in help or a passive notice.
- **Pros**: Fewer steps before a request.
- **Cons**: Developers can send workspace context without knowingly acknowledging the destination.
- **Why rejected**: Explicit acknowledgement is the minimum trust bar for a feature that may send a natural-language request off-device.

### Alternative 3: Show preset layouts as the primary interface

- **Description**: Put fixed layouts alongside the conversational request from the first screen.
- **Pros**: Faster path for common needs.
- **Cons**: Dilutes the intended learning about whether natural-language personalization is valuable and makes the first surface busier.
- **Why rejected**: Keep presets as a reliable recovery path while keeping the requested conversational experience primary.

## Consequences

### Positive

- Developers see the exact effect of their confirmed choice immediately.
- The first LLM interaction is explicit and informed.
- Recovery remains available without turning MVP into a visual editor or a preset picker.
- Success can be assessed through focused usability evaluation without collecting statusline content.

### Negative

- First use has one acknowledgement step before a request can be made.
- Immediate application raises the expectation that preview and active result always agree.
- The recovery path must be explained clearly so users understand when and why it appears.

### Risks

- **Acknowledgement discourages use** — Keep it concise, specific, and limited to the first request.
- **Users expect unlimited expression** — Describe the supported fields and explain unsupported requests before confirmation.
- **The LLM is unavailable** — Offer recovery presets without losing the developer's requested outcome.
- **A compact terminal hides information** — Preview the actual single-line result and make omission predictable.

## Implementation Notes

- The PRD intentionally specifies user-visible behavior and trust boundaries only. The later TechSpec determines the delivery mechanism for immediate application and the exact LLM request contract.
