# ADR-002: Validate supervised parallel progress before autonomous orchestration

## Status

Accepted

## Date

2026-07-15

## Context

The approved idea establishes a provider-neutral way for a primary coding agent to create and observe Kitten-owned child work. PRD clarification refined the intended user promise: solo developers need supervised parallel progress, a direct recovery path when a child needs attention, and a clear attribution between an initiating task and every child it creates.

Comparable agent products make background status, ownership, and human intervention visible. At the same time, adoption and trust in autonomous coding agents remain uneven. The first product release must therefore validate whether developers repeatedly choose visible parallel delegation, rather than imply that Kitten provides a broad autonomous workforce.

## Decision

Position the MVP as **supervised parallel progress**. A primary coding agent may launch bounded, independent child work and observe its state, while the developer retains immediate visibility and a direct path into any child conversation that needs attention.

The MVP succeeds when developers voluntarily repeat parallel launches in normal work. It does not promise autonomous completion, hidden background execution, team coordination, or a complete child-management console. The product will make phase boundaries explicit and use observed repeat use, attention recovery, and trust signals to decide which additional control to add next.

## Alternatives Considered

### Alternative 1: Autonomous completion-first

- **Description**: Optimize the first release for agents completing work without developer supervision.
- **Pros**: Stronger autonomy narrative and less active monitoring in the ideal case.
- **Cons**: Increases trust, expectation, and recovery burdens before the product has validated that users accept hidden or unattended work.
- **Why rejected**: It conflicts with the chosen solo-developer promise of visible, attributable child work.

### Alternative 2: Rich child-management experience first

- **Description**: Deliver steering, cancellation, input forwarding, profiles, handoffs, and other controls with launch and status.
- **Pros**: Closer parity with mature orchestration products and fewer manual transitions later.
- **Cons**: Delays learning, broadens the user mental model, and creates expectations that every child state is controllable from every context.
- **Why rejected**: It turns the MVP into a platform instead of testing the core launch-and-supervise behavior.

### Alternative 3: UI-only delegation

- **Description**: Require the developer to create every child manually in the existing Kitten UI.
- **Pros**: Familiar supervision with no new agent-facing workflow.
- **Cons**: The primary coding agent cannot initiate the selected parallel-work loop, reducing the product to an existing manual interaction.
- **Why rejected**: It fails the core outcome of agent-initiated but developer-supervised delegation.

## Consequences

### Positive

- Gives solo developers a simple, trustworthy reason to try parallel child work.
- Keeps visibility, attribution, and recovery at the center of the product experience.
- Creates an observable adoption signal before further controls compete for roadmap capacity.

### Negative

- Developers must use the normal child conversation when intervention is needed.
- The first release will not match products that already expose comprehensive child management.
- Some users will want automation sooner than the MVP provides.

### Risks

- **Low repeat use**: Treat the MVP as a learning loop and stop expansion until users repeatedly choose it.
- **Invisible-work anxiety**: Keep every child visible, attributable, and explicit about attention or terminal state.
- **Expectation debt**: Describe absent management capabilities as phased follow-ups rather than implied behavior.
- **Conflicting parallel edits**: Start with independent tasks and keep isolation policy as an explicit future product decision.

## Implementation Notes

The product must preserve a direct, recognizable path from an active or attention-blocked child to its normal Kitten conversation. Product telemetry and research must measure aggregate adoption and recovery outcomes without collecting task content.
