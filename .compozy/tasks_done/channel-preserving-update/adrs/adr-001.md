# ADR-001: Preserve Verified Installation Channels with Fail-Closed Updates

## Status

Accepted

## Date

2026-07-20

## Context

Kitten is distributed through two co-primary channels: a global npm package and a checksum-verified standalone binary. Issue #36 introduces an explicit `kitten --update` command, but an executable named `kitten` does not prove who owns it. A permissive updater could overwrite a copied binary, update a different npm prefix, follow an unsafe path, or silently switch the user's installation channel.

The council found that the value of self-update is trustworthy convenience, not maximal update coverage. The command must therefore prove both the source installation and the exact replacement before it performs any mutation. Checksum verification protects against corruption and artifact mismatch, but does not establish trust if the release publisher itself is compromised; signed provenance and release attestation remain outside this slice.

## Decision

Ship one explicit, synchronous `kitten --update` command for the latest stable release. It runs before normal Cockpit boot and classifies the running installation before selecting exactly one channel-specific transaction:

1. A proved global npm installation updates only `@matheusbbarni/kitten@latest` through npm.
2. A proved standalone installation updates only when its canonical executable path matches the versioned XDG-state installation record written by the verified installer. It resolves the shipped host artifact, requires exactly one valid matching `SHA256SUMS` row, validates the downloaded bytes, compares the current hash first, and atomically replaces the target only after all checks pass.
3. Every source run, local dependency, `npx` invocation, copied binary, malformed or stale record, unsupported platform, and inconclusive provenance exits non-zero without mutation and prints the two supported recovery commands.

The standalone transaction uses a secure temporary file in the target directory, rejects unsafe or changed targets, retains the previous executable on every failure, serializes competing updates by canonical target, and never executes the downloaded bytes. The command does not add background checks, prompts, relaunch, TUI UI, version/prerelease selection, rollback UI, other package managers, or an automatic channel fallback.

## Alternatives Considered

### Alternative 1: Provenance detection plus reinstall guidance only

- **Description**: Detect the invocation channel and print its reinstall command without self-updating either channel.
- **Pros**: Smallest implementation and mutation surface; avoids filesystem replacement complexity.
- **Cons**: Does not test the self-update value hypothesis and leaves standalone users without an equal in-product update path.
- **Why rejected**: It is a safe fallback for unverified invocations, but not the agreed two-channel V1 value.

### Alternative 2: Permissive self-update with a standalone-download fallback

- **Description**: Treat executable name or path heuristics as sufficient and download a standalone binary whenever npm provenance is uncertain.
- **Pros**: More invocations appear to succeed and the control flow is superficially simple.
- **Cons**: Can overwrite an unowned executable, update a different npm installation, or silently change the user's channel.
- **Why rejected**: Violates the channel-preservation and no-unverified-mutation contract.

### Alternative 3: Expand V1 to complete update management

- **Description**: Add automatic checks, notifications, release notes, version selection, additional package managers, or attestation work.
- **Pros**: Broader convenience and a stronger long-term release story.
- **Cons**: Adds lifecycle, UX, compatibility, and trust-boundary complexity before the core decision table is proven.
- **Why rejected**: The user chose a surgical, explicit V1; publisher-compromise resistance is a separate decision.

## Consequences

### Positive

- Both supported installation channels receive equal, channel-preserving update behavior.
- Ambiguous invocation is safe by default and ends with actionable recovery guidance.
- The decision table and injectable seams make high-risk failure cases testable without live network access.

### Negative

- Valid but nonstandard installations may be refused and require a manual reinstall.
- Provenance, canonical-path, manifest, filesystem, and concurrency checks add more work than a generic downloader.
- Checksum verification cannot defend a compromised release publisher.

### Risks

- **Wrong npm owner or prefix** — prove the installed package belongs to npm's global root before invoking npm; never fall back to a binary download.
- **Path substitution, symlink, or concurrent-update race** — use canonical targets, safe same-directory temporary files, target revalidation, serialization, and atomic rename.
- **Manifest ambiguity or release mismatch** — require exactly one valid host-artifact entry and hash the bytes before replacement.
- **Failure damages the current executable** — test every download, validation, permission, write, and rename failure for byte-identical preservation.

## Implementation Notes

- Keep provenance classification, npm invocation, release retrieval, manifest parsing, hashing, replacement, and output behind narrow injectable seams.
- Preserve existing `--version` before `--help` metadata precedence; do not introduce a general argument parser.
- Treat successful process exit as insufficient evidence when the intended executable was not actually updated; success reporting must reflect the classified channel and final outcome.
