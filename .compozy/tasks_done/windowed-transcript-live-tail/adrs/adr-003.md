# ADR-003: Separate transcript projection from semantic session state

## Status

Accepted

## Date

2026-07-15

## Context

`SessionState.turns` is the complete normalized transcript and the session reducer is its sole writer. The focused conversation currently maps that full array directly into UI rows. A windowed presentation must survive focus changes, retain the existing no-transcript-persistence contract, and account for tool updates that can target any historical turn. React-local state cannot survive a focused-session remount, while adding presentation state to `SessionState` would couple transient viewport behavior to semantic state and persisted-run expectations.

## Decision

Add a pure core transcript-projection module that maps complete turns plus explicit protection input and a per-session window request into typed presentation rows. Store only the per-session, non-persisted window request and detached reading state in `AppState`, exposed through narrow selectors and actions. Keep `SessionState`, the session reducer's event semantics, ACP translation, and run-record persistence unchanged. `ConversationView` consumes the projection and owns renderer-specific scroll offset capture and restoration through its scrollbox ref.

## Alternatives Considered

### Alternative 1: Keep window state inside ConversationView

- **Description**: Use React-local state for history depth and detached reading state.
- **Pros**: Fewer store changes and local ownership of renderer details.
- **Cons**: Focus changes remount the view and lose session-specific state; no narrow selector contract exists for other command dispatchers.
- **Why rejected**: The PRD requires independent state across session switches.

### Alternative 2: Add presentation state to SessionState

- **Description**: Persist history depth and scroll-related state with the semantic transcript model.
- **Pros**: One state object and reducer path.
- **Cons**: Pollutes the protocol-free session domain with UI behavior and risks changing the deliberate no-transcript-persistence boundary.
- **Why rejected**: Presentation state is transient, renderer-facing, and must not change semantic or persistence ownership.

### Alternative 3: Introduce generalized virtualization first

- **Description**: Replace transcript rendering with a reusable virtual list platform.
- **Pros**: Potential broad reuse for future large lists.
- **Cons**: Adds a larger abstraction without defining protected live content or proving the immediate bounded-view value.
- **Why rejected**: It exceeds the V1 YAGNI boundary.

## Consequences

### Positive

- Preserves a single authoritative transcript while making presentation policy deterministic and testable.
- Retains per-session user context without persisting it or coupling it to ACP events.
- Lets unchanged projected turns retain their original references and stable keys.

### Negative

- Introduces a small transient state model and selector/action surface.
- Requires explicit reconciliation when an older tool row changes.

### Risks

- **Stale collapsed content after an old tool update**: recompute the affected projection from authoritative turns and test deterministic marker counts.
- **Scroll jump after reveal**: capture detached scroll offset before the projection change and restore it after layout.
- **False recovery expectations**: leave unavailable-restoration branches unchanged and keep windowing disabled or inert there.

## Implementation Notes

Place the dependency-free projection module under `src/core/`; keep per-session window state in `src/store/`; keep OpenTUI refs and layout timing in `src/ui/ConversationView.tsx`. A bounded rendered view is the V1 goal; this ADR does not promise a generalized virtualizer or alter the reducer's existing derived-state cost.
