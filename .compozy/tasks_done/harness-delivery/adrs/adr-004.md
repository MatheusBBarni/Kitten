# ADR-004: Gate harness encoding through exact certified runtime profiles

## Status

Accepted

## Date

2026-07-14

## Context

The installed `@agentclientprotocol/sdk` exposes `session/prompt` as user content blocks and does not provide a portable hidden host-instruction field. Its `_meta` field cannot be assumed to carry interoperable semantics. A universal tagged composite would therefore risk treating hidden host guidance as user content, leaking it into provider-visible history, or behaving differently across adapters.

The feature must support the built-in Claude Code, Codex, and Cursor recipes, then admit new providers without weakening the same contract. Existing capability gating demonstrates an appropriate repository pattern: exact recipe identity, version-aware evidence, and unsupported-by-default behavior.

## Decision

Pass a protocol-free `HarnessPromptEnvelope` from the controller to the agent boundary. It keeps visible user blocks separate from the optional rendered harness. `src/agent/` is solely responsible for translating that envelope into a provider-specific request.

Add an exact runtime-profile capability registry. A profile is eligible only when its recipe identity, adapter version range, and credentialed contract test prove that its encoding preserves the harness/user separation required by this feature. V1 must certify the built-in Claude Code, Codex, and Cursor recipes. New providers are added only by a new explicit registry entry and passing contract evidence; unknown or changed recipes are unsupported and cause the fresh conversation to fail closed before its visible first task is sent.

Do not define a universal tagged composite fallback. The profile contract decides the selected encoding, but no profile may expose the harness through Kitten's transcript, prompt history, handoff bundle, persisted user content, or telemetry. The core, store, UI, and config remain free of ACP request types.

## Alternatives Considered

### Alternative 1: Send one tagged composite text block for every provider

- **Description**: Concatenate tagged harness and user instruction text into a universal ACP prompt.
- **Pros**: One implementation path and immediate broad coverage.
- **Cons**: ACP treats the blocks as user content; providers may display, store, or interpret the harness inconsistently.
- **Why rejected**: It cannot meet the hidden-content contract across unverified adapters.

### Alternative 2: Let each adapter silently prepend a string without a registry

- **Description**: Add local adapter-specific logic with no certification metadata.
- **Pros**: Small per-adapter change.
- **Cons**: Recipe upgrades can change behavior silently and there is no evidence gate for new providers.
- **Why rejected**: It makes support status implicit and hard to audit.

### Alternative 3: Expose the harness as an editable user prompt

- **Description**: Display the harness so all providers receive the same user message format.
- **Pros**: Portable and visibly inspectable.
- **Cons**: Violates the PRD's hidden-content and user-authorship boundaries.
- **Why rejected**: It changes the product contract and creates transcript, handoff, and privacy leakage.

## Consequences

### Positive

- The feature is enabled only where its core safety claim has evidence.
- New providers can join through a repeatable certification path.
- Provider-specific details remain at the adapter boundary.

### Negative

- Coverage expands more slowly than a universal fallback.
- Certified profiles need maintenance when recipes or adapter versions change.

### Risks

- **A provider update invalidates a profile** — include exact recipe/version evidence in contract tests and fail closed on mismatch.
- **Credentialed tests are unavailable in some environments** — keep them opt-in outside ordinary unit runs and block release enablement until evidence exists.
- **Profile metadata leaks content** — restrict all diagnostics to fixed profile IDs, harness version, state, and failure category.

## Implementation Notes

Add capability metadata in the existing config area, modeled after the repository's narrow capability gates. Keep adapter profile selection deterministic; the registry must default to unsupported. Contract tests use synthetic harness and prompt values and assert wire separation plus content-free observability.
