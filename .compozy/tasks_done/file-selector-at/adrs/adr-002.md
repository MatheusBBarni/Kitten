# ADR-002: Limit V1 to Normal Repository Files and Preserve Composition on No Match

## Status

Accepted

## Date

2026-07-12

## Context

The V1 selector must help every Kitten developer reference a known file faster without presenting a confusing or brittle experience. The product promise is an explicit pointer to a file, not proof that its contents were attached to an agent.

The candidate set and the no-match state shape trust. Showing generated, ignored, or otherwise non-normal files would add noise to the common workflow. Blocking or dismissing the prompt when search has no result would turn a convenience into a composition interruption.

## Decision

V1 will show normal repository files only. It will exclude ignored, generated, and outside-repository files from the user-facing candidate list.

When a query has no matching candidate, the selector will keep ordinary typing available and show a brief, clear no-match message. Users may refine the query, delete it, or continue composing manually without losing their draft.

## Alternatives Considered

### Alternative 1: Show all files reachable from the session

- **Description**: Include ignored, generated, and other non-normal files in the selector.
- **Pros**: Maximizes possible coverage for exceptional tasks.
- **Cons**: Adds noise, exposes confusing paths, and makes common search results less useful.
- **Why rejected**: The MVP validates everyday prompt composition, not exhaustive file access.

### Alternative 2: Hide the selector whenever there is no match

- **Description**: Dismiss the file-selection state and leave the user to infer why it disappeared.
- **Pros**: Keeps the interface visually minimal.
- **Cons**: Removes useful feedback and can make the interaction feel unreliable.
- **Why rejected**: Users need a legible outcome while retaining control of their draft.

### Alternative 3: Require a manual-path recovery flow

- **Description**: Interrupt no-match searches with instructions to enter a full path.
- **Pros**: Offers a directed fallback.
- **Cons**: Adds friction to the exact workflow the feature is intended to simplify.
- **Why rejected**: Manual entry remains available naturally; V1 should not add another decision point.

## Consequences

### Positive

- Keeps results relevant for the common developer workflow.
- Protects prompt composition when discovery cannot help.
- Reinforces the product's honest, low-friction file-reference promise.

### Negative

- Some legitimate exceptional files will require manual path entry.
- Users may need clearer future controls if real usage reveals a demand for broader coverage.

### Risks

- **Candidate policy surprises**: state the normal-file boundary in product guidance and gather feedback before broadening it.
- **No-match confusion**: make the message brief, visible, and non-blocking.

## Implementation Notes

- Treat normal repository files as the default candidate category.
- Keep no-match feedback within the prompt-composition flow and never clear or submit the draft.
- Reassess the boundary only after opt-in usage data and user feedback show a recurring unmet need.
