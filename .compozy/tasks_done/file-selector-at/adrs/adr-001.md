# ADR-001: Keep @ File Selection as an Honest, On-Demand Single-File Reference

## Status

Accepted

## Date

2026-07-12

## Context

Kitten users need a faster way to reference a known repository file while composing a prompt. The product already has a prompt-local slash-command completion flow, but it has no repository-file discovery capability. Sessions can have different working directories, and referencedFiles is intentionally derived only from agent tool calls.

Competing tools make @ file references familiar, but the meaning of a typed path can differ by agent provider. A broad context feature would add token cost, unclear attachment semantics, and durable indexing complexity before demand is validated.

## Decision

Ship V1 as a provider-neutral, keyboard-first selector that lets a user type @, choose exactly one repository-relative file for the focused session, and insert an explicit file reference into the prompt.

V1 will use bounded, on-demand discovery rooted in the focused session's repository context. It will not claim guaranteed file-content attachment, change tool-derived file-reference state, build a persistent index, or add folders, symbols, multi-select, configuration, or provider-specific attachment behavior.

## Alternatives Considered

### Alternative 1: Manual path entry only

- **Description**: Keep the prompt editor unchanged and ask users to type or paste paths.
- **Pros**: No new discovery behavior, lowest implementation cost, no selector semantics to explain.
- **Cons**: Does not solve path discovery or materially reduce composition friction.
- **Why rejected**: It cannot credibly test whether integrated file selection creates habitual value.

### Alternative 2: Guaranteed provider-specific file attachment

- **Description**: Build an adapter-level attachment contract that ensures every selected file is supplied as context to each provider.
- **Pros**: Strongest semantic guarantee and potentially greater agent quality.
- **Cons**: Larger cross-provider scope, slower learning loop, higher maintenance burden, and divergent provider behavior.
- **Why rejected**: Premature before evidence shows that users repeatedly value the smaller composition workflow.

### Alternative 3: Broad context selector

- **Description**: Support folders, symbols, multiple files, generated-file options, and persistent indexing in the first release.
- **Pros**: Closer feature parity with broad-context coding tools.
- **Cons**: Expands token-cost, performance, privacy, and interaction risks; obscures the core hypothesis.
- **Why rejected**: Violates V1 scope discipline and introduces infrastructure before demand is proven.

## Consequences

### Positive

- Tests a familiar @ interaction with minimal provider coupling.
- Keeps the user in a keyboard-first prompt-composition flow.
- Preserves the existing invariant that referencedFiles reflects agent-observed activity only.
- Makes expansion contingent on observed adoption, completion, correction, and latency data.

### Negative

- A selected path may not mean identical file-context behavior across providers.
- On-demand discovery can be slower than a persistent index in very large repositories.
- The V1 deliberately does not solve multi-file, folder, or symbol context needs.

### Risks

- **Misleading context expectations**: label the selected item as a file reference and avoid attachment-like claims; validate behavior with each supported provider.
- **Incorrect session scope**: resolve candidates against the focused session's working directory.
- **Large-repository latency or failure**: bound discovery time and results, preserve ordinary typing on failure, and show a clear empty/error state.
- **Unsafe or noisy candidates**: stay within repository boundaries, respect established ignore behavior, and exclude unsupported file types.

## Implementation Notes

- Reuse the prompt-local completion interaction: typed filter, keyboard navigation, explicit accept, and dismiss without mutating the draft.
- Use an on-demand, session-aware repository file source with no watcher, persistent index, or cross-session cache in V1.
- Treat selected references as prompt text; do not mutate tool-derived session file metadata.
- Instrument opt-in, content-free interaction metrics: invocation, selection, correction, latency, and repeat use.
