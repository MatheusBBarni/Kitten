# ADR-003: Shell Runtime and Rendering Architecture

## Status

Accepted

## Date

2026-07-09

## Context

The integrated shell needs a real PTY-backed process, a terminal emulator to interpret its byte stream, and a place in Kitten's layered architecture (pure Domain Core -> store -> controller -> UI, ADR-003 of the original app).
Kitten already runs an imperative process runtime per agent (`AgentConnection`) owned by the `SessionController`, translates its stream into pure `DomainSessionEvent`s, reduces them into an immutable store via `sessionReducer`, and coalesces high-frequency output with a per-frame (~16 ms) scheduler.
Bun 1.3.13 is in use, which includes native PTY support (`Bun.Terminal`, since 1.3.5). No terminal-emulation dependency exists yet.

## Decision

Model the shell as a new imperative runtime that mirrors the existing agent pattern, and split its state into a render representation and a semantic representation.

1. **PTY backend:** spawn one long-lived interactive `$SHELL` through `Bun.Terminal` (native PTY). Persistent cwd, env, tab completion, and history come from the real shell process, not from Kitten code.
2. **ShellRuntime (imperative, controller-owned):** a `ShellRuntime` owns the PTY, the `@xterm/headless` emulator instance, the scrollback ring, and the semantic command ring. It is created and disposed by `SessionController`, exactly like `AgentConnection`, and is obtained through an injectable `ShellRuntimeFactory` seam (mirroring `TransportFactory`) so tests drive it without a real PTY.
3. **VT emulation:** feed PTY bytes to `@xterm/headless` (the VS Code terminal core, pure JS/WASM). A thin bridge reads its active screen buffer and renders styled cells into OpenTUI each frame, driven by the existing frame scheduler.
4. **Two representations:**
   - Render state: the emulator's screen + a bounded scrollback ring, kept imperative in the `ShellRuntime` (never spread cell-by-cell through the immutable store).
   - Semantic state: a pure `shell` slice on `AppState` holding `cwd`, a bounded ring of command records (command, output, exit code), running status, and a monotonically increasing render revision. A pure `shellReducer` writes it from `ShellEvent`s, matching the single-writer reducer discipline.
5. **Bounded memory:** scrollback and the command ring are fixed-size ring buffers; the UI renders only the visible window.

## Alternatives Considered

### Alternative 1: Put the full screen grid in the immutable store

- **Description**: Keep the terminal grid as immutable state in `AppState`, spread on every chunk.
- **Pros**: One uniform state model; the UI reads the grid like any other slice.
- **Cons**: O(n) array spreads per output chunk; a `yes`-style flood repaints the whole store; violates the "narrow slice" performance rule the store exists to enforce.
- **Why rejected**: The render grid is high-frequency pixel-like data with no cross-agent consumers; it belongs in the imperative runtime, with only a revision counter crossing into the store.

### Alternative 2: node-pty for the PTY backend

- **Description**: Use the `node-pty` native addon instead of `Bun.Terminal`.
- **Pros**: Battle-tested; supports Windows ConPTY.
- **Cons**: Native addon build step, historically imperfect on Bun; redundant now that Bun ships a first-party PTY.
- **Why rejected**: `Bun.Terminal` is first-party, matches the stack, and V1 is POSIX-only (Windows is out of scope), so ConPTY is not needed.

## Consequences

### Positive

- Reuses proven patterns (imperative runtime in the controller, pure reducer, frame scheduler, injectable factory), so the layered architecture stays coherent.
- The semantic slice makes the hand-off snapshot a purely additive read, not a reverse-engineering of pixels.
- `@xterm/headless` is a mature, well-maintained emulator, avoiding a bespoke or young native VT engine.

### Negative

- Introduces a buffer-to-OpenTUI render bridge Kitten owns and must maintain.
- Two representations of shell state to keep consistent (render revision vs semantic slice).

### Risks

- **Render throughput** under output floods. *Mitigation:* frame-coalesced rendering, bounded scrollback ring, visible-window-only draw.
- **`@xterm/headless` + Bun interop** is unproven in this repo. *Mitigation:* a first-step integration spike before deeper work (see build order).

## Implementation Notes

- New files under `src/agent/` (or a new `src/shell/`) for `ShellRuntime` + `ShellRuntimeFactory`; `src/core/shellReducer.ts` + `shell` types in `src/core/types.ts`; a `ShellPane` view in `src/ui/`.
- `SessionController` gains ownership and disposal of the `ShellRuntime`, alongside the agent runtimes.
