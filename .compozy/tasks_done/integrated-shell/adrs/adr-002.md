# ADR-002: Ship the Full Cockpit Shell in One Release, With Interactive-App Takeover in the MVP

## Status

Accepted

## Date

2026-07-09

## Context

The `_idea.md` for the integrated shell set V1 as a persistent PTY shell that feeds a curated hand-off snapshot, and ADR-001 deferred both in-pane interactive apps and any full-window takeover.
During PRD brainstorming the product owner made two decisions that tighten and expand that scope:

- The shell is a **toggle full-width pane** (matching Kitten's "focused pane, not a split" design), not a split or an ephemeral overlay.
- When a user launches a full-screen interactive app (vim, lazygit, htop), V1 does a **full-window takeover** (hand the whole terminal to the app, restore the cockpit on exit) rather than declining with a redirect message.

The owner then chose **Approach A**: ship the persistent shell, the takeover, the curated hand-off snapshot, and discovery affordances as one release, instead of phasing the shell and the differentiator apart.

The forces: the council concluded that a shell without the hand-off ride-along is Toad parity that cannot justify reversing the prior deferral, so the shell and the snapshot belong together.
Separately, the feature's headline promise is "never leave the cockpit," and a shell that refuses vim/lazygit breaks that promise at the take-the-wheel moment (the loudest preserved dissent).
The takeover closes that gap without building in-pane VT emulation.

## Decision

Adopt **Approach A**. The MVP is a single release containing:

1. A toggle full-width persistent PTY shell (cwd/env persist, full-color, tab completion, navigable scrollback), running while hidden.
2. A generalized pane-focus model and a fixed `Ctrl+C` that interrupts the focused shell's foreground command instead of quitting the app.
3. **Full-window takeover** for full-screen interactive apps: on detecting the alt-screen escape, Kitten suspends its own rendering, hands the whole terminal to the app, and restores the cockpit on exit.
4. A curated hand-off snapshot (cwd + recent command records) auto-suggested in the existing hand-off preview, dropped per item with `Space`, redacted, env excluded.
5. Discovery via a status-strip affordance and the F1 help panel.

This **amends ADR-001**: the full-window takeover moves from "deferred (V1.5/V2 open question)" into the MVP. ADR-001's other deferrals (in-pane VT emulation of interactive apps, the shared substrate, env in the snapshot, Windows) stand.

## Alternatives Considered

### Alternative 1: Approach B - Shell first, differentiator second

- **Description**: Phase 1 ships the shell + takeover + discovery; the hand-off snapshot lands in Phase 2.
- **Pros**: Smaller Phase 1; validates "never leave the cockpit" before investing in the ride-along; ships a usable terminal sooner.
- **Cons**: Phase 1 is Toad parity that does not test the moat, which is exactly what the prior idea deferred; delays the only differentiated part.
- **Why rejected**: The council held that the ride-along is the sole justification for building the shell now; deferring it ships parity and postpones the learning that matters.

### Alternative 2: Approach C - Differentiator first, decline interactive

- **Description**: Phase 1 ships the shell + hand-off snapshot but only declines interactive apps; takeover comes in Phase 2.
- **Pros**: Smallest Phase 1; gets the unique moat feature out fastest.
- **Cons**: Reopens the take-the-wheel gap the owner chose to close; breaks the "never leave the cockpit" promise for interventions that need vim/lazygit.
- **Why rejected**: Contradicts the owner's explicit decision to include takeover in the MVP.

## Consequences

### Positive

- The feature ships whole: users get a real terminal that also feeds the hand-off, and the moat test runs from day one.
- The "never leave the cockpit" promise holds even for interactive take-the-wheel moments, defusing the strongest council dissent.
- A single coherent release avoids a half-feature intermediate state.

### Negative

- Largest MVP: the focus/`Ctrl+C` rework, the takeover, and the hand-off wiring all land together, increasing release size and coordination.
- Raises the bar for the first release to be correct across more surfaces at once (rendering, focus, interrupt, takeover restore, redaction).

### Risks

- **Takeover restore correctness** - suspending and restoring the cockpit around an external app must leave the UI uncorrupted. *Mitigation:* treat restore-on-exit and resize as explicit acceptance criteria; the app already owns raw-mode/alt-screen control it can hand off and reclaim.
- **Scope pressure** - a large MVP can slip. *Mitigation:* the phased rollout still sequences internally (focus/interrupt first, then shell, then takeover, then snapshot), even though all ship in one release.

## Implementation Notes

Deferred to the TechSpec. Product-level only: the takeover reuses the terminal-ownership primitives the TUI already controls; the snapshot reuses the existing hand-off preview curation and the arbitrary-text redactor.
