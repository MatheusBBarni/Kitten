# ADR-005: In-Pane Interactive Apps, Pane Focus, and Ctrl+C Routing

## Status

Accepted

## Date

2026-07-09

## Context

The PRD (ADR-002) put interactive-app support in the MVP and originally framed it as a full-window takeover, with in-pane rendering deferred to Phase 2.
During technical design the owner chose **in-pane alt-screen emulation** for the MVP instead.
Because the shell already renders through `@xterm/headless`, the emulator can model the alternate-screen buffer directly, so a full-screen app can render inside the full-width shell pane without handing the window off and restoring it.
This ADR also settles two coupled decisions the shell forces: how keyboard focus routes to the shell, and what `Ctrl+C` does (today `exitOnCtrlC: true` makes it quit the whole app, `src/index.ts:60-63`).

## Decision

1. **Interactive apps render in-pane.** All PTY output, including alt-screen apps, feeds `@xterm/headless`; the bridge renders whichever buffer is active (primary or alternate) into the full-width shell pane. There is no window handover and no OpenTUI suspend/restore. When an app enters the alternate screen, the pane may expand to full height (hiding the prompt/status strip) to maximize space.
2. **Pane focus union.** Replace the agent-only focus model with a discriminated focus: `focusedPane: { kind: "agent"; agentId } | { kind: "shell" }`, keeping the active agent id for the conversation behind the shell. A toggle chord (VS Code convention `Ctrl+` `` ` ``) shows/focuses the shell and returns focus to the agent.
3. **Input routing.** While the shell is focused, key (and later mouse) events are encoded to terminal byte sequences and written to the PTY; the global cockpit chords stand down, mirroring the overlay stand-down pattern. Exactly one key stays Kitten's while the shell is focused: the toggle chord, so the user can always leave.
4. **Ctrl+C via the PTY line discipline.** Set `exitOnCtrlC: false`. When the shell is focused, forward the `0x03` byte to the PTY, whose termios line discipline delivers SIGINT to the foreground process group; Kitten never quits. When the shell is not focused, `Ctrl+C` retains the existing app-quit teardown.
5. **Resize.** On terminal or pane resize, resize both the PTY (`Bun.Terminal.resize`) and the emulator so output reflows correctly.

## Alternatives Considered

### Alternative 1: Full-window raw PTY passthrough (the PRD's original mechanism)

- **Description**: On alt-screen entry, suspend OpenTUI, resize the PTY to the full terminal, pipe raw bytes both ways, restore on exit.
- **Pros**: The app draws itself to the real tty; simplest rendering path; robust.
- **Cons**: Suspend/restore dance around OpenTUI; the app briefly owns the whole screen; Kitten chrome disappears.
- **Why rejected**: The owner chose in-pane emulation, which keeps Kitten chrome and avoids the suspend/restore lifecycle. Recorded here because it remains the safer fallback if in-pane emulation proves unreliable.

### Alternative 2: Special-case `isShell` boolean beside `focusedAgentId`

- **Description**: Add a boolean rather than a pane union.
- **Pros**: Smallest diff.
- **Cons**: Forks the concept of focus; every future focusable surface pays the tax.
- **Why rejected**: The Architect flagged this as accidental debt; a discriminated union generalizes cleanly.

## Consequences

### Positive

- "Never leave the cockpit" holds for interactive tools with Kitten chrome intact.
- `Ctrl+C` behaves like a real terminal when the shell is focused, and the rage-quit bug is removed.
- The focus model extends to future panes without rework.

### Negative

- Takes on the hardest part of terminal emulation (full cursor addressing, resize) in the MVP.
- The global keymap is no longer literally global: it stands down while the shell is focused, a behavior change to document.
- **Supersedes the PRD's stated MVP mechanism** (full-window takeover) and pulls the PRD's Phase 2 "in-pane rendering" into the MVP; the PRD Phased Rollout should be reconciled.

## Implementation Notes

- Mouse forwarding is deferred (keyboard-only for V1); `vim`/`lazygit`/`htop` are usable by keyboard.
- The key-to-VT encoder is part of the bridge; the toggle chord's exact binding is confirmed as `Ctrl+` `` ` `` with a terminal-reporting fallback noted as an open question.
