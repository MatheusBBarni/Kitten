# ADR-001: Ship a Bounded, Attention-Safe Session-Tab Lifecycle

## Status

Accepted

## Date

2026-07-12

## Context

Kitten already manages an ordered collection of independent live agent conversations, but presents only one focused conversation at a time. Users need to supervise parallel work without losing context, and requested tab creation, renaming, keyboard and mouse navigation, and close behavior.

Market research shows that parallel agent threads and tabbed conversations are becoming expected in coding-agent tools. The core product risk is not displaying tabs; it is hiding approvals or errors, confusing a tab close with cancellation of live work, or making background conversations unreachable on narrow terminals. Legacy terminals may also collapse `Ctrl+H` into Backspace and reserve `Ctrl+L` for redraw/clear behavior.

The council debated a smaller read-only session switcher versus a full tab lifecycle, and whether overflow and attention visibility are V1 concerns. The user prioritizes a complete workflow and approved exact navigation keys with a safe terminal fallback.

## Decision

Ship a bounded V1 session-tab lifecycle in which each tab represents one live agent conversation.

V1 includes:
- creating a conversation tab;
- user-controlled tab renaming;
- selection by mouse and previous/next keyboard navigation, using `Ctrl+H`/`Ctrl+L` when safely distinguishable plus a documented fallback;
- visible active, approval, error, and activity/attention state;
- deterministic access to off-screen tabs and direct navigation to sessions needing attention;
- an explicit close flow that never silently cancels or terminates active agent work;
- persistence of the minimum tab metadata needed to restore identity, display name, order, focus, and recoverable attention state.

V1 excludes reordering, pinning, grouping, detachable panes, multi-row tab layouts, automatic title generation, automatic workspace/worktree isolation, and silent agent termination.

## Alternatives Considered

### Alternative 1: Read-only session switcher experiment

- **Description**: Show existing configured sessions as tabs with keyboard/mouse switching and attention badges, but no creation, rename, close, or persistence.
- **Pros**: Lower delivery risk; validates switching demand quickly; reuses the current fixed session model.
- **Cons**: Does not meet the requested complete workflow; creates misleading tab expectations without lifecycle ownership; cannot validate naming or safe close semantics.
- **Why rejected**: Useful as a fallback experiment, but too narrow for the user-approved V1 and does not establish the intended product model.

### Alternative 2: Full workspace manager

- **Description**: Combine tabs with reordering, groups, pinning, panes, worktree isolation, richer automation, and advanced organization.
- **Pros**: Stronger long-term supervision and conflict-management story.
- **Cons**: Expands scope substantially, delays learning, and mixes tab navigation with workspace orchestration before demand is proven.
- **Why rejected**: Overbuilds beyond the immediate switching and conversation-management problem.

## Consequences

### Positive

- Delivers the end-to-end workflow users expect from tabs rather than navigation chrome alone.
- Makes parallel conversations easier to supervise while preserving per-session identity and state.
- Treats approval/error reachability and lifecycle safety as release requirements.
- Leaves a clear product boundary for future workspace-management features.

### Negative

- Requires product-level lifecycle and persistence semantics beyond the current fixed config-seeded session collection.
- Requires careful terminal compatibility behavior for the requested navigation chords.
- Adds UI density and interaction complexity in narrow terminal windows.

### Risks

- **Accidental agent cancellation:** separate tab removal/closure from cancellation or termination, require explicit confirmation for destructive actions, and preserve a clear state transition.
- **Hidden urgent work:** retain visible or aggregate approval/error indicators and deterministic access to every tab, including off-screen tabs.
- **Concurrent edits in one checkout:** surface shared-workspace context and defer automatic isolation to a later release.
- **Ambiguous labels:** retain stable identity and disambiguate duplicate display names.
- **Terminal key ambiguity:** provide a discoverable fallback whenever the terminal cannot safely distinguish `Ctrl+H` or `Ctrl+L`.

## Implementation Notes

The product boundary is tab metadata and tab lifecycle on one side, and independent agent execution, permission handling, and cancellation on the other. Future technical design must preserve Kitten's core/store/controller/UI layering and define the behavior when the final visible tab is closed.
