# ADR-002: Prioritize a Restorable, Fast-Switching Conversation Tab Workspace

## Status

Accepted

## Date

2026-07-12

## Context

The Session Tabs idea establishes that Kitten users need a safe way to supervise parallel live agent conversations. Fresh PRD research confirms that adjacent products treat separate agent conversations, quick task switching, clear labels, reachable overflow, and non-destructive exit behavior as expected workflow capabilities.

The primary V1 user is a solo developer coordinating two to four conversations. They value fast context switching above all else, but must not miss approvals, errors, or finished work. They expect a new tab to begin a fresh conversation, an active-work close action to offer explicit choices, and a saved run to restore the full tab workspace.

The product decision must preserve that focused workflow without reducing tabs to a passive session picker or expanding V1 into a general workspace-management product.

## Decision

Ship V1 as a **restorable, fast-switching conversation tab workspace**.

Each tab represents one live agent conversation. Users can create a fresh conversation tab, rename it, select it by mouse, move to adjacent tabs with `Ctrl+H`/`Ctrl+L` when safely available plus a discoverable fallback, and understand which tabs need attention.

V1 has three user-visible lifecycle states:
- **Visible** — represented in the immediate tab strip and selectable directly.
- **Background-running** — no longer in the immediate strip, but the conversation remains live, appears in the background-work and attention summary, and remains reachable and restorable.
- **Closed** — removed from the workspace; it has no continuing live work and does not return on restore.

Closing a tab with active work presents clear user choices: keep it running in the background, cancel it deliberately, or leave it open. If the final visible tab is moved to the background or closed, Kitten remains open on an empty workspace with a primary new-conversation action and any reachable background-work summary; it does not exit the run.

This ADR amends ADR-001's restoration language. Restoring a saved run returns the full user workspace: visible and background-running conversation state, names, order, selected tab, and retained conversation context. V1 prioritizes immediate switching and reliable awareness over advanced workspace organization.

## Alternatives Considered

### Alternative 1: Static session switcher

- **Description**: Present only existing configured sessions as selectable tabs and defer tab creation, renaming, close behavior, and restoration.
- **Pros**: Smaller initial scope and faster validation of switching demand.
- **Cons**: Does not meet the requested fresh-conversation or lifecycle workflow; leaves users without meaningful ownership of their tab workspace.
- **Why rejected**: It tests navigation but not the complete user value proposition.

### Alternative 2: Workspace manager

- **Description**: Add organization, workspace isolation, richer multi-task management, and advanced tab controls in V1.
- **Pros**: Addresses broader coordination and parallel-work concerns.
- **Cons**: Delays the core switching benefit, increases cognitive load, and expands beyond the primary solo-developer workflow.
- **Why rejected**: It is disproportionate to the V1 learning goal.

### Alternative 3: Close always cancels active work

- **Description**: Treat closing a working tab as a confirmed cancellation action.
- **Pros**: Simple mental model and fewer visible background conversations.
- **Cons**: Makes an ordinary organization action destructive and undermines trust in long-running agent work.
- **Why rejected**: The user explicitly requires a choice between background execution, deliberate cancellation, and leaving the tab open.

## Consequences

### Positive

- Gives solo developers an end-to-end workflow for managing a small parallel fleet.
- Aligns the product with common conversation-tab expectations while keeping Kitten's provider-neutral focus.
- Makes attention and lifecycle clarity part of the core value rather than optional polish.
- Supports continuity across saved runs instead of forcing repeated setup.

### Negative

- Requires clear terminology to distinguish hiding a conversation, keeping it running, cancelling it, and ending a tab.
- Adds user-facing state to explain when a restored conversation is unavailable or needs attention.
- Requires a discoverable fallback when a terminal cannot safely support the requested shortcut pair.

### Risks

- **Accidental loss of work:** use explicit active-work choices and confirmation language that states the consequence.
- **Hidden urgent state:** retain visible approval/error/finished signals and a direct path to every conversation needing attention, including background-running work.
- **Background-work confusion:** define visible, background-running, and closed states in the product and show background work in a reachable summary.
- **Shortcut inconsistency:** document the fallback and ensure no user is blocked from tab navigation.
- **Overgrown V1:** defer advanced organization and workspace-management features until repeat-use evidence supports them.

## Implementation Notes

The PRD will define user-visible capabilities and acceptance outcomes only. Detailed interaction design and delivery mechanics belong in the subsequent TechSpec.
