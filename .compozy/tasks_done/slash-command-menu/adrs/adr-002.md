# ADR-002: Bundle the slash-menu V1 into a single release

## Status

Accepted

## Date

2026-07-10

## Context

The hybrid V1 (ADR-001) has three deliverables: the ACP command slice (surfacing the focused agent's advertised commands), the unified `/` palette, and the always-visible Ctrl+T footer hint.
These can ship together or be staged across releases.

ADR-001 advised landing the ACP slice as its own PR first, partly to avoid a merge conflict with a live parallel task (the Ctrl+S sessions overview) that was editing `keymap.ts`, `appStore.ts`, and `selectors.ts`.
That task has since landed and committed (`b34ab45`), and the working tree is clean of that churn, so the coordination risk that motivated staging is largely gone.

The open decision is the rollout strategy: one bundled release, a cheap footer-first bet that measures before building the palette, or a Kitten-actions-first palette with the agent commands as a fast-follow.

## Decision

Ship all three deliverables as a single bundled V1 release.

Internally, sequence the work in the ADR-001 order (the ACP command slice first as its own PR, since it is pure-core testable and lowest risk, then the palette, then the footer), but release them together as one feature.
The agent-command surface stays on the critical path because it is the single biggest user win: those commands are 100% unreachable in Kitten today, and any approach that defers them delays the highest-value half of the feature.

## Alternatives Considered

### Alternative 1: Footer-first staged bet (Approach B)

- **Description**: Ship only the Ctrl+T footer, instrument first-session hand-off rate, then build the palette in a second release once data confirms the discoverability thesis.
- **Pros**: Near-zero risk first step; validates the north-star metric cheaply; hedges the "is a palette worth it" dissent.
- **Cons**: Two releases; the agent-command unlock stays unavailable longer; the footer alone is a thin experience that may under-represent the palette's value.
- **Why rejected**: Delays the highest-value half for data we can also gather after a bundled launch. The footer is retained inside V1 as the cheap complement, just not shipped alone.

### Alternative 2: Kitten-actions palette first, agent commands fast-follow (Approach C)

- **Description**: Ship the palette with cockpit actions plus the footer first (no ACP slice), then add the agent-command surface as a fast-follow.
- **Pros**: Defers the only greenfield/risky piece; the discoverability-of-cockpit-actions win ships fastest.
- **Cons**: The palette debuts as a single-source menu like every rival, so it re-opens the "just catch-up" critique until the fast-follow lands; the differentiator is absent at launch.
- **Why rejected**: The two-agent grouping is what makes the palette more than parity; launching without it wastes the first impression.

## Consequences

### Positive

- The biggest single user win (agent commands going from unreachable to reachable) ships immediately.
- The differentiator (two-agent grouped palette + discoverable hand-off) is present at launch, not deferred.
- One coherent release and one thing for users to learn.

### Negative

- Largest single release of the three options.
- The greenfield ACP slice sits on the critical path.

### Risks

- **ACP slice complexity.** Mitigation: build it first as its own PR, tested in the pure core, before any UI depends on it.
- **Palette jank on the high-traffic prompt editor.** Mitigation: token-begin trigger guard plus no-match disarm, shipped behind tests, with the modal fallback from ADR-001 held in reserve.

## Implementation Notes

Internal landing order: (1) ACP command domain slice, (2) palette overlay + trigger + key capture, (3) footer affordance.
Release them together as V1.
