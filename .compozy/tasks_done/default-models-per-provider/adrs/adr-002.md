# ADR-002: Restore configured defaults on each intentional provider selection

## Status

Accepted

## Date

2026-07-13

## Context

The approved idea lets an individual developer declare a default model and reasoning effort for each provider. The remaining product decisions were whether a manual, in-session change should override that default on a future `/model` selection, and how Kitten should behave when the saved effort is not currently available.

## Decision

Each time a developer intentionally selects a provider-backed session in `/model`, Kitten will restore that provider's configured default. A manual change is a temporary session choice and does not survive the next intentional provider selection. If the configured model is available but the configured reasoning effort is not, Kitten will use the model, retain the last confirmed effort, and mark the result as partially applied in the existing picker and status strip. Kitten will never silently choose a different effort.

## Alternatives Considered

### Alternative 1: Keep manual changes for the lifetime of a live session

- **Description**: Let a manual model or effort choice override the configured default until that session ends.
- **Pros**: Preserves an ad hoc choice without reapplication.
- **Cons**: Makes the result of returning to a provider less predictable and weakens the meaning of a configured default.
- **Why rejected**: The user chose predictable reapplication on every intentional `/model` provider selection.

### Alternative 2: Leave both values unchanged when the saved effort is unavailable

- **Description**: Treat the saved model and effort as all-or-nothing.
- **Pros**: Avoids a partial result.
- **Cons**: Discards an available, requested model preference and leaves repetitive work unresolved.
- **Why rejected**: The user chose to retain the model change and communicate a partial result clearly.

### Alternative 3: Substitute another available effort automatically

- **Description**: Choose a nearby reasoning-effort level when the saved value is unavailable.
- **Pros**: Maximizes automatic configuration.
- **Cons**: Hides a consequential cost and quality decision from the developer.
- **Why rejected**: It would undermine trust through silent substitution.

## Consequences

### Positive

- Gives each deliberate provider selection a predictable, repeatable outcome.
- Reduces repeated model and effort setup without asking for another confirmation.
- Makes unavailable effort visible without blocking use of an available model.

### Negative

- A manual change is intentionally temporary and may surprise users who expect it to persist.
- A partial result requires concise, clear wording in existing UI surfaces.

### Risks

- **Expectation mismatch**: explain the temporary nature of manual changes in configuration documentation and in the picker state.
- **Stale preferences**: show the actual confirmed model and effort rather than silently substituting a value.
- **Configuration error**: retain strict validation so users receive actionable feedback for invalid entries.

## Implementation Notes

The technical specification must define how Kitten determines the actual confirmed result and how the existing picker and status strip communicate full, partial, and unavailable outcomes. It must preserve the user-authored nature of `config.json`.
