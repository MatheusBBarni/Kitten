# ADR-004: Reactive, persisted configuration - preferences slice, atomic read-modify-write, and a file-watcher

## Status

Accepted

## Date

2026-07-09

## Context

Kitten loads config once at boot (`loadAppConfig` in `createCockpitSession`) and never writes it back; nothing about config is reactive.
The settings modal needs three capabilities the codebase lacks: a config value that a view can change and see applied live, a safe way to persist that change to `~/.config/kitten/config.json`, and reconciliation so a change made in the modal and a hand-edit to the file do not diverge.
Constraints from the existing architecture: the store (`appStore.ts`) is the single reactive source and patches state with structural sharing; the config file is zod-`strict()` and expresses deltas over defaults; only the config layer may touch disk (ADR-003); a broken config is a hard error, never a silent fallback.
The PRD left the hot-reload policy open; during clarification the product owner chose to include a file-watcher in V1.

## Decision

Introduce a reactive, persisted configuration path built from three parts.

1. **Reactive slice.** Extend `AppConfig` and `USER_CONFIG_SCHEMA` with a `theme` field. Add a `preferences: { theme: ThemePreference }` slice to `AppState`, seeded from the loaded config at store creation, mutated by a `setThemePreference` action using the same structural-sharing patch as the overlay slots. `usePalette` reads it through a selector, so a change re-renders the cockpit live.
2. **Atomic read-modify-write.** Add a config writer in the config layer. On each save it re-reads the on-disk user config (deltas), sets only the changed key, validates the result against `USER_CONFIG_SCHEMA`, then writes atomically (write a temp file in the same directory, `mkdir -p` the parent, then rename over the target). Concurrent hand-edits to keys the modal does not own (agents, telemetry) survive. A validation or write failure keeps the in-memory live change and surfaces the failure without crashing.
3. **File-watcher.** Watch the config path and, on external change, re-read and apply the theme preference into the store (debounced). The reload is idempotent: if the reloaded value equals the current one (the common case after the modal's own write) it is a no-op, which prevents a write-reload feedback loop. A file that fails to parse mid-edit is treated as transient - the watcher keeps the current preference rather than throwing.

The UI triggers persistence through a controller action / config service, never writing disk from a view.

## Alternatives Considered

### Alternative 1: Serialize known deltas from memory (no re-read)

- **Description**: Track the delta shape in memory and write the whole set on each save.
- **Pros**: Simpler; no read step.
- **Cons**: Clobbers any key the in-memory model does not represent, for example a hand-added agent override.
- **Why rejected**: The read-modify-write preserves unmodeled keys, which is the whole point of the delta design.

### Alternative 2: Defer the file-watcher

- **Description**: Make the modal the only live write path; hand-edits apply on next launch.
- **Pros**: Smallest V1; matches how every other config field behaves today.
- **Cons**: A hand-edit while running is invisible until relaunch; the council's dissent flagged the split-brain smell.
- **Why rejected**: The product owner chose one live source of truth for V1.

### Alternative 3: A separate preferences store

- **Description**: A second reactive store for preferences.
- **Pros**: Isolates settings state.
- **Cons**: A parallel store and subscription path the codebase does not have.
- **Why rejected**: Reusing the one store (ADR-004's reactive model) is simpler and consistent.

## Consequences

### Positive

- Theme changes apply live and persist durably, and hand-edits reconcile through one path.
- The write path preserves keys the modal does not model, so it is safe against a partially hand-edited file.
- The reactive slice reuses the store's existing subscription and structural-sharing machinery.

### Negative

- Instant-apply navigation can fire the writer rapidly, so writes must be debounced or coalesced.
- The watcher adds a moving part with a self-write guard to reason about.

### Risks

- Watcher feedback loop: mitigated by the idempotent, compare-before-apply reload.
- Write thrash under live navigation: mitigated by debouncing the persist step (the store update stays synchronous, only the disk write is debounced).
- A mid-edit invalid file surfacing through the watcher: mitigated by ignoring parse failures in the watcher and keeping the current preference.

## Implementation Notes

- The writer and watcher target `resolveConfigPath()` and honor `KITTEN_CONFIG`, mirroring the loader.
- Follow the `recorder.ts` precedent for `mkdirSync` and path resolution; use rename-based atomicity rather than the recorder's append.
- The open-settings binding is `Ctrl+,`; on terminals without the Kitty keyboard protocol this chord may not be delivered, recorded here as a known limitation (see the TechSpec Known Risks).
