# ADR-001: Settings modal V1 scope - theme-first on a reactive-config foundation

## Status

Accepted

## Date

2026-07-09

## Context

Kitten is an early-stage TUI (Bun, TypeScript, OpenTUI, React) that runs Claude Code and Codex side by side over the Agent Client Protocol.
Its core value is a keyboard-driven hand-off flow that passes a live task between the two agents.
The proposed feature is an in-app settings modal so users can customize Kitten without hand-editing `~/.config/kitten/config.json`.

The product owner set the intent up front: the core value is unlocking *new* customization (theme selection and keymap remapping), not merely surfacing config that already exists.
The initial proposal was four categories in one V1 - theme selection, keymap remapping, agent launch-command plus telemetry editing, and a link-out to a separate (not-yet-built) model-provider-selector - all "live and persisted," framed as a Quick Win.

Codebase reality contradicts the "Quick Win for all four" framing:

- A reusable modal-overlay pattern already exists (store slot, keyboard capture, absolutely-positioned box), used by the approval prompt and hand-off preview.
- There is **no config write-back** anywhere; config is read once at boot and discarded.
- Config is **not reactive**; nothing applies live today.
- There is **no theme override** (the palette is auto-derived from the terminal's reported dark/light mode; two palettes are hard-coded).
- There is **no keymap override** (keymaps are frozen module constants; the matcher closes over them at import).
- The telemetry recorder is immutable at construction.
- Changing an agent launch command tears down and rebuilds that agent's connection, which **discards the live session** - so "live" cannot apply uniformly to all four categories.

A five-advisor council (pragmatic-engineer, architect-advisor, security-advocate, product-mind, devils-advocate) debated the scope and converged.

## Decision

Ship the settings modal in stages built on one shared foundation, rather than four categories at once.

1. **Foundation first (the real deliverable).** Build a reactive config slice in the store, a delta-only, atomic, schema-validated config write-back in the config layer, and config-file hot-reload so modal-edits and hand-edits reconcile through a single write/read path (no split-brain).
2. **V1 = the settings modal shell with theme as its one fully-built category.** The shell follows the existing overlay pattern (store slot joined into `selectHasOpenOverlay`, `useKeyboard` capture, a `keymap.ts` binding, absolutely-positioned box) and ships tabs, reset-to-default, a which-key hint footer, and a **live preview that repaints the real dual-agent cockpit** as the user moves through themes. Theme preference is `auto | light | dark` (named presets optional), consulted by `usePalette` ahead of the terminal-reported mode.
3. **Keymap remapping is the committed immediate fast-follow (V1.1)** on the same foundation: a serializable binding model, a predicate compiler, terminal-aware conflict validation, and hint derivation from the binding table.
4. **Defer from V1:** agent launch-command editing and the telemetry toggle (both surface existing config rather than new customization; launch-command additionally cannot be live and has the highest blast radius). The config file continues to cover them.
5. **Omit from V1:** the model-provider link-out, since that selector is not built (a door to an unbuilt room); at most a disabled placeholder.

## Alternatives Considered

### Alternative 1: All four categories, uniformly "live and persisted" (the original proposal)

- **Description**: Theme, keymap, launch-command + telemetry, and a model-provider link-out, all in one V1, all applied live and persisted.
- **Pros**: Feels complete; one shipping event; matches the raw ask.
- **Cons**: Not internally consistent - launch-command edits cannot be live (relaunch discards the session); hides four net-new subsystems plus a reactivity refactor under a "Quick Win" label; forces the keymap differentiator to ship inside a four-for-one sprint, risking a half-built headline.
- **Why rejected**: The "live" promise is physically false for one category and the scope is mislabeled; shipping the differentiator rushed is worse than sequencing it.

### Alternative 2: Config file-watcher only, no modal (Devil's Advocate)

- **Description**: Hot-reload `config.json` on change (the k9s pattern); users keep editing the file they already edit; no settings UI.
- **Pros**: Near-zero UI cost; serves the terminal-native power user; delivers "live + persisted" for the categories where live is possible.
- **Cons**: No discoverability (must know the schema), no capture-a-key rebinding, no live theme preview, no differentiated frame; loses the in-app surface that gemini-cli and toad use to stand out.
- **Why rejected as the primary surface, adopted as the foundation**: The file-watcher is the correct single source of truth and is built underneath; the modal is a thin writer on top of it. Rejected only as a *replacement* for the modal.

### Alternative 3: Theme as a bare `/theme` command (not a modal)

- **Description**: Ship theme switching as a slash command with no settings shell.
- **Pros**: Cheapest possible theme delivery.
- **Cons**: Looks like OpenCode's `/theme`, not like a dual-agent tool; discards the modal frame that keymap and later settings need to slot into.
- **Why rejected**: Theme is the least differentiated of the four; shipping it without the frame wastes the one increment that plants the differentiated surface.

## Consequences

### Positive

- Proves config write-back and reactive config end to end on the cheapest payload (a single theme enum).
- Lands the differentiated modal frame now, so keymap drops in next with no shell work.
- One write/read path (modal + hot-reload) prevents the split-brain bug between in-memory state and the on-disk file.
- Honest live-vs-restart labeling protects the live hand-off session the product is built around.

### Negative

- Theme alone is table-stakes parity (OpenCode already ships an in-app theme picker), so the first public increment is not itself a differentiator.
- The foundation work (reactive slice, delta writer, hot-reload) is larger than a naive "theme picker" and precedes visible payoff.
- Users wanting launch-command or telemetry control from the UI must still use the config file until a later slice.

### Risks

- **Config corruption on write** - mitigated by writing to a temp file then atomic rename, emitting deltas over defaults only, and validating the serialized bytes against the same zod schema before commit; on failure keep the in-memory change but surface the persistence error loudly.
- **Split-brain between modal and hand-edits** - mitigated by building hot-reload as the single source of truth and making the modal write through it.
- **Over-promising "live"** - mitigated by labeling which settings apply live vs require restart, and by deferring the non-live categories out of V1 entirely.
- **Keymap footguns (in the fast-follow)** - mitigated by terminal-aware conflict validation: reject bare printables for global bindings, reject legacy-ambiguous chords (Ctrl+H/I/M/[), reject cross-keymap collisions, and keep the old binding when a capture is rejected.
- **Effort spent off the moat** - mitigated by making V1's own telemetry measure whether theme and settings are actually used, so the dissent (this is decoration, not need) is tested rather than assumed.

## Implementation Notes

- Reuse the overlay machinery verbatim: add a `settings` slot to the store's overlay state, expose it through selectors, join it into `selectHasOpenOverlay`, add an `open-settings` binding to `COCKPIT_KEYMAP`, and mount the component below the approval prompt so a permission request still outranks it.
- The write-back must invert `mergeAppConfig` (serialize the diff against `defaultAppConfig()`), not dump the merged object, to preserve the delta-over-defaults design and any keys the modal does not own.
- Honor existing conventions: no hard-coded colors (palette tokens only), the keymap table stays the single source of truth for dispatch, help, and hints, side-effect-free module imports, and the strict layering where only the config layer writes disk.
- Keep every config-write path human-driven and unreachable from any ACP/agent code path.
