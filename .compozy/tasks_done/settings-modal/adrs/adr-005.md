# ADR-005: Theme override via a palette registry consulted ahead of terminal mode

## Status

Accepted

## Date

2026-07-09

## Context

Today the palette is derived purely from the terminal's reported mode: `usePalette()` calls `paletteFor(useThemeMode())`, which branches between two hard-coded constants, `DARK_PALETTE` and `LIGHT_PALETTE`.
There is no user preference input, and `syntaxStyleFor` caches its syntax style per `ThemeMode`.
V1 must let a user pin `light` or `dark`, keep `auto` (follow the terminal), and pick one of 1-2 named presets (ADR-003), applied live.

## Decision

Introduce a user theme preference and a small palette registry.

- Define `ThemePreference = "auto" | "light" | "dark" | <presetId>`, persisted in config (ADR-004).
- Introduce a keyed palette registry: the built-in `dark` and `light` palettes plus 1-2 preset palettes, each palette carrying a stable `id`.
- `usePalette` resolves the effective palette from the preference and the live terminal mode: `auto` resolves through `paletteFor(terminalMode)` (unchanged behavior); any other value resolves through the registry, falling back to the terminal-derived palette if a persisted preset id is unknown.
- It reads the preference from the store's `preferences` slice, so it re-renders on a preference change (live) and still re-renders on a `theme_mode` flip while the preference is `auto`.
- Key `syntaxStyleFor` by the effective palette `id` rather than by `ThemeMode`, so a preset gets its own syntax style.

## Alternatives Considered

### Alternative 1: Keep the two-palette light/dark branch

- **Description**: Extend `paletteFor` with more `if` branches.
- **Pros**: No new structure.
- **Cons**: Cannot express named presets cleanly; every new preset edits the resolver.
- **Why rejected**: A registry is the natural shape once there are more than two palettes.

### Alternative 2: Per-preset ad hoc wiring (no registry)

- **Description**: Special-case each preset where it is consumed.
- **Pros**: None meaningful.
- **Cons**: Scatters palette knowledge; violates the single-source palette convention.
- **Why rejected**: Defeats the no-hard-coded-color, one-palette-source discipline.

## Consequences

### Positive

- Adding a future preset is a registry entry, not a resolver edit.
- The `auto` path is unchanged, so today's terminal-follow behavior is preserved exactly.

### Negative

- The syntax-style cache key changes from mode to palette id, a small refactor of `syntaxStyleFor`.

### Risks

- A preset may read poorly on some terminal backgrounds; mitigated by keeping presets to 1-2 in V1 and reviewing each against light and dark backgrounds.
- A stale or removed preset id left in a hand-edited config must not break rendering; mitigated by the fallback to the terminal-derived palette for an unknown id.

## Implementation Notes

- Preset palettes must satisfy the existing legibility rule (foreground colors chosen to read against whichever terminal background is present) and use the palette token structure, not raw hex at call sites.
- The specific preset(s) to ship are an open content decision recorded in the TechSpec and PRD.
