# ADR-004: Recheck only the selected unavailable Cursor session

## Status

Accepted

## Date

2026-07-16

## Context

The PRD requires a user to recheck Cursor after completing a recovery action. Kitten's configured-session start is currently internal and no controller action exposes a recheck. A full application restart would disrupt healthy Claude Code and Codex sessions, contradicting the product's per-agent resilience guarantee.

The existing store and controller already own per-session availability and runtime state. ACP lifecycle calls stay in the adapter; the UI must operate through `ControllerActions` rather than a connection directly.

## Decision

Add one narrow `ControllerActions` recheck operation for a selected unavailable configured Cursor session. It restarts that session's existing readiness and connection lifecycle, replaces only its availability/runtime state, and catches failures into the existing normalized Cursor readiness presentation. It does not restart the app, mutate a healthy sibling session, create a new core event type, or expose adapter internals to the UI.

The UI offers recheck only when the normalized recovery state identifies a user-remediable condition. An unreviewed or mismatched profile remains visible and unsupported; recheck may reconfirm it but does not bypass certification.

## Alternatives Considered

### Alternative 1: Require a full Kitten restart

- **Description**: Ask users to relaunch the application after any recovery action.
- **Pros**: Avoids a new action surface.
- **Cons**: Interrupts healthy sessions and makes recovery unnecessarily disruptive.
- **Why rejected**: It violates per-agent isolation and provides a worse recovery path than the existing architecture can support.

### Alternative 2: Defer in-app recheck

- **Description**: Show guidance but omit any recheck until a later phase.
- **Pros**: Smallest code change.
- **Cons**: Leaves the PRD's recovery journey incomplete and forces users to guess when state has changed.
- **Why rejected**: The selected-session action is narrow and directly completes the promised user flow.

### Alternative 3: Generic provider restart action

- **Description**: Add a cross-provider restart abstraction for all agent sessions.
- **Pros**: Could be reused by future providers.
- **Cons**: Broadens scope and risks changing mature provider behavior for a Cursor-specific need.
- **Why rejected**: V1 needs only Cursor recheck; a generic framework is premature.

## Consequences

### Positive

- Completes the recovery journey without disturbing ready providers.
- Preserves the controller as the only UI-facing orchestration seam.
- Keeps unsupported-profile behavior fail-closed and explicit.

### Negative

- Adds a small lifecycle path that requires focused controller and UI coverage.
- A recheck can repeat a known unsupported result, which must remain clearly distinguishable from a user-fixable failure.

### Risks

- **Cross-session disruption**: constrain the action to one selected configured session.
- **Unbounded retries**: expose only deliberate user-triggered recheck; no automatic loop.
- **Misleading recovery**: show recheck only for states where it is meaningful and preserve maintainer-owned certification wording.

## Implementation Notes

- Place orchestration in `src/app/controller.ts` and expose it through `src/app/actions.ts`.
- Reuse existing readiness normalization and store availability updates.
- Keep ACP calls within `src/agent/agentConnection.ts`; do not let UI code access a connection.
