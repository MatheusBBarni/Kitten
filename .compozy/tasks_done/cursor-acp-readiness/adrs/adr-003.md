# ADR-003: Keep Cursor certification compiled and gate it on reviewed native evidence

## Status

Accepted

## Date

2026-07-16

## Context

Kitten already models a Cursor runtime profile as an exact built-in recipe with strict command, ordered arguments, environment, and semantic-version matching. The production registry is intentionally empty because no reviewed native Cursor evidence exists. The PRD requires one exact profile to be supported only after it completes the full lifecycle, while keeping all other configurations fail-closed.

The certification decision must not create a new user-configurable trust surface or imply that synthetic coverage alone proves compatibility. The normal test suite must remain credential-free and deterministic, while native Cursor behavior requires a reviewed local installation.

## Decision

Keep the first reviewed Cursor profile in the existing compiled registry. Add a literal exact profile only after the focused deterministic suite and a reviewed opt-in native contract run both pass for the observed local runtime. Do not add a profile manifest, user override, version range, runtime discovery acceptance, or guessed version literal.

The compiled profile remains the only source of a `cursor-certified` runtime decision. Any mismatch stays in the existing fail-closed readiness path. Renewal follows the same mock-plus-native gate before a changed profile is committed.

## Alternatives Considered

### Alternative 1: Repository-owned certification manifest

- **Description**: Move reviewed profiles into a separate checked-in data file.
- **Pros**: Separates profile data from configuration code and could support more profiles later.
- **Cons**: Adds a new loader, validation path, and abstraction for one profile.
- **Why rejected**: The existing compiled registry is the smallest trusted boundary for V1.

### Alternative 2: Dynamic profile discovery

- **Description**: Accept a local Cursor runtime when discovery or a loose version rule appears compatible.
- **Pros**: Broadens reach and reduces explicit renewals.
- **Cons**: Converts an unverified runtime into a support claim and weakens fail-closed behavior.
- **Why rejected**: Presence or a version string cannot replace reviewed lifecycle evidence.

### Alternative 3: Synthetic tests only

- **Description**: Commit a profile after mocks pass, deferring the native run.
- **Pros**: Fast and deterministic.
- **Cons**: Does not prove the credentialed local Cursor lifecycle.
- **Why rejected**: It would certify behavior that has not been observed in the native provider.

## Consequences

### Positive

- Reuses the narrowest existing trust boundary and keeps certification easy to audit.
- Makes the release gate explicit: focused mocks plus an opt-in reviewed native run.
- Prevents accidental broadening through user configuration or version ranges.

### Negative

- Maintainers must renew evidence whenever the reviewed local profile changes.
- A missing local Cursor installation blocks certification rather than allowing a best-effort release.

### Risks

- **Stale profile evidence**: require the same gate for renewal and retain revocation behavior from ADR-001.
- **Guessed version literals**: prohibit registry changes until the native contract records the observed runtime.
- **Slow native verification**: keep it opt-in and separate from the normal deterministic suite.

## Implementation Notes

- Keep exact profile matching in `src/config/configLoader.ts` and preflight in `src/config/readiness.ts`.
- Keep the normal suite free of credentials and external Cursor dependencies.
- Treat a failed or unavailable native run as a certification blocker, not as permission to weaken matching.
