# ADR-003: Cockpit-Run Persistence - One JSON File Per Run, Debounced Store Subscription

## Status

Accepted

## Date

2026-07-09

## Context

Session resume needs a durable record of each cockpit run.
The record is small - two ACP session ids, light metadata (last prompt, git branch, project, timestamps, message counts, per-agent status), and the curated `HandoffBundle` - because transcripts are rehydrated live from each agent rather than stored (PRD "pointers + bundle only").
The record is written on by default, kept until the user deletes it, scoped to the current project in the picker, and deletable per-session and globally.

Kitten already has the two patterns this needs.
The telemetry recorder resolves an XDG-state path and appends JSONL through `createJsonlFileSink`, and it subscribes to the app store via `recorder.watch(store)`.
Configuration is a deltas-from-default file with a single `telemetryEnabled` boolean gate, defaulted, schema-validated, and merged.

## Decision

Persist each run as **one JSON file per run**, and drive writes by **subscribing to the app store, debounced**.

- **Layout:** `~/.local/state/kitten/sessions/<project>/<runId>.json`, where `<project>` is a stable encoding of the run's `cwd` and `<runId>` is a Kitten-minted id independent of ACP session ids. Resolve the base path with the same XDG-state logic as `resolveTelemetryPath`, honoring an explicit `KITTEN_SESSIONS_PATH` override.
- **Record shape:** `{ version, runId, cwd, gitBranch, focusedAgentId, createdAt, updatedAt, agents: Record<AgentId, { sessionId, lastPrompt, messageCount, status }>, handoffBundle }`. The `version` field gates future migrations.
- **Write trigger:** a persistence writer subscribes to the store (mirroring `recorder.watch`), coalesces changes with a short debounce (~250 ms), overwrites the run file atomically, and flushes once more on dispose/exit. Overwrite-in-place is safe because the record is small and last-write-wins.
- **Consent gate:** a new `persistenceEnabled` config flag, **defaulting to true**, mirroring `telemetryEnabled` exactly (default const, `defaultAppConfig` field, `z.boolean().optional()` schema key, `??` merge). When disabled, the writer is a no-op that opens no file, mirroring the telemetry `NOOP_RECORDER`.
- **Redaction:** route every persisted free-text field through the existing secret redactor. The `HandoffBundle` is already redacted by the assembler; metadata such as `lastPrompt` gets a fresh `createSecretRedactor()` pass before it is written.
- **Delete:** per-session delete unlinks the run file; global delete clears the `sessions` tree.

## Alternatives Considered

### Alternative 1: Append-only JSONL log per project

- **Description**: One JSONL file per project; each autosave appends a new run snapshot, last-wins on replay.
- **Pros**: Mirrors the telemetry sink verbatim; append is cheap and crash-safe.
- **Cons**: Per-session delete means rewriting the log; the picker must replay and dedupe to show current state; unbounded append for a record that changes many times per run.
- **Why rejected**: Delete and "show the current record" are the common operations, and both are trivial with one file per run and awkward with an append log.

### Alternative 2: Per-run files plus a separate index

- **Description**: Per-run JSON files plus an `index.json` for fast picker listing.
- **Pros**: Faster listing when a project accumulates many runs.
- **Cons**: Two writes per change and an index that can drift out of sync with the files.
- **Why rejected**: Premature optimization; a directory read is fast enough for realistic run counts, and one source of truth is simpler. An index can be added later without changing the record format.

### Alternative 3: Lifecycle-event or interval autosave

- **Description**: Write only on enumerated events (turn complete, focus change, hand-off, exit), or on a periodic timer.
- **Pros**: Fewer writes.
- **Cons**: Event-based risks missing a state change and leaving a stale record; interval-based loses the most recent turns on a crash between ticks.
- **Why rejected**: A debounced store subscription is already the crash-safe, single-code-path option and the store is the single source of truth.

## Consequences

### Positive

- Delete and list are filesystem-trivial (unlink, readdir); project scope is a subdirectory.
- Small at-rest footprint; the heavy transcripts stay in the agents' own stores.
- Reuses two proven in-repo patterns (the JSONL sink's path resolution and the store-subscription of `recorder.watch`), and the config-flag shape of `telemetryEnabled`.

### Negative

- Keep-forever retention lets the tree grow unbounded; mitigated by cheap per-session and global delete and deferred retention policy.
- Overwrite-in-place must be atomic (write-temp-then-rename) to avoid a torn file on crash mid-write.

### Risks

- **Project-key collisions or churn** if `cwd` encoding is naive. *Mitigation*: encode the absolute `cwd` deterministically, the way Claude Code encodes its project directories.
- **Redaction miss on new metadata fields.** *Mitigation*: funnel all persisted strings through one redaction helper so a new field cannot bypass it.

## Implementation Notes

- New file under `src/telemetry/` or a sibling `src/persistence/` module; reuse `resolveTelemetryPath`'s XDG logic rather than reimplementing it.
- The writer is constructed in `createCockpitSession` from `config.persistenceEnabled` and handed the store, next to `createTelemetryRecorder`.
