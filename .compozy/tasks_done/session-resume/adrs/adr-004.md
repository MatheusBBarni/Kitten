# ADR-004: Live Restore via loadSession Replay, with Two-State Degradation

## Status

Accepted

## Date

2026-07-09

## Context

Restoring a run must rebuild both agent panes, focus, and the hand-off relationship.
Three facts from the codebase shape how:

1. The store's replay path already rebuilds a transcript from streamed updates: ACP `session/update` becomes a `DomainSessionEvent` that flows through `store.applyEvent` into `sessionReducer`. `startSession` seeds an empty `turns: []`, so a `loadSession` that re-streams full history repopulates the pane through the exact path `newSession` already uses - no new reducer code.
2. The controller's `startAgent` establishes a load-bearing order: `startSession` (which resets the slice) must run **before** `onUpdate` subscribes, or early streamed events are discarded.
3. `agentCapabilities` from `initialize` is currently read then discarded at `agentConnection.ts:170`, and the `AgentConnection` adapter exposes no `loadSession`. The SDK's `ClientSideConnection` does expose `loadSession`, `listSessions`, and `deleteSession`.

Because Kitten stores only pointers plus the bundle (ADR-002), there is no transcript to display when an agent cannot reload a session, so the three restoration states named in ADR-001 (restored / history-only / unavailable) cannot all be delivered.

## Decision

Restore live through ACP `loadSession` and its streamed replay, and model degradation with **two states**.

- **Surface the capability.** Add `loadSession(sessionId, cwd)` to the `AgentConnection` adapter, mirroring `newSession` but calling `connection.loadSession({ sessionId, cwd, mcpServers: [] })`. Widen `ReadyState` to carry `agentCapabilities.loadSession` (and `sessionCapabilities.delete`) out past the ACP boundary so the controller can branch and so delete can reach the agent's own store when supported.
- **Restore orchestration** mirrors `startAgent`: `connect()` -> if the agent advertises `loadSession` and a stored session id exists, `loadSession(sessionId, cwd)`, else `newSession(cwd)` -> `store.startSession(agentId, restoredSessionId)` to bind and reset the slice **before** subscribing -> `onUpdate -> store.applyEvent` so the re-streamed history repopulates -> then `store.setFocus(persistedFocusedAgentId)`. Each agent degrades independently, mirroring `failAgent`, so one dead session never blocks the other.
- **Two-state model.** `live` = the agent reloaded the session and it is promptable (history re-streams from the agent itself). `unavailable` = the agent lacks `loadSession`, the reload failed, or the underlying transcript was purged; the pane renders the persisted bundle and metadata as read context and offers to start a fresh session seeded from that context. Surface the per-agent state through a small `AppState.restoration: Record<AgentId, "live" | "unavailable" | null>` field set by a `setRestoration` action and read by a selector, so the pane can show a badge without touching the reducer. This refines ADR-001's restoration model under the pointers-only storage decision.
- **Hand-back is unchanged.** Direction derives from focus via `nextAgentId`, so restoring `focusedAgentId` plus a re-streamed (non-empty) transcript makes `handoff.begin()` work with no changes.

## Alternatives Considered

### Alternative 1: Persist a transcript snapshot to keep three states

- **Description**: Store a bounded redacted transcript so an unreloadable session can still be browsed read-only, preserving ADR-001's "history-only" state.
- **Pros**: A degraded pane still shows prior conversation.
- **Cons**: Enlarges the at-rest content surface the PRD deliberately minimized; duplicates state the agent already owns; adds a redaction and size-bounding burden.
- **Why rejected**: The PRD chose pointers + bundle only; the bundle already carries the curated context needed to continue, so a full snapshot is not worth the surface.

### Alternative 2: New store rehydrate action that seeds turns from stored state

- **Description**: Add an action that seeds `SessionState.turns` from a persisted snapshot instead of relying on streamed replay.
- **Pros**: Independent of the agent's replay timing.
- **Cons**: Requires storing the transcript (see Alternative 1) and duplicates the reducer's job.
- **Why rejected**: `startSession` + `loadSession` streaming already rebuilds the transcript through the existing path; a rehydrate action is unnecessary.

### Alternative 3: Read the agents' own JSONL transcript files directly

- **Description**: Parse `~/.claude/projects/*.jsonl` and `~/.codex/sessions/*` to restore history.
- **Pros**: Works even if the adapter mishandles `loadSession`.
- **Cons**: The formats are internal and change between versions; couples Kitten to each agent's private storage.
- **Why rejected**: ACP `loadSession` is the durable, supported contract; direct parsing is brittle.

## Consequences

### Positive

- No new reducer logic: restore reuses the streamed-replay path that already works.
- Per-agent independent degradation keeps a partial resume useful and honest.
- Two states match what pointers-only storage can actually deliver, so the UI never promises history it does not have.

### Negative

- Ordering is load-bearing: `startSession` must precede subscription and `loadSession`, or replayed history is lost.
- A new `AppState.restoration` field, `setRestoration` action, and selector are added purely to surface per-side status.

### Risks

- **Stale or unsupported session id.** *Mitigation*: gate on the advertised `loadSession` capability, catch a failed load, and fall through to `unavailable` cleanly.
- **Replay latency.** History arrives asynchronously after `loadSession` resolves; *mitigation*: show a restoring state per pane until the stream settles, which the coalescing scheduler already smooths.

## Implementation Notes

- The confirmation probe (PRD Phase 1) that a pinned adapter reloads a days-old id belongs in the existing `selfCheck` harness.
- `deleteSession` on the adapter is only needed if Kitten later offers to also purge the agent's own copy; V1 delete removes only Kitten's run file.
