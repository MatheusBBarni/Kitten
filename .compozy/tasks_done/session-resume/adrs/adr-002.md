# ADR-002: V1 Rollout Shape - Whole-Cockpit Resume Delivered End-to-End

## Status

Accepted

## Date

2026-07-09

## Context

ADR-001 settled the *scope* of session resume: a two-layer whole-cockpit model (a reliable relationship layer plus best-effort per-agent liveness) with a data-at-rest gate and an up-front adapter-capability probe.
The PRD phase had to settle the *shape* of the V1 release, and two things changed the calculus.

First, focused research retired the top feasibility risk.
Both pinned adapters - `claude-code-acp@0.57.0` and `codex-acp@1.1.0` - durably persist their own sessions to disk and resolve `session/load` by id from those on-disk stores (verified at source level), so a days-old session id reloads as long as the underlying transcript still exists.
The residual risk is narrower: the agent's own transcript may already be gone (Claude Code auto-purges after 30 days by default; Codex keeps sessions indefinitely), so a resume must degrade gracefully to "session no longer available" rather than error.

Second, three product decisions were locked during clarification:

1. Persistence is **on by default** with a first-run disclosure, an always-available delete, and a config off-switch - matching the universal norm across Claude Code, Codex, aider, Cursor, and Zed, none of which gate persistence behind opt-in.
2. Kitten stores only **pointers plus the curated bundle** - the two ACP session ids, light metadata (last prompt, branch, cwd, timestamps, message counts), and the hand-off bundle - and rehydrates transcripts live via `session/load`. The heavy transcripts already live in `~/.claude` and `~/.codex`, so this keeps Kitten's at-rest surface small.
3. Retention is **keep-forever until manual delete**, with the picker marking a session "unavailable" when the agent's own transcript is gone.

## Decision

Adopt **Approach A: deliver whole-cockpit resume end-to-end in V1**, built in internal phases rather than split across separate user-facing releases.

V1 ships, as one coherent capability:

- A `Ctrl+R` session picker, scoped to the current project with a one-key widen, live fuzzy filtering, a `Space` preview before committing, and rows showing summary-or-first-prompt, relative time, message count, and git branch.
- On-by-default persistence of pointers, metadata, and the curated hand-off bundle, with a first-run disclosure of what is stored and where, per-session and global delete, and an off-switch.
- Live per-side restore via `session/load`, with explicit per-side status (`restored` / `history-only` / `unavailable`), never silent and never an all-or-nothing block, and a graceful "session no longer available" path.
- Hand-back after resume through the existing `Ctrl+T` redacted-preview-confirm flow, with no new egress path.
- A "resume last run" fast-path alongside the picker.

The internal build order front-loads the persistence engine and a confirmation probe against the pinned binaries, then the picker and live restore, then the degradation and preview polish.

## Alternatives Considered

### Alternative 1: Autosave first, picker as a fast follow (Approach B)

- **Description**: V1 is continuous autosave plus an on-startup "resume last run in this project?"; the cross-run `Ctrl+R` picker ships in a later release.
- **Pros**: Smallest initial surface; fastest to crash recovery and same-project continuity; proves the persistence engine before any picker UI.
- **Cons**: Defers the `Ctrl+R` picker the idea centered on and the park-across-days job ("which of my past runs?"); reads as `claude --continue` and does not differentiate at first.
- **Why rejected**: The autosave engine and the "resume last run" fast-path are folded into Approach A as the first internal phase, so Approach B's value is captured without stopping short of the picker. With the feasibility risk retired, there is no reason to ship the undifferentiated half alone.

### Alternative 2: Read-only history floor + single focused-agent live (Approach C, the preserved dissent)

- **Description**: V1 ships best-effort read-only history for both panes plus live resume of only the focused agent; whole-cockpit live and the hand-off-edge are deferred.
- **Pros**: Smallest promise; lowest risk if adapters could not reliably reload.
- **Cons**: Spends the V1 budget on the commodity half and defers the differentiator.
- **Why rejected**: The research retired the exact risk that justified this caution - both adapters reload days-old ids from disk - so the honest-floor trade is now a pure loss: the same effort buys a weaker product. It remains the documented fallback only if a pre-build confirmation probe contradicts the source-level finding.

## Consequences

### Positive

- Ships the differentiating relationship restore and the picker in one release, so the first thing users see is the whole cockpit coming back, not a `--continue` clone.
- On-by-default persistence with a small pointers-plus-bundle footprint delivers park-across-days with zero setup while keeping the at-rest surface minimal.
- Explicit per-side status and a graceful "unavailable" path keep the trust brand intact even when an agent's transcript has been purged.

### Negative

- Largest single V1 surface of the three options, with the most UX states to design and test (per-side degradation, unavailable sessions, preview).
- Keep-forever retention lets the picker and store grow unbounded and can list sessions whose transcripts the agent has already purged, which the "unavailable" state must handle cleanly.

### Risks

- **Purged underlying transcript.** A session older than Claude Code's 30-day window (or otherwise deleted) will fail `session/load`. *Mitigation*: mark the row "unavailable," still restore the relationship layer and the bundle, and offer to start fresh from that context.
- **Confirmation gap.** The durability finding is source-verified but not yet exercised against Kitten's own client. *Mitigation*: a confirmation probe against the pinned binaries is the first internal build step; a negative result reopens Approach C as the fallback.

## Implementation Notes

- Persistence can mirror the existing `telemetryEnabled` config pattern for its off-switch, and reuse the JSONL state sink under `~/.local/state/kitten/`.
- The first-run disclosure attaches to the existing boot guidance flow (`src/config/firstRun.ts`).
- Apply the existing secret redactor to the persisted bundle; hand-back reuses the existing preview-confirm path, so no new egress code is introduced in V1.
