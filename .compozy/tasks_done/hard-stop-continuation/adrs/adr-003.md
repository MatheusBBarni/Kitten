# ADR-003: Model continuation state in the core and effects in the controller

## Status

Accepted

## Date

2026-07-20

## Context

Hard Stop Continuation needs one private, generation-bound draft to survive cancellation settlement, second Escape, and uncertain-session recovery. The draft must be visible to the composer at the right time, yet it must not become a transcript turn, persisted run content, telemetry payload, diagnostic value, or handoff material.

Kitten already separates pure, reducer-owned live steering state from controller-owned cancellation, timers, connection calls, and dispatch. The same layering must be preserved here, while avoiding reuse of steering's queue and prefixed follow-up behavior. A controller-only object would make UI recovery and exact lifecycle transitions opaque to the store; a UI-local object would lose state across focus and lifecycle changes.

## Decision

Add a dedicated TypeScript post-interrupt continuation state and event union to the protocol-free core. Each session owns at most one generation-fenced continuation request containing its `PromptBlock[]`, explicit phase, and one optional recovery payload. The reducer is the only writer of this live-only state. Generic selectors expose only phase, queue count, and recovery availability; a narrowly scoped recovery selector exposes the payload only for the composer to copy once and acknowledge.

Add a controller-owned coordinator that observes the reducer state and owns provider cancellation, settlement waiting, generation checks, safe-dispatch admission, timeout handling, and cleanup. The UI reaches it only through new `ControllerActions` for queuing and restoring a continuation. The coordinator dispatches reducer events and sends an ordinary prompt only after safe admission; it never lets the UI call a connection or advance lifecycle state directly.

## Alternatives Considered

### Alternative 1: Controller-only mutable continuation state

- **Description**: Keep the queue, recovery draft, and phase entirely in controller closures.
- **Pros**: Fewer core types and no raw draft in session state.
- **Cons**: Makes recovery visibility, identity changes, and exact-once transitions harder to derive and test through the normal store contract.
- **Why rejected**: The feature needs deterministic live state visible across controller and UI boundaries without creating a second mutable state system.

### Alternative 2: Reuse the steering state machine

- **Description**: Store a Hard Stop continuation as a special steering request.
- **Pros**: Reuses existing queue and recovery mechanics.
- **Cons**: Steering cancels to send a prefixed follow-up during an active turn, while this feature waits to send an ordinary next turn after an explicit interruption.
- **Why rejected**: The semantic mismatch would blur interaction ownership and make future maintenance unsafe.

### Alternative 3: Keep recovery state in PromptEditor

- **Description**: Use component-local state for queued and restored drafts.
- **Pros**: Smallest apparent UI change.
- **Cons**: State is lost on focus changes or remounts and cannot safely coordinate with session generation or transport settlement.
- **Why rejected**: The safety boundary belongs outside the renderer and must survive view changes during the live run.

## Consequences

### Positive

- Preserves the core/store/controller/UI layering and makes state transitions pure and testable.
- Gives the composer a deterministic one-time recovery path without exposing raw queued content to generic observers.
- Keeps all provider effects under the controller's existing session and generation authority.

### Negative

- Adds a focused state machine beside steering rather than sharing one generic abstraction.
- Requires explicit cleanup on delivery, recovery acknowledgement, session replacement, close, and controller disposal.

### Risks

- **Raw blocks leak through a generic selector** — expose only content-free status outside the focused recovery selector and add negative privacy assertions.
- **Stale callbacks advance a new generation** — bind every coordinator action to session, generation, request identity, and active lifecycle identity.
- **UI dispatches while recovery owns the draft** — keep the existing composer blocking rule until copied recovery is acknowledged.

## Implementation Notes

- Place the TypeScript contracts and event union in `src/core/types.ts`, pure transitions beside existing core lifecycle helpers, reducer cases in `src/core/sessionReducer.ts`, and narrow selectors in `src/store/selectors.ts`.
- Keep controller effects in `src/app/controller.ts` and expose only typed action methods from `src/app/actions.ts`.
- Do not create a new package or introduce ACP types outside `src/agent/`.
