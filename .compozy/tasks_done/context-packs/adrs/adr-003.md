# ADR-003: Keep Context Packs in a session-keyed AppStore slice with manifest-and-sealed persistence

## Status

Accepted

## Date

2026-07-16

## Context

Kitten's AppStore is the single mutable owner of application state, while `SessionState` is reducer-owned transcript and agent state. Context Packs require independent session-owned lifecycle state: operator and child revisions, selected-item identities, a draft brief, freshness, materialized review candidates, and an immutable sealed payload. This state must survive restart without persisting raw source content or live child authority.

## Decision

Add a session-keyed `contextPacks` slice to AppStore. Model the pack lifecycle and deterministic validation in a new protocol-free core module, but keep the mutable current projection in AppStore rather than extending `SessionState` or retaining it in a controller map.

Persist a metadata-only Draft Manifest and the exact redacted Sealed payload through the next strict RunRecord version. Restored drafts revalidate every selection before review or sealing. Restored sealed packs remain inspectable, but every consumption attempt recomputes recipient fit. Never persist raw selected source content, the review candidate before sealing, a live Context Build child, capability attestation, reservation, or provider error text.

## Alternatives Considered

### Alternative 1: Extend `SessionState`

- **Description**: Place draft and sealed pack fields beside transcripts and reducer-owned session events.
- **Pros**: Keeps all session-associated data in one type.
- **Cons**: Mixes artifact lifecycle and persistence concerns with ACP-derived session state and expands the main reducer unnecessarily.
- **Why rejected**: Context Pack mutations have a distinct authority and materialization lifecycle; a store slice preserves SessionState's narrow ownership.

### Alternative 2: Keep packs in a controller runtime map

- **Description**: Maintain context packs imperatively beside live connections.
- **Pros**: Short initial path for controller operations.
- **Cons**: Loses selector visibility, atomic UI updates, restart persistence, and a single mutable owner.
- **Why rejected**: It would create a second mutable state owner and make review or restore behavior race-prone.

### Alternative 3: Persist full draft source content

- **Description**: Store copied file and diff content in every draft so restoration needs no reread.
- **Pros**: Fast resume and no source lookup before review.
- **Cons**: Retains unreviewed raw content, increases privacy exposure, and obscures drift.
- **Why rejected**: The product requires explicit freshness revalidation and allows only redacted sealed content to become durable.

## Consequences

### Positive

- Preserves AppStore as the sole mutable owner while keeping the domain model pure and testable.
- Makes draft mutation, UI review state, and persistence restoration selector-visible and atomic.
- Enforces the distinction between a revalidatable manifest and an exact reviewed sealed payload.

### Negative

- Adds a strict persistence version and migration boundary.
- Materializing a restored draft requires a fresh workspace read before review.
- Store actions and selectors must be added deliberately instead of reusing transient overlay state.

### Risks

- **Manifest/schema drift** — use strict allowlisted serializers and migration tests.
- **Stale selection reuse** — fence review and sealing on current identity, digest, size, and revision.
- **Accidental raw-content persistence** — test serializers negatively and persist sealed bytes only after redaction.

## Implementation Notes

- Add a focused `src/core/contextPack.ts` domain module and colocated tests; do not create a generic artifact framework.
- Add `contextPacks: Readonly<Record<SessionId, ContextPackState>>` and narrow actions/selectors in existing store files.
- Extend `src/persistence/runRecord.ts`, `runStore.ts`, and fixtures with a versioned Context Pack projection.
