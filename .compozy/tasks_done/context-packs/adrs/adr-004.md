# ADR-004: Use a separate generation-bound Context Pack bridge for `explore-v2`

## Status

Accepted

## Date

2026-07-16

## Context

The current binary child bridge registers both scoped `ask_user` and `agent_run`. Context Build needs a child to inspect and revise only one parent-owned Draft Context Pack, but it must not gain agent control, workspace write access, shell or general Git access, external MCP, cross-session access, sealing, sending, exporting, or approval authority. Hiding unrelated controls in UI or prompt text would not enforce that boundary.

## Decision

Introduce a separate same-binary Context Pack bridge and a separately certified `explore-v2` profile. Register it only for an accepted Context Build and bind every invocation to the parent session id, child session id, launch generation, and draft revision it read.

The bridge exposes only a closed allowlist: scoped `ask_user`; draft summary and status reads; bounded workspace tree/file/diff reads; and revision-fenced add, remove, brief, and mode-permitted instruction mutations. It rejects all operations outside the parent Session Workspace and all stale, oversized, unauthorized, or generation-mismatched requests. The child never receives a direct seal, consume, export, or control operation.

## Alternatives Considered

### Alternative 1: Extend the shared child bridge and hide `agent_run`

- **Description**: Add Context Pack tools to the current shared bridge and remove unrelated tools from presentation.
- **Pros**: Reuses existing registration and lifecycle plumbing.
- **Cons**: Leaves an agent-control handler in the same capability surface and relies on filtering rather than structural separation.
- **Why rejected**: The product requires enforcement below tool advertisement; a hidden handler is not an acceptable authority boundary.

### Alternative 2: Give the child prompt-only curation instructions

- **Description**: Ask the child to report suggested selections for the operator to apply manually.
- **Pros**: Requires no scoped mutation capability.
- **Cons**: Reverts to report-only Explore and cannot produce a revision-fenced, reviewable draft.
- **Why rejected**: It does not deliver the core Context Pack curation loop.

### Alternative 3: Expose general workspace and Git tools

- **Description**: Let the child inspect any repository data through existing general tools.
- **Pros**: Broad discovery flexibility.
- **Cons**: Expands authority, bypasses bounded artifacts, and creates an unreviewable attack surface.
- **Why rejected**: It violates the least-authority product contract.

## Consequences

### Positive

- Makes the Context Build capability attestable, narrow, and independently testable.
- Gives the operator a revision-fenced draft rather than a prose recommendation.
- Preserves `explore-v1` as an unchanged report-only capability.

### Negative

- Requires distinct bridge registration, provider evidence, and failure states.
- Adds bounded file and diff artifact services that the existing prompt-local selector does not provide.
- An unavailable attestation must block Context Build rather than fall back.

### Risks

- **Tool-route confusion** — test direct invocation of every forbidden operation against the bridge.
- **Stale child mutation** — require the revision read by every mutation and make operator edits win.
- **Path escape or oversized reads** — enforce realpath containment and fixed byte limits before content leaves the host.

## Implementation Notes

- Keep bridge registration and execution-time authorization in the controller/application layer; no ACP types enter the core or store.
- Reuse parent-generation lifecycle fencing and safe repository containment patterns, but do not reuse the mixed-purpose child bridge.
- Close and discard bridge authority when the Context Build settles, fails, or its parent generation changes.
