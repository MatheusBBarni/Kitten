# ADR-002: V1 rollout as a compose-complete MVP

## Status

Accepted

## Date

2026-07-09

## Context

ADR-001 committed V1 to a model and effort selector plus an effort-tagged hand-off.
The open question was sequencing: ship the whole hybrid at once, ship the selector first and add the hand-off tagging as a fast-follow, or ship only a thin selector slice.

Two facts constrain the choice.
The council established that a model/effort selector on its own is parity with tools that already exist (aider, OpenCode, Crush), and that the differentiator is effort composed with hand-off.
The success metrics the user chose are behavioral: kept effort-changes and hand-offs paired with an effort change, plus a confirmed-applied reliability floor.
Those value metrics cannot be measured without both the effort-tagged hand-off and the switch telemetry in place.

## Decision

Ship V1 as one compose-complete MVP: the selector, verified confirmed-state UI, the capability allowlist, the always-visible status-strip display, the effort-tagged hand-off, and opt-in content-free telemetry.

Internally the build order is selector-and-verify first, then the hand-off tagging on top, because the hand-off composes on a switch the user must be able to trust.
But the first public release is the differentiated whole rather than a parity dropdown, and every chosen KPI is measurable from day one.

## Alternatives Considered

### Alternative 1: Selector-first, hand-off fast-follow

- **Description**: MVP is the selector, verify-applied, allowlist, status strip, and telemetry; the effort-tagged hand-off lands as a phase-2 fast-follow.
- **Pros**: Smaller first release; hardens the trust-critical switch in isolation before composing it with the moat.
- **Cons**: The first release reads as parity until the hand-off tagging arrives; the value KPIs cannot be measured yet.
- **Why rejected**: Delays the actual differentiator and leaves the headline metrics unmeasurable at launch, for a modest reduction in first-release size.

### Alternative 2: Thin safe slice

- **Description**: MVP is only the selector, verify-applied, and allowlist. Status strip, telemetry, and hand-off all deferred.
- **Pros**: Absolute minimum to prove the switch works reliably.
- **Cons**: No differentiation, no discoverability, and none of the chosen KPIs can be measured.
- **Why rejected**: Effectively walks back the approved hybrid scope from ADR-001.

## Consequences

### Positive

- The first release is differentiated on effort and hand-off, not just a dropdown.
- All chosen KPIs (confirmed-applied, kept effort-change, effort-linked hand-offs, adoption) are measurable at launch.
- Reuses existing overlay, hand-off, and telemetry patterns, so the larger scope is mostly assembly rather than new surface.

### Negative

- The first release is larger than a selector-only slice.

### Risks

- **Verify-applied must be solid before the hand-off composes on top** - mitigated by the internal build order (selector-and-verify first).
- **Larger scope could slip** - mitigated by reusing the existing modal-overlay, hand-off-bundle, and telemetry-recorder patterns rather than building new ones.

## Implementation Notes

The PRD's Phased Rollout Plan reflects this: MVP carries all six V1 features; Phase 2 is the provider/agent swap on the same rails; Phase 3 is the effort-aware advisor.
