# ADR-004: Live in-place switching with confirmed-state UI and a category allowlist

## Status

Accepted

## Date

2026-07-09

## Context

The PRD requires changing a pane's model and effort on the live session, with no restart, and with the cockpit showing only the setting the agent confirms is running.
The pinned SDK (`@agentclientprotocol/sdk@1.2.1`) exposes `ClientSideConnection.setSessionConfigOption(sessionId, configId, value)`, which mutates the running session and returns the full refreshed `configOptions`.
Agent-initiated changes arrive as a `config_option_update` notification.
Two hazards shape the design: the same config surface also carries the `mode` category (whose Claude values include `bypassPermissions`, the switch that disables Kitten's approval prompts); and some ACP agents acknowledge a switch the backend never applies ("ack != applied").

Today Kitten discards `newSession`'s `configOptions` (`agentConnection.ts:176-180`) and drops `config_option_update` (`acpTranslate.ts:55`).

## Decision

Switch by calling `setSessionConfigOption` on the live session; never tear the session down.

Drive the UI purely from the agent-reported option set. The response of `setSessionConfigOption` and the `config_option_update` notification both translate into the single `config_options` domain event (ADR-003), and the reducer replaces the stored set wholesale. The overlay and status strip render the returned `currentValue`, never the optimistically requested value. If the call errors, or the returned `currentValue` does not match what was requested, the pane shows an `unverified` state and keeps the last confirmed value.

Seed initial state by capturing `newSession`'s `configOptions`. Translate `config_option_update`; keep dropping `current_mode_update` (modes are permission controls, not model/effort, and are out of scope).

The UI allowlists categories `model` and `thought_level`. `mode`, `model_config`, and any other category are filtered out before rendering (fail-closed allowlist, not a denylist), so the overlay can never surface `bypassPermissions`.

## Alternatives Considered

### Alternative 1: Optimistic UI

- **Description**: Show the requested model/effort immediately on send.
- **Pros**: Feels instant.
- **Cons**: Can assert a state the agent never applied, misleading the user about which model, cost, and data profile is live.
- **Why rejected**: A selector that misreports state corrodes the exact fidelity promise the hand-off depends on.

### Alternative 2: Restart-based switching

- **Description**: Re-spawn the agent with new settings and carry context across with the hand-off bundle.
- **Pros**: Works even if an adapter offers no live config options.
- **Cons**: Lossy (a bounded excerpt, not the full session) and slow.
- **Why rejected for V1**: The pinned adapters support live switching. Kept as a documented fallback only for an agent that advertises no config options.

### Alternative 3: Render every advertised category

- **Description**: Show whatever categories the adapter advertises, including `mode` and `model_config`.
- **Pros**: No per-category UI work.
- **Cons**: Surfaces `bypassPermissions` and Fast mode next to the model picker.
- **Why rejected**: Unacceptable safety footgun; the generic surface is a control plane, so presentation must be allowlisted.

## Consequences

### Positive

- The cockpit is honest: it shows only agent-confirmed state, and the switch keeps transcript, files, and diffs.
- Additive and greenfield: capture data already received, add adapter and action methods, translate one dropped notification.
- Safe by construction against permission-mode exposure.

### Negative

- On a flaky adapter the UI may show an `unverified` state rather than a crisp success.
- Panes whose adapter advertises nothing show no picker.

### Risks

- **ack != applied** - mitigated by confirmed-state rendering and the `unverified` fallback; tracked as the confirmed-applied KPI.
- **Runtime advertisement unverified** - a live-handshake confirmation that both adapters expose `model`/`thought_level` is owed before build; the design already degrades to "no picker" if absent.

## Implementation Notes

Add `setSessionConfigOption` to `AgentConnection` (calls `requireConnection().setSessionConfigOption`), a `setSessionConfigOption` controller action mirroring `sendPrompt`/`cancel`, and seed captured `configOptions` via `store.applyEvent` in `startAgent`.
