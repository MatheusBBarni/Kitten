# ADR-001: V1 scope for the in-app model & reasoning-effort selector

## Status

Accepted

## Date

2026-07-09

## Context

Kitten drives two AI coding agents (Claude Code, Codex) over ACP.
Today a pane's model and reasoning effort are fixed by whatever the spawned adapter defaults to; changing them means editing `~/.config/kitten/config.json` and relaunching, which discards the live session.
The user wants an in-app selector to change a pane's model and reasoning effort mid-session without a restart.

Research confirmed this is fully supported by the pinned dependencies: ACP exposes `session/set_config_option` with categories `model` and `thought_level` (effort), the pinned SDK (`@agentclientprotocol/sdk@1.2.1`) implements it, and both pinned adapters advertise it.
The switch mutates the live session in place, so the transcript, files, and diffs survive ("seamless continue").
Kitten currently discards the `configOptions`/`modes` returned by `session/new` and drops the `config_option_update` notifications, so the work is additive and greenfield with no dependency bump.

Two facts shaped the scope. First, the same generic `set_config_option` surface also carries `mode` (Claude Code permission modes, including `bypassPermissions`) and `model_config` (Fast mode) - so rendering whatever an adapter advertises would put a "skip every human approval" toggle next to the model picker.
Second, some ACP agents acknowledge a switch the backend never applies ("ack != applied"), which for a product whose moat is fidelity-of-state-across-a-boundary would mean the cockpit lying about the exact property the hand-off depends on.

A five-advisor council debated the scope. Model-switching alone is table stakes (aider, OpenCode, Crush already do it); reasoning-effort as a first-class in-terminal knob is rare, and it is the axis that differentiates - especially composed with Kitten's hand-off.

## Decision

Ship V1 as an in-app, keyboard-driven overlay that changes a pane's **model** and **reasoning effort** on the live ACP session, for the two existing agents.

- **Generic data model, narrow surface.** Model the generic ACP config-option channel in the protocol-free domain core (`SessionState`), keyed by opaque category ids; the V1 overlay renders only the `model` and `thought_level` categories.
- **Strict allowlist in code.** The overlay surfaces only `model` + `thought_level`. `mode` (incl. `bypassPermissions`), `model_config`, and any future category are excluded by an allowlist (fail-closed), not a denylist.
- **Confirmed-state UI, never optimistic.** The returned full `configOptions` list is the single source of truth. The UI shows the agent-confirmed applied value; on unconfirmable or errored switches it shows an `unverified` state and keeps the prior confirmed value (fail closed). Applied state is verified for both model and effort.
- **Effort positioned as the wedge and designed to compose with hand-off.** The selected effort/model is carried into the existing hand-off bundle so "hand this over, bumped to high effort" is a native cockpit move, tying the feature to Kitten's moat rather than to parity.
- **Per-agent capability discovery.** The picker is driven entirely by what each adapter advertises at runtime; if an adapter advertises nothing, no picker is shown for that pane.
- **Deferred to V2+:** swapping the provider/agent a pane runs, `model_config`/Fast mode and permission-mode controls, and the signal-watching auto-effort advisor.

## Alternatives Considered

### Alternative 1: Model-only selector (drop effort)

- **Description**: Ship a mid-session model picker and leave reasoning effort out of V1.
- **Pros**: Smaller; effort is where "ack != applied" is hardest to eyeball.
- **Cons**: Model-switch alone is exactly the parity slice competitors already ship; drops the one axis (effort) that differentiates.
- **Why rejected**: The council (including the advisor who first proposed it) converged that effort is the wedge, not me-too; cutting it keeps the distraction and throws away the differentiation.

### Alternative 2: Generic renderer exposing every advertised config option

- **Description**: Render whatever categories the adapter advertises, including `mode` and `model_config`.
- **Pros**: Zero per-category UI work; V2 controls appear for free.
- **Cons**: Inverts the security posture from default-deny to default-allow and drags `bypassPermissions` and Fast mode into the overlay.
- **Why rejected**: Unacceptable safety footgun. The generic surface is a control plane, not a model menu; presentation must be allowlisted.

### Alternative 3: Full verified compose-loop advisor in V1

- **Description**: Kitten watches pane signals (stall, retry, token burn) and suggests "bump effort" or "hand off at high effort," one keystroke to accept, with verified apply.
- **Pros**: The genuinely 10x, hard-to-copy version; makes the moat demonstrable.
- **Cons**: A quarter of work, not a Quick Win; needs signal detection and a suggestion engine that do not exist yet.
- **Why rejected**: Correct direction, wrong horizon. It is the V2+ stretch; V1 lays the rails (generic channel + hand-off-composable effort) it will build on.

## Consequences

### Positive

- Removes the config-edit-and-relaunch friction with additive, greenfield work on an already-received ACP surface.
- Elevates reasoning effort - the cheapest cost/quality lever - to a first-class, visible control few competitors expose.
- Composes with the hand-off moat rather than competing for roadmap time with it.
- The generic-channel data model makes V2 (provider swap, more categories) a rendering/policy change, not new plumbing.

### Negative

- Model-switching on its own is parity; the feature must be positioned on effort + compose-with-hand-off to avoid reading as "me too."
- Adds a trust surface: a selector that misreports state would corrode the hand-off's core promise, so verify-applied is mandatory, not optional.

### Risks

- **ack != applied**: mitigated by confirmed-state-only UI, an `unverified` state, and fail-closed behavior; tracked as a KPI (`> 98% confirmed applied`).
- **`bypassPermissions` exposure**: mitigated by the code-level allowlist; the overlay cannot reach `mode`.
- **Adapter advertises nothing**: hide the picker for that pane rather than render an inert control.
- **Vanity metric**: adoption-as-clicks is demoted to a secondary signal in favor of confirmed-applied (reliability floor) and kept-change / effort-linked hand-offs (value).

## Implementation Notes

- Capture `NewSessionResponse.configOptions`/`.modes` (currently discarded at `src/agent/agentConnection.ts:176-180`).
- Translate the dropped `config_option_update` / `current_mode_update` notifications (`src/agent/acpTranslate.ts:50-58`) into a new domain event; the reducer does a wholesale replace of the option set (the agent always returns the full updated list).
- Mirror the existing modal-overlay pattern (`ApprovalPrompt.tsx` / `HandoffPreview.tsx`): store slot + `useKeyboard` capture + Enter/Esc, a new `keymap.ts` binding, and current model/effort surfaced in `StatusStrip.tsx`.
- Add controller actions that call `ClientSideConnection.setSessionConfigOption` and apply the returned option list.
