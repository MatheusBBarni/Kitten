# ADR-001: MCP Propagation Mechanism - ACP Session-Injection vs Writing Agent Config Files

## Status

Accepted

## Date

2026-07-09

## Context

Kitten runs Claude Code and Codex side by side over the Agent Client Protocol (ACP).
Users who rely on MCP (Model Context Protocol) servers must configure the same servers separately in each agent, which duplicates work and lets the two agents drift apart.
Drift is especially damaging for Kitten because its core value is a clean cross-agent hand-off: if one agent exposes a tool the other lacks, the hand-off breaks.

The feature declares MCP servers once in Kitten's config and makes both agents use them.
Two mechanisms can deliver this, and they trade off cleanly against each other.

Codebase research surfaced the load-bearing fact: ACP's `session/new` request already accepts a required `mcpServers` array, and Kitten already calls `newSession({ cwd, mcpServers: [] })` with a hardcoded empty list at `src/agent/agentConnection.ts:178`.
The ACP SDK fully supports the `McpServer` union (stdio, http, sse), stdio is mandatory for every ACP agent, and http/sse are gated behind an advertised capability.
No code anywhere in Kitten touches Claude Code's or Codex's own config files, so the write-to-disk path is entirely greenfield.

A five-advisor council debated the mechanism and reached a strong recommendation, with preserved dissent on the V1 scope boundary.

## Decision

V1 uses ACP session-injection as the sole propagation mechanism.
Kitten holds one global `mcpServers` list in `~/.config/kitten/config.json`, resolves it into ACP `McpServer` descriptors, and injects that list into every `session/new` (and `session/load`) call for every configured agent.
V1 supports stdio transport only, which is guaranteed for all ACP agents; http/sse are deferred behind per-agent capability detection.
The mapping invariant is "every agent gets every compatible MCP," filtered by each agent's advertised transport capabilities, with room for a per-agent exclude as an overridable default.

Injection ships behind two non-negotiable guardrails:
- A CI smoke test that injects a trivial stdio server into each pinned adapter (`claude-agent-acp@0.57.0`, `codex-acp@1.1.0`) and asserts the tool actually appears in the session. A red gate means the injection feature does not ship and a setup document is the V1 fallback.
- A visible per-agent "loaded N servers" readout so a silent miss (an adapter that accepts the array but ignores it, or a server that fails to spawn) is loud rather than invisible.

Persistence into each agent's own config file - the "works standalone outside Kitten" capability - is deferred to V2 as an explicit, user-consented, opt-in action, not a silent default on launch.

## Alternatives Considered

### Alternative 1: Write MCPs into each agent's own config file

- **Description**: On startup, serialize the global MCP list into Claude Code's `.mcp.json`/`~/.claude.json` (JSON) and Codex's `~/.codex/config.toml` (TOML), merging without clobbering user-defined entries.
- **Pros**: Persists across restarts; MCPs also work when the user runs the agent standalone outside Kitten; matches the user's originally stated preference.
- **Cons**: Two divergent serializers permanently coupled to foreign schemas that churn (see codex issue #3441); merge-never-clobber must reconcile two different identity and precedence models; risk of corrupting a user's hand-maintained config; Claude project-scope `.mcp.json` triggers an approval prompt, which is itself manual setup and contradicts the "zero manual setup" goal.
- **Why rejected**: High and permanent maintenance surface plus real blast radius (trust-ending config corruption) in exchange for a secondary want, when injection already satisfies the core job by construction. Retained as an explicit opt-in V2.

### Alternative 2: Hybrid - inject by default plus write-to-disk in V1

- **Description**: Do both mechanisms in the first release: inject for in-cockpit parity and also persist to each agent's config for standalone use.
- **Pros**: Satisfies both the in-Kitten and standalone contexts immediately.
- **Cons**: Carries the full write-to-disk risk and scope into V1, defeating the quick-win framing; the merge/clobber problem lands on the critical path immediately.
- **Why rejected for V1**: The write-to-disk safety stack (atomic writes, backups, reference-only secrets, marker-enforced never-clobber, per-file consent) is a subsystem that should not gate the first release. Sequenced as V2.

### Alternative 3: Documentation only

- **Description**: Ship a setup guide showing how to configure MCPs in each agent; build nothing.
- **Pros**: Zero maintenance, zero foreign-file writes, ships today.
- **Cons**: Hands the manual, drift-prone burden back to the user, which is the exact pain the feature exists to remove; makes drift the default rather than preventing it.
- **Why rejected**: A cockpit whose value is cross-agent parity cannot rely on the user hand-maintaining two configs in sync. Retained only as the fallback if the smoke-test gate fails.

## Consequences

### Positive

- Delivers "declare once, zero manual setup, no drift" by construction: a single source of truth feeds both agents identically every session.
- Minimal, well-bounded implementation that routes entirely through the ACP anti-corruption layer (ADR-003) already owned under `src/agent`; the domain model stays pure and free of foreign file formats.
- No mutation of the user's existing agent config files, so no clobber or corruption risk in V1.
- Reinforces Kitten's differentiator: it owns both agents' session lifecycle and is the only tool that unifies MCP across these two ACP agents automatically on launch.

### Negative

- MCP servers are available only inside Kitten-launched sessions; they do not appear when the user runs Claude Code or Codex standalone. This does not satisfy the user's stated standalone preference in V1 and is a deliberate deferral, not a resolution.
- The "quick win" is a focused subsystem (resolver, ACP translation, capability detection, smoke test, readout), not a literal one-line change.

### Risks

- **Adapter ignores the injected array (silent no-op).** Worst failure: a healthy session with none of the declared tools. Mitigation: the CI smoke test gates the release; the per-agent readout surfaces real-world misses; the smoke test must re-run whenever a pinned adapter version changes, since honoring is proven only at pin-time.
- **ACP wire-shape mismatch.** stdio `command` must be an absolute path (so `npx` must be resolved via `Bun.which`) and env is an array of `{name,value}` rather than a map. Mitigation: a translation layer inside `src/agent` with a loud skip-and-warn when a command cannot be resolved.
- **Capability divergence.** An http/sse server injected into a stdio-only agent falsifies a naive "every agent gets every MCP" mapping. Mitigation: capability-gate per transport and warn; treat the invariant as "every compatible MCP."
- **Green readout, wrong tools.** A "loaded N servers" count can be green while a specific tool is still absent. Mitigation: report server identities in the readout, not just a count; the future smoke test can assert on named tools.

## Implementation Notes

- Add a top-level `mcpServers` field to `USER_CONFIG_SCHEMA` and `AppConfig`, seeded to `[]` in `defaultAppConfig()` and merged in `mergeAppConfig()` (`src/config/configLoader.ts`, `src/core/types.ts`).
- Keep resolution pure in `src/core` (declarations to domain descriptors, no I/O, no SDK import); do ACP wire translation and capability gating inside `src/agent`.
- Widen `AgentConnection.newSession` to accept the resolved server list and thread it from the controller (`src/app/controller.ts:136`) down to the `newSession` call that currently passes `[]`.
- Follow the injectable-seam and pure-core-plus-thin-shell conventions (ADR-003, ADR-005); mirror the test patterns in `configLoader.test.ts` and `firstRun.test.ts`.
- The final V1 boundary (injection-only vs pulling standalone persistence forward) is confirmed with the user during the idea's direction selection; revise this ADR if that scope changes.
