# ADR-004: Environment-Reference Resolution and Failure Semantics

## Status

Accepted

## Date

2026-07-09

## Context

The PRD requires MCP secrets to be expressed as environment-variable references (`${VAR}`) rather than stored as plaintext in Kitten's config.
No `${VAR}` expansion helper exists in the codebase today; this is greenfield.

Two different failure modes need distinct handling, and they pull in opposite directions:
- The config loader's established stance is that a broken config is louder than a silent fallback: structural problems throw a `ConfigError` that names the offending field and blocks boot.
- The PRD's runtime stance for provisioning is "warn, never block": one bad MCP server must not stop the cockpit from opening.

An unresolved `${VAR}` sits between these: it is not a structural config error (the config is well-formed), but it does mean one server cannot be provisioned.

## Decision

Add a pure environment-reference resolver: SDK-free, taking an injectable environment map that defaults to `process.env`, expanding `${VAR}` occurrences inside MCP server env values.

Split failure semantics by phase:
- **Parse/load time (structural):** a non-stdio transport, a missing command, or a malformed entry throws a `ConfigError` naming the field, blocking boot. This matches the loader's existing fail-loud stance.
- **Boot time (value resolution):** an unresolved `${VAR}` is not a parse error. The affected server is dropped from injection and recorded on the per-agent readout with a reason; the cockpit still boots and the remaining servers load.

Resolution runs at boot (per launch), not at parse, because environment values are a runtime input.
Parse validates structure; boot resolves values.

Every resolved value passes through the existing secret redactor before any log or telemetry emission, so secrets never appear in output.

## Alternatives Considered

### Alternative 1: Fail loud on an unresolved variable

- **Description**: Treat an unresolved `${VAR}` as a `ConfigError` that blocks boot.
- **Pros**: Consistent with the loader's structural fail-loud rule; fully predictable.
- **Cons**: One missing token in one server would stop the whole cockpit from opening, which contradicts the PRD's warn-never-block promise.
- **Why rejected**: Provisioning is additive; a single unresolvable server should degrade to a warning, not a boot failure.

### Alternative 2: Resolve a missing variable to an empty string

- **Description**: Substitute an empty value and still inject the server.
- **Pros**: Never blocks; least code.
- **Cons**: Produces a silently misconfigured server (for example an empty API key) that fails later in a confusing way.
- **Why rejected**: A visibly skipped server with a clear reason is more actionable than a server that loads broken.

## Consequences

### Positive

- The user's warn-never-block choice is honored for the runtime failure that actually occurs in practice (a missing token), while structural mistakes still fail loud.
- Secrets never land in Kitten's config or in logs.

### Negative

- Two failure paths exist (parse-time throw, boot-time skip), which the readout and tests must both cover.

### Risks

- **A skipped server looks like a working one.** Mitigation: the readout names each skipped server and its reason (unset variable), not just a count.
- **Redaction gaps.** Mitigation: route every resolved env value through the redactor at the single emission boundary; test that a known token is redacted.

## Implementation Notes

- The resolver is a pure helper (default env `process.env`, injectable for tests), most naturally alongside `configLoader.ts` or as a small `src/core` utility, throwing nothing on an unset variable but reporting it upward so boot can skip-and-warn.
- Structural validation stays in the strict config schema at parse time.
