# ADR-004: Persist editor preferences as validated direct argument vectors

## Status

Accepted

## Date

2026-07-16

## Context

The File Explorer must support the operating-system default editor and an explicitly saved custom editor. A flexible command string would force shell parsing and make quoting, interpolation, and injection behavior ambiguous. The preference must survive restart, preserve unrelated strict user-config fields, support Save and Cancel in Settings, and react safely to validated external config edits.

The existing config loader, atomic writer, and watcher already enforce strict delta configuration and live application of valid changes. The launcher can report that a process dispatch succeeded or failed, but cannot accurately promise that an external GUI is fully ready for interaction.

## Decision

Represent the editor preference as a strict discriminated config value: `system-default` or `custom` with one executable and a discrete argument array containing exactly one `{file}` placeholder. The Settings Editor tab keeps a component-local draft; only a valid explicit Save persists one whole preference, while Cancel leaves the saved value unchanged. A validated watcher reload updates the preference used for the next user-initiated open.

Launch custom editors and system defaults through direct argument-vector process spawning on supported macOS and Linux platforms. Never invoke a shell, interpolate environment variables, or accept additional placeholders. When custom dispatch fails, attempt the system default once. Surface only the final dispatch outcome and record only fixed, content-free outcome categories.

## Alternatives Considered

### Alternative 1: Store a shell command string

- **Description**: Let users enter an arbitrary shell command that receives a filename.
- **Pros**: Familiar for power users and compact in the settings UI.
- **Cons**: Requires parsing and escaping semantics, expands injection risk, and makes validation and cross-platform behavior ambiguous.
- **Why rejected**: Direct executable and argument vectors provide a smaller, auditable contract.

### Alternative 2: Persist every valid field edit immediately

- **Description**: Save the preference as the developer types in Settings.
- **Pros**: No dedicated Save action.
- **Cons**: Partial edits can change the next file opening unexpectedly and Cancel cannot reliably restore the prior preference.
- **Why rejected**: The product requires explicit Save and Cancel semantics.

### Alternative 3: Require a restart for external config edits

- **Description**: Ignore valid editor-preference reloads until the next Kitten launch.
- **Pros**: Simpler runtime update path.
- **Cons**: Conflicts with existing config-watch behavior and delays an explicitly changed preference.
- **Why rejected**: The next user-initiated open can safely use the latest validated preference.

## Consequences

### Positive

- Keeps editor invocation shell-free and unambiguous.
- Gives Settings a clear draft, validation, Save, and Cancel lifecycle.
- Preserves strict config behavior and applies live valid changes predictably.
- Aligns user-facing notice language with the observable dispatch contract.

### Negative

- Users cannot express arbitrary shell wrappers, environment interpolation, or multiple filename substitutions.
- A successfully dispatched external application may still fail after process launch, which remains outside Kitten's observable promise.

### Risks

- **Invalid persisted preference blocks future opens**: validate schema and exact placeholder cardinality before Save or reload application.
- **Fallback is mistaken for a successful custom launch**: report a distinct final fallback outcome and never suppress the failure path internally.
- **Telemetry drifts toward sensitive configuration**: define closed outcome types that exclude executable, arguments, paths, names, and error text.

## Implementation Notes

Extend the existing strict config schema, merger, atomic writer, watcher, Settings view, and telemetry recorder. Reuse their dependency-injection and fail-soft conventions; do not add a shell abstraction or new persistence store.
