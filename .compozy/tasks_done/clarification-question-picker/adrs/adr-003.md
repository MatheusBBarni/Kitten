# ADR-003: Fail closed on a verified ACP elicitation allowlist

## Status

Accepted

## Date

2026-07-12

## Context

Kitten pins the ACP TypeScript SDK at 1.2.1. The SDK exposes structured user input through the experimental `elicitation/create` client callback, but Kitten does not currently advertise or implement that callback. Its configured provider recipes are user-overridable, and the existing connection handshake does not prove that a provider can complete a structured clarification round trip.

Enabling the callback for every provider would make the PRD's capability-transparency promise unreliable. The feature must protect the adapter boundary from an unstable protocol surface while allowing verified configurations to opt in.

## Decision

Classify structured clarification capability at configuration/readiness time using a fail-closed allowlist of exact, integration-tested built-in adapter package and version combinations. Only a configuration that matches a verified entry may advertise ACP elicitation support and register the adapter callback.

All custom recipes, overridden commands, unknown versions, and untested built-in combinations are `unsupported` in V1. The resolved capability result is passed into the connection boundary; the rest of the application consumes only a protocol-free supported/unsupported result.

## Alternatives Considered

### Alternative 1: Per-provider user opt-in flag

- **Description**: Allow users to enable structured clarification for any configured provider.
- **Pros**: Fastest route to broader compatibility.
- **Cons**: Users can enable an incomplete or incompatible ACP lifecycle and encounter runtime failure.
- **Why rejected**: A manual flag cannot substitute for contract evidence and violates the fail-closed PRD boundary.

### Alternative 2: Enable elicitation for every provider

- **Description**: Advertise the SDK capability globally and handle failure after a request arrives.
- **Pros**: Minimal configuration logic.
- **Cons**: Treats provider kind as proof of support and exposes experimental protocol failure to users.
- **Why rejected**: Adapter support is not negotiated or verified today.

### Alternative 3: Wait for a future negotiated capability

- **Description**: Defer the feature until ACP defines an agent-advertised elicitation capability.
- **Pros**: Stronger protocol certainty.
- **Cons**: Blocks a bounded V1 even when concrete adapter/version combinations can be tested.
- **Why rejected**: Exact contract-tested combinations provide a safe incremental path now.

## Consequences

### Positive

- Makes unsupported configurations explicit before use.
- Confines unstable ACP behavior to the adapter boundary.
- Ties every enabled provider configuration to repeatable contract evidence.

### Negative

- V1 may initially enable no provider until a real adapter contract test passes.
- Allowlist maintenance is required when a verified adapter changes version.

### Risks

- **Version drift**: exact matching disables the feature rather than silently assuming compatibility.
- **False confidence from mock tests**: require actual adapter/version contract tests before allowlisting.
- **Future protocol change**: isolate ACP mapping and reevaluate the allowlist when upgrading the SDK.

## Implementation Notes

Keep ACP request, response, and experimental capability types inside `src/agent/`. Resolve capability before connection initialization and expose a protocol-free value to configuration, controller, and UI surfaces.
