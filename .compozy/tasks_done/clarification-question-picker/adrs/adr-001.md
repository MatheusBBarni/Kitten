# ADR-001: Scope the clarification picker around explicit structured requests

## Status

Accepted

## Date

2026-07-12

## Context

Kitten is a terminal cockpit that hosts multiple live coding-agent sessions. Agents can become blocked when they need user clarification, while the current product supports only permission requests. Research shows that Claude Code and Codex expose structured user-question capabilities, but availability through Kitten's current adapter and transport path must be verified before release; unanswered questions can also look like stalled sessions. The proposed feature needs to improve answer clarity without confusing clarification with permission or relying on unsafe detection of ordinary prose questions.

The key forces are fast validation, multi-session attribution, provider differences, reliable waiting-state visibility, and the need to preserve a free-text escape path.

## Decision

Deliver V1 only for provider adapters verified to support a complete request/response lifecycle for **explicit structured clarification requests**. Support single-select, multi-select, and text responses through a dedicated clarification experience that is visually and semantically separate from permissions.

V1 will define thin shared interaction-routing primitives for request identity, session ownership, queue ordering, terminal lifecycle states, and visible waiting state. Permission and clarification domains retain distinct payloads, policies, cancellation semantics, actions, and UI terminology.

## Alternatives Considered

### Alternative 1: Detect clarification questions in arbitrary agent prose

- **Description**: Infer that a normal agent message is a clarification and turn it into a picker.
- **Pros**: Broad coverage without provider support.
- **Cons**: False positives, ambiguous response routing, disruptive interruptions, and no reliable way to recover the agent's intended answer contract.
- **Why rejected**: It would make V1 unreliable and distort both user experience and measurement.

### Alternative 2: Build a full decision inbox before the first release

- **Description**: Include persistent history, cross-session triage, prioritization, notifications, and reusable answers.
- **Pros**: Stronger long-term multi-agent differentiator.
- **Cons**: Delays learning, increases interaction complexity, and builds unvalidated infrastructure.
- **Why rejected**: Reliability and visibility are required now; advanced inbox capabilities are not.

### Alternative 3: Reuse permission flow without distinct clarification semantics

- **Description**: Present questions as another type of permission request.
- **Pros**: Lowest initial implementation cost.
- **Cons**: Blurs user intent and risk levels, creates incorrect terminology, and couples different response/cancellation policies.
- **Why rejected**: Clarifications and permissions are distinct user decisions and must remain so.

## Consequences

### Positive

- Delivers a narrow, measurable path to clearer agent responses.
- Maintains provider independence and session attribution.
- Establishes reliable lifecycle and attention-routing foundations without a premature platform.
- Preserves user agency through selectable options and text input.

### Negative

- V1 coverage depends on provider adapters exposing a verified structured request/response path; the picker is not released for an unsupported adapter.
- The product cannot automatically convert every natural-language question into a picker.
- Thin shared primitives require deliberate boundary design even for a small release.

### Risks

- **Provider schema drift or unequal capability**: normalize at the adapter boundary and degrade per session.
- **Stale, duplicate, or misrouted requests**: use request identity, terminal states, deterministic ordering, and single-answer guards.
- **Unnoticed background request**: show session-attributed waiting state and validate zero unnoticed requests in usability tests.
- **Poor option quality**: require descriptions and retain a text-response path; monitor response-mode distribution and option-fit satisfaction in user research.

## Implementation Notes

Before implementation, confirm at least one provider adapter can expose and resolve structured clarification requests through Kitten's transport. The technical specification should define provider capability detection, response mapping, request lifecycle, queue arbitration, focus behavior, accessibility, content-free telemetry, and behavior on cancellation, disconnect, and session restoration. It should not define prose detection for V1.
