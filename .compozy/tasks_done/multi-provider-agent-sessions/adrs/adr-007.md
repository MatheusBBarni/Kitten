# ADR-007: Layered Attention Notifications - Per-OS Shell-Out with Focus Gating

## Status

Accepted

## Date

2026-07-09

## Context

ADR-003 committed V1 to notifying the developer at the operating-system level, not only in the terminal, when a session needs them while their attention is elsewhere.
Kitten is a terminal UI with a deliberately small dependency set (`@opentui`, `react`, the ACP SDK, `ws`, `zod`) and a no-server, bring-your-own-tools ethos.
The notification must be content-free, fire on the transition into a needs-you state, and reach the developer only when Kitten is not the focused window, or it becomes noise.

## Decision

Deliver attention on three layers, with a per-OS shell-out and terminal-focus gating.

- In-app: the status strip reflects fleet state and the Ctrl+S overview routes the developer. Always present.
- Terminal bell: write BEL on the transition into a needs-you state; most terminals raise an alert or badge even when unfocused. This is the universal fallback.
- Native OS notification: detect the platform and shell out to a built-in tool - `osascript -e 'display notification'` on macOS, `notify-send` on Linux, a PowerShell toast on Windows - with no added dependency. The message carries only the session title, provider, working directory, and state.
- Focus gating: enable terminal focus reporting (DECSET 1004) and treat a focus-out as "Kitten unfocused." The OS notification fires only while unfocused. Where focus reporting or a native channel is unavailable, fall back to notifying on every needs-you transition, with the bell always as the last resort.
- Firing discipline: one notification per transition into needs-you, deduplicated per session, never repeated while the session stays in that state.

## Alternatives Considered

### Alternative 1: Shell-out without focus gating

- **Description**: Same per-OS shell-out, but fire on every needs-you transition regardless of focus.
- **Pros**: No focus tracking; simplest.
- **Cons**: A desktop notification appears even while the developer is looking straight at Kitten, which trains them to ignore it.
- **Why rejected**: Un-gated notifications erode the signal the feature exists to provide.

### Alternative 2: Cross-platform notification dependency

- **Description**: Add a library such as `node-notifier` plus focus gating.
- **Pros**: One notification code path across operating systems.
- **Cons**: Adds a dependency and its bundled per-OS binaries, and inherits its own platform quirks, against the minimal-deps, no-server ethos.
- **Why rejected**: The built-in per-OS tools cover the same ground with zero new dependencies.

## Consequences

### Positive

- Reaches an away developer natively while adding no dependency.
- Degrades gracefully: native channel when present and unfocused, bell everywhere else.

### Negative

- Introduces a small per-OS surface (three shell-out paths) and a focus-tracking listener, each needing its own test seam.

### Risks

- A shell-out can fail or be blocked by the OS. Mitigation: treat the native channel as best-effort and always keep the bell.
- DECSET 1004 focus events are not universal. Mitigation: when focus state is unknown, notify on transition rather than suppressing, so a missed focus signal never silences a real need.
- Shelling out to a system command is a surface to keep clean. Mitigation: pass no prompt content, and never interpolate untrusted text into the command; the message is assembled from the session's own title, provider, directory, and state.

## Implementation Notes

New component under `src/` (for example `src/notify/`) that subscribes to session-status transitions via the store's `subscribeSelector`, holds the OS detection and the three shell-out paths behind an injectable seam, and reads focus state from an OpenTUI focus listener. Wired at boot in `src/index.ts` alongside the telemetry recorder.
