# ADR-001: N-Session Model as Infrastructure Beneath the Hand-off Wedge, Not a Ctrl+S Headline

## Status

Accepted

## Date

2026-07-09

## Context

Kitten today runs exactly two ACP agents (Claude Code + Codex) and its differentiator is a curated, editable, redacted context hand-off between them.
The raw idea for this iteration was "launch and run multiple agents from potentially different providers; hit Ctrl+S to see the state of all agents on one screen."

Research surfaced a decisive fact: the reference screenshot is Toad's own project directory, and Toad already ships exactly that Ctrl+S multi-provider overview.
The space is crowded and consolidating (Claude Squad ~8.1k stars, Warp's Agent Management Panel, a built-in agent view in Claude Code, Vibe Kanban shut down despite 18.6k stars).
The business scorecard rated the overview screen Differentiation: Pass and Defensibility: Maybe, because the grid itself is trivially copyable.
Kitten's own prior thesis is explicit: do not race Toad on breadth; own the cross-agent memory/hand-off layer Toad leaves siloed.

A five-advisor council (pragmatic-engineer, architect-advisor, product-mind, devils-advocate, the-thinker) debated whether to build this feature, where its center of gravity should sit, whether to supersede the two-pane layout, and how large V1 should be.
The council converged, and a focused rebuttal round between the Product Mind and the Architect produced two matching partial concessions that define the cut line below.

## Decision

Build the N-session model as **infrastructure** beneath Kitten's existing hand-off differentiator. Do not make the Ctrl+S overview the product's headline.

V1 substrate (prerequisite, ships together):

- Replace the fixed `sessions: Record<AgentId, SessionState>` with a growable session collection.
- Introduce a `SessionDescriptor` that splits **instance identity** (`sessionId`) from **provider kind** (`providerKind`), and carries per-session `cwd`, `task`, `status`.
- Move the working directory onto the per-session descriptor (per-session `cwd` is non-negotiable, because the moat is cross-**project** hand-off and a shared-cwd demo cannot exercise it).
- Extend the status model with `waiting` / `done` / `error` so the overview cannot show stale or zombie rows.
- Re-express the existing hand-off as descriptor-to-descriptor addressing. The curation mechanism and redacted preview stay untouched; only the target addressing changes, forced by the identity split. The moat must not go dark during this change.
- Ship a thin, read-mostly Ctrl+S overview: status at a glance plus attention-routing ("jump to the agent that needs you") plus focus-switch, over a statically seeded fleet (sessions seeded from config at launch).

Deferred until validation justifies it:

- Runtime spawn/kill lifecycle UI and the `terminated` status.
- Grid chrome/polish, provider-switch UI, session grouping, durable/cross-restart persistence.

The two-pane layout is **generalized, not deleted**: it becomes the N=2 view over the session collection.

## Alternatives Considered

### Alternative 1: Ctrl+S multi-provider overview as the headline feature (the original framing)

- **Description**: Treat the fleet overview as the product's marquee capability; build lean-core (spawn/kill + grid) and supersede the two-pane layout, deferring hand-off rewiring.
- **Pros**: Matches the raw idea and the reference screenshot; visually impressive; immediately "feels" like a multi-agent product.
- **Cons**: Directly copies Toad's shipped headline into a consolidating market; competes on the exact axis the prior thesis said to avoid; defers the one differentiated part (hand-off) while shipping the commodity part; risks regressing the moat as collateral of the layout change.
- **Why rejected**: Low differentiation and low defensibility; spends a small project's scarce attention arriving second on a competitor's home turf.

### Alternative 2: Full triage board over a shared cross-agent memory (the ambitious version)

- **Description**: Build the fleet as a triage board ranked by who-needs-you, over a shared, curated, redacted memory that sessions lend to each other (the "newsroom assignment desk with a shared morgue" frame).
- **Pros**: Strongest possible differentiation; only Kitten's hand-off heritage could own it; turns N sessions from a switcher into a shared-context system.
- **Cons**: Large, partly unvalidated; the shared-memory primitive is a research bet; premature before the attention-routing pain is measured.
- **Why rejected**: Deferred as the V2+ direction, not V1. It is the destination the V1 substrate deliberately makes reachable.

## Consequences

### Positive

- Pays down an accidental boundary (the two-agent assumption calcified into a root type) that compounds monthly.
- Fixes the real codebase gap (single shared working directory) and unlocks the genuinely underserved angle: a cross-project fleet of N independent repos.
- Keeps the differentiator (hand-off) center stage and lands it on the new model in the same release, so the moat never regresses.
- Reversible and cheap to validate; avoids funding fleet-management surface before the fleet pain is confirmed.

### Negative

- The overview ships deliberately thin (no spawn/kill, no grid polish), which may feel less complete than Toad on first glance.
- The `SessionDescriptor` refactor touches the store, controller, config, and hand-off addressing at once.

### Risks

- **Unvalidated pain**: "agents sit idle waiting" is asserted, not observed. Mitigation: statically seeded 3-session validation loop with ~10 multi-agent power users; measure attention latency and 7-day return before funding lifecycle.
- **Moat regression during rewire**: hand-off addressing changes with the identity split. Mitigation: keep curation/preview untouched; land hand-off on the descriptor model in the same release; treat any hand-off regression as a release blocker.
- **Concurrency/permission safety**: N live agents across N repos multiplies approval-context confusion (approving a shell command in the wrong repo). Mitigation: every approval and status row is labeled with its session title and `cwd`; no cross-session auto-approve in V1.
- **Intentional-debt guardrail**: do not key the collection by `AgentId` (provider-kind) as a shortcut; that silently caps the fleet at one session per provider and contaminates the validation signal. Key by `sessionId`.

## Implementation Notes

- Concentrated refactor sites (from codebase exploration): `src/core/types.ts` (`AgentId` union), `src/store/appStore.ts` (`sessions` record, `AGENT_IDS`), `src/app/controller.ts` (already `Map`-based; add per-session `cwd`), `src/app/actions.ts` + `src/app/handoff.ts` (`nextAgentId` / "the other agent" targeting), `src/config/configLoader.ts` (fixed two-key `agents` object), `src/ui/keymap.ts` (add Ctrl+S), and a new `SessionsOverlay` modeled on the existing approval / hand-off preview overlays.
- Unlock signal for spawn/kill in a later release: users manually restart or juggle sessions, hit the seeded-fleet wall (want a fourth session or a different `cwd` mid-run), or idle-time only collapses when they can add/kill agents on the fly.
