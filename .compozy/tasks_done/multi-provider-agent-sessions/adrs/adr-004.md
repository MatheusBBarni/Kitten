# ADR-004: N-Session Identity Model - Split Provider Kind from Instance Identity

## Status

Accepted

## Date

2026-07-09

## Context

The store keys all session state by the provider union: `AppState.sessions: Record<AgentId, SessionState>` with `focusedAgentId: AgentId`, where `AgentId = "claude-code" | "codex"` (`src/core/types.ts:14`, `src/store/appStore.ts:66`).
`SessionState.sessionId` is the ACP session id assigned after the handshake.
This shape cannot hold two sessions of the same provider (two Claude Code sessions on two repositories), and it conflates the kind of an agent with the identity of a running session.
The controller is already collection-shaped (`Map<AgentId, AgentRuntime>`), so the structural blocker is the store's fixed record and the identity type, not the orchestration.

## Decision

Introduce a Kitten-assigned session instance identity distinct from provider kind.

- Rename the union to `ProviderKind = "claude-code" | "codex"` (the spawn-recipe kind), and add `SessionId` as an opaque string instance identity assigned at config load, before any handshake.
- The store holds `sessions: Record<SessionId, SessionState>`, an explicit `order: SessionId[]` for stable display order, and `focusedSessionId: SessionId`.
- `SessionState` becomes `{ id: SessionId; providerKind: ProviderKind; title: string; cwd: string; task?: string; acpSessionId: string; turns; status; referencedFiles; pendingDiffs; plan }`. The ACP session id is a separate field (`acpSessionId`), empty until the session hands shake, so a not-ready session still exists in the collection.
- The collection is a plain object plus an order array, not a `Map`, to preserve the store's immutable structural-sharing and `Object.is` selector-equality invariants (ADR-004 of the base app, `subscribeSelector`).

## Alternatives Considered

### Alternative 1: `Map<SessionId, SessionState>` for the collection

- **Description**: Use a JavaScript `Map` keyed by session id, ordered natively.
- **Pros**: Cleaner iteration and insertion order without a separate array.
- **Cons**: A `Map` is mutable and does not structurally share; the store relies on new-object-per-change and `Object.is` slice equality so an untouched session's subscribers stay silent. A `Map` would force reworking those invariants.
- **Why rejected**: It trades a working, well-tested store contract for iteration sugar.

### Alternative 2: Reuse the ACP session id as the instance key

- **Description**: Key the collection by the ACP session id, adding no new identifier.
- **Pros**: No id to assign; one identity concept.
- **Cons**: The ACP id exists only after a successful handshake, so a session that fails to come up would have no key and could not appear in the overview - exactly the not-ready row the overview must show.
- **Why rejected**: The overview must render not-ready sessions, which have no ACP id yet.

## Consequences

### Positive

- Two sessions may share a provider, which is the whole point of an independent fleet.
- Not-ready sessions live in the collection and render in the overview with their reason.
- The store keeps its existing immutability and targeted-subscription guarantees.

### Negative

- `AgentId` is referenced across roughly 114 sites; the rename to `ProviderKind` plus the key change from provider to `SessionId` is a wide mechanical edit.
- `SessionState` carries more fields (identity, cwd, title, task), so construction and test fixtures grow.

### Risks

- The two names `sessionId` (ACP) and `SessionId` (instance) invite confusion. Mitigation: rename the ACP field to `acpSessionId` and reserve `SessionId`/`id` for the instance identity.
- A wide rename can silently break selectors. Mitigation: land the rename behind the existing typecheck and test suite before adding behavior.

## Implementation Notes

Primary edit sites: `src/core/types.ts` (union rename, `SessionState` fields, `SessionId`), `src/store/appStore.ts` (`sessions`, `order`, `focusedSessionId`, `initialSessions`, `setFocus`), `src/store/selectors.ts` (key by `SessionId`), `src/app/controller.ts` (`Map<SessionId, AgentRuntime>`), `src/app/actions.ts` (`nextAgentId` -> session-based targeting).
