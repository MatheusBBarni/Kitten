# ADR-002: Ship the Full Attention Cockpit as a Single V1

## Status

Accepted

## Date

2026-07-09

## Context

ADR-001 set the strategic frame: build the N-session model as infrastructure beneath the hand-off differentiator, with a deliberately thin attention-routing overview.
The PRD phase had to choose how much of that slice ships in the first release, and in what order.
The council had flagged one sequencing hazard: generalizing the two-pane layout while deferring the hand-off re-addressing opens a window where the hand-off (Kitten's one differentiator) can regress or break as collateral of the layout change.

Three phasings were on the table: ship the whole coherent slice at once (A), ship the attention loop first and re-address hand-off in a fast follow (B), or ship a throwaway validation slice before committing to the real overview (C).

## Decision

Ship the full attention cockpit as one coherent V1 (Approach A).

The single release includes: the multi-session model with a per-session working directory; the Ctrl+S overview with distinct session states and a jump-to-the-next-that-needs-you action; attention nudges (in-app status strip, terminal bell, and native OS notifications per ADR-003); the existing curated, redacted hand-off re-addressed to name an explicit target session; safe per-session approvals that label every prompt and status row with its session and working directory; and the two-pane view generalized to the N=2 case.
The unproven "agents sit idle" assumption is validated with a small cohort after this slice ships, not before.

## Alternatives Considered

### Alternative 1: Attention loop first, hand-off re-addressing as a fast follow (Approach B)

- **Description**: Phase 1 ships the session model, the overview, and the bell with hand-off left on the current two-agent path; Phase 2 re-addresses hand-off to sessions.
- **Pros**: Smaller, faster first ship; validates the attention loop sooner.
- **Cons**: Generalizing the two-pane layout while the hand-off still assumes a single "other agent" is the exact window the council warned about, where the differentiator regresses or breaks.
- **Why rejected**: It ships the commodity part first and puts the moat at risk to save a short amount of time.

### Alternative 2: Validation-first thin slice (Approach C)

- **Description**: A minimal, instrumented throwaway build (a few seeded sessions, the needs-you signal, existing hand-off, just enough overview) put in front of about ten users before committing to the full overview.
- **Pros**: Cheapest possible learning loop; de-risks the "agents sit idle" assumption before more build.
- **Cons**: Produces scaffolding that gets replaced; delays a shippable public V1; less complete for early adopters.
- **Why rejected**: The substrate refactor (identity split, per-session working directory, status model) has to be built for any real version anyway, so a throwaway slice saves little and delays the real thing.

## Consequences

### Positive

- One coherent release delivers the whole value proposition, and the hand-off moat never regresses because it is re-addressed in the same slice.
- Nothing that works today is lost: the two-pane view becomes the N=2 case of the session model.
- The validation cohort tests a real, shippable product rather than throwaway scaffolding.

### Negative

- The first release is larger: the identity split, per-session working directory, status model, and hand-off re-addressing all land together.

### Risks

- Landing several changes at once raises release risk. Mitigation: treat any hand-off regression as a release blocker, and keep the hand-off curation and redacted preview untouched so only the addressing changes.
- The "agents sit idle" pain is still unproven at ship time. Mitigation: instrument attention latency and run the small-cohort validation immediately after launch, before funding runtime lifecycle management.

## Implementation Notes

Deferred out of V1 (revisited after validation): runtime session spawn/kill, session grouping, ranked triage queues beyond a single needs-you signal, durable cross-restart persistence, non-ACP providers, and a provider-switch UI.
