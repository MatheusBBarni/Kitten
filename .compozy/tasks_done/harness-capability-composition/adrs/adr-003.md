# ADR-003: Compose Capabilities in Core and Make the Adapter Envelope-Only

## Status

Accepted

## Date

2026-07-16

## Context

The approved product requires optional harness guidance to be selected only from facts that are true for a fresh Kitten session. The existing harness renderer is a pure static-block contract, the controller owns fresh-generation lifecycle, and the agent adapter owns ACP encoding. The adapter currently adds bridge guidance when it sees a bridge server, which can affect restored sessions and bypass the intended fresh-session selection boundary.

V1 must prove the bridge and child-control capability slice without overloading certified harness-profile eligibility, leaking ACP or bridge details into the domain, or duplicating hidden guidance.

## Decision

Add a focused TypeScript composition module in `src/core/` beside the existing harness renderer. It accepts a closed, protocol-free capability context and returns reviewed static `HarnessBlock`s plus a content-free result. The module owns neither capability discovery nor delivery state.

The controller constructs the context from its current fresh generation and recomputes it immediately before the first dispatch. A bridge/child-control fact is confirmed only for the live generation after bridge registration and successful fresh-session setup; stale, absent, unknown, or conflicting facts select no optional block. The controller passes the composed blocks to the existing renderer, then passes the resulting envelope to the adapter.

Remove adapter-side automatic bridge guidance injection. The adapter encodes only guidance explicitly supplied in the envelope and retains its existing exact certified-profile validation. Loaded sessions preserve their existing context and never receive a newly composed optional fragment.

## Alternatives Considered

### Alternative 1: Extend the static renderer with lifecycle facts

- **Description**: Put capability discovery and fragment selection inside `harnessPrompt.ts`.
- **Pros**: Fewer files and one apparent prompt entrypoint.
- **Cons**: Mixes the pure reviewed renderer with controller runtime state and makes its contract harder to test and review.
- **Why rejected**: The renderer must remain deterministic from explicit static inputs; context construction has different ownership.

### Alternative 2: Build bridge guidance directly in the controller

- **Description**: Let the controller concatenate bridge wording before rendering.
- **Pros**: Keeps all session inspection close to lifecycle code.
- **Cons**: Couples wording and deterministic selection to mutable orchestration logic and makes future fragments inconsistent.
- **Why rejected**: A small pure composer gives the controller a narrow input/output seam without creating a generic prompt platform.

### Alternative 3: Keep adapter-side automatic bridge injection

- **Description**: Preserve server-name-based adapter guidance and deduplicate against the composed catalog.
- **Pros**: Smaller immediate adapter change.
- **Cons**: Applies to loaded sessions, duplicates ownership, and treats bridge provisioning as sufficient proof of an optional user promise.
- **Why rejected**: Envelope-only adapter behavior is the only clear fresh-session boundary.

## Consequences

### Positive

- Preserves ACP isolation, core purity, and controller lifecycle ownership.
- Gives V1 one authoritative source for bridge and child-control wording.
- Keeps restored sessions free from newly injected optional guidance.
- Lets future capability sources participate through the same closed input and static output contract.

### Negative

- Requires coordinated changes across core, controller, adapter, and their tests.
- The first bridge fact has a stricter generation-valid definition than server-name presence alone.

### Risks

- **Stale lifecycle evidence** — Recompute from the current generation immediately before first dispatch and fail closed to base-only.
- **Boundary drift** — Add source-boundary tests proving core imports no ACP, controller keeps discovery, and adapter does not autonomously add guidance.
- **Guidance duplication** — Remove the adapter injection and assert exact selected block IDs in envelope tests.

## Implementation Notes

Use repository-native TypeScript interfaces and colocated Bun tests. Do not add a new package, protocol type, persistence schema, or user-facing UI surface.
