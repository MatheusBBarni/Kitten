# ADR-001: Compose Fresh Harnesses from Confirmed Capability Snapshots

## Status

Accepted

## Date

2026-07-16

## Context

Issue #20 needs Kitten to tailor the hidden harness supplied to a fresh ACP session without claiming tools, roles, worktrees, steering, handoff, or permissions that the host has not actually confirmed. A static prompt either omits useful, session-specific guidance or makes false promises. The existing base harness contract and fresh-session delivery lifecycle deliberately have separate owners, so composition must not absorb either responsibility.

The first release must demonstrate a real conditional path while several potential capability sources remain unavailable, incomplete, or independently scheduled. Capability evidence can also become stale before the first dispatch, and the context must never contain prompt content, user or repository content, paths, credentials, bridge endpoints, or raw errors.

## Decision

Create a protocol-free, closed capability context that distinguishes confirmed, absent, and unknown facts. The composer selects only reviewed, static fragments from a deterministic catalog. Unknown, conflicting, stale, absent, or provider-name-inferred facts select no optional fragment; a valid base-only harness is always the safe result.

Capture composition for each new, fallback, or replacement fresh generation. A loaded session never receives a newly composed or silently mutated hidden harness. If an optional capability loses authoritative confirmation before the first dispatch, discard that selection and compose only the remaining currently confirmed fragments (or the base-only harness) before delivery. Once delivered, the harness remains immutable for that session generation.

V1 includes one independently confirmed, generation-valid active capability slice and the generic catalog shape for future staged integrations. It preserves the base contract's bounds and ordering, keeps delivery and provider encoding outside the composer, and emits only content-free result metadata such as contract version, selected fragment IDs, count, and base-only fallback status.

## Alternatives Considered

### Alternative 1: Infer fragments from provider identity or configured defaults

- **Description**: Use provider names, recipe labels, or likely defaults as a proxy for session capabilities.
- **Pros**: Fast initial coverage with little integration work.
- **Cons**: Breaks for custom, changed, or partial providers and produces false capability claims.
- **Why rejected**: A provider label is not current session evidence and violates the product's truthfulness promise.

### Alternative 2: Ship only a generic catalog with no active fragment

- **Description**: Define the model and catalog but always compose the base-only harness in V1.
- **Pros**: Lowest immediate risk and smallest implementation effort.
- **Cons**: Does not prove the selected-fragment path or its integration contract.
- **Why rejected**: One independently confirmed vertical slice provides the required end-to-end evidence without broad speculative activation.

### Alternative 3: Activate every listed capability family in V1

- **Description**: Require clarification, bridge, roles, worktrees, steering, and handoff guidance to be active immediately.
- **Pros**: Broad apparent coverage.
- **Cons**: Couples the card to unfinished features, encourages guessed facts, and delays a reviewable release.
- **Why rejected**: Each family should activate only after its own authoritative source and lifecycle evidence exist.

## Consequences

### Positive

- Fresh agents receive concise guidance that matches the runtime facts Kitten can stand behind.
- Base-only fallback works for custom and minimally capable providers without special cases.
- Stable IDs, ordering, and a content-free result make capability-matrix tests and diagnostics reviewable.
- The catalog can grow independently as capability sources become authoritative.

### Negative

- V1 exposes fewer optional behaviors than a broad, speculative catalog.
- Maintainers must define confirmation lifetime, invalidation, conflicts, and ownership for every future fragment.
- A capability that becomes unavailable before first delivery may intentionally lose its optional guidance rather than preserving a stale claim.

### Risks

- **Stale or conflicting evidence** — Bind confirmation to the relevant session generation, model unknown/conflict explicitly, and test invalidation paths.
- **Prompt drift and contradiction** — Keep static fragment ownership explicit, deterministic ordering fixed, and representative matrix goldens mandatory.
- **Content leakage** — Restrict inputs and diagnostics to bounded metadata; assert that no prompt text, path, token, secret, repository content, or raw error crosses the boundary.

## Implementation Notes

The composition domain consumes the bounded extension seam from issue #18 and supplies the composed result to the delivery path owned by issue #19. It does not own ACP transport, provider-specific encoding, permissions, orchestration, persistence of rendered text, or user-facing prompt editing.
