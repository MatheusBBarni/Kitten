# ADR-002: Make Truthful Capability Guidance a Silent Fresh-Run Default

## Status

Accepted

## Date

2026-07-16

## Context

Developers need fresh Kitten agents to receive accurate host guidance without a new status surface competing for attention. The approved idea establishes one proven V1 capability slice and a safe base-only experience, but the PRD must set the user-facing operating model: how normal starts feel, how changed availability takes effect, what counts as release success, and which later user need earns the next phase.

Research shows that comparable coding-agent products apply scoped instructions at session start and caution against conflicting active context. The primary user also values reliable work continuity over a dashboard of invisible host details. Existing Kitten behavior distinguishes fresh starts from loaded conversations and keeps successful hidden-harness delivery quiet.

## Decision

V1 will be silent during healthy fresh-session starts. Its product promise is zero false capability claims: every agent receives either the stable base guidance or the base plus only independently confirmed optional guidance. When a capability-specific start is degraded, Kitten may provide concise, content-free recovery or diagnostic information; it will not expose prompt text, selected operational wording, or sensitive runtime details in routine UI.

Changed capability guidance takes effect only in a newly started Kitten run, never by silently changing an active conversation. V1 proves the Kitten MCP bridge and child-control guidance slice. Structured clarification is the preferred Phase 2 user outcome once it has independently confirmed availability.

## Alternatives Considered

### Alternative 1: Show a routine capability-composition indicator

- **Description**: Display a badge or session summary whenever optional guidance is selected.
- **Pros**: Gives developers immediate visibility into host behavior.
- **Cons**: Adds cognitive overhead to healthy starts and risks making hidden operational guidance appear user-configurable or authoritative.
- **Why rejected**: The primary value is trustworthy agent behavior, not a new dashboard; normal success should remain quiet.

### Alternative 2: Apply changed guidance to the next fresh session in an already running Kitten instance

- **Description**: Let availability changes affect newly created agents without restarting Kitten.
- **Pros**: Faster access to new guidance.
- **Cons**: Creates an inconsistent experience across sessions in the same run and makes the effective user promise harder to explain.
- **Why rejected**: A new-run boundary is clearer, preserves continuity, and matches the current product lifecycle.

### Alternative 3: Prioritize broad capability activation over claim accuracy

- **Description**: Activate several planned capability families in V1 to maximize apparent coverage.
- **Pros**: More visible feature breadth.
- **Cons**: Increases the risk that an agent receives guidance for a capability that is unavailable or not yet verified.
- **Why rejected**: Zero false claims is the primary release bar; breadth follows independently confirmed evidence.

## Consequences

### Positive

- Developers receive a predictable, low-noise startup experience.
- Active and restored conversations retain stable operating assumptions.
- A valid base-only start remains a successful product outcome for minimally capable or custom providers.
- Phase 2 has a clear user-centered priority: structured clarification when available.

### Negative

- Developers do not receive routine visual proof of which optional guidance was selected.
- Newly available capability guidance may require starting a new Kitten run before it is usable.
- V1 demonstrates narrow depth before broader user-visible capability coverage.

### Risks

- **Users misread silence as missing functionality** — explain degraded states clearly and keep developer documentation precise about the healthy default.
- **Demand for broad immediate activation** — preserve the staged catalog and publish the rule that each capability needs independent confirmation.
- **Phase 2 dependency delay** — retain a complete base-only experience until structured clarification satisfies its availability criteria.

## Implementation Notes

This ADR sets user-visible behavior and rollout priority only. It does not prescribe composition, delivery, transport, or provider implementation details.
