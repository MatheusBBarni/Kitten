# ADR-002: Make in-Kitten review the primary child-workspace completion loop

## Status

Accepted

## Date

2026-07-15

## Context

The managed-child-worktrees idea establishes safe, isolated child workspaces. PRD clarification shows that the MVP must optimize first for developer confidence: a developer should be able to review and safely resolve every child workspace without manual recovery. Opening an external tool automatically, optimizing launch speed, or ending with only a summary would weaken the product's supervised-work promise.

Existing Kitten session surfaces already let users find child sessions and understand lifecycle. The product decision is how a finished child workspace should be presented and resolved from that familiar context.

## Decision

The MVP will make in-Kitten review the primary completion experience. A child session will clearly present its retained managed-workspace identity, current review availability, and the status needed for the developer to decide the next action. Kitten will preserve the workspace until the developer explicitly requests a safe cleanup; it will not merge, delete, or promote work automatically.

External editor opening, richer diff comparison, and worktree rebinding are later enhancements. The MVP succeeds when a developer can locate, review, and safely resolve every child workspace from Kitten without manually reconstructing where the work happened.

## Alternatives Considered

### Alternative 1: Automatically open the child workspace in an editor

- **Description**: Send the developer to an external editor when a child finishes.
- **Pros**: Uses familiar review tools and can expose a full project view quickly.
- **Cons**: Breaks the Kitten completion loop, assumes an editor preference, and obscures the child-workspace relationship.
- **Why rejected**: It makes an external tool the default answer before Kitten establishes provenance and status.

### Alternative 2: Show a concise completion summary only

- **Description**: Report that a child finished without a retained-workspace review surface.
- **Pros**: Minimal new user interface.
- **Cons**: Developers still need to find the path, branch, and disposition manually.
- **Why rejected**: It does not meet the user-selected no-manual-recovery success criterion.

### Alternative 3: Optimize primarily for the fastest possible launch

- **Description**: Minimize visible workspace context and review controls in favor of a shorter start flow.
- **Pros**: Reduces perceived startup friction.
- **Cons**: Does not address the central trust problem of parallel code-changing work.
- **Why rejected**: Confidence in correct, reviewable isolation is more valuable than shaving a small amount of launch interaction.

## Consequences

### Positive

- Developers keep a clear chain from child session to workspace to review decision.
- Retained work becomes a visible product artifact rather than hidden branch/path clutter.
- Explicit actions protect users from accidental loss or unreviewed promotion.

### Negative

- The MVP intentionally defers automatic editor handoff and advanced visual review.
- Users must consciously resolve retained workspaces rather than relying on automatic cleanup.

### Risks

- **Review surface is not discoverable** — Use the existing session and child lifecycle surfaces, clear labels, and concise action copy.
- **Retained workspaces accumulate** — Make status and safe cleanup visible without deleting work automatically.
- **Users expect work to be merged** — State that a finished child is review-ready, not promoted, throughout the completion experience.

## Implementation Notes

The subsequent TechSpec will define the detailed interaction design and system boundaries. This decision only fixes the user-facing completion loop and its explicit-control policy.
