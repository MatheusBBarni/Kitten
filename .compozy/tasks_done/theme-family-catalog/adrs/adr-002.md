# ADR-002: Preserve instant selection and documentation-first provenance in V1

## Status

Accepted

## Date

2026-07-20

## Context

The theme-family catalog expands a five-entry Settings list into an accessibility-gated, 18-preset experience. The existing interaction applies a selected theme immediately, and the product needs a clear choice about whether broader browsing should introduce a separate confirmation step. It also needs a user-facing provenance approach that informs trust without turning the catalog into an in-app marketplace or credits product.

The PRD clarification round confirmed three priorities: developers expect a direct keyboard interaction that commits the chosen theme immediately; public catalog documentation is sufficient for source and license discovery in V1; and the release decision must depend on complete accessibility, provenance, reachability, and preference-integrity evidence rather than broad usage analytics that conflict with Kitten's local, opt-in privacy posture.

## Decision

V1 retains instant apply: moving to a selectable theme immediately makes it the developer's active and durable preference. The picker clearly communicates the active row and preserves a visible, non-color-only keyboard focus state, but adds no confirm, cancel, undo, preview-only, or separate save interaction.

The public theme-catalog documentation is the authoritative place to discover each preset's upstream source and license/attribution. V1 adds no inline source details or dedicated in-app credits view.

The catalog ships only when the entire agreed roster meets the quality gate: every preset has verifiable public provenance, rendered text and states meet the accessibility threshold, all choices are usable from the keyboard in a short terminal, and saved preferences remain trustworthy across ordinary use. Optional local usage signals can inform future work, but cannot substitute for the launch gate.

## Alternatives Considered

### Alternative 1: Live preview followed by explicit confirmation

- **Description**: Let users browse a temporary preview, then require a separate action to keep the theme.
- **Pros**: Makes experimentation reversible and matches some graphical theme pickers.
- **Cons**: Adds an interaction mode and extra decision to a workflow users already understand as direct selection.
- **Why rejected**: The established Settings behavior is immediate and durable; V1 optimizes for fast keyboard choice rather than a new preview workflow.

### Alternative 2: Inline source metadata or a dedicated credits view

- **Description**: Surface attribution next to each row or in a catalog screen inside the application.
- **Pros**: Makes provenance visible at the moment of selection.
- **Cons**: Expands the picker and adds a separate runtime product surface without improving the primary readability outcome.
- **Why rejected**: Maintained public documentation provides the required transparency within the bounded V1 scope.

### Alternative 3: Release on usage or feedback evidence

- **Description**: Treat selection volume, retention, or subjective feedback as the primary release decision.
- **Pros**: Centers observed user behavior.
- **Cons**: Kitten intentionally keeps telemetry local and opt-in, so broad analytics would be incomplete and cannot prove accessible, trustworthy baseline behavior.
- **Why rejected**: Product signals guide later iteration, but the complete quality gate is the credible V1 launch criterion.

## Consequences

### Positive

- Developers retain a familiar, low-friction keyboard selection flow.
- The picker stays focused on choice and readability rather than becoming an attribution browser.
- V1 has a clear, privacy-compatible release decision that protects user trust.

### Negative

- Users cannot browse themes without changing their saved choice.
- Provenance requires following the public documentation rather than reading it in the application.
- The all-or-nothing quality bar can defer the full catalog when one preset is not ready.

### Risks

- **Accidental selection while exploring** — make the active row and keyboard focus unmistakable; preserve the reset-to-Auto path.
- **Source information feels hard to find** — link the documentation from the existing configuration guidance and keep names exact.
- **Quality gate delays perceived value** — communicate the catalog as one verified release rather than exposing incomplete families.

## Implementation Notes

The follow-on TechSpec must preserve the existing product behaviors named here without prescribing a particular implementation. It must also preserve the stable-ID, source-faithful, local-privacy, and accessibility boundaries in ADR-001 and the repository's theme catalog contract.
