# ADR-003: Make the core theme catalog the identity and compatibility authority

## Status

Accepted

## Date

2026-07-20

## Context

The current theme feature duplicates the allowed preset list in `ThemePresetId`, strict configuration validation, the UI palette registry, and the Settings list. Those copies currently cover only Catppuccin Mocha and Latte. Expanding to 18 durable presets by extending each copy would make configuration, telemetry, palette resolution, labels, and future migrations drift independently.

ADR-001 requires a protocol-free core catalog with stable IDs, source metadata, presentation order, and explicit aliases. The UI still owns terminal color data and rendering behavior, while configuration must remain strict and user-owned preferences must not be rewritten merely because a known legacy ID is canonicalized for a live session.

## Decision

Add `src/core/themeCatalog.ts` as the protocol-free authority for canonical preset IDs, family and variant metadata, exact display names, alphabetical presentation metadata, public source URL, license/attribution, and a typed explicit alias-to-canonical map. Derive `ThemePresetId` and the canonical preference list from this catalog; retain Auto, Light, and Dark as built-in preferences outside the catalog.

The configuration boundary accepts canonical IDs and recognized aliases, resolves a recognized alias to its canonical live `ThemePreference`, and continues to reject unknown values. The application persists only an explicit later canonical selection, so loading a legacy value never causes a startup write. Telemetry continues to accept only the derived closed `ThemePreference` union.

`src/ui/theme.ts` remains the owner of `CockpitPalette` values, palette resolution, terminal-mode behavior, and syntax-style caching. Its named palette registry must be exhaustive over canonical catalog IDs; core metadata must not import OpenTUI, React, filesystem, or telemetry types.

## Alternatives Considered

### Alternative 1: Keep all catalog metadata in the UI palette registry

- **Description**: Extend `src/ui/theme.ts` so colors, IDs, labels, source records, ordering, and aliases live in one UI registry.
- **Pros**: Fewer files and one visually obvious palette table.
- **Cons**: Makes strict configuration and telemetry depend on UI ownership and leaks rendering concerns into the durable preference contract.
- **Why rejected**: The product needs a protocol-free, long-lived identity boundary independent of terminal rendering.

### Alternative 2: Make the configuration enum authoritative

- **Description**: Keep canonical IDs and aliases in `configLoader.ts` and derive UI behavior from configuration values.
- **Pros**: Centralizes validation close to persistence.
- **Cons**: Couples core identity and UI presentation to file-loading concerns and cannot naturally carry source metadata.
- **Why rejected**: Configuration consumes the catalog; it should not define the catalog.

### Alternative 3: Add compatibility cases only when a preset retires

- **Description**: Omit alias data until a concrete rename or retirement occurs.
- **Pros**: Smaller immediate implementation.
- **Cons**: Leaves a documented durable-ID contract without an enforced migration shape and invites ad hoc resolver branches later.
- **Why rejected**: The explicit typed alias map makes future compatibility a testable normal path rather than an emergency exception.

## Consequences

### Positive

- Every durable theme consumer derives identity and metadata from one protocol-free source.
- Alias behavior, strict validation, and fixed-enum telemetry remain consistent and testable.
- The UI keeps color and native rendering ownership without becoming a configuration authority.

### Negative

- A new core module and exhaustive UI parity checks add disciplined coordination across existing files.
- Configuration parsing must distinguish recognized aliases from unknown invalid values.

### Risks

- **A catalog entry lacks a UI palette** — enforce exhaustive canonical-ID coverage before release.
- **An alias is accepted but persisted prematurely** — exercise startup, watcher, and explicit-selection lifecycle paths separately.
- **Metadata reaches protocol or rendering boundaries incorrectly** — keep the core module dependency-free and review imports under the existing layering rule.

## Implementation Notes

Keep the catalog data immutable and source-faithful. Do not add dynamic loading, custom palettes, or an additional persistence store. `resolvePalette` may accept runtime strings for defensive fallback, but canonicalization must use the catalog contract before named lookup.
