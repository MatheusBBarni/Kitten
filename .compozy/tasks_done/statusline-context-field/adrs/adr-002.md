# ADR-002: Make CONTEXT a voluntary, omission-first handoff-awareness cue

## Status

Accepted

## Date

2026-07-20

## Context

Developers who keep coding agents active through extended work need a compact cue for deciding when a voluntary handoff may be useful. Kitten already offers a configurable statusline, and the proposed **CONTEXT** item can surface the selected session's remaining context headroom inside that familiar surface.

The user clarified that V1 must optimize for long-running sessions, omit the item when no valid value is available, and succeed only when users can use the cue for voluntary handoff timing while understanding that it is informational. Context visibility is established in comparable agent CLIs, but user concern about accuracy means the feature must not imply a guaranteed countdown or become automated workflow policy.

## Decision

V1 will provide **CONTEXT** only as an optional statusline field. It presents the selected session's current remaining-context cue as **ctx <remaining>%** when valid, and disappears when no valid value is available.

The field is voluntary, neutral, and informational. It does not show placeholders or estimates, create alerts or threshold states, label freshness or provider provenance, recommend a handoff, or automate one. The product measures success through moderated long-session scenarios that test both handoff usefulness and correct user understanding.

## Alternatives Considered

### Alternative 1: Persistent unavailable placeholder or estimated value

- **Description**: Keep a visible context slot when no current value exists, using an unavailable label or estimated percentage.
- **Pros**: Gives the statusline a stable visual shape and suggests that context is a first-class concern.
- **Cons**: Creates false certainty or invites users to infer meaning from missing data.
- **Why rejected**: Omission is more honest and better matches the product promise of a trustworthy cue.

### Alternative 2: Passive low-context guidance

- **Description**: Add visual warnings, threshold states, or explanatory freshness labels beside the field.
- **Pros**: Could reduce confusion about what the number means.
- **Cons**: Adds policy and visual semantics before the base cue has demonstrated value.
- **Why rejected**: The user chose field-only V1; qualitative evidence should justify any richer guidance later.

### Alternative 3: Context-aware handoff assistance

- **Description**: Recommend or automate a handoff when remaining context falls below a threshold.
- **Pros**: Could turn awareness into workflow action.
- **Cons**: Treats an approximate informational signal as a decision authority and changes the user's workflow.
- **Why rejected**: Voluntary handoff timing is the V1 outcome; automation is a separate future product bet.

## Consequences

### Positive

- Long-running-session users gain a compact cue without interrupting active work.
- The product remains honest when a current value is unavailable.
- The V1 is easy to evaluate through scenario-based research before it expands.

### Negative

- Users who want a continuous context slot will see it disappear when no valid value exists.
- The cue cannot explain every provider-specific nuance in V1.
- It improves awareness but does not remove the judgment required to decide whether to hand work off.

### Risks

- **Users mistake the percentage for a guaranteed limit** — use neutral copy, omit unavailable data, and evaluate comprehension explicitly.
- **The cue is not valuable enough to earn statusline space** — measure usefulness in long-session handoff scenarios before expanding the feature.
- **Requests expand into alerts or automation** — preserve this ADR's voluntary field-only boundary.

## Implementation Notes

The PRD must retain the field-only product boundary, name omission as the expected absent-data experience, and make handoff clarity the primary V1 success metric.
