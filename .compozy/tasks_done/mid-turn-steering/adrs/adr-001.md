# ADR-001: Adopt a Lossless, Provider-Neutral Steering Contract for V1

## Status

Accepted

## Date

2026-07-15

## Context

Kitten currently permits an ordinary prompt submission while a session is working. This can replace local in-flight prompt tracking before the existing ACP turn settles, creating concurrent prompts, interleaved outcomes, and orphaned interaction callbacks. Developers supervising long-running agent tasks need to redirect work without losing their instruction or obscuring whether the prior turn completed.

The feature must preserve Kitten's layering: ACP mechanics remain in `src/agent/`, the core is protocol-free, the controller orchestrates effects, and the store is the only mutable-state writer. Providers may eventually offer native steering, but V1 must behave safely even when they do not.

## Decision

Deliver steering as a first-class, reliability-first V1 contract rather than as another ordinary prompt. The core defines prompt and steering identities plus legal lifecycle transitions; the controller serializes effects and commits terminal outcomes only after confirmation; the adapter uses a verified native steering capability when available and otherwise performs the fallback.

The fallback preserves queued user intent in chronological order, waits for permission and clarification work to reach a safe boundary, requests cancellation, waits for a bounded terminal settlement, and sends one coalesced follow-up that explicitly marks the preceding turn as interrupted. Every queued request has one visible terminal outcome: delivered or restored for the developer. Generation fencing and idempotent settlement protect close, crash, timeout, late-terminal-event, and session-replacement races.

## Alternatives Considered

### Alternative 1: Treat steering as an ordinary prompt

- **Description**: Allow `sendPrompt` to invoke another ACP prompt while a turn is active.
- **Pros**: Lowest immediate implementation effort.
- **Cons**: Permits the concurrent-send hazard, gives no reliable interruption semantics, and can orphan user intent.
- **Why rejected**: It fails the feature's reliability premise and contradicts user expectations established by current agent tools.

### Alternative 2: Support only providers with native steering

- **Description**: Expose steering only when an adapter advertises a native capability.
- **Pros**: Smaller fallback surface and provider-optimal behavior where available.
- **Cons**: Produces inconsistent product behavior and leaves unsupported adapters exposed to the current hazard.
- **Why rejected**: Kitten's value is provider-neutral supervision; capability differences must stay behind the adapter boundary.

### Alternative 3: Build a configurable provider-policy platform

- **Description**: Add policies, per-provider queue strategies, and broad user configuration in V1.
- **Pros**: Maximum flexibility for future providers.
- **Cons**: Expands lifecycle states, testing cost, and user decisions before evidence of need.
- **Why rejected**: It delays the core reliability fix and violates the chosen minimal V1 scope.

## Consequences

### Positive

- Prevents more than one active ACP prompt per session.
- Gives developers visible, lossless steering semantics across providers.
- Keeps provider-specific optimization private to the adapter while retaining a testable protocol-free contract.
- Establishes reliable lifecycle primitives for future native steering enhancements.

### Negative

- Adds explicit queue, interruption, settlement, and failure states to the product.
- A bounded terminal wait can delay follow-up dispatch when an adapter is slow or ambiguous.
- The UI and transcript must explain interruption without presenting duplicated user turns.

### Risks

- **Late or stale terminal events** — fence requests by identity and session generation; make terminal settlement idempotent.
- **Permission or clarification races** — do not begin fallback cancellation until active interaction work reaches a safe boundary.
- **Ambiguous delivery or provider failure** — do not silently retry; restore the queued text visibly for the developer.
- **Hidden state divergence** — test every named race, including timeout, close, crash, and session replacement.

## Implementation Notes

V1 excludes provider-policy configuration, automatic retries after ambiguous delivery, persistent steering history, and advanced multi-author steering controls. It must retain Kitten's fail-soft UI callbacks and content-free telemetry boundary. A later version may add optimized native capability paths only after the fallback contract is proven.
