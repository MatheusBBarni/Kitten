# ADR-003: Model Steering as a Protocol-Free State Machine with a Controller Effect Runner

## Status

Accepted

## Date

2026-07-15

## Context

Kitten's current `AgentConnection.prompt()` begins a new in-flight record by completing the old one. That permits a second prompt to displace tracking for an unfinished turn. The product requires one active turn, ordered steering, generation-safe settlement, and visible recovery while preserving the repository boundary: the core is pure, the store is the only session-state writer, the controller orchestrates effects, and ACP remains in `src/agent/`.

The existing interaction coordinator already proves the value of stable request identity, connection generation, queued lifecycle, and idempotent settlement. Steering needs the same truth model without mixing transport promises into the reducer.

## Decision

Add a protocol-free steering state machine in `src/core/` and expose its projection through `SessionState` and `DomainSessionEvent`. It owns steering request identity, generation binding, queue order, legal phase transitions, and a recoverable terminal outcome. Queued prompt blocks remain live state only and are never added to run persistence or telemetry.

Add a controller-owned steering effect runner per session. It reads the core state, waits for a safe interaction boundary, invokes the adapter, and dispatches the next pure event only after the effect settles. `src/agent/` remains the only layer that calls ACP. The adapter must reject an attempted concurrent `prompt()` even if a future caller bypasses the action surface.

## Alternatives Considered

### Alternative 1: Keep all steering state private to the controller

- **Description**: Store the queue, phases, and recovery outcome in controller-local mutable structures.
- **Pros**: Fewer reducer and selector changes.
- **Cons**: UI state, test assertions, and effect ownership can diverge; transition validity is difficult to test without transport fakes.
- **Why rejected**: It creates split lifecycle authority and weakens the store's single-writer contract.

### Alternative 2: Make the ACP adapter own the entire queue

- **Description**: Let each adapter serialize, cancel, and resend prompts independently.
- **Pros**: Keeps transport handling close to the provider.
- **Cons**: Leaks provider behavior into product semantics and duplicates cross-provider recovery rules.
- **Why rejected**: Kitten needs one provider-neutral control experience and a protocol-free domain model.

## Consequences

### Positive

- Lifecycle transitions become deterministic, reducer-testable, and visible to narrow selectors.
- The controller can fence asynchronous work by session generation without owning a second truth source.
- New providers can optimize transport behind the same core contract.

### Negative

- Adds explicit state and event vocabulary beyond the current coarse session status.
- Requires careful UI acknowledgement when a recovery draft is copied into the composer.

### Risks

- **Raw queued text survives longer than intended in memory** — clear it on every terminal transition and exclude it from persistence and telemetry.
- **Late completion mutates replacement state** — require request id and generation on all controller-issued settlement events.
- **A direct adapter caller bypasses actions** — reject concurrent `prompt()` calls at the adapter boundary.

## Implementation Notes

The effect runner may hold only timers and in-flight promises. It must not retain a second authoritative queue. Core transitions, session state, and UI projection remain TypeScript domain types; ACP request and notification types remain in `src/agent/`.
