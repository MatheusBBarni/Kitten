# ADR-003: Layered Architecture with an ACP Anti-Corruption Layer

## Status

Accepted

## Date

2026-07-07

## Context

Kitten depends on two fast-moving, pre-1.0 dependencies on its critical path: the ACP TypeScript SDK and OpenTUI.
ADR-001 and ADR-002 require an anti-corruption boundary around ACP and a strategy seam for hand-off assembly, so the deterministic bundle can later be swapped for an LLM-curated one without touching the UI.
The design must isolate the volatile dependencies, keep the hand-off logic pure and testable, and coalesce high-frequency streaming updates so the terminal stays flicker-free.

## Decision

Kitten uses a layered design that separates a functional core from an imperative shell:

1. **Agent Adapter Layer (imperative shell).** One `AgentConnection` per agent wraps a `Bun.spawn` subprocess and an ACP `ClientSideConnection`, implements the ACP `Client` interface (`requestPermission`, `sessionUpdate`), and translates the ACP `SessionNotification` union into Kitten's own normalized domain events. No ACP wire types leak past this layer. This is the anti-corruption boundary.
2. **Domain Core (pure).** The `SessionState` model, the `HandoffEngine` (a `BundleAssembler` strategy interface with a deterministic implementation), the `SecretRedactor`, and the telemetry heuristics live here. Pure, no I/O, fully unit-testable.
3. **Reactive Store.** In-memory application state (per-agent sessions, focus, status) exposed to the UI, batching streaming updates so only affected views re-render.
4. **UI Shell (OpenTUI + React).** Renders store state and dispatches user intents (prompt, switch focus, hand-off, approve) back through the core and adapters.

## Alternatives Considered

### Alternative 1: ACP types throughout (no anti-corruption layer)

- **Description**: Let ACP SDK types flow directly into the store and UI components.
- **Pros**: Less code; no translation layer.
- **Cons**: Couples the UI to a pre-1.0 protocol that ships frequent breaking changes; every SDK bump ripples across the app.
- **Why rejected**: Directly contradicts ADR-001/002; the churn isolation is the whole point.

### Alternative 2: Event-sourced / CQRS core

- **Description**: Model sessions as an append-only event log with projections.
- **Pros**: Strong auditability; natural fit for replayable transcripts.
- **Cons**: Heavy machinery for an in-memory, single-run MVP; slows delivery.
- **Why rejected**: YAGNI; in-memory reactive state is sufficient for V1 (ADR-002).

## Consequences

### Positive

- Churn from ACP and OpenTUI is confined to the adapter and shell layers; the core stays stable.
- The hand-off `BundleAssembler` strategy swaps deterministic to LLM-curated (ADR-002) without UI changes.
- The pure core is fast and simple to unit-test, matching the chosen testing strategy.

### Negative

- Requires upfront boundary discipline and a translation layer between ACP types and domain events.
- Two representations of a session (ACP-side and domain-side) must be kept in sync at the adapter seam.

### Risks

- The normalized domain event model could miss a field some agent relies on - mitigate by building it for the two concrete agents first and extending only when a real third agent exposes a gap.

## Implementation Notes

- Keep the `BundleAssembler` interface tiny so the Phase 2 LLM implementation is a drop-in.
- The adapter layer owns all subprocess and JSON-RPC concerns; the core never imports the ACP SDK.
