# ADR-003: Use a Certified Native Cursor ACP Profile with Adapter-Owned Login

## Status

Accepted

## Date

2026-07-13

## Context

Kitten currently treats Claude Code and Codex as closed, built-in provider recipes. The shared ACP adapter performs `initialize`, while Cursor's documented local ACP flow also requires the `cursor_login` authentication method before a session is created. The PRD requires Cursor to be a dependable default third session, while its absence, authentication failure, or incompatibility must leave sibling sessions usable.

The local workspace does not currently have the Cursor `agent` binary installed, so an exact supported Cursor version cannot be claimed without a credentialed contract run. Optional Cursor features must remain unavailable until independently verified.

## Decision

Add Cursor as a built-in resolved provider profile whose native launch command is `agent acp`. The profile carries an exact certified Cursor CLI version and the `cursor_login` ACP method. The config loader attaches that runtime-only profile only when the final resolved Cursor recipe exactly matches the certified recipe; a modified recipe is not treated as certified.

`AgentConnection.connect()` will perform the shared ACP initialization followed by the profile's authentication request inside `src/agent/`. Readiness will verify the Cursor CLI version before opening the long-lived session and will surface version mismatch or authentication failure as Cursor-only not-ready results. The certified version literal is added only in the reviewed change that includes a passing, opt-in credentialed contract result.

## Alternatives Considered

### Alternative 1: Generic user-supplied ACP command

- **Description**: Let any command be configured as a Cursor session and rely on the common adapter handshake.
- **Pros**: Smallest code change and broad nominal compatibility.
- **Cons**: Cannot safely determine authentication behavior, version compatibility, or supported optional capabilities.
- **Why rejected**: The requested V1 is a dependable first-class Cursor integration, not a generic provider platform.

### Alternative 2: External authentication before Kitten starts

- **Description**: Require the user to run Cursor login manually and never send an ACP authentication request.
- **Pros**: Leaves the generic adapter unchanged.
- **Cons**: Contradicts Cursor's documented ACP lifecycle and makes readiness failures less actionable.
- **Why rejected**: Authentication is protocol interaction and belongs at the adapter boundary, where it can degrade safely.

### Alternative 3: Advertise all Cursor capabilities after initialization

- **Description**: Treat a successful handshake as proof that every optional Cursor feature is usable.
- **Pros**: Faster feature surface.
- **Cons**: Risks offering untested clarification, restoration, or configuration behavior.
- **Why rejected**: Optional capabilities require separate credentialed evidence and otherwise fail closed.

## Consequences

### Positive

- The common adapter lifecycle remains protocol-contained while Cursor's documented login is handled reliably.
- A release names an exact compatibility profile instead of promising support for every Cursor CLI version.
- Unavailable or incompatible Cursor sessions are legible and isolated from Claude Code and Codex.

### Negative

- The first release is blocked on a credentialed contract run that records an exact Cursor version.
- Provider recipe overrides can deliberately move a user outside the certified profile.
- Readiness gains a small version/authentication taxonomy that must be covered by tests and user-facing copy.

### Risks

- **Cursor changes authentication or ACP behavior**: re-run the contract before updating the certified version.
- **The local binary has an unexpected version format**: fail closed with an actionable version-mismatch message.
- **An authentication error is ambiguous**: preserve the agent-provided reason in the Cursor-specific recovery message.
- **A later capability is assumed safe from login success**: keep each optional capability behind its own exact-profile evidence.

## Implementation Notes

- Keep ACP SDK calls, including `authenticate`, inside `src/agent/`.
- Keep the runtime profile out of user-authored JSON; strict config continues to express only provider recipe deltas.
- Reuse the existing readiness and controller degradation paths instead of adding a Cursor-specific boot flow.
- The real contract test is opt-in and never runs in the normal test suite or telemetry path.
