# ADR-004: Gate Explore Launches on Provider-Specific Capability Attestation

## Status

Accepted

## Date

2026-07-15

## Context

Current delegated children clone their parent provider configuration. Permission prompts and tool advertisement do not prove that an adapter cannot write, execute an unrestricted shell, or reach attached services. The default Codex recipe is explicitly full access, so treating an arbitrary resolved provider as safe would violate the approved fail-closed contract.

Different provider adapters expose different launch and permission guarantees. V1 must support only providers whose restricted child recipe can be evidenced by a closed, versioned contract.

## Decision

Introduce a closed provider-specific `ExploreCapabilityVerifier` registry at the app/config boundary. A verifier receives the resolved provider recipe and confirmed runtime metadata and returns either a versioned attestation with the exact V1 restrictions or a typed denial reason. The controller accepts an `explore` launch only after this result is current, then creates a fresh child using the attested recipe.

No user-controlled configuration flag can declare a runtime safe. Missing, stale, unknown, or partially verified evidence denies the launch. The resulting child receives only Kitten’s scoped `ask_user` bridge; global external MCP servers and agent-control tools are excluded.

## Alternatives Considered

### Alternative 1: Trust a provider configuration flag

- **Description**: Let a user or workspace mark a provider as safe for `explore`.
- **Pros**: Flexible and quick to expose.
- **Cons**: Converts a runtime security claim into unverified mutable configuration.
- **Why rejected**: It cannot establish the product’s required safety proof.

### Alternative 2: Infer safety from permission prompts and hidden tools

- **Description**: Launch normally and rely on the approval overlay or advertised-tool filtering.
- **Pros**: Reuses existing mechanisms.
- **Cons**: Adapters can retain authority without requesting a prompt, and hidden tools are not an execution barrier.
- **Why rejected**: It is a best-effort convention rather than a fail-closed boundary.

### Alternative 3: Require cross-provider parity before V1

- **Description**: Delay until all supported providers present identical evidence.
- **Pros**: Uniform initial availability.
- **Cons**: Blocks a proven-safe subset and forces a generalized design before real differences are known.
- **Why rejected**: A closed allowlist can be safe and useful while retaining a path to future parity.

## Consequences

### Positive

- The controller has one explicit eligibility gate and typed, explainable denial outcomes.
- Unsupported providers cannot silently approximate `explore`.
- MCP capability selection is tied to the accepted policy instead of a global default.

### Negative

- V1 may expose no eligible provider until a restricted recipe can be demonstrated.
- Each new provider requires its own evidence, versioning, and regression coverage.

### Risks

- **Attestation becomes stale after adapter changes**: include a verifier/recipe version and deny on mismatch.
- **Global MCP leakage**: build the child server list from accepted policy, not the parent’s generic server list.

## Implementation Notes

Keep the verifier registry closed and internal for V1. It is not the user-configurable role-profile surface proposed in the original issue.
