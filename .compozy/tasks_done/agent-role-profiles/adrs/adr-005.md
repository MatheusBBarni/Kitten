# ADR-005: Reserve Explore Capacity Atomically at Child Registration

## Status

Accepted

## Date

2026-07-15

## Context

The existing registry prevents nested delegation but does not enforce per-parent or global live-child limits. Launch is asynchronous, so a controller-only preflight can observe capacity and then race with another launch before either child is registered. The product requires capacity to be a safety guarantee, not a best-effort notice.

## Decision

Treat an accepted `explore` registration as a capacity reservation in the same synchronous core/store transition that inserts the child snapshot and visible session. A child in startup consumes capacity until its launch fails or reaches a terminal lifecycle state. The controller must never await between the authoritative capacity decision and registration; failed startup, cancellation, terminal completion, and parent cascade paths release the reservation through the existing delegation lifecycle.

The core decision returns a typed capacity denial rather than silently registering a child or relying on an asynchronous reconciliation queue.

## Alternatives Considered

### Alternative 1: Controller preflight before registration

- **Description**: Count children in the controller, then start a child if the count is below limits.
- **Pros**: Small apparent change.
- **Cons**: Concurrent requests can pass the same preflight and exceed the cap.
- **Why rejected**: It does not provide a race-safe product guarantee.

### Alternative 2: Reconcile excess children after startup

- **Description**: Start requests freely and cancel children when a later reconciliation finds a limit violation.
- **Pros**: Simple launch path.
- **Cons**: Allows transient authority and resource overuse, and creates confusing operator outcomes.
- **Why rejected**: Safety limits must prevent, not repair, excess launches.

## Consequences

### Positive

- Per-parent and global limits remain correct under concurrent calls.
- UI, lifecycle, and telemetry can distinguish capacity denial from provider ineligibility.
- Startup failure paths have a defined reservation-release responsibility.

### Negative

- Registration and cleanup tests must cover more lifecycle states.
- A child may consume a slot briefly while startup is in progress.

### Risks

- **Leaked startup reservation**: require cleanup on every failed or stale launch path.
- **Incorrect terminal accounting**: reuse reducer-owned terminal transitions and assert idempotent release.

## Implementation Notes

Do not add a queue, database, or background reconciler. Existing immutable delegation state and store commits are sufficient for V1.
