# ADR-003: Resolve Explore Policy in Core and Snapshot It on Registration

## Status

Accepted

## Date

2026-07-15

## Context

Delegation state is protocol-free and reducer-owned in `src/core/`, while the store is the only mutable owner of session and workspace state. The `explore` feature needs one deterministic policy definition that the controller, store, selectors, UI, telemetry, and future agent-control path can share. Computing the effective policy independently in those layers would allow the displayed role, launch behavior, and recorded child state to diverge.

The policy must remain fixed for V1, survive ordinary runtime publication while a child is live, and disappear with the existing ephemeral delegation state after restore.

## Decision

Add a pure `src/core/explorePolicy.ts` module that owns closed `explore` policy types, denial reasons, capacity rules, and pure launch-decision helpers. Extend `DelegatedChildSnapshot` and the registration event with an immutable `ExplorePolicySnapshot` that contains only role, verified restrictions, limits, attestation version, and confirmed provider/model/effort display values. The store writes that snapshot in the same commit that adds the child; selectors project it to views without recomputation from mutable configuration.

The snapshot is not persisted across restore. It is a live delegation fact, not user configuration or historical authorization.

## Alternatives Considered

### Alternative 1: Resolve policy only in the controller

- **Description**: Keep policy objects in controller runtime maps and derive UI state from those maps.
- **Pros**: Fewer core type changes.
- **Cons**: Duplicates store state, weakens selector determinism, and makes atomic capacity and UI projection harder to prove.
- **Why rejected**: The controller is an I/O orchestration seam, not a durable domain-state authority.

### Alternative 2: Derive policy directly in UI from configuration

- **Description**: Render a role label from current configuration without storing the launch-time decision.
- **Pros**: Minimal runtime state.
- **Cons**: Can display a policy that differs from the launched child after configuration changes and cannot support audit-safe lifecycle reasoning.
- **Why rejected**: A safety label needs a launch-time fact, not a fresh UI guess.

## Consequences

### Positive

- One pure source of truth makes role behavior testable without ACP or React.
- Child views receive stable, immutable effective-policy data through existing selectors.
- Future entry points can reuse the same decision and denial vocabulary.

### Negative

- Delegation types, registration events, fixtures, selectors, and snapshot tests need coordinated updates.
- The snapshot deliberately represents a narrow V1 role rather than a reusable arbitrary-profile schema.

### Risks

- **Snapshot drift from launch configuration**: construct it only from an accepted attestation and never recompute it after registration.
- **Accidental persistence**: retain the existing restore behavior that initializes delegation state empty.

## Implementation Notes

Keep the module free of ACP, configuration I/O, React, and telemetry. The controller supplies attestation facts; core validates and freezes the resulting policy snapshot.
