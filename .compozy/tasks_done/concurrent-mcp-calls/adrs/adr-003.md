# ADR-003: Admit independently authenticated sockets within route capacity

## Status

Accepted

## Date

2026-07-16

## Context

The bundled `kitten-ask-user` MCP child opens a private local socket for every
`ask_user` or `agent_run` invocation. `src/app/kittenMcpBridge.ts` authenticates
each frame with a capability, then currently makes the first socket the route's
exclusive `boundSocket`. A second valid socket for the same session generation is
rejected with `connection_stream_limit` before the existing four-call route limit
is evaluated. This makes valid mixed supervised work fail solely because the
child uses short-lived connections.

The bridge must keep its private endpoint, capability-derived authority,
generation fence, strict frame validation, 256-call lifetime limit, and four
pending-call limit. Closing one child socket must not turn a per-connection
transport failure into cancellation of unrelated calls on the same authorized
route.

## Decision

Replace the singleton `Route.boundSocket` ownership model with a route-local set
of authenticated connection states. A socket may bind only to one route, but a
route may have several bound sockets. After capability and generation validation,
each frame reserves the existing per-route capacity immediately; there is no
queue, fairness policy, or shared cross-session capacity.

`PendingCall` continues to own its originating socket. A normal idle socket close
only unregisters that socket. A socket error or close settles/cancels only its
own pending clarification through a targeted request handle and lets an
`agent_run` result become unavailable to that disconnected caller; it does not
close the route or replay the request. Route invalidation (`session_replaced`,
conversation close, controller disposal) still cancels every outstanding
clarification, stops the listener, removes the endpoint, and prevents further
admission.

## Alternatives Considered

### Alternative 1: Add a FIFO queue above the existing limit

- **Description**: Queue same-route calls until one of the four active calls settles.
- **Pros**: Could smooth bursts without an immediate tool error.
- **Cons**: Requires queue lifetime, fairness, cancellation, and retry semantics.
- **Why rejected**: The defect is false socket exclusivity, not absence of a scheduler; a queue expands the MVP and obscures admission timing.

### Alternative 2: Reuse one persistent child socket

- **Description**: Change the bundled MCP child to multiplex every invocation through one process-local connection.
- **Pros**: Preserves the bridge's singleton-socket assumption.
- **Cons**: Adds client-side lifecycle and multiplexing state and does not help independently spawned clients.
- **Why rejected**: The server already authenticates each invocation and has bounded route capacity; server-side admission is the smaller, more general repair.

### Alternative 3: Permit unbounded same-route sockets and calls

- **Description**: Remove both stream and call admission limits.
- **Pros**: Eliminates immediate contention failures.
- **Cons**: Allows resource exhaustion and changes the trust and reliability contract.
- **Why rejected**: Bounded capacity and fail-closed behavior are explicit PRD constraints.

## Consequences

### Positive

- `ask_user` and `agent_run` can overlap from separate valid sockets in one parent session.
- The existing four-call maximum becomes the authoritative source of temporary-capacity failures.
- A failed child connection no longer cancels unrelated same-route work.
- Cross-session authority remains capability and generation scoped.

### Negative

- The bridge must track socket membership and per-socket pending work precisely.
- Targeted clarification cancellation is needed in the controller interaction seam.
- More lifecycle race cases require focused bridge tests.

### Risks

- **A lost socket leaves a clarification visible** — cancel the exact request through a handle-level cancellation operation and assert it is removed from the coordinator.
- **A route teardown misses a socket** — retain listener-wide shutdown and route-level cancellation as the only route teardown path.
- **A duplicate or cross-session request is admitted** — preserve capability lookup, per-socket route binding, generation validation at dispatch, and existing call-ID checks.
- **Capacity becomes ambiguous** — reject the fifth pending call immediately with the existing bounded `busy` response; do not queue it.

## Implementation Notes

Modify only the existing bridge, controller clarification-handle seam, and their tests. Keep endpoint creation private (Unix socket or Windows named pipe), do not add configuration, persistence, or a scheduler, and remove the now-invalid `connection_stream_limit` admission expectation.
