# ADR-002: Center the MVP on mixed supervised work and deliberate recovery

## Status

Accepted

## Date

2026-07-16

## Context

Issue #27 affects an agent author when two valid supervised actions overlap within one parent session. The highest-value user moment is not abstract parallelism: an agent needs a developer decision while it starts or observes delegated work. Today, a false contention can look like a generic failure, leaving the developer unable to tell whether work can continue or what action is safe.

The product must improve continuity and clarity without expanding the release into a broader orchestration surface or making a potentially duplicate child-start action appear safely repeatable.

## Decision

Make the MVP promise two things to the agent author: valid overlapping supervised work continues, and a genuine temporary capacity limit gives a concise, truthful, deliberately actionable state. The existing tool outcome experience communicates whether a request is temporarily constrained or unavailable. It offers recovery guidance only after the original action has a known terminal outcome; the product does not automatically resume or replay an ambiguous action.

The primary acceptance scenario is a developer decision overlapping with delegated work from one parent session. The release remains focused on this moment and the associated privacy, isolation, and bounded-resource guarantees.

## Alternatives Considered

### Alternative 1: Deliver continuity only

- **Description**: Remove the false failure but leave generic outcome presentation unchanged.
- **Pros**: Smaller user-facing scope.
- **Cons**: Developers still cannot distinguish a real limit from unavailability or determine safe next steps.
- **Why rejected**: The selected value proposition is continuity and truth, not silent transport improvement.

### Alternative 2: Automatically resume constrained work

- **Description**: Retry a constrained action without developer involvement when capacity appears available.
- **Pros**: Reduces visible friction in simple cases.
- **Cons**: An ambiguous delegated-work start could be repeated, producing duplicate work and eroding trust.
- **Why rejected**: The user chose clear, safe manual recovery; automatic replay requires a separately proven product contract.

### Alternative 3: Build a generalized parallel-work management product

- **Description**: Add configurable scheduling, priority controls, durable history, and a dedicated management surface.
- **Pros**: Addresses broader future scale and visibility needs.
- **Cons**: Delays correction of the immediate reliability defect and introduces a larger adoption surface.
- **Why rejected**: The MVP needs a narrow, testable trust repair first.

## Consequences

### Positive

- Gives agent authors a concrete mixed-work scenario in which Kitten is dependable.
- Makes recovery understandable without requiring users to infer transport state.
- Keeps the product aligned with supervised, privacy-conscious agent work.

### Negative

- Deliberately defers automation and richer work-management controls.
- Requires concise user-facing wording that avoids implying an unsafe replay is available.

### Risks

- **Developers misread a temporary state as permission to repeat an unknown action** — distinguish known retryable limits from ambiguous outcomes and make manual recovery explicit.
- **Scope expands into a general agent scheduler** — keep the user stories and rollout criteria tied to the mixed supervised-work scenario.
- **Operational detail leaks into the user experience** — present only bounded, actionable states and preserve the product's content-free privacy posture.

## Implementation Notes

Use the established tool outcome surface for the user-facing state. Preserve the product boundaries recorded in ADR-001; implementation mechanics belong in the follow-on TechSpec.
