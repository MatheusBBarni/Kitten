# ADR-004: Project closed MCP failures without replaying ambiguous work

## Status

Accepted

## Date

2026-07-16

## Context

The bridge already returns bounded `busy` and `unavailable` errors to the bundled
MCP child, but the controller does not wire the bridge's existing failure callback
to the opt-in recorder. The agent-facing transcript also reduces failed tool calls
to a generic `failed` label. As a result, an agent author cannot distinguish an
authoritative capacity limit from an unavailable route, and maintainers cannot
observe the closed outcome category without collecting private request content.

MCP error results are forwarded as text content through ACP tool-call updates.
`src/agent/acpTranslate.ts` currently retains diffs only, while the core tool-call
model and `ToolCallRow` are protocol-free. A design must classify only Kitten's
fixed error envelope, never retain or display arbitrary tool output, raw errors,
prompts, task text, route endpoints, capabilities, or identifiers.

## Decision

Add the protocol-free `ToolCallFailureKind` union `"temporary_capacity" |
"unavailable"` as an optional field on `ToolCallRecord` and `ToolCallUpdate`.
At the ACP boundary, remember whether a tool-call ID belongs to the bundled
`kitten-ask-user` MCP server and parse only its exact, one-field JSON error
envelope. Map `busy` to `temporary_capacity` and `unavailable` to `unavailable`;
discard the source text after classification. Unknown, malformed, or unrelated
tool content remains a normal generic failure.

The reducer preserves and explicitly clears this optional domain field. The
existing `ToolCallRow` adds plain-language status detail: temporary capacity tells
the author to wait for a known terminal outcome before deliberately trying again;
unavailable states that the action cannot be completed. Neither state renders a
retry control or triggers automatic replay. A disconnected or otherwise ambiguous
`agent_run.start` is shown as unavailable and is never repeated by Kitten.

Wire `KittenMcpBridgeOptions.onFailure` in the controller to a new opt-in,
content-free recorder event. The controller maps bridge reasons to a small
semantic vocabulary (`capacity_limited`, `unavailable`, or `invalid_request`) and
records no route, capability, session, call, error string, prompt, or task data.

## Alternatives Considered

### Alternative 1: Show every raw MCP error in the transcript

- **Description**: Forward all ACP text content to the tool-call row.
- **Pros**: Requires little classification logic and can aid debugging.
- **Cons**: Leaks provider text, request context, and unstable implementation details into the UI and local telemetry paths.
- **Why rejected**: The PRD requires bounded, content-free failure states.

### Alternative 2: Treat all failed tool calls as temporary capacity

- **Description**: Add one generic retry-oriented label for every MCP failure.
- **Pros**: Simple presentation.
- **Cons**: Makes unavailable and ambiguous delegated starts look safely retryable.
- **Why rejected**: It violates the selected no-replay safety rule and is not truthful.

### Alternative 3: Add durable execution history before explaining failures

- **Description**: Persist tool results and deduplicate starts across reconnects.
- **Pros**: Could support future idempotent replay.
- **Cons**: Adds retention, identity, and privacy scope beyond the immediate defect.
- **Why rejected**: V1 must remain a bounded reliability repair; durable replay needs its own product and privacy decision.

## Consequences

### Positive

- Agent authors see distinct, actionable bounded states in the existing tool outcome surface.
- The core remains independent of ACP and MCP types.
- Opt-in local telemetry observes meaningful bridge categories without content.
- Unknown work is never silently replayed.

### Negative

- The adapter needs small per-tool-call classifier state because ACP updates may omit a title.
- Presentation depends on a controlled child error envelope and needs end-to-end coverage.
- Generic third-party MCP failures intentionally retain the existing generic display.

### Risks

- **An arbitrary tool output is misclassified** — require both remembered bundled-server identity and an exact bounded error envelope.
- **A later update loses classification** — merge `failureKind` by tool-call ID and remove adapter classifier state when terminal updates settle.
- **Telemetry expands over time** — make the recorder input a closed type and test that no free-text field can cross the API.
- **A user repeats ambiguous work** — do not expose automatic retry; use wording that ties manual retry to a known terminal outcome.

## Implementation Notes

Keep ACP and JSON parsing in `src/agent`; expose only the closed domain union from
`src/core`. Keep `agent_run_control` telemetry intact and add a separate bridge
outcome event rather than overloading it. No database, remote telemetry, public
HTTP API, dashboard, configuration option, or retry scheduler is introduced.
