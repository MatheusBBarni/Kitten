# ADR-005: First-Run Persistence via a Runtime State File plus a Read-Only Config Setting

## Status

Accepted

## Date

2026-07-09

## Context

The PRD requires the welcome banner to show in full on the first ever launch and auto-quiet to a one-line greeting on subsequent launches, plus an explicit way to disable it.
Auto-quiet requires remembering that the app has run before.
Codebase exploration found that Kitten has **no persisted user-state** today: `src/config/configLoader.ts` is strictly read-only (no code writes config), and `src/config/firstRun.ts` only reads the filesystem.
Config lives at `$KITTEN_CONFIG` / `$XDG_CONFIG_HOME/kitten/config.json` and is a user-authored, delta-over-defaults, `strict()` schema.

## Decision

Split machine-written state from user-authored config.

- **Runtime state file** (app-written): a small JSON file at `$XDG_STATE_HOME/kitten/state.json` (falling back to `~/.local/state/kitten/state.json`), holding a `firstRunSeenAt` timestamp written on the first successful boot. This is the only thing the app persists, and it keeps `config.json` read-only.
- **Config setting** (user-authored): an optional `welcomeBanner: "auto" | "always" | "off"` field added to the config schema. `auto` (default) uses the state file to decide full-vs-quiet; `always` always shows the full banner; `off` suppresses it. The user hand-edits this; the app never writes it.
- Banner-shown decision: `off` -> nothing; `always` -> full; `auto` -> full when `firstRunSeenAt` is absent, else quiet.

## Alternatives Considered

### Alternative 1: Config writer

- **Description**: Add a config write path and persist `firstRunSeen` and the banner preference in `config.json`.
- **Pros**: Single file.
- **Cons**: Breaks the read-only-config invariant; mixes app-written state into a user-authored file, risking clobbered hand edits/comments; needs safe-write/merge logic.
- **Why rejected**: The read-only config boundary is a deliberate, valuable invariant; state does not belong in the user's config.

### Alternative 2: No persistence (session-only)

- **Description**: Do not persist; quiet only within a session or via an env var/flag each run.
- **Pros**: Simplest.
- **Cons**: Every fresh launch shows the full banner - the exact behavior the PRD's auto-quiet decision avoids.
- **Why rejected**: Fails the PRD requirement.

## Consequences

### Positive

- `config.json` stays read-only and user-owned; app state is isolated and disposable.
- Follows the XDG base-directory convention (state vs config separation).

### Negative

- Introduces Kitten's first app-written file and a new (tiny) read/write module.

### Risks

- **State file unwritable** (permissions, read-only FS). Mitigation: treat write failure as non-fatal - fall back to showing the full banner and continue; never block boot on it.
- **Schema drift** in `state.json`. Mitigation: validate with a small zod schema and ignore/reset on parse failure.

## Implementation Notes

- Add a `welcomeBanner` optional field to the config schema in `src/config/configLoader.ts` (default `"auto"`); keep the loader read-only.
- Add a minimal state read/write module (e.g. `src/config/appState.ts`) with `readFirstRunSeen()` and `markFirstRunSeen()`, using the XDG state path; both must fail soft.
- The banner-shown decision is computed at boot (transient banner) and reused by the idle-screen banner.
