# ADR-002: Chrome-First, Data-Additive Rollout for the TUI Reskin

## Status

Accepted

## Date

2026-07-09

## Context

The `_idea.md` and ADR-001 established the V1 scope of the Claude Code-style TUI reskin: a Kitten-branded welcome banner, a rich dual-agent status bar, and a prompt + palette restyle, keeping the dual-agent layout and hand-off unchanged.
PRD research then confirmed the key delivery tension.
The status bar's highest-value signals - per-agent context-usage % (the most acute, most-requested signal, because users get blindsided by silent auto-compaction) and model + effort - are not fully plumbed today.
Two of them already have accepted ADRs and pending task packets in sibling feature folders (`agent-usage-gauge`, `model-effort-selector`); the rest of the reskin (palette, prompt, banner, git branch, focus, run-state) depends on no unbuilt work.
The question is how to sequence delivery so the reskin ships value without either blocking on unrelated features or shipping a bar that lies about data it does not have.

## Decision

Adopt a **chrome-first, data-additive** rollout in three phases:

1. **Phase 1 - Visual layer.** Warm-accent palette preset + rounded borders + chevron prompt restyle, and the branded welcome banner that also fills the currently-blank boot/handshake wait as a loading state. No data dependency. Ships immediately.
2. **Phase 2 - Dual-agent status bar (real signals).** The restructured always-on bar rendering the signals Kitten owns today: per-agent focus and run-state (kept as two orthogonal signals), model when the session reports it, plus shared git branch and cwd, and the always-visible hand-off affordance. Governed by hide-when-absent and the 80-column priority-collapse budget.
3. **Phase 3 - Additive clarity.** Per-agent context-usage % (color thresholds green -> amber ~70% -> red ~85%) and model + effort light up in their pre-built slots as the `agent-usage-gauge` and `model-effort-selector` seams land. No layout reflow.

The reskin never blocks on the pending data features; each phase is independently shippable and honest about what it shows.

## Alternatives Considered

### Alternative 1: Clarity-gated bundle

- **Description**: Hold the entire reskin until model+effort and usage-gauge are ready, then ship one complete, data-rich release.
- **Pros**: One coherent "wow" release; no partial or empty states; the flagship context-% is present from day one.
- **Cons**: Delays all the polish (banner, palette, prompt) behind unrelated data work; couples this effort's timeline to two other features; contradicts the council's "ship honest value now."
- **Why rejected**: The visual polish and the real-signal bar deliver value immediately and have no reason to wait; coupling the whole reskin to external timelines is unnecessary risk.

### Alternative 2: Polish-only, defer the bar

- **Description**: Ship only the visual layer (palette/prompt/boot-banner) now; defer the entire status-bar restructure to a later effort.
- **Pros**: Smallest, fastest, lowest-risk; zero data coupling.
- **Cons**: Drops the signature differentiator (the dual-agent status bar); delivers only the first-impression half of the stated goal, not operational clarity.
- **Why rejected**: The owner's goal is explicitly both polish and clarity, and the dual-agent bar is the product's most differentiating surface. Dropping it forfeits the core value.

## Consequences

### Positive

- Polish ships in Phase 1 with no data dependency; the signature bar ships in Phase 2 on signals that are real today.
- The flagship context-% and model/effort snap into pre-built, tested slots when their seams land, with no second layout reflow.
- Each phase is independently valuable and demoable, so progress is visible and reversible.

### Negative

- The highest-value clarity signal (context-%) arrives in Phase 3; until then the bar is honest but partial.
- Delivery value is spread across three increments rather than one headline release.

### Risks

- **Sequencing risk**: if `agent-usage-gauge` and `model-effort-selector` stall, Phase 3 stalls and the bar stays partial. Mitigation: Phases 1-2 stand alone and already beat today's experience; the bar is designed hide-when-absent so a missing signal is invisible, not broken.
- **Perceived incompleteness**: users may notice missing signals between phases. Mitigation: hide-when-absent means empty slots collapse to zero width; nothing advertises a signal it cannot show.

## Implementation Notes

- Phase boundaries map to the PRD's Phased Rollout Plan.
- The banner-as-boot-screen shows agents as "connecting..." during the handshake, then becomes the idle screen until the first prompt; it auto-quiets to a one-line greeting after first run and can be disabled via a setting.
- Context-% color thresholds and the hand-off affordance's "why it can't run yet" states are user-facing requirements carried in the PRD.
