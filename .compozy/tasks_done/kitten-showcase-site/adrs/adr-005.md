# ADR-005: Resolve GitHub star count client-side with resilient fallback

## Status

Accepted

## Date

2026-07-12

## Context

The PRD requires an accurate GitHub star control and a non-fabricated fallback when data is unavailable. Build-time prefetch is an option, but it increases workflow coupling. We need a simple, deterministic behavior with graceful degradation.

## Decision

Implement a client-side fetch from GitHub REST endpoint
`https://api.github.com/repos/{owner}/{repo}` on page load with:

- async star count update,
- accessible loading and fallback text,
- and no zero fabrication if retrieval fails.

If the fetch fails, the control shows a safe fallback label and keeps the star CTA usable.

## Alternatives Considered

### Alternative 1: Build-time star prefetch in CI

- **Description**: Fetch star count in a workflow and inject into site JSON.
- **Pros**: Fully deterministic static value and no browser API calls.
- **Why rejected**: Requires an extra deploy-side mutation step and can hide stale counts longer than acceptable for launch messaging.

### Alternative 2: No star number in V1

- **Description**: Display only a “Star” link without count.
- **Pros**: Eliminates API dependency.
- **Why rejected**: Contradicts PRD requirement to include accurate secondary social proof signal.

## Consequences

### Positive

- Always-on current public count for visitors (within GitHub API availability).
- No build pipeline coupling for every view.

### Negative

- Page includes a small external dependency at runtime.
- Rate limits and network errors are possible.

### Risks

- Temporary unavailable API or request failure.
- Mitigation: show non-deceptive fallback text and avoid a zero value.

## Implementation Notes

Add `client:load` script that:

- requests repo metadata once on mount,
- applies text + `aria-live` update only when successful,
- caches nothing locally (no identifiers).

Use explicit repository owner/name constants in site config; avoid reading from user input.
