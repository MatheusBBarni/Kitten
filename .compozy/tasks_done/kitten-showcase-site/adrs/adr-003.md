# ADR-003: Keep showcase delivery as a separate Astro subproject in `site/`

## Status

Accepted

## Date

2026-07-12

## Context

Kitten is a terminal application with a Bun/OpenTUI stack and strict layering rules. The website is static, browser-only public content. Bundling it into the root Bun stack would mix incompatible concerns (CLI runtime tooling, test setup, and terminal-only packages) and blur release artifacts.

## Decision

Create a dedicated `site/` directory as a standalone Astro project with its own `package.json`, build script, and output folder. The site source and compiled artifacts live entirely outside the CLI runtime path and are deployed to GitHub Pages independently from the Kitten binary release pipeline.

The root task remains unchanged except for:

- a top-level GitHub Actions workflow under `.github/workflows/` that runs in `site/`
- optional reference links from root docs to the public site URL.

## Alternatives Considered

### Alternative 1: Embed a docs site under `docs/` using the root package toolchain

- **Description**: Reuse the root `package.json` and build flow for website content.
- **Pros**: Fewer dependency files, no separate `npm` lockfile.
- **Why rejected**: Increases coupling between CLI release checks and website dependencies, and raises risk of breaking either artifact with changes for the other.

### Alternative 2: Keep site as a second static folder under root without its own dependency graph

- **Description**: Add raw HTML/CSS/JS directly under `public-site/` with no Astro.
- **Pros**: Fewer moving pieces.
- **Why rejected**: Drops composability for reusable sections and degrades accessibility/testing discipline versus Astro componentization.

## Consequences

### Positive

- Clear ownership boundary between CLI runtime and web marketing surface.
- Independent dependency and lockfile lifecycle for site tooling.
- Independent deployment cadence from binary release artifacts.

### Negative

- Two lockfiles and two build scripts to keep aligned.
- Slightly more repository overhead for path navigation.

### Risks

- Drift between docs and CLI versioning if release links and install messaging are not synchronized in automation.
- Mitigation: define install command and repo metadata through a shared static config file consumed by the site build.

## Implementation Notes

Use an explicit subproject layout:

- `site/package.json` and `site/astro.config.mjs`
- `site/src/pages/index.astro`
- `site/src/components/*` for sections (`Hero`, `Proof`, `Faq`, `Trust`, `Cta`, `StarBadge`)
- `site/public/*` for media and assets
- `site/.github` is not required; all CI/CD remains in repo root workflow files.
