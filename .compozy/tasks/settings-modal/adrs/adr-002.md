# ADR-002: Instant-apply, live-preview interaction model for the settings modal

## Status

Accepted

## Date

2026-07-09

## Context

ADR-001 set the V1 scope: a settings modal shell with theme, built on a reactive-config foundation, keymap as the fast-follow.
It left the modal's interaction and commit model open.
The PRD clarification fixed the surrounding constraints: the primary persona is the existing daily power user, the product owner's stance is "live and persisted," and the app already has two modal overlays (`ApprovalPrompt`, `HandoffPreview`) with settled arrow / Enter / Esc conventions.
The decision is how a user moves through settings and when a change takes effect and is saved.

## Decision

The settings modal is an instant-apply, live-preview, tabbed overlay.

- Categories are tabs (Theme in V1, Keys added in Phase 2).
- Moving the selection to a theme repaints the real cockpit live and persists the change immediately; there is no separate save step.
- Each setting offers reset-to-default.
- Esc closes the modal; the last applied value stays in effect.
- Only settings that can genuinely apply live are presented as instant; anything that cannot (deferred to later phases) is labeled, never shown as instant.

## Alternatives Considered

### Alternative 1: Preview-then-confirm

- **Description**: Same tabbed shell and live preview, but a change persists only on an explicit Enter/Apply; Esc reverts to the prior value.
- **Pros**: Guards against an accidental persisted change; echoes the approval/hand-off "confirm with Enter" muscle memory.
- **Cons**: Adds friction for a pick-and-go power user; introduces an ambiguous "previewing but not saved" state; requires holding and restoring revert state.
- **Why rejected**: The safety benefit is marginal because a theme change is trivially reversible (pick another, or reset-to-default), and the friction works against the primary persona.

### Alternative 2: Searchable command palette

- **Description**: No tabbed screen; a searchable list of settings that the user filters and edits inline (gemini-cli `/settings` style).
- **Pros**: Scales to many settings; suits keyboard-first users.
- **Cons**: Weakens the browse-and-preview theme experience and the differentiated "settings home" frame; a flat list is a worse fit for live preview than a tabbed browse.
- **Why rejected**: ADR-001 already argued that shipping settings as a bare command undersells the surface; the palette is the same trade at a different grain.

## Consequences

### Positive

- Lowest friction for the power-user persona: pick a theme and move on.
- Matches the "live and persisted" stance directly.
- Reuses the existing overlay conventions, so the modal feels native immediately.
- Tabs extend cleanly to the keymap category in Phase 2.

### Negative

- Navigating onto a theme persists it, so an exploratory move is a real change; mitigated by reset-to-default and by the change being trivially reversible.
- Instant persistence fires the write path frequently, so write-back must stay cheap (and likely debounced); this is an implementation concern the TechSpec must handle, not a product one.

### Risks

- A slow or failing write mid-navigation could surface errors while the user browses; mitigated by keeping the in-memory live change and surfacing a persistence failure without blocking the browse.
- The "applies live" promise must never extend to a setting that cannot be live; mitigated by the labeling rule above and by deferring non-live categories out of V1 per ADR-001.

## Implementation Notes

- Follow the overlay pattern verbatim: store slot joined into `selectHasOpenOverlay`, `useKeyboard` capture with `preventDefault`, an `open-settings` binding in `COCKPIT_KEYMAP`, mounted below the approval prompt so a permission request still outranks it.
- Debounce or coalesce writes so live navigation does not thrash the config file.
