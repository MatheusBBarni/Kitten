# ADR-002: V1 Product Scope - Guarded Parity With a Single Global MCP List

## Status

Accepted

## Date

2026-07-09

## Context

ADR-001 fixed the mechanism: V1 propagates MCP servers by injecting them into each agent's ACP session, not by writing to agent config files.
That still leaves a product-scope decision about how much control and how many guardrails ship in the first release.

Clarifying answers during PRD brainstorming set four boundaries:
users author MCP servers by hand-editing Kitten's config with documentation, secrets are expressed as environment-variable references rather than raw values, the user sees a per-agent readout of what loaded and a warning on anything skipped without startup ever being blocked, and config changes take effect on the next launch.

The open axis was how complete V1 should be: minimal injection, guarded parity with one global list, or guarded parity plus per-agent control.

## Decision

V1 ships "guarded parity" with a single global MCP list.

The user declares one `mcpServers` list in Kitten's config.
Kitten injects the compatible subset into every agent session, resolves `${VAR}` environment references, and never writes raw secrets anywhere.
A per-agent readout reports which servers loaded and warns on any that were skipped or failed, but startup is never blocked.
Transport is gated by each agent's advertised capability, with stdio as the guaranteed baseline.
A CI smoke test gates the release by proving each pinned adapter honors an injected server.

Per-agent differences (excluding a server from a specific agent) are deferred to a later phase.

## Alternatives Considered

### Alternative 1: Minimal injection, defer guardrails

- **Description**: Ship config schema, injection, documentation, and the smoke test only; defer the loaded-servers readout and capability gating to a fast-follow.
- **Pros**: Fastest path to a first release; least surface.
- **Cons**: Reintroduces the silent-no-op risk the council flagged; contradicts the user's explicit choice of a visible per-agent readout.
- **Why rejected**: The readout and capability gating are the guardrails that make injection trustworthy; dropping them trades away the feature's credibility for marginal speed.

### Alternative 2: Guarded parity plus per-agent exclude

- **Description**: The chosen scope plus the ability to exclude a specific server from a specific agent.
- **Pros**: Honors intentional divergence (a user who deliberately wants different tools in each agent); addresses the devils-advocate's held-firm point.
- **Cons**: Adds config surface, documentation, and a second mapping model in the first release, for a flexibility most users will not need on day one.
- **Why rejected for V1**: A single global list is the simplest expression of the "declare once" goal; the exclude is a clean, additive refinement that can ship once the core is proven. Retained as an early V2 candidate.

## Consequences

### Positive

- The V1 scope maps one-to-one onto the four clarifying answers, so there is no ambiguity to resolve during the TechSpec.
- Guardrails (readout, capability gating, smoke test, env references) are all in the first release, so the feature is trustworthy from day one rather than after a hardening pass.
- A single global list is the smallest honest expression of "declare once," which keeps documentation and mental model simple.

### Negative

- Users who want different tool sets per agent cannot express that in V1; they get every compatible server in both agents.
- The per-agent readout and capability gating add modest scope beyond the barest possible injection.

### Risks

- **Users expect per-agent control sooner than planned.** Mitigation: the global list is designed as an overridable default so a per-agent exclude is additive, not a rewrite; track demand and pull it forward if it surfaces.
- **The "warn, never block" stance hides a genuinely broken setup.** Mitigation: the readout names servers, not just a count, and the smoke test guarantees the mechanism itself works before release.

## Implementation Notes

- The four clarifying answers (hand-edit authoring, env-reference secrets, readout with warn-never-block, apply-on-next-launch) are product requirements captured in the PRD, not implementation choices.
- The per-agent exclude should be modeled now as an overridable-default shape so it can be layered on without reworking the global list.
