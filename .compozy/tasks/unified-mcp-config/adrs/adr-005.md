# ADR-005: Adapter-Honor Smoke Test as a Gated Real-Subprocess Integration Test

## Status

Accepted

## Date

2026-07-09

## Context

The council made one requirement non-negotiable: before the injection feature ships, prove that each pinned adapter actually honors client-provided MCP servers.
The worst failure mode is a silent no-op, where an adapter accepts the `mcpServers` array, ignores it, and returns a healthy session with none of the declared tools.

An in-process ACP stub (the existing in-memory transport pair) can prove that Kitten sends a correctly shaped `mcpServers` array, but it cannot prove that the real adapters (`claude-agent-acp@0.57.0`, `codex-acp@1.1.0`) honor it, because the stub is Kitten's own code.
Only a real adapter subprocess can prove real honoring.
Real adapters are launched through `npx` and may need to be installed, which is a CI-environment constraint.

## Decision

Build a real-subprocess integration test that, for each pinned adapter, spawns the adapter, injects a trivial local stdio MCP server (a small fixture server that exports one known tool), creates a session, and asserts that the injected tool appears.

Run this test under a dedicated script/tag (for example `bun run test:mcp-smoke`), separate from the default unit run, so it executes in environments where the adapters are installed rather than on every unit invocation.

This test is the release gate: a red result blocks shipping the injection feature, and the documented setup fallback becomes the interim answer.

Keep a fast in-memory contract test in the default unit suite as well, asserting Kitten builds and sends the correct `mcpServers` array, so wiring regressions are caught cheaply; the real-subprocess test remains the honoring gate.

## Alternatives Considered

### Alternative 1: In-memory contract test only

- **Description**: Rely solely on the in-process ACP stub to assert the array is sent correctly.
- **Pros**: Fast, dependency-free, runs everywhere.
- **Cons**: Tests Kitten's wiring, not whether the real adapter honors injected servers, which is exactly the risk the council flagged.
- **Why rejected**: It cannot detect the silent no-op that motivates the gate.

### Alternative 2: In-memory contract plus documented manual smoke

- **Description**: Ship the contract test for CI and a written manual procedure a maintainer runs before release.
- **Pros**: Pragmatic if authenticated adapters cannot run in CI.
- **Cons**: A manual step is skippable and drifts; it is not a real gate.
- **Why rejected as the primary**: An automated real-subprocess test is a stronger gate. The manual procedure is retained only as a fallback if a specific CI environment cannot run the adapters.

## Consequences

### Positive

- The single worst failure mode (silent no-op) is caught before release rather than by a user mid-task.
- The fast contract test still guards wiring on every unit run.

### Negative

- The test tree needs a minimal fixture stdio MCP server.
- The gated test depends on the adapters being installed in the environment that runs it, and must re-run whenever a pinned adapter version changes.

### Risks

- **The gate does not run in a given CI environment.** Mitigation: the dedicated script makes it explicit and runnable wherever adapters exist; the documented manual procedure is the fallback, and honoring is proven at pin-time only, so the gate is tied to adapter version bumps.
- **A tool loads but is not callable.** Mitigation: assert on the named tool's presence, and prefer a fixture whose tool can be invoked to confirm end-to-end.

## Implementation Notes

- Add a fixture stdio MCP server (a tiny script exporting one deterministic tool) under the test tree.
- Add the `test:mcp-smoke` script; keep it out of the default `bun test` path.
- The in-memory contract test uses the existing in-process transport pair seam.
