# ADR-002: Prioritize fast, explicit child launch in the MVP

## Status

Accepted

## Date

2026-07-14

## Context

The approved idea defines reliable, provider-neutral delegation of independent child work within one active coding task. The PRD clarification round established the first product moment users value most: a developer explicitly states a child task and desired outcome, remains focused on the parent task, and immediately sees that the child is running.

Existing session visibility and market research make supervision, interaction handling, terminal outcomes, and safe cancellation essential trust requirements. They must support the launch experience, but they should not compete with it for MVP focus.

## Decision

Make fast, explicit child launch the primary V1 experience. A developer starts independently scoped child work from an active parent without being taken away from that parent, and receives an immediate, unambiguous Running indication for the new child.

V1 still includes visible child identity, attributable attention and terminal status, outcome inspection, and safe group cancellation as the minimum trust baseline. It does not make automatic decomposition, role-template selection, advanced supervision, or orchestration automation prerequisites for launch.

## Alternatives Considered

### Alternative 1: Treat every lifecycle capability as an equal primary experience

- **Description**: Give launch, supervision, collection, cancellation, and long-term monitoring equal prominence in MVP design.
- **Pros**: Presents a broad, complete orchestration story from day one.
- **Cons**: Dilutes the first-use moment and increases the chance of delaying a usable delegation loop.
- **Why rejected**: The user identified fast delegation as the core value to validate; other lifecycle capabilities remain necessary safeguards, not the lead product promise.

### Alternative 2: Offer automatic task decomposition

- **Description**: Let Kitten suggest or create child tasks from a parent request.
- **Pros**: Reduces initial user effort and can feel more autonomous.
- **Cons**: Makes delegation quality less predictable and expands the product into planning.
- **Why rejected**: V1 must validate intentional delegation of independent work, with the developer retaining explicit scope control.

### Alternative 3: Start with reusable role prompts

- **Description**: Require developers to select a child role or template before defining its work.
- **Pros**: May improve consistency for repeat workflows.
- **Cons**: Adds an extra decision to the launch path and can constrain task-specific intent.
- **Why rejected**: The desired V1 flow is direct task-and-outcome entry; reusable roles can follow after real usage reveals stable patterns.

## Consequences

### Positive

- Creates a clear, measurable first-use value proposition.
- Keeps the developer anchored in the parent task while parallel work begins.
- Establishes a visible Running state as an immediate trust signal.
- Preserves the essential lifecycle safeguards agreed in ADR-001.

### Negative

- Advanced supervision and reusable launch shortcuts are deferred.
- Users with highly repetitive delegation patterns may initially want more guidance.
- The product must communicate that a Running child is active work, not a completed result.

### Risks

- **Launch feels fast but work is poorly scoped**: prompt the developer for an explicit task and desired outcome, and position V1 for independent work.
- **Users miss child state after launch**: retain visible child identity and a clear Running indicator in normal workspace surfaces.
- **Cancellation surprises users**: explain the number and status of affected children before a group cancellation.

## Implementation Notes

This is a product-priority decision. The subsequent technical specification must preserve the user-facing guarantees without adding automatic planning or role-template selection to the MVP launch path.
