# ADR-001: Use a flat, host-owned delegation registry for V1

## Status

Accepted

## Date

2026-07-14

## Context

Kitten can create and keep several ACP sessions live, but they are peers: it has no host-owned relationship between a parent task and delegated child work. Provider-native subagents would make identity, lifecycle, visibility, interaction handling, and safety vary by adapter.

The selected product outcome is reliable delegation of parallel, independent work within one active coding task. V1 must preserve Kitten's existing provider-neutral boundaries: children remain normal focusable Kitten sessions, while the host owns their delegation lifecycle. The codebase already provides a dynamic session factory, generation fencing for stale runtime events, attention routing, and exact-once interaction teardown. It does not yet provide a parent-child graph or a general completion waiter.

## Decision

Build V1 as a bounded, in-memory, controller-owned registry for a single flat set of independent child sessions under one active parent. The registry is the authoritative owner of delegation identity, parent-child ownership, child lifecycle, terminal outcome, and aggregate wait/inspect state; the controller remains the authoritative owner of each normal Kitten session runtime.

V1 includes atomic child registration before launch, generation-fenced state publication, isolated child failure, exact-once terminal settlement, child steer/cancel/respond actions, and a minimal aggregate join contract: the parent can inspect children, wait for terminal outcomes, and find each child's stable terminal result reference. A parent close with live children requires clear confirmation; confirming it invokes controller-owned cascade cancellation. V1 never implicitly detaches a child.

## Alternatives Considered

### Alternative 1: Provider-native subagents

- **Description**: Let each ACP provider own its own child agents and expose their behavior through Kitten.
- **Pros**: Potentially less host-side lifecycle code; can use provider-specific features.
- **Cons**: Inconsistent lifecycle, safety, visibility, and interaction semantics; children may not be normal Kitten sessions.
- **Why rejected**: It violates Kitten's ACP anti-corruption boundary and prevents a reliable cross-provider product contract.

### Alternative 2: Generic workflow platform in V1

- **Description**: Start with nesting, dependency graphs, scheduling, retries, persistence, and provider-specific extensions.
- **Pros**: Broader future capability and a larger automation surface.
- **Cons**: Delays validation, multiplies lifecycle states, and risks building a scheduler before proving the core delegation loop.
- **Why rejected**: The chosen active-task use case needs trustworthy bounded delegation, not a general workflow engine.

### Alternative 3: Visibility-only child grouping

- **Description**: Create and label child sessions but require users to manually poll tabs and reconstruct completion.
- **Pros**: Smallest apparent implementation.
- **Cons**: Does not provide reliable delegation, obscures terminal outcomes, and makes the product experiment inconclusive.
- **Why rejected**: A minimal wait/inspect/result contract is the completion half of delegation, not optional orchestration scope.

## Consequences

### Positive

- Delivers provider-neutral, inspectable parallel delegation using normal workspace sessions.
- Establishes a durable host-owned control-plane boundary for later MCP or automation consumers.
- Reuses generation fencing, attention routing, and teardown patterns rather than creating a parallel runtime.
- Keeps the initial product loop small enough to validate with active coding tasks.

### Negative

- V1 cannot recover live ownership after restart or support long-lived detached children.
- A confirmation step adds friction when closing a parent with live children.
- Flat independent child sets limit some workflows that would benefit from dependencies or nesting.

### Risks

- **Split ownership between registry and session controller**: define a narrow lifecycle interface and keep session runtime ownership in the controller.
- **Stale or duplicate terminal events**: require generation checks and exactly-once settlement for every mutation and waiter.
- **Orphaned work on parent close**: offer only cancel-and-close or abort-close; never detach implicitly in V1.
- **Overbuilding orchestration**: capability-gate the vertical slice and defer scheduling, retries, and automation until user value is measured.

## Implementation Notes

This is a product-boundary decision, not a detailed technical design. A later specification must define the registry's state model, lifecycle transition matrix, and concrete controller/store/UI changes. Its acceptance coverage must include parallel start, one-child failure, cancellation races, stale publication, parent close, and cleanup.
