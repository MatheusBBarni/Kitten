# ADR-004: Derive delegation completion from store selectors in V1

## Status

Accepted

## Date

2026-07-14

## Context

The PRD requires a parent to see whether child work is active or settled and inspect terminal outcomes. Technical clarification selected UI-derived completion rather than a controller action that resolves a completion promise. The current UI already subscribes to narrow immutable store slices, and the controller's interaction coordinator is intentionally reserved for agent-blocking permission and clarification promises.

## Decision

Expose delegation group state through pure selectors over `AppState`. The UI derives active, needs-input, and settled group presentation from child snapshots and existing session status. V1 does not add a public controller `waitForChildren` promise, auto-inject a completion message into the parent transcript, or expose a new external orchestration API.

The controller still settles each child lifecycle transition exactly once. Selectors receive immutable terminal snapshots and remain valid if the parent is no longer focused.

## Alternatives Considered

### Alternative 1: Controller completion promise

- **Description**: Provide a `waitForChildren(parentId)` action that resolves when all children settle.
- **Pros**: Convenient for future automation consumers and imperative callers.
- **Cons**: Adds waiter ownership, cancellation, and cleanup semantics beyond the selected V1 UI need.
- **Why rejected**: UI polling through existing selectors is sufficient and keeps the controller free of a second promise coordinator.

### Alternative 2: Automatic parent transcript summary

- **Description**: Insert a completion summary into the parent conversation when children settle.
- **Pros**: Makes completion visible without opening workspace UI.
- **Cons**: Blends host lifecycle notices into agent transcripts and can create noisy or misleading context.
- **Why rejected**: The workspace is the existing visibility surface; terminal snapshots belong there in V1.

## Consequences

### Positive

- Reuses narrow selector subscriptions and immutable state updates.
- Avoids unresolved completion waiters during parent close, replacement, and shutdown.
- Keeps the parent transcript exclusively about the developer and agent conversation.

### Negative

- A future agent-facing API cannot await groups without a follow-up contract.
- UI consumers must handle state transitions rather than one imperative completion callback.

### Risks

- **Excessive re-renders**: expose memo-friendly narrow selectors keyed by parent id.
- **Ambiguous terminal display**: preserve an immutable terminal snapshot with a clear final status and timestamp.
- **Future API pressure**: add a waiter only when a real non-UI consumer exists, with its own lifecycle ADR.

## Implementation Notes

Add selectors for a parent’s ordered children, aggregate state, and terminal snapshots. Keep the existing interaction coordinator unchanged.
