# ADR-003: Boot Banner via a Transient Pre-Controller Render Root

## Status

Accepted

## Date

2026-07-09

## Context

The PRD requires the welcome banner to fill the currently-blank boot wait while the two agents complete their ACP handshake (seconds, longer on a cold first run that `npx`-downloads the adapters).
Today `src/index.ts` creates the renderer (`createRenderer()`) but does not call `renderCockpit()` until after `createSessionController` resolves, and the controller `await`s `Promise.all(plan.map(startSession))` before returning.
The React tree therefore mounts only after both agents are ready, and during the handshake the terminal shows nothing.
The store and controller do not exist yet during that window, so nothing store-driven can paint there.
The PRD also lists "in-app setup wizard / rewiring boot diagnostics" as a non-goal, so the existing readiness gate and stderr diagnostics must stay intact.

## Decision

Render a lightweight, self-contained boot banner directly into the renderer between `createRenderer()` and controller readiness in `src/index.ts`, then swap to the real cockpit once agents are up.

- The boot banner is a standalone renderable that does not depend on the store or controller; agent rows are static "connecting..." placeholders.
- The existing readiness gate is unchanged: if boot is blocked (no agent ready), the banner is torn down and the current stderr diagnostics + non-zero exit still fire.
- On success, the boot banner is replaced by `renderCockpit(renderer, controller, recorder)`.
- The banner's visual content (mascot, greeting, layout) is shared with the in-tree idle-screen banner so the two look identical and the handshake-to-ready transition is seamless.

## Alternatives Considered

### Alternative 1: Mount-early with a store "connecting" state

- **Description**: Refactor boot so the full cockpit mounts immediately with sessions seeded in a "connecting" state, and the controller updates readiness reactively as agents come up.
- **Pros**: Unifies rendering; the banner becomes the app's store-driven not-ready state; no transient second root.
- **Cons**: Touches the controller lifecycle and the readiness gate, converting the stderr-exit path into an in-TUI state - which edges into the "boot diagnostics" non-goal; larger, more delicate change.
- **Why rejected**: Disproportionate risk for a chrome reskin, and it collides with a stated non-goal.

### Alternative 2: Idle-screen only

- **Description**: Do not paint during the handshake; render the banner only as the idle/empty screen once mounted.
- **Pros**: Smallest change; no boot-path work.
- **Cons**: The pre-ready window stays blank, walking back the PRD's "banner as boot screen" decision.
- **Why rejected**: Fails the explicit PRD requirement to fill the boot wait.

## Consequences

### Positive

- Fills the blank handshake window with a branded first impression while leaving the readiness gate and diagnostics untouched.
- Contained to `src/index.ts` plus a shared banner renderable; no controller-lifecycle risk.

### Negative

- Two render passes into one renderer (boot banner, then cockpit) with a swap; the boot banner cannot show live per-agent detail (store not built yet).
- If readiness fails, the user briefly sees the banner and then the error moves to stderr - a small discontinuity.

### Risks

- **Flicker on swap** between the boot banner and the cockpit. Mitigation: reuse the same renderer and a matching layout so the transition is a content change, not a teardown/rebuild; verify no flash in tests.
- **Banner outlives a hung handshake**. Mitigation: the banner is purely presentational; the existing handshake timeout still governs progression to the gate.

## Implementation Notes

- Add a `renderBootBanner(renderer)` that mounts a minimal banner root and returns a disposer; call it right after `createRenderer()` and dispose it immediately before `renderCockpit()`.
- Keep the banner component pure and prop-driven so both the boot root and the idle screen render it from the same source.
