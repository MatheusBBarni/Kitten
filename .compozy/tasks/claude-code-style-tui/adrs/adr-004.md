# ADR-004: Extend the Existing Palette Instead of Building the Theme Registry

## Status

Accepted

## Date

2026-07-09

## Context

The reskin needs Kitten's own warm accent plus new chrome colors (banner tones, per-agent run-state colors, and context-usage thresholds).
`src/ui/theme.ts` today exposes a fixed `CockpitPalette` with two constants, `DARK_PALETTE` and `LIGHT_PALETTE`, resolved by `paletteFor(mode)` and `usePalette()`.
A separate accepted decision, the settings-modal ADR-005, specifies a palette registry keyed by preset id, a `ThemePreference` type, and a store `preferences` slice.
Codebase exploration confirmed that ADR-005 is **not implemented**: there is no `ThemePreference` type, no registry, no `id` on `CockpitPalette`, and no `preferences` slice on `AppState`.
The reskin must not block on, or partially build, that registry.

## Decision

Add the reskin's colors as new keys on the existing `CockpitPalette` interface and its two constants (`DARK_PALETTE`, `LIGHT_PALETTE`), and retune the `accent` value to Kitten's warm brand accent.

- New palette keys cover: banner/mascot tones, per-agent run-state colors (reusing the existing `status` tones where they already fit), and context-usage threshold colors (ok / warn / critical).
- All colors continue to be resolved through `usePalette()`; no component inlines hex.
- The `theme.test.tsx` invariants (distinct tones per mode, light/dark parity, no-truecolor legibility) are extended to the new keys.
- If the settings-modal registry (ADR-005) later lands, these keys migrate into whatever preset palettes it introduces; the single-source-through-`theme.ts` discipline makes that a mechanical move.

## Alternatives Considered

### Alternative 1: Build the ADR-005 palette registry now

- **Description**: Implement the registry, `ThemePreference`, and `preferences` slice as part of the reskin, then add the accent as a preset.
- **Pros**: The reskin's accent would arrive as a first-class user-selectable preset; avoids a later migration.
- **Cons**: Pulls a whole settings-modal feature into a chrome reskin; adds a store slice and config surface out of scope here; large scope creep.
- **Why rejected**: The registry is owned by the settings-modal effort; duplicating it here inflates scope and risks divergence.

### Alternative 2: Inline the accent where needed

- **Description**: Hard-code the new colors in the banner and status-bar components.
- **Pros**: Fastest.
- **Cons**: Violates the no-hard-coded-color rule and the theme test invariants; breaks light/dark parity.
- **Why rejected**: Direct conflict with an existing, enforced project constraint.

## Consequences

### Positive

- Minimal, on-convention change entirely within `theme.ts`; light/dark parity and no-truecolor legibility preserved by the existing test harness.
- No coupling to the unbuilt registry; the reskin ships independently.

### Negative

- When ADR-005 lands, the new keys must be migrated into preset palettes (a small, mechanical follow-up).

### Risks

- **Color-count growth** makes the palette unwieldy. Mitigation: group related keys (e.g. a nested `context` threshold object) and keep names semantic.

## Implementation Notes

- Extend `CockpitPalette` with grouped keys, e.g. a `context: { ok; warn; critical }` object and any banner-specific tones; reuse `status` for run-state where the mapping is exact.
- Update `theme.test.tsx` to assert the new keys are present and distinct in both modes.
