# ADR-001: Use a bounded, proof-gated same-session continuation

## Status

Accepted

## Date

2026-07-20

## Context

Issue #32 exposes a broken interruption path: cancelling a fresh, harness-bearing prompt can mark harness delivery indeterminate and leave an otherwise healthy ACP session at "Safe start unavailable." The developer's next draft remains in the composer and `/new` is the only route that can dispatch it.

The selected V1 serves iterative coding users. It must recover from every explicit Hard Stop without turning a cancellation into steering, concurrent prompting, durable retry, or an optimistic claim about provider behavior. The main uncertainty is transport semantics: a cancellation response, an open socket, or a late callback alone do not prove that the interrupted provider turn is safely settled for a subsequent prompt.

The existing controller owns prompt settlement, active lifecycle identity, and generation fencing. Existing steering is a useful sequencing reference but has incompatible semantics: it creates a steering follow-up rather than an ordinary next prompt. The established Kitten privacy boundary also requires pending draft text to remain absent from persistence, telemetry, diagnostics, and handoff material.

## Decision

Introduce a named, controller-owned interruption-recovery coordinator with one live-only continuation slot. After an explicit Hard Stop, it may queue exactly one continuation for the captured session generation. The coordinator dispatches that continuation as an ordinary next prompt only after the controller has affirmative, current-generation safe-settlement evidence from the active adapter. If that proof is unavailable or any cancellation, timeout, connection, replacement, or close outcome is indeterminate, Kitten retains the draft visibly and offers `/new`; it never dispatches into uncertainty.

The coordinator exposes only a narrow, content-free phase/recovery projection to the UI. The UI cannot infer or advance the lifecycle. A second Escape clears the queued continuation and restores it to the composer locally without another provider cancellation. The first harness-bearing interruption records `settled_interrupted` as a content-free terminal checkpoint, distinct from both delivered and failed; it permits continuation without duplicating the harness or claiming that the provider consumed it.

## Alternatives Considered

### Alternative 1: Limit recovery to a fresh first prompt

- **Description**: Repair only the harness-bearing first-turn failure and retain the current behavior for later interruptions.
- **Pros**: Smaller immediate change and limited initial test matrix.
- **Cons**: Creates an arbitrary distinction between first and later turns, leaving iterative users unable to rely on Hard Stop as a general recovery action.
- **Why rejected**: The selected user outcome is safe continuation after every explicit Hard Stop, not a narrow harness exception.

### Alternative 2: Allow a prompt after any generic terminal callback

- **Description**: Treat local completion or an open connection as sufficient to release the continuation.
- **Pros**: Broadest same-session coverage with less adapter-specific proof work.
- **Cons**: A stale, duplicate, or semantically weak callback can send withdrawn text, overlap provider work, or make harness consumption appear known when it is not.
- **Why rejected**: It converts ambiguous provider state into a user-visible delivery guarantee and violates the lossless, fail-closed recovery contract.

### Alternative 3: Build durable interruption/retry management

- **Description**: Persist queued drafts and add retries, configurable queues, or a general interruption platform.
- **Pros**: Could support richer recovery and orchestration later.
- **Cons**: Expands privacy, persistence, lifecycle, and operational scope before the narrow recovery behavior is validated.
- **Why rejected**: A single-slot, live-only coordinator is the smallest credible learning loop and preserves the existing privacy boundary.

## Consequences

### Positive

- Restores an ordinary same-session continuation where the controller can prove it is safe.
- Keeps ownership with the existing controller and generation-fenced prompt lifecycle rather than coupling interruption behavior to steering.
- Makes every ambiguous path lossless: the user keeps a visible draft and has an explicit `/new` recovery route.
- Preserves privacy by construction: no pending continuation content enters durable or cross-session surfaces.

### Negative

- Same-session continuation coverage is intentionally conditional on adapter-safe settlement evidence; unsupported or uncertain adapters fall back to `/new`.
- The coordinator adds a safety-critical state machine and adversarial race-test burden.
- Product measurement can use only opt-in, content-free outcomes, limiting diagnostic detail.

### Risks

- **Stale or duplicate completion dispatches withdrawn text** — fence every asynchronous transition by session, generation, and interrupted prompt identity; atomically remove the slot before ordinary dispatch.
- **Provider cancellation semantics are weaker than Kitten observes** — require affirmative adapter-safe settlement evidence; otherwise retain the draft and require `/new`.
- **Queued draft leaks through an adjacent surface** — keep draft content outside persisted session state and assert negative persistence, telemetry, diagnostics, and handoff cases with sentinels.
- **Escape changes approval or clarification ownership** — preserve the current interaction precedence and exclude those states from Hard Stop cancellation.

## Implementation Notes

- Reuse controller-owned settlement and generation authority, but do not reuse steering transport or its prefixed follow-up semantics.
- Define the closed `settled_interrupted` harness state alongside other persistence/schema projections; it records only lifecycle metadata.
- Prove the behavior with delayed, duplicated, missing, stale-generation, close, replacement, timeout, and second-Escape-at-dispatch-boundary tests before claiming provider coverage.
