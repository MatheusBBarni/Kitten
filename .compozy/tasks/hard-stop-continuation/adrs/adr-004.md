# ADR-004: Gate continuation on attested settlement and persist only closed metadata

## Status

Accepted

## Date

2026-07-20

## Context

The controller's local prompt settlement and an open connection do not by themselves prove that an interrupted provider is ready for a new ordinary turn. Sending on that inference can replay withdrawn content, overlap provider work, or misrepresent first-harness consumption. The product must fail closed where an adapter cannot provide affirmative safety evidence.

At the same time, a confirmed first-harness interruption needs the durable, content-free `settled_interrupted` checkpoint so restore logic can distinguish it from both delivered and indeterminate failure. The queued continuation itself remains private live content and must never survive a restart or enter telemetry, diagnostics, or handoff material.

## Decision

Introduce an explicit, provider-attested safe-settlement capability for Hard Stop continuation. The controller may release a queued continuation only when the current provider profile attests to the required cancellation-and-terminal-settlement semantics and the matching current-generation lifecycle settles. All absent, stale, failed, timed-out, replaced, closed, or disconnected cases recover the draft and require `/new`.

Extend the closed harness-delivery checkpoint vocabulary with `settled_interrupted`. Persist and restore only its version, generation, and state through every existing checkpoint schema and projection. Do not persist continuation blocks, request identifiers, phase, raw errors, or recovery text. Add only allowlisted opt-in telemetry outcomes and bounded timing data; no event may contain continuation text, route/capability values, raw errors, or draft identifiers.

## Alternatives Considered

### Alternative 1: Treat generic prompt settlement as safe for every adapter

- **Description**: Release a continuation whenever the local prompt promise settles.
- **Pros**: Broadest immediate feature coverage and less provider-specific configuration.
- **Cons**: Confuses local observation with provider quiescence and risks duplicate or unsafe dispatch.
- **Why rejected**: The feature's value depends on trustworthy recovery, so ambiguous adapters must fail closed.

### Alternative 2: Persist the queued continuation for cross-restart recovery

- **Description**: Store draft content and replay or restore it after restart.
- **Pros**: Richer recovery continuity.
- **Cons**: Changes Kitten's privacy and persistence contract and creates delivery ambiguity across sessions.
- **Why rejected**: V1 needs only live-run recovery; durable content requires a separate product and security decision.

### Alternative 3: Disable all continuation until every provider is attested

- **Description**: Ship only manual draft restoration until complete provider parity exists.
- **Pros**: Uniform behavior and no partial rollout.
- **Cons**: Delays safe value for providers that can already prove the required lifecycle semantics.
- **Why rejected**: Attested profiles and an explicit fallback make incremental coverage honest and safe.

## Consequences

### Positive

- Makes same-session dispatch conditional on evidence rather than optimism.
- Preserves truthful restore behavior for a confirmed first-harness interruption.
- Keeps all continuation content structurally outside durable and cross-session surfaces.

### Negative

- Initial same-session coverage may differ by provider.
- Checkpoint schema, writer, and restore projections must change together.
- The release gate needs a broader adversarial test matrix than an ordinary prompt feature.

### Risks

- **Capability drifts from adapter behavior** — require exact attestation tests and fail closed for unknown profiles.
- **Checkpoint schema diverges across persistence paths** — validate strict schema, writer, and restore round trips together.
- **Observability adds a content-bearing field later** — define a closed event union and test sentinel content against every output surface.

## Implementation Notes

- Keep adapter capability declaration and ACP-specific evidence inside `src/agent/`; pass only a protocol-free closed verdict to app/controller layers.
- Update `src/app/harnessDelivery.ts`, `src/store/appStore.ts`, `src/persistence/runRecord.ts`, and `src/persistence/runWriter.ts` in one compatibility change.
- Require layered core, controller, UI, persistence, telemetry, handoff, and attested-adapter integration tests before enabling a profile.
