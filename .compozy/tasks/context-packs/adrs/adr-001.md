# ADR-001: Plan the full Context Packs contract with evidence-gated vertical delivery

## Status

Accepted

## Date

2026-07-16

## Context

Issue #24 defines Context Packs as a session-owned, review-first artifact: a mutable Draft Context Pack may be curated by one verified `explore-v2` child, then materialized, redacted, reviewed, sealed, persisted, and explicitly sent to an eligible recipient. The product contract includes Send Here, Start Child, handoff attachment, and Markdown export, but it explicitly preserves the report-only `explore-v1` policy.

The current repository has the required ownership principles but not the required primitives. In particular, there is no Context Pack materializer, digest/slice model, recipient profile, or separately attested `explore-v2` profile; the present binary child MCP mode also exposes `agent_run` alongside `ask_user`. Shipping any capability as available before its authority, provenance, freshness, accounting, and confirmation boundaries are enforced would weaken the product's core trust claim.

## Decision

The Context Packs PRD and TechSpec will preserve the complete Issue #24 product contract as the V1 destination. Its task graph will deliver that contract through dependency-ordered vertical slices, and each operational capability remains unavailable until its exact supporting evidence is implemented and verified.

The immutable sealed artifact, deterministic review materialization, false-negative-biased redaction, source freshness, budget enforcement, explicit confirmation, and content-free telemetry are non-negotiable gates. `explore-v1` remains report-only and production-closed; Context Build requires a separately certified `explore-v2` capability whose allowlisted bridge is bound to the parent session and launch generation. Missing or stale material, source identity, capability, accounting, or recipient-fit evidence fails closed. A task may define the data model or UI for a later destination, but it must not expose that destination as usable until its gate is complete.

## Alternatives Considered

### Alternative 1: Ship every destination as one undifferentiated implementation slice

- **Description**: Build curation, sealing, persistence, recipient fit, Send Here, fresh child start, handoff, and export concurrently.
- **Pros**: Delivers the visible end-to-end workflow in one release cycle.
- **Cons**: Multiplies unproven authority, lifecycle, persistence, and recipient paths around missing primitives.
- **Why rejected**: It would make capability and fit checks optimistic placeholders, producing a broad workflow before the exact-review trust boundary is proven.

### Alternative 2: Replace the full contract with an ephemeral Send Here-only product

- **Description**: Limit the product permanently to a session-local build, review, and current-session send.
- **Pros**: Smallest initial operational surface and fastest validation loop.
- **Cons**: Loses the issue's explicit portable sealed artifact, persistence, handoff, fresh-child, and export outcomes.
- **Why rejected**: The narrow path is a useful delivery slice but not the selected product destination; it would turn Context Packs into a less differentiated temporary prompt helper.

### Alternative 3: Retrofit Context Pack authority into `explore-v1`

- **Description**: Extend the existing report-only Explore child with curation and delivery privileges.
- **Pros**: Reuses an established role name and launch path.
- **Cons**: Breaks the accepted `explore-v1` safety and capability contract; conflates report generation with artifact mutation.
- **Why rejected**: Issue #24 requires a separately certified `explore-v2` profile and fail-closed attestation for this authority.

## Consequences

### Positive

- Preserves the complete user outcome while preventing unverified paths from becoming product promises.
- Keeps the core/store/controller and capability boundaries testable and independently reviewable.
- Allows the task graph to validate durable artifact guarantees before extending them to more recipients.

### Negative

- The PRD and TechSpec must be more explicit about dependencies, evidence, and disabled states.
- Some visible actions may remain unavailable until a later dependency is verified.
- The task graph is larger than a simple single-destination feature and needs strict sequencing.

### Risks

- **Capability escape through the shared child MCP mode** — create an `explore-v2`-specific registrar and enforce authorization below tool advertisement.
- **Stale or oversized source material** — require bounded materialization, identities/digests, deterministic assembly, and explicit refresh/reselection.
- **False confidence from provider-neutral estimates** — distinguish Pack Estimate from recipient fit and block sends without current certified evidence.
- **Persistence of sensitive or live runtime state** — persist only a metadata-only draft manifest and the exact redacted sealed payload; exclude raw source, child ownership, attestations, reservations, and provider errors.

## Implementation Notes

- Split tasks by core artifact/revisions, store and persistence, scoped bridge and source artifacts, review and sealing, recipient profiles and fit, UI, and handoff/export composition.
- Preserve the existing confirmation-only handoff route; an attached pack remains immutable and must participate in one combined review.
- Treat every selected repository item as untrusted data, not instruction authority.
