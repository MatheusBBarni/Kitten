# ADR-005: Gate every Context Pack consumption on a shared fail-closed Recipient Fit decision

## Status

Accepted

## Date

2026-07-16

## Context

A Context Pack's displayed Pack Estimate is provider-neutral and can guide curation, but it does not prove that a particular existing session or fresh child has capacity for the exact sealed payload. Current session usage reports only live used and total context values; no existing recipient profile or certified payload counter proves a safe future send. A partial send, silent trim, or operator override would break the exact-review promise.

## Decision

Define a closed, protocol-free Recipient Fit decision and require one shared controller gate immediately before Send Here, Start Child, and handoff composition. Existing sessions require current live headroom plus a certified provider/model count or conservative upper bound for the exact sealed payload. Fresh children require a current versioned Recipient Profile tied to the exact recipe/model, usable fresh-session capacity, counter version, and reserve.

Missing, stale, invalid, or insufficient evidence yields `unavailable` or `insufficient` and blocks the action. The product preserves the sealed payload unchanged, offers no manual override, and never trims, substitutes, or partially sends it. Export remains an explicit operator action over the exact sealed payload and does not claim recipient fit.

## Alternatives Considered

### Alternative 1: Treat Pack Estimate as delivery authorization

- **Description**: Allow delivery whenever the estimate is below the selected Pack Budget.
- **Pros**: Simple and provider-neutral.
- **Cons**: A budget is not a recipient tokenizer or capacity guarantee.
- **Why rejected**: It would present a best-effort estimate as proof and could send a reviewed pack a recipient cannot accept.

### Alternative 2: Trim or partially send oversized packs

- **Description**: Silently remove material until the recipient accepts the request.
- **Pros**: Maximizes delivery success.
- **Cons**: Changes the reviewed artifact and hides omitted context from the operator.
- **Why rejected**: It violates immutability and exact-review custody.

### Alternative 3: Permit a manual operator override

- **Description**: Allow an operator to force delivery after an unavailable-fit warning.
- **Pros**: Provides an escape hatch for advanced users.
- **Cons**: Converts a safety gate into a warning and makes outcomes unpredictable.
- **Why rejected**: The selected policy is fail closed with no unsafe workaround.

## Consequences

### Positive

- Keeps curation guidance distinct from delivery authorization.
- Gives all recipient actions one consistent eligibility rule and blocked-state vocabulary.
- Preserves the exact sealed bytes throughout review, persistence, and consumption.

### Negative

- Some destinations remain unavailable until counting evidence and profiles exist.
- Provider support expands only with maintained recipient evidence.
- The controller needs to re-evaluate fit at each action rather than caching a prior green result.

### Risks

- **Stale profile evidence** — version profiles and invalidate them when recipe/model/counter/reserve changes.
- **Headroom changes between review and send** — evaluate immediately before the explicit confirmation path dispatches.
- **Inconsistent gates across destinations** — route all consumption through one decision service and integration-test every path.

## Implementation Notes

- Keep `RecipientProfile`, `RecipientFit`, and exact payload count inputs protocol-free in the core; the controller supplies live usage and profile evidence.
- Add no provider fallback, generic tokenizer dependency, or warning-only flag in V1.
- Use the same fit decision before a handoff preview confirms its combined payload.
