# ADR-001: Deliver a finite, accessibility-gated 18-preset catalog atomically

## Status

Accepted

## Date

2026-07-20

## Context

Issue #31 proposes expanding Kitten from Auto, Light, Dark, and two Catppuccin presets to a source-attributed 18-preset catalog. The current preference IDs, configuration validation, palette registry, and Settings rows duplicate their small preset lists. This makes a staged expansion prone to drift across persistence, restoration, telemetry, and presentation.

The product goal is accessible readability for all terminal developers, not an unbounded theme marketplace. Market research shows that terminal applications already offer large or importable scheme collections; Kitten's differentiated value is a finite catalog whose provenance, stable identity, keyboard access, and rendered readability are explicit and verified. The user selected the complete 18-preset catalog as a strategic V1 and prioritized accessible readability.

The council found that the atomic release has a large correlated failure surface: attribution, palette fidelity, xterm-256 contrast, aliasing, persistence, telemetry, and short-terminal navigation must all work together. A smaller Auto/Light/Dark contrast improvement remains a valid fallback if the full evidence bar cannot be met.

## Decision

Ship the agreed 18 source-verified presets as one finite, built-in catalog. The catalog is the single authoritative product boundary for durable preset identity, family and display metadata, source and license/attribution records, presentation order, and compatibility aliases. Auto, Light, and Dark remain first; catalog families and variants sort alphabetically; headings are non-selectable; and selecting a preset applies it immediately.

The atomic catalog is releaseable only when every included preset has public provenance and compatible attribution, all rendered foreground roles meet a 4.5:1 contrast floor in both truecolor and xterm-256 rendering, Settings remains keyboard-usable in a short terminal, aliases canonicalize live state without a startup write, and telemetry stays local, opt-in, content-free, and limited to fixed IDs. If any of those gates cannot be proven for the full catalog, defer the catalog and prioritize the narrower baseline readability alternative rather than weakening the guarantees or releasing a partial family set.

V1 excludes custom or imported themes, a theme marketplace, runtime palette downloads, an in-app credits view, and Gruvbox Light variants.

## Alternatives Considered

### Alternative 1: Stage a smaller preset subset

- **Description**: Add a few popular themes first, then extend families over later releases.
- **Pros**: Lower immediate palette and UI-test surface; faster feedback.
- **Cons**: Breaks the family-level promise, preserves duplicate knowledge, and creates temporary contracts and migration churn.
- **Why rejected**: The durable catalog boundary and complete source-verified family experience need to land together; otherwise the product sacrifices its differentiator without avoiding the eventual integration work.

### Alternative 2: Improve Auto, Light, and Dark contrast only

- **Description**: Focus first on contrast controls and selection safeguards in the existing built-in themes.
- **Pros**: Directly targets accessibility with a smaller release and fewer compatibility concerns.
- **Cons**: Does not provide familiar palette choice, provenance, or the durable catalog contract requested by issue #31.
- **Why rejected**: Retained as the explicit fallback if the full catalog cannot satisfy its evidence gates, but it does not fulfill the selected V1 outcome.

### Alternative 3: Build an extensible marketplace or import system

- **Description**: Support user-supplied palettes, runtime downloads, or a broad theme gallery.
- **Pros**: Maximum choice and a potentially larger customization surface.
- **Cons**: Introduces unbounded provenance, accessibility, compatibility, support, and security obligations while diluting the curated product promise.
- **Why rejected**: V1 differentiates through verified curation, not catalog size or user-generated theming infrastructure.

## Consequences

### Positive

- Every theme consumer can derive from one stable catalog instead of maintaining parallel preset lists.
- Terminal developers gain recognizable, source-attributed palette choice with a measurable readability guarantee.
- Stable IDs and explicit aliases preserve user preferences through future catalog evolution.
- The finite scope remains auditable and avoids marketplace or import-system obligations.

### Negative

- Atomic scope increases the pre-release verification matrix and may delay delivery if any one preset fails a mandatory gate.
- The catalog is a strategic UX investment whose sustained value still needs privacy-safe post-release evidence.
- Source fidelity sometimes requires narrowly documented foreground adjustments for readability.

### Risks

- **Incorrect source or license data** — require a direct public source and compatible attribution record for every preset before inclusion.
- **Unreadable xterm approximation or selection surface** — enforce role-by-role 4.5:1 checks in truecolor and xterm-256, plus distinct non-color state surfaces.
- **Persistence or telemetry drift** — derive valid IDs and aliases from the catalog and retain end-to-end restoration, no-startup-write, and fixed-enum telemetry tests.
- **Short-terminal picker failure** — test grouped keyboard navigation at the minimum supported terminal dimensions; headings must never be selectable.

## Implementation Notes

This decision defines product scope and acceptance boundaries, not the implementation design. The follow-on PRD and TechSpec must preserve the existing Settings modal ownership, immediate-apply behavior, strict config validation, local opt-in telemetry policy, and the protocol-free core boundary described in ADR 0019.
