# ADR-004: Project catalog metadata into typed scrollable Settings rows

## Status

Accepted

## Date

2026-07-20

## Context

`SettingsView` currently derives a flat selectable list from UI palette keys and clamps movement by array index. The catalog requires non-selectable family headings, selectable variants, alphabetical grouping, immediate selection, active-family context, and reachability in short terminals. A rendering-only heading layer would leave navigation and visibility separate from what users see.

Existing OpenTUI views already use a bounded `scrollbox` with stable child IDs and `scrollChildIntoView` to keep arrow-selected rows visible. Settings must preserve its current modal priority, keyboard ownership, immediate apply semantics, and Reset-to-Auto behavior.

## Decision

Build one typed UI row projection from the core catalog: built-in preference rows, non-selectable family-heading rows, and selectable canonical-preset rows. Use that same projection for rendering, visible-family context, keyboard movement, and active-row identity. Navigation steps only across selectable rows; headings never receive the active marker or a persisted preference.

Render the grouped rows in a bounded vertical OpenTUI `scrollbox` inside the existing Settings dialog. Give selectable rows stable IDs and call `scrollChildIntoView` when the active preference changes. Preserve horizontal-scroll suppression and the established modal listener behavior. Auto, Light, and Dark remain the first selectable rows; Reset sets Auto; arrow movement continues to apply the highlighted selectable row immediately.

## Alternatives Considered

### Alternative 1: Keep a flat selectable array and render headings separately

- **Description**: Retain the current `ThemePreference[]` navigation list while inserting decorative family labels during rendering.
- **Pros**: Minimal change to current movement logic.
- **Cons**: The navigation model can drift from visible rows and cannot reliably preserve family context or scroll selected children into view.
- **Why rejected**: The product requires headings to be a first-class non-selectable part of the list model.

### Alternative 2: Maintain Settings-specific catalog data

- **Description**: Hand-maintain families, labels, and navigation order in `SettingsView` separate from the core catalog.
- **Pros**: Localizes the UI work.
- **Cons**: Reintroduces the duplicated knowledge the core catalog is intended to remove.
- **Why rejected**: The UI should project authoritative metadata, not recreate it.

### Alternative 3: Add picker state to the application store

- **Description**: Persist selected row, scroll position, and family context in global store state.
- **Pros**: Could restore picker position across re-renders.
- **Cons**: Adds mutable application state for a transient modal interaction without a product requirement.
- **Why rejected**: The current component-local selection derived from the active preference is sufficient; renderer-owned scrolling already solves visibility.

## Consequences

### Positive

- What users navigate is exactly what they see.
- Short-terminal behavior reuses established OpenTUI scrolling patterns and remains testable by stable row IDs.
- The Settings modal keeps its immediate, keyboard-only interaction and existing precedence.

### Negative

- The view gains a small projection layer and must separately distinguish active selection from visible headings.
- Family context needs explicit rendering behavior when the active row scrolls away from its heading.

### Risks

- **A heading is accidentally selectable** — make row kind drive navigation and persistence eligibility.
- **The selected variant scrolls out of view** — call `scrollChildIntoView` on active-row changes and verify a constrained viewport.
- **Horizontal scrollbar layout steals a row** — reuse the existing hidden-horizontal-scrollbar pattern from compact lists.

## Implementation Notes

Keep the row projection in the existing UI package, adjacent to `SettingsView`, rather than adding a new package or store slice. Reuse the existing Settings keymap; do not add confirmation, cancellation, or new global bindings.
