# ADR-004: Trustworthy Shell State via OSC 133 + OSC 7 Shell Integration

## Status

Accepted

## Date

2026-07-09

## Context

The hand-off snapshot's entire value is that it carries *trustworthy* working state to the receiving agent: the real working directory, real command boundaries, and real exit codes.
The council concluded that a fake shell "ships a lie" on exactly these fields, which would poison the moat test.
A raw PTY byte stream does not announce where a command starts or ends, what its exit code was, or what the shell's cwd is after a `cd` (the child's cwd changes, but Kitten's own process cwd does not).

## Decision

Derive command boundaries, exit codes, and cwd from **shell-integration escape sequences**, the same mechanism VS Code, iTerm2, WezTerm, and Kitty use.

1. **Inject integration hooks** into the spawned shell so it emits:
   - **OSC 133** markers: prompt start (`A`), command start (`B`), command executed, and command finished with exit code (`D;<code>`).
   - **OSC 7** to report the current working directory on each prompt.
   Injection is done by prepending shell-specific setup to the interactive shell (a sourced snippet for bash/zsh via `--rcfile`/`ZDOTDIR` or an equivalent env hook), never by editing the user's own dotfiles.
2. **Parse the sequences** at the emulator boundary: `@xterm/headless` surfaces OSC handlers, so the `ShellRuntime` registers handlers that emit `ShellEvent`s (`command_started`, `command_finished` with exit code, `cwd_changed`).
3. **The `shellReducer`** folds those events into the semantic `shell` slice: it closes the current command record with its exit code and updates `cwd`.
4. **Graceful degradation:** a shell without the integration hook (an unsupported shell, or a user who disabled it) still renders and runs; it simply falls back to raw scrollback with no per-command boundaries and no reliable cwd, and the snapshot advertises only what it can prove.

## Alternatives Considered

### Alternative 1: Heuristic prompt parsing

- **Description**: Infer command boundaries and cwd by pattern-matching the prompt and output.
- **Pros**: No shell modification.
- **Cons**: Breaks on custom prompts/themes; cannot know exit codes; can report the wrong cwd. This is the "ships a lie" failure mode.
- **Why rejected**: Unreliable on precisely the fields the moat depends on.

### Alternative 2: Raw scrollback only, no structure

- **Description**: Snapshot the last N raw lines plus Kitten's process cwd.
- **Pros**: Simplest.
- **Cons**: No exit codes, no boundaries, wrong cwd after `cd`.
- **Why rejected**: Too weak to carry trustworthy context; retained only as the degraded fallback.

## Consequences

### Positive

- Real exit codes, command boundaries, and cwd, so the snapshot carries fact rather than guess.
- Standard, well-understood mechanism with prior art across major terminals.
- Degrades safely instead of failing when integration is unavailable.

### Negative

- Per-shell integration snippets to author and maintain (bash and zsh for V1).
- The snapshot's quality depends on the integration being active, so the UI must be honest when it is not.

### Risks

- **Integration hook conflicts** with a user's exotic prompt or existing OSC 133 setup. *Mitigation:* additive hooks that do not replace the user's prompt; detect an existing integration and skip.
- **Non-bash/zsh shells** (fish, nushell) unsupported at launch. *Mitigation:* documented; degrade to raw scrollback; add shells later.

## Implementation Notes

- Shell-integration snippets live as static assets shipped with Kitten; the `ShellRuntime` selects one by the configured/detected shell and injects it at spawn.
- The command ring stores redaction-agnostic raw output; redaction happens only at hand-off assembly time (ADR-001, reusing `secretRedactor`).
