# ADR-004: Sequence defaults from agent-confirmed model state

## Status

Accepted

## Date

2026-07-13

## Context

Model and reasoning-effort choices are advertised by a live provider, and selecting a model can change the available effort values. Kitten already updates session options only from the agent's confirmed response. The feature must report full, partial, and unavailable default applications in both `/model` and the status strip without inventing or silently substituting a value.

## Decision

The controller applies a configured model first. Only after the agent confirms the refreshed option set does it resolve and apply the configured effort. If the model is unavailable or rejected, it leaves the verified session configuration unchanged and records an unavailable outcome. If the model is confirmed but the effort is unavailable or rejected, it retains the agent-confirmed post-model state and records a partial outcome. It never rolls back optimistically or selects a substitute effort. The resulting outcome is stored as protocol-free, session-scoped state through the reducer so the picker and status strip render the same result.

## Alternatives Considered

### Alternative 1: Use cached pre-model effort options

- **Description**: Resolve both values from the option set observed before changing the model.
- **Pros**: Fewer reads and a simpler sequence.
- **Cons**: Can apply an effort that is no longer available for the confirmed model.
- **Why rejected**: Model changes may refresh capability choices; only the refreshed confirmed response is authoritative.

### Alternative 2: Roll back the model when effort fails

- **Description**: Treat model and effort as an all-or-nothing pair.
- **Pros**: Avoids a partial user-visible result.
- **Cons**: Requires another best-effort agent mutation and may itself fail or produce another changed option set.
- **Why rejected**: It would make the confirmed state less trustworthy and contradicts the chosen product behavior.

### Alternative 3: Keep the outcome only in overlay-local state

- **Description**: Let `/model` render a temporary message without changing session state.
- **Pros**: Avoids extending core state.
- **Cons**: The status strip cannot render the result consistently, and closing the overlay loses the explanation.
- **Why rejected**: The user selected a session-scoped result shared by existing feedback surfaces.

## Consequences

### Positive

- Keeps displayed configuration aligned with the provider-confirmed state.
- Makes partial application explicit instead of hiding a changed capability set.
- Lets UI surfaces share one reducer-owned result without ACP awareness.

### Negative

- Default application is deliberately sequential rather than transactional.
- A partial result may show an effort different from the value that existed before the model change if the provider confirms such a change.

### Risks

- **Concurrent tab input**: retain the existing modal behavior and serialize the controller action per selected session.
- **Misleading copy**: derive display values from confirmed options only and make partial/unavailable labels explicit.
- **Unsupported option categories**: continue using the existing model and reasoning-effort allowlist.

## Implementation Notes

Add a `DefaultApplyResult | null` field to `SessionState` and a `default_apply_result` domain event. The result describes the terminal attempt only; it must never replace or patch `configOptions`. A narrow selector supplies it to `ModelSelect` and `StatusStrip`.
