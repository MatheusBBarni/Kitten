# ADR-003: Keep provider defaults declarative and controller-owned

## Status

Accepted

## Date

2026-07-13

## Context

Kitten already separates strict personal configuration, controller actions, protocol-free state, and presentation-only UI. The feature needs a per-provider default model and reasoning effort that can change through external configuration edits, but must not change a live session until the developer intentionally selects that provider through `/model`.

## Decision

Add an optional top-level, provider-keyed defaults map to the strict user configuration. The controller owns the in-memory snapshot of those defaults and exposes one action that applies a selected session's provider default. The configuration watcher updates only this controller snapshot. `ModelSelect` invokes the action only after its existing explicit tab/session selection; it does not sequence individual option changes itself. The feature does not call the configuration writer.

## Alternatives Considered

### Alternative 1: Add defaults to provider spawn recipes

- **Description**: Mix model and effort preferences with the provider command, arguments, and environment.
- **Pros**: Keeps all provider-related values together.
- **Cons**: Conflates a user preference with process-launch configuration and makes validation and future changes less clear.
- **Why rejected**: A separate top-level map preserves the existing spawn-recipe boundary and the user-preference lifecycle.

### Alternative 2: Let `ModelSelect` orchestrate the default sequence

- **Description**: Have the UI call individual model and effort actions after switching tabs.
- **Pros**: Requires fewer controller changes.
- **Cons**: Puts policy, error handling, and ordered live-agent interaction in a presentation component.
- **Why rejected**: It violates Kitten's UI-to-agent action boundary and makes the behavior hard to reuse or test in isolation.

### Alternative 3: Apply defaults while loading or reloading configuration

- **Description**: Immediately mutate matching live sessions when a file is loaded or changes.
- **Pros**: Makes the running app reflect file contents sooner.
- **Cons**: Causes hidden runtime changes without a deliberate user selection.
- **Why rejected**: The PRD requires new defaults to take effect only on a later intentional `/model` selection.

## Consequences

### Positive

- Keeps configuration user-authored and strictly validated.
- Preserves controller ownership of agent interactions and UI presentation boundaries.
- Makes reload behavior predictable and side-effect-free for live sessions.

### Negative

- Adds a small controller-owned preference snapshot in addition to the loaded configuration object.
- Requires an explicit bridge from the configuration watcher to the controller.

### Risks

- **Stale snapshot**: replace it only after the watcher delivers a fully valid resolved configuration.
- **Accidental UI orchestration**: expose one controller action rather than reusing raw per-option calls from the view.
- **Configuration writes**: keep this feature isolated from the existing theme writer.

## Implementation Notes

Implement the preferences as `providerDefaults: Partial<Record<ProviderKind, ProviderModelDefault>>` with optional non-empty model and effort values. The controller action receives a session identifier, resolves its provider, and uses its internal snapshot. No preference belongs in the app store because the store owns live session state, not configuration source state.
