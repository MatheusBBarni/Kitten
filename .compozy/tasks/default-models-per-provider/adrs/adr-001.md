# ADR-001: Apply per-provider defaults on intentional model-session selection

## Status

Accepted

## Date

2026-07-13

## Context

Kitten users can choose a model and reasoning effort for live provider-backed sessions through `/model`, but that choice must currently be repeated when moving to another provider session. The proposed personal `config.json` preference supplies an optional model-and-effort default for each provider. Runtime options are agent-advertised, opaque, and may change after a model switch; Kitten must continue to show only agent-confirmed state.

## Decision

When a user intentionally tabs to a session of another provider inside `/model`, Kitten will attempt that provider's configured model and reasoning-effort default. It will apply the model first, use the agent-confirmed refreshed options to determine whether the saved effort remains valid, and display only the confirmed outcome. A missing, unavailable, or rejected default never becomes an optimistic value. Defaults are read from user-authored configuration only; manual changes remain session-local and configuration reloads do not mutate active sessions.

## Alternatives Considered

### Alternative 1: Seed defaults only when creating a new session

- **Description**: Apply defaults during session creation and leave `/model` tab changes side-effect-free.
- **Pros**: Avoids changing established session state during navigation.
- **Cons**: Does not deliver the requested automatic provider-switch experience.
- **Why rejected**: The user explicitly selected automatic application when changing provider/session in `/model`.

### Alternative 2: Apply model and effort independently without compatibility checks

- **Description**: Send both saved values whenever the user tabs to a provider session.
- **Pros**: Smallest apparent implementation.
- **Cons**: Can send stale or unsupported effort values and misrepresent the active configuration.
- **Why rejected**: It violates Kitten's agent-confirmed-state and fail-closed constraints.

## Consequences

### Positive

- Removes repetitive model-and-effort setup for personal multi-provider workflows.
- Keeps configuration ownership with the user and preserves honest runtime state.
- Limits automatic behavior to an explicit user action inside `/model`.

### Negative

- A model change can refresh effort capabilities, so applying a saved pair requires ordered confirmation.
- A failed effort may leave only the model confirmed; the UI must describe the actual result rather than claim pair-level success.

### Risks

- **Stale configuration values**: retain the last confirmed session state and surface the rejected/default-not-applied result.
- **Manual override surprise**: limit automatic application to intentional provider/session changes within `/model` and document the behavior.
- **Permission or unsafe options leaking into configuration**: accept model and reasoning-effort categories only.

## Implementation Notes

The later technical specification must define the compatibility resolver, ordered confirmation behavior, partial-result copy, and tests. It must preserve the existing `ControllerActions` seam and must not add any path that writes `config.json`.
