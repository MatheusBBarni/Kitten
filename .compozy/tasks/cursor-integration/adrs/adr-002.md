# ADR-002: Launch Cursor by Default as an Independently Available Third Session

## Status

Accepted

## Date

2026-07-13

## Context

The approved idea establishes Cursor as a first-class local alternative to Claude Code and Codex. PRD clarification confirmed the primary outcome: Cursor must be a dependable third live session. Users also chose a zero-configuration first launch that includes Cursor and exposes its own availability state.

Cursor availability depends on a user's local installation and authenticated account. Treating an unavailable Cursor session as a global launch failure would deprive users of the agents that are ready, while silently hiding it would make setup problems difficult to resolve.

## Decision

Kitten will include Cursor beside Claude Code and Codex on a zero-configuration launch. Cursor will be presented as an equal session whenever it is available. When it is unavailable, the cockpit will name the Cursor-specific issue and recovery action while leaving every ready sibling session usable.

The product will prioritize reliable session availability over handoff-adoption or growth metrics in its initial success decision. Cursor remains a local, developer-directed session; reviewed handoffs and explicit confirmation remain mandatory.

## Alternatives Considered

### Alternative 1: Require explicit Cursor opt-in

- **Description**: Keep the default launch to Claude Code and Codex, with Cursor enabled only by a user setting.
- **Pros**: Avoids showing an unavailable third session to users who do not use Cursor.
- **Cons**: Hides the new capability and creates a configuration hurdle for its target users.
- **Why rejected**: The requested product value is a first-class third alternative, not an advanced opt-in integration.

### Alternative 2: Block launch until Cursor is ready

- **Description**: Treat a missing or unauthenticated Cursor installation as a launch blocker.
- **Pros**: Ensures all visible sessions are available.
- **Cons**: Makes Cursor's local setup problem prevent users from working with their ready agents.
- **Why rejected**: It contradicts Kitten's independent-agent availability promise and the user's stated preference.

### Alternative 3: Hide Cursor when unavailable

- **Description**: Omit Cursor from the cockpit until it is ready.
- **Pros**: Keeps the interface minimal for users without Cursor.
- **Cons**: Obscures the integration, deprives users of recovery guidance, and makes availability unpredictable.
- **Why rejected**: Clear, actionable status is more trustworthy than silent omission.

## Consequences

### Positive

- Cursor users see its value immediately without an extra setup decision.
- Ready agents remain productive when Cursor needs local attention.
- Reliability has an explicit product priority and measurable launch outcome.

### Negative

- Users who do not have Cursor will initially see an unavailable session and recovery guidance.
- The release must keep its compatibility promise current as Cursor evolves.
- Three active sessions can increase a developer's local resource use.

### Risks

- **Confusing setup messaging**: use a concise agent-specific explanation and one clear next action.
- **External Cursor changes**: certify the supported local experience and state limitations honestly.
- **Default-session overload**: keep focus, switching, and status labeling consistent across all three sessions.
- **Over-trusting transfers**: retain preview, redaction, curation, and explicit confirmation before every handoff.

## Implementation Notes

- Treat Cursor as an equal user-visible session in the default launch experience.
- Preserve the current local-first, developer-controlled interaction model.
- Keep unverified Cursor-only behavior outside the initial release.
