# ADR-001: Ship Cursor as a Certified Local Third ACP Session

## Status

Accepted

## Date

2026-07-13

## Context

Kitten already runs Claude Code and Codex as long-lived ACP sessions and has evolved its session and hand-off model to support a fleet. Existing users who also use Cursor need a first-class local Cursor alternative without manually rebuilding context outside Kitten.

Cursor documents a native `agent acp` stdio server, so a third local session is viable. Its CLI and ACP surface remain an external, fast-moving dependency, and authentication or capability failures must not affect the existing providers. The user selected a minimal, clean V1 and explicitly declined a council debate.

## Decision

Add Cursor as a first-class, version-certified local ACP provider. It will run as a third persistent session beside Claude Code and Codex; it will participate in Kitten's existing session picker and explicit hand-off target picker. All Cursor-targeted hand-offs retain the mandatory redacted preview and explicit human confirmation.

The implementation must pin a verified Cursor launch recipe, use the existing per-agent readiness gate, and degrade Cursor independently when its binary, authentication, or ACP handshake is unavailable. Cursor-specific optional capabilities remain unavailable until an exact, credentialed contract test verifies them.

## Alternatives Considered

### Alternative 1: Generic custom ACP provider configuration

- **Description**: Let users supply an arbitrary command and arguments instead of defining Cursor as a provider.
- **Pros**: Small initial surface and nominal support for many ACP tools.
- **Cons**: Moves compatibility, versioning, setup guidance, and safety validation to every user.
- **Why rejected**: It does not deliver a dependable Cursor integration and weakens Kitten's known-good provider contract.

### Alternative 2: Generic provider platform or marketplace

- **Description**: Build an extensible provider registry before adding Cursor.
- **Pros**: Broad future-provider potential.
- **Cons**: Substantially wider scope and delays a requested, already feasible integration.
- **Why rejected**: The user chose a clean third-provider feature, not a platform initiative.

### Alternative 3: Cursor cloud/background-agent integration

- **Description**: Integrate Cursor's remote, asynchronous agent products.
- **Pros**: Adds long-running and parallel cloud workflows.
- **Cons**: Introduces remote lifecycle, repository, credential, and security concerns outside Kitten's local-session model.
- **Why rejected**: V1 is explicitly local and persistent; cloud agents require a separate product and safety decision.

## Consequences

### Positive

- Existing Cursor users gain a first-class third agent without abandoning the Kitten workflow.
- The established multi-session target picker, readiness gate, redaction, and confirmation safety contracts are reused.
- A certified launch recipe gives users clearer recovery guidance than an unbounded custom command.

### Negative

- Starting Cursor adds another local process and may increase launch time and resource use.
- Kitten must maintain a compatibility contract against Cursor CLI/ACP changes.
- The closed provider identity and a few provider-specific UI and telemetry branches must grow deliberately.

### Risks

- **Cursor version or ACP drift**: pin and credential-test the exact recipe; mark it not-ready rather than guess.
- **Authentication prompts or missing installation**: surface the existing per-agent readiness reason while leaving siblings usable.
- **Unverified optional behavior**: fail closed for structured clarification, restoration special cases, and other Cursor-specific extensions.
- **Unsafe transfer of work**: preserve redacted preview and explicit confirmation for every target, including Cursor.

## Implementation Notes

- Add the `cursor` provider identity, display metadata, configuration validation, and pinned native `agent acp` recipe.
- Verify real `initialize`, session, prompt, permission, and cancellation behavior before declaring the profile supported.
- Reuse the existing N-session hand-off target selection; do not add automatic routing or sending.
- Keep cloud/background agents, generic provider registration, and new Cursor-only capabilities outside this scope.
