# ADR-001: Capability-gated multi-language syntax highlighting

## Status

Accepted

## Date

2026-07-13

## Context

Kitten developers need to recognize code produced in streamed Markdown quickly. The requested V1 language set is JavaScript, TypeScript, Rust, Go, OCaml, ReScript, JSON, Bash, Python, Markdown, and diff, including common fence aliases. The existing shared Markdown renderer has theme-aware syntax styling, but the current OpenTUI parser bundle covers only a subset of this set. Compiled binaries add a load-bearing requirement: parser assets must be available before the shared Tree-sitter client starts.

The council agreed that language breadth is valuable only if it is trustworthy. Silent alias misses, binary-only failures, unreadable theme output, or fallback behavior that changes copyable source would undermine the product benefit.

## Decision

Pursue the requested V1 language set through one shared, declarative highlighting capability. A language is considered supported only when its aliases, parser availability, theme rendering, source-build behavior, compiled-binary behavior, and byte-preserving plaintext fallback have passed the same release gate. Unknown, unavailable, malformed, or failed fences must remain readable and copy-safe plaintext rather than receiving guessed or partial highlighting.

The implementation may prove the capability with representative languages first, but the V1 outcome remains the user-selected core set plus common formats. Per-surface handling and ad hoc regex highlighting are excluded.

## Alternatives Considered

### Alternative 1: Ship only the highest-frequency languages

- **Description**: Limit V1 to JavaScript, TypeScript, Python, Bash, and Go.
- **Pros**: Smaller parser package and faster initial validation.
- **Cons**: Misses the user-requested Rust, OCaml, ReScript, Markdown, JSON, and diff experience; weakens the cross-language product hypothesis.
- **Why rejected**: A capability-gated full manifest retains the bounded V1 value without promising unreliable support.

### Alternative 2: Add syntax handling separately to each UI surface

- **Description**: Patch transcript, hand-off preview, and other code surfaces independently.
- **Pros**: Local changes may appear faster.
- **Cons**: Behavior, aliases, streaming treatment, and fallbacks can drift.
- **Why rejected**: The existing shared Markdown leaf is the intentional ownership boundary.

### Alternative 3: Best-effort regex highlighting

- **Description**: Color common tokens without parser registration or binary verification.
- **Pros**: Small apparent implementation cost.
- **Cons**: Misleading output, fragile streaming behavior, and incorrect copying.
- **Why rejected**: It violates the trustworthy fallback contract and creates compounding maintenance debt.

## Consequences

### Positive

- One capability boundary keeps language aliases, availability, and rendering behavior consistent.
- The requested language set can expand without new Markdown-surface ownership.
- Release verification makes packaged binaries as trustworthy as source runs.

### Negative

- Parser packaging, startup ordering, and release tests increase V1 effort.
- ReScript needs an explicit grammar decision or remains deliberately plaintext.

### Risks

- **Grammar and binary assets drift** — verify the declared set in compiled artifacts before release.
- **Alias mistakes select the wrong parser** — maintain an explicit, tested alias map.
- **Fallback changes source semantics** — require byte-preserving copy tests for unknown and failed fences.
- **Theme contrast regresses** — validate the shared semantic token palette across supported terminal modes.

## Implementation Notes

The capability must remain in the shared Markdown/highlighting boundary. Preserve the current streaming behavior and avoid rewriting complete fenced source.
