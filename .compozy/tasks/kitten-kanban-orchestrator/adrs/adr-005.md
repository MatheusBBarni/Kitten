# ADR-005: Own queued follow-ups and Attention Blockers in the desktop attempt coordinator

## Status

Accepted

## Date

2026-07-21

## Context

The PRD requires a persistent composer, no silent interruption of a running agent, and answer-first Attention Blockers. Cockpit's current steering path cancels an active prompt before a later follow-up, so it cannot satisfy the desktop contract. Technical clarification selected explicit operator confirmation after the active turn settles rather than automatic dispatch.

## Decision

The desktop attempt coordinator owns a FIFO queue of ordinary follow-up drafts per live attempt. When the active turn reaches a normal terminal outcome, the next draft remains visible and requires explicit operator confirmation before it becomes the next prompt in the same attempt. The coordinator never uses ordinary composer input to cancel, fork, or duplicate a turn.

An attempt permits one active Attention Blocker. A scoped `ask_user` request persists the blocker, moves the card to `needs_attention`, locks stage movement, and produces a card-scoped notification. Its submitted, skipped, timed-out, or cancelled result is appended before the coordinator releases the blocked attempt. The required answer takes priority over ordinary queued submission.

## Alternatives Considered

### Alternative 1: Reuse Cockpit cancellation-based steering

- **Description**: Send active composer input through the existing steer-and-cancel flow.
- **Pros**: Reuses existing behavior.
- **Cons**: Violates the no-silent-interruption contract and changes the active turn unexpectedly.
- **Why rejected**: The desktop must provide an explicit, non-cancelling follow-up experience.

### Alternative 2: Automatically dispatch each queued draft

- **Description**: Prompt again immediately after the active turn settles.
- **Pros**: Fewer user actions.
- **Cons**: Can send stale intent after the user has reviewed the settled response.
- **Why rejected**: The operator selected confirmation after the turn as the safer V1 contract.

## Consequences

### Positive

- Keeps follow-up intent visible and reversible.
- Makes Attention Blocker priority, lifecycle, and audit evidence deterministic.

### Negative

- Adds one confirmation action before a queued follow-up is sent.

### Risks

- Users may expect automatic sending. Mitigate with explicit queued and ready-to-send states in the inspector.

## Implementation Notes

- Reuse the scoped `ask_user` protocol and capability model, but bind it to desktop attempt identity and durable events.
- Do not reuse Cockpit steering coordinator state or its cancellation semantics.
