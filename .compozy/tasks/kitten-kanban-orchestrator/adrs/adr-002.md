# ADR-002: Make Attention Blockers the V1 supervision priority

## Status

Accepted

## Date

2026-07-21

## Context

The Workflow Board's V1 value must be continuous, trustworthy supervision of coding-agent work. The approved product scope already requires every card to retain its conversation and attempt history, and to expose a composer at every stage. Clarification established that the most important MVP proof is that users do not miss an agent question that leaves work blocked.

Users need an unambiguous experience when an agent requires input: the run must stop, the affected card must stand out on the board, and the desktop must notify the user. The user must answer before the attempt continues. This attention flow must not erase the persistent composer or hide the card's historical context.

## Decision

V1 will treat an Attention Blocker as the primary active-supervision event. An outstanding blocker changes the card to `needs_attention`, prevents execution from continuing, highlights the card, and creates a desktop notification. Opening the card keeps its attempt history and composer visible, but the required structured answer receives primary focus and ordinary follow-up submission waits until that answer receives a submitted, skipped, timed-out, or cancelled outcome.

The board will measure supervision success first by whether every blocker becomes visible and reaches a recorded outcome. Stage movement remains locked while the card is running or needs attention; resolving a blocker returns control to the same attempt without changing its workflow stage.

## Alternatives Considered

### Alternative 1: Passive history with no board-level attention signal

- **Description**: Keep agent questions only in a card transcript for users to discover manually.
- **Pros**: A quieter board and fewer interruptions.
- **Cons**: Users can miss blocked work and cannot reliably supervise several cards.
- **Why rejected**: It fails the V1 success criterion of no missed blockers.

### Alternative 2: Keep ordinary composer input equally actionable during a blocker

- **Description**: Let the user submit unrelated follow-ups before answering the agent's required question.
- **Pros**: Maximizes apparent flexibility.
- **Cons**: Makes the execution state harder to understand and can bury the action that unblocks work.
- **Why rejected**: The agent's question has a clear priority; answering it first provides a more trustworthy experience.

### Alternative 3: Automatically resolve or dismiss unanswered questions

- **Description**: Allow the board to infer a default answer or resume work without user involvement.
- **Pros**: Fewer visible interruptions.
- **Cons**: Removes the user control and recorded decision that distinguish governed work.
- **Why rejected**: Human input must remain explicit and attributable.

## Consequences

### Positive

- Gives users a reliable, visible path from an agent question to a recorded outcome.
- Makes active supervision concrete without requiring users to continuously watch every conversation.
- Preserves the persistent card context while prioritizing the action that unblocks work.

### Negative

- Users may experience more board and desktop notifications during busy periods.
- Ordinary follow-up input temporarily waits behind the required answer.

### Risks

- Repeated notifications may become ignorable. Mitigate with clear, actionable notification copy and one coherent attention state per card.
- Users may confuse a blocked card with a failed one. Mitigate with distinct labels, explanation, and visible next action.

## Implementation Notes

- Preserve all submitted, skipped, timed-out, and cancelled outcomes in the card's Run Transcript.
- Accessibility must not rely on highlight color alone; every attention state needs text, focus, and keyboard access.
