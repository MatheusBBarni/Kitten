# ADR-003: Establish the packages-only workspace before desktop delivery

## Status

Accepted

## Date

2026-07-21

## Context

ADR-0023 defines `packages/tui`, `packages/desktop`, and `packages/engine` as the only application and capability boundaries. The repository is still a single root Bun package, so the desktop cannot be introduced without first making that boundary real. The technical clarification selected a packages-first migration rather than a transitional root application.

## Decision

Convert the repository into a private Bun workspace rooted at `packages/*` before desktop feature work begins. Move the existing Cockpit into `packages/tui` while preserving its published CLI, configuration, adapter behavior, test suite, build, installer, and release contracts. Add `packages/desktop` as the new Electrobun application and `packages/engine` only for UI-free shared capability contracts.

`packages/desktop` owns board scheduling, worktrees, Direct ACP attempt lifecycle, persistence, notifications, and host RPC. `packages/engine` must not own board state, application stores, persistence, worktrees, renderer code, or a universal controller. Existing Cockpit controller, store, UI, and JSON run persistence remain TUI-owned.

## Alternatives Considered

### Alternative 1: Scaffold the desktop beside a temporary root Cockpit

- **Description**: Add workspace metadata and desktop code while retaining Cockpit at the root until later.
- **Pros**: Starts visible desktop work sooner.
- **Cons**: Keeps two competing application layouts and delays proof of the intended boundary.
- **Why rejected**: It weakens ADR-0023 and expands migration ambiguity across every later desktop task.

### Alternative 2: Build the desktop at the root first

- **Description**: Implement the desktop in the existing root application and migrate packages afterward.
- **Pros**: Avoids an initial relocation.
- **Cons**: Violates the packages-only decision and couples incompatible lifecycles.
- **Why rejected**: It creates a known structural rewrite immediately after feature delivery.

## Consequences

### Positive

- Makes ownership and dependency direction explicit before desktop state exists.
- Keeps Cockpit behavior independently testable and releasable.

### Negative

- Delays desktop-visible work until Cockpit parity is established in `packages/tui`.

### Risks

- Relocation can regress Cockpit release behavior. Mitigate with its existing contract matrix before desktop work advances.

## Implementation Notes

- Preserve exact dependency pinning and the repository minimum-release-age policy.
- Keep application imports within `packages/`; shared imports flow from applications to `packages/engine`, never the reverse.
