# ADR-001: Constrain V1 to a linear governed workflow with queued active-run input

## Status

Accepted

## Date

2026-07-21

## Context

Issue #40 proposes a local-first macOS Workflow Board. Users need to create and reorder stages, assign a default local Workflow Skill after creating a stage, inspect every card's attempt history, and keep a prompt composer visible for the entire attempt lifecycle.

The canvas interaction could imply a general workflow graph, while human and agent movement plus active ACP conversations create race, recovery, and transcript-integrity risks. Existing ACP behavior does not establish a universal safe contract for injecting a new message into an in-flight turn.

## Decision

V1 provides a canvas-like editor for one ordered Workflow Stage path. Users may add, remove, and drag-reorder stages, then connect each stage to its immediate successor with visible arrows. Validation permits only one start, one end, and one predecessor/successor per stage; branches, joins, cycles, conditional transitions, and arbitrary graph routing are excluded. The starter sequence is `Backlog -> To-do -> Refinement -> Ready -> Doing -> Finished -> Closed`; every new stage is visibly incomplete until its default validated local Workflow Skill is selected.

A selected card always exposes its append-only Run Transcript history and persistent composer. When no attempt is live, submitting the composer starts a new Direct ACP Run Attempt using a fresh immutable Run Context. During a live attempt, submission is explicitly queued as the next message; it never silently cancels, forks, or duplicates the current turn. A later certified native-steering capability may replace this behavior only behind explicit provider evidence.

Every transition is versioned and idempotent. Human movement is permitted only for settled cards; agent success may advance exactly one stage. `running` and `needs_attention` impose Stage Lock. Final-stage success becomes `ready_for_review`, and no path auto-pushes, opens a pull request, merges, or completes the card.

## Alternatives Considered

### Alternative 1: General workflow graph in V1

- **Description**: Allow arbitrary stage nodes, edges, branches, joins, cycles, and transition conditions.
- **Pros**: Matches a literal canvas metaphor and supports complex processes immediately.
- **Cons**: Introduces graph validation, ambiguous progression, more failure modes, and a much larger visual and persistence surface.
- **Why rejected**: No current evidence requires non-linear routing, and it would delay the governed attempt model that differentiates Kitten.

### Alternative 2: Immediate live-message injection for every active ACP attempt

- **Description**: Treat the persistent composer as a universal interrupt or steering channel.
- **Pros**: Feels responsive and maximizes live control.
- **Cons**: ACP providers do not supply one proven injection contract; implicit cancellation can lose, duplicate, or misorder work.
- **Why rejected**: The safe universal behavior is a visible queued message, with live steering deferred until certified per-runtime evidence exists.

### Alternative 3: Fixed uneditable columns with no Skill requirement

- **Description**: Ship a conventional Kanban board with predefined stages and optional free-text prompts.
- **Pros**: Lowest initial visual complexity.
- **Cons**: Fails the user need to create/reorder stages and make a stage's default Workflow Skill explicit and locally validated.
- **Why rejected**: It removes the workflow governance that makes the board useful for coding-agent work.

## Consequences

### Positive

- Establishes a small, understandable workflow model that users can configure directly.
- Preserves continuous operator input without making unsafe promises about active-turn delivery.
- Keeps safety and review boundaries auditable through exact-once, versioned transitions and immutable Run Contexts.

### Negative

- Users who require branching workflows must wait for a later release.
- Queued active-run input is less immediate than certified native steering.
- Stage editing and card transition handling still require careful versioning and recovery behavior.

### Risks

- Stale, duplicate, or out-of-order completion events could move a card incorrectly. Mitigate with workflow-version fencing, idempotency keys, and deterministic recovery tests.
- A workflow edit during an attempt could invalidate the next stage. Mitigate by snapshotting the attempt's workflow version and rejecting stale transitions.
- Users may misunderstand queued messages as immediate steering. Mitigate with explicit queue state, removal control, and transcript events.

## Implementation Notes

- The desktop owns durable append-only attempt events and board state; the existing Cockpit persistence model is insufficient for full transcript history.
- Reuse the scoped `ask_user` pattern per attempt, but keep it generation-fenced and live-session scoped. Persist its audit record; after a crash, mark the live attempt interrupted rather than pretending it resumed.
- Preserve the separation between a user-managed Workflow Stage and system-managed Execution Status.
