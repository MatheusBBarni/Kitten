# ADR-007: Stage the Cockpit workspace relocation behind compatibility gates

## Status

Accepted

## Date

2026-07-21

## Context

ADR-003 chose a packages-first workspace before desktop delivery and originally described an atomic Cockpit relocation. Codebase exploration established that the current root Cockpit spans far more than the task workflow's permitted task size. Keeping one oversized migration task would make execution, review, rollback, and verification opaque.

The operator selected a split migration rather than an exception to the task-size rule. The workspace boundary remains mandatory; only the migration delivery shape changes.

## Decision

Supersede ADR-003's atomic-relocation delivery detail. Migrate Cockpit into `packages/tui` through dependency-ordered, behavior-preserving slices. Each slice owns one coherent source/test or public-delivery surface, includes the necessary import/configuration bridge, and ends with fresh targeted compatibility evidence. A temporary bridge may exist only when required by the next declared migration slice and must be removed by its dependent completion task.

The root becomes a private workspace before desktop code begins. No slice may establish a second application lifecycle outside `packages/tui`; the final layout remains the packages-only boundary from ADR-0023.

## Alternatives Considered

### Alternative 1: One atomic relocation task

- **Description**: Move the entire Cockpit source/test/release surface in one task.
- **Pros**: No transitional internal layout.
- **Cons**: Exceeds the task-size rule and makes failures difficult to isolate or review.
- **Why rejected**: The structural migration is too broad to be an independently executable task.

### Alternative 2: Defer all desktop planning until a separate migration packet

- **Description**: Stop this packet and create an unrelated migration plan first.
- **Pros**: Fully isolates migration planning.
- **Cons**: Splits one prerequisite graph and obscures the dependency between TUI relocation and desktop delivery.
- **Why rejected**: The packages-first prerequisite belongs in this product delivery graph.

## Consequences

### Positive

- Keeps every task reviewable and verification-scoped.
- Surfaces migration regressions at the responsible source or delivery slice.

### Negative

- Requires short-lived compatibility bridges and more dependency edges.

### Risks

- A bridge can become permanent. Mitigate with explicit removal deliverables and final packages-only validation.

## Implementation Notes

- Preserve the current published CLI, build, installer, release, and configuration contracts throughout the staged move.
- Re-run the relevant Cockpit test/self-check/build evidence after each affected slice.
