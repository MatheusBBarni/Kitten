# ADR-004: Persist desktop work as an append-only SQLite journal with projections

## Status

Accepted

## Date

2026-07-21

## Context

The PRD requires durable Workflow Boards, cards, immutable Run Contexts, complete Run Transcripts, Attention Blocker outcomes, and review evidence. Cockpit's JSON run writer intentionally persists only summary metadata and cannot reconstruct full card history. Technical clarification selected an event journal with derived current views.

## Decision

`packages/desktop` will own one local SQLite database. Every board mutation, card transition, attempt lifecycle change, normalized agent/user/tool activity, queued message operation, Attention Blocker outcome, and review disposition appends one ordered immutable event. The same transaction updates read projections for current board, stage, card, attempt, attention, queue, and review views.

Each event carries a stable event identifier, attempt/card/board identity where applicable, monotonic attempt sequence, actor, event type, and timestamp. The transaction rejects duplicate identities and stale workflow or attempt versions. A Run Context and resolved Skill snapshot are stored once at attempt creation and never rewritten. Startup marks an unclosed live attempt interrupted; it never claims to restore an ACP turn.

## Alternatives Considered

### Alternative 1: Mutable current rows with a small audit log

- **Description**: Store current state as the authority and append only selected audit records.
- **Pros**: Fewer rows and direct reads.
- **Cons**: Makes complete transcript reconstruction and exact transition recovery unreliable.
- **Why rejected**: Full durable history is a product contract, not optional diagnostics.

### Alternative 2: JSON snapshots per board

- **Description**: Persist periodically rewritten board files.
- **Pros**: Resembles the existing Cockpit run store.
- **Cons**: Does not provide transactional multi-entity updates, ordered immutable history, or safe concurrent lifecycle projection.
- **Why rejected**: It cannot provide the required attempt and blocker evidence.

## Consequences

### Positive

- Makes history, auditability, idempotency, and restart recovery explicit.
- Supplies efficient current-state reads without mutating the evidence record.

### Negative

- Requires schema migrations and projection invariants from the first release.

### Risks

- Incorrect projection logic can diverge from the journal. Mitigate with transactional writes, rebuild tests, and strict event validation.

## Implementation Notes

- The renderer accesses projections only through typed host RPC and never opens the database.
- Use transaction boundaries for an event append plus all affected projection updates.
