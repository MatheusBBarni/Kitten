# ADR-006: Resolve Workflow Skills from deterministic project and user catalog roots

## Status

Accepted

## Date

2026-07-21

## Context

Every Workflow Stage requires a default local Skill and every Run Context must retain the exact resolved Skill. The repository has project-local agent Skill directories, while users also maintain local Skill locations. Technical clarification selected both root classes with deterministic precedence.

## Decision

The desktop Skill Catalog will scan configured trusted-repository roots before configured user-level roots. A project Skill shadows user-level Skills with the same metadata name, so repository-owned workflow instructions take precedence. Distinct same-name Skills within the same root class remain a visible collision and cannot be selected by display name alone. The catalog canonicalizes paths before discovery so linked directories do not create duplicates. Each catalog entry has a stable identity derived from its canonical location and validated `SKILL.md` content digest.

At Run Attempt creation, the coordinator resolves a card override or stage default to one catalog entry and snapshots its identity, canonical location, digest, and exact validated content in the immutable Run Context. Later catalog changes affect future attempts only.

## Alternatives Considered

### Alternative 1: Project roots only

- **Description**: Discover Skills only inside the trusted repository.
- **Pros**: Simple and highly portable.
- **Cons**: Excludes a developer's established local workflow library.
- **Why rejected**: It does not meet the selected project-plus-user catalog scope.

### Alternative 2: User roots only

- **Description**: Discover Skills only from global local configuration.
- **Pros**: Centralized personal setup.
- **Cons**: Ignores repository-specific workflow rules.
- **Why rejected**: Project workflows need controlled local precedence.

### Alternative 3: Resolve a free-text name at launch

- **Description**: Ask an agent or shell to resolve a name dynamically.
- **Pros**: Minimal catalog UI.
- **Cons**: Non-deterministic and not auditable.
- **Why rejected**: Violates the PRD's validated local-catalog requirement.

## Consequences

### Positive

- Supports personal and repository-owned workflows while preserving exact attempt provenance.

### Negative

- Requires catalog validation, same-precedence collision handling, and clear root diagnostics.

### Risks

- A linked root can change after discovery. Mitigate by canonicalization and per-attempt content snapshotting.

## Implementation Notes

- Catalog root configuration is user-visible and read-only from the renderer through typed host RPC.
- A missing or invalid root makes the affected catalog entry unavailable; it never falls back to a free-text Skill.
