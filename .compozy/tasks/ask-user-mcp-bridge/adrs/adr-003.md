# ADR-003: Use a controller-owned bridge with per-session authenticated local IPC

## Status

Accepted

## Date

2026-07-14

## Context

Kitten must expose a local `ask_user` MCP tool to each eligible agent session without allowing an MCP caller to select, impersonate, or interfere with another Kitten session. The current controller owns session identity, connection generation, clarification queueing, and lifecycle cancellation. ACP configuration already provisions stdio MCP servers into each session, but Kitten does not yet host an MCP server.

RepoPrompt CE demonstrates the useful host-owned pattern: validate the MCP request, derive the target from the active connection, route it to the host UI, wait for one response, and serialize a structured result. Kitten must adapt that pattern to its multi-session TUI and existing layer boundaries.

## Decision

Create a controller-owned `AskUserBridge` application service. For each live eligible session generation, it registers a route containing the Kitten `SessionId`, generation, a private local IPC endpoint, and a fresh unguessable capability. The controller appends one generated, reserved MCP stdio server declaration after the already-resolved user MCP server list for that session. User declarations retain their order unchanged.

The generated declaration launches the same Kitten executable in a dedicated `--ask-user-mcp` mode. That child uses the direct, exact `@modelcontextprotocol/sdk` dependency and `StdioServerTransport` for its provider-facing MCP stream. It communicates with the parent only through the private local endpoint: a Unix-domain socket on POSIX and a named pipe on Windows. Every IPC request carries the capability; the host derives the session and generation from the registration and rejects malformed, stale, duplicate, oversized, or unauthorized requests.

The bridge adapter and MCP wire schema remain under `src/agent/`; the app service remains under `src/app/`; the normalized clarification payload and outcome remain protocol-free in `src/core/`. The store and UI receive only the existing protocol-free projection. Route removal, session replacement, provider failure, conversation close, and controller disposal cancel pending bridge requests and close their endpoints. The V1 guarantee is at most one accepted settlement while the generation is live; no cross-process recovery is implied.

## Alternatives Considered

### Alternative 1: Put the bridge inside `AgentConnection`

- **Description**: Make each ACP connection own MCP hosting, routing, and request lifecycle.
- **Pros**: Keeps provider process concerns near the adapter.
- **Cons**: Duplicates controller-owned session-generation logic and couples provider transport to application interaction policy.
- **Why rejected**: The controller is the sole owner of the multi-session lifecycle and clarification queue.

### Alternative 2: Use loopback TCP with a capability

- **Description**: Bind a localhost port per bridge registration.
- **Pros**: Familiar protocol and easy diagnostics.
- **Cons**: Enlarges the listener surface and makes local-port lifecycle and collision handling unnecessary V1 work.
- **Why rejected**: A local socket or named pipe provides the narrower required process boundary.

### Alternative 3: Make the external store the bridge router

- **Description**: Store capabilities and pending requests as reactive application state.
- **Pros**: Makes route state visible to UI selectors.
- **Cons**: Leaks security credentials and transport lifecycle into a user-facing state layer.
- **Why rejected**: The store must project interaction state, not own secret-bearing I/O routes.

## Consequences

### Positive

- Binds every tool call to the real owning session and generation.
- Reuses the existing clarification coordinator and cancellation semantics.
- Preserves ACP and MCP as adapter edges around a protocol-free core.
- Keeps the same-binary delivery and local-first product model.

### Negative

- Adds parent/child IPC lifecycle ownership and platform-specific endpoint handling.
- Requires a direct MCP SDK dependency and a dedicated executable mode.
- Cannot recover an in-flight answer after the parent process exits.

### Risks

- **Capability leakage or endpoint impersonation**: use fresh capability values, private endpoint permissions, bounded frames, and fail-closed validation.
- **Orphaned child processes**: close endpoints and terminate bridge children during every controller teardown path.
- **Duplicate or stale replies**: bind routes to generation and settle each request through one coordinator identity.

## Implementation Notes

The parent must create the registration before passing the generated MCP declaration into ACP `session/new` or `session/load`. The child must emit MCP protocol only on stdout; it must never log question or answer content to stdout or telemetry.
