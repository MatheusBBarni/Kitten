# ADR-003: Use a Pure TypeScript Renderer with Caller-Supplied Static Blocks

## Status

Accepted

## Date

2026-07-14

## Context

The PRD requires a concise, versioned, reviewable harness contract with explicit unsupported-version behavior, no dynamic base content, and a bounded future extension seam. Kitten's layering rule confines ACP SDK types to `src/agent/` and requires `src/core/` to be pure. Existing core domains validate unknown input strictly and return discriminated results rather than relying on exceptions or silent recovery.

The renderer needs to compose the V1 base with future static capability blocks without owning capability discovery, provider behavior, configuration, persistence, or ACP encoding. It must make invalid versions and block input legible to #19 and #20 while preserving deterministic output for review.

## Decision

Add `src/core/harnessPrompt.ts` with colocated `harnessPrompt.test.ts`. The module owns reviewed `v1` constants, the protocol-free TypeScript data model, validation, escaping, deterministic rendering, and typed outcomes. It exports a renderer that accepts a requested version plus caller-supplied static blocks and returns either rendered text or a fixed rejection code.

The module recognizes only reviewed version constants and never falls back to another version. It sorts valid optional blocks by stable identifier after the base, rejects malformed or duplicate identifiers and configured count or budget violations, and escapes block text before placing it inside fixed text delimiters. The base remains private reviewed text and is checked by a deterministic whitespace-token counter against the 150-token limit.

No optional block registry, capability selection, ACP type, transport state, configuration, persistence, telemetry emission, or UI is introduced in this card. #20 later owns selection from a reviewed static registry; #19 later owns conversion of already-rendered text into ACP prompt blocks and lifecycle delivery.

## Alternatives Considered

### Alternative 1: Place rendering in the ACP adapter

- **Description**: Build the contract directly as ACP prompt blocks in `src/agent/`.
- **Pros**: The immediate transport caller receives its final shape.
- **Cons**: Couples stable policy wording to provider transport, prevents pure review tests, and violates the protocol-free domain requirement.
- **Why rejected**: The contract must be reusable by later delivery work without importing ACP types or lifecycle state.

### Alternative 2: Make the core renderer own a capability registry

- **Description**: Have the renderer resolve optional block identifiers from a built-in or configured registry.
- **Pros**: One place to find all fragment text.
- **Cons**: Pulls #20 selection policy into #18 and encourages configuration or dynamic-content paths.
- **Why rejected**: The renderer needs only validated static blocks; capability evidence and selection have their own card and lifecycle.

### Alternative 3: Throw for invalid versions or blocks

- **Description**: Treat malformed requests as exceptional control flow.
- **Pros**: Small apparent API surface.
- **Cons**: Makes expected rejection harder to handle consistently and risks errors crossing asynchronous UI boundaries later.
- **Why rejected**: A fixed discriminated result mirrors existing core validation and keeps unsupported behavior explicit.

## Consequences

### Positive

- One pure TypeScript module owns the contract without changing the ACP boundary.
- Callers receive deterministic output or a fixed, content-free rejection reason.
- #20 can supply reviewed static blocks later without changing base rendering.
- Colocated golden, semantic, validation, and source-boundary tests make behavior reviewable.

### Negative

- The type system cannot prove that a future caller's string originated from reviewed static text; #20 must uphold that provenance boundary.
- The module deliberately does not produce runtime agent behavior until #19 consumes it.

### Risks

- **Extension seam expands into a registry**: mitigate by accepting only supplied blocks and keeping V1's block list empty.
- **Delimiter escaping changes wording**: mitigate with exact golden fixtures for base-only and escaped extension cases.
- **Token terminology varies by model**: mitigate with one documented deterministic whitespace-token measure for the static base rather than a model dependency.

## Implementation Notes

- Use repository-native TypeScript for the core interface and tests.
- Keep the base contract and supported-version constants in the new core module; do not add a package, config file, persistent model, or directory.
- The source-boundary test may inspect the module source, but `harnessPrompt.ts` itself must not import ACP, adapters, runtime APIs, or telemetry.
