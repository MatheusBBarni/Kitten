# ADR-001: Expose a bounded start-and-poll MCP surface

## Status

Accepted

## Date

2026-07-15

## Context

Issue #11 established Kitten's protocol-free, host-owned registry for visible child sessions. Issue #12 needs to let a provider session create and observe that child work through the bundled Kitten MCP bridge without delegating authority to ACP implementations or provider-native subagent features.

The target V1 user is a solo developer who asks a primary coding agent to launch independent work in parallel. The product value is quick, trustworthy delegation with every child remaining a normal visible Kitten conversation. Market research confirms that visibility and intervenability are baseline expectations for background coding-agent work, while adoption remains uneven enough that a broad orchestration platform is not justified for V1.

The current registry and UI already provide lifecycle projection and a human recovery path. The bundled MCP transport already provides an authenticated private capability bound to one session generation. They do not yet provide an agent-facing orchestration protocol, an imperative waiter, interaction forwarding, or result handoffs.

## Decision

Provide one provider-neutral `agent_run` MCP tool with two V1 operations:

- `start` accepts a bounded batch of independent task/outcome requests and returns stable Kitten child IDs with initial snapshots.
- `poll` returns snapshots only for child IDs owned by the parent derived from the authenticated MCP capability route.

The route, not tool input, supplies the parent session and generation. The bridge and controller fail closed for unknown or stale routes, fabricated or cross-parent child IDs, closed or unready parents, recursive child spawning, and invalid or over-limit input. A controller-owned MCP entrypoint preserves the registry's ownership and generation checks without inheriting the UI dialog's selected-parent requirement.

Each successful start creates an ordinary, visible Kitten child conversation. Polling must make active, needs-input, and terminal states explicit and include a reliable identity that lets the developer continue recovery in the existing session UI. The protocol remains intentionally detached and human-supervised.

## Alternatives Considered

### Alternative 1: Full control plane in V1

- **Description**: Ship wait, cancel, steer, respond, exploration, management, role profiles, transcripts, and handoffs with start and poll.
- **Pros**: Closer to the complete RepoPrompt CE control surface; fewer manual UI transitions for a parent agent.
- **Cons**: Defines complex authorization, interaction, timeout, result-retention, and promise-lifecycle semantics before their host primitives and user demand are proven.
- **Why rejected**: It would turn the validation slice into a broad orchestration platform and substantially enlarge the security and lifecycle-race surface.

### Alternative 2: Add a bounded non-cancelling `wait` in V1

- **Description**: Let a parent wait for one or more child snapshots until completion or timeout, without cancelling the children on timeout.
- **Pros**: More ergonomic completion observation than repeated polls.
- **Cons**: Adds a second promise lifecycle, timeout semantics, aggregation rules, and resource-ownership tests.
- **Why rejected**: It is a convenience layer that can later wrap a stable polling contract; it is not required to validate the selected parallel-launch value.

### Alternative 3: Provider-native delegation or a UI-only registry

- **Description**: Either defer to each ACP provider's subagent model or require the developer to launch every child through Kitten's existing UI.
- **Pros**: Less initial MCP bridge work.
- **Cons**: Provider behavior would fragment, or the primary agent could not directly initiate the selected workflow.
- **Why rejected**: Both paths fail the provider-neutral, agent-initiated product outcome.

## Consequences

### Positive

- Validates the core solo-developer workflow with the smallest provider-neutral MCP surface.
- Retains one authoritative host-owned lifecycle model and normal Kitten visibility for Codex and Claude Code children.
- Limits authorization and data-exposure risk through route-derived ownership, strict limits, and fail-closed validation.
- Leaves future `wait` as a thin adapter over stable snapshots rather than a competing execution model.

### Negative

- Parent agents must poll; they cannot wait, steer, cancel, forward input, or retrieve handoffs through MCP in V1.
- Human supervision remains necessary for a child that needs attention or must be stopped.
- The V1 evidence may reveal demand for controls that need separate design work before they can be added safely.

### Risks

- **Invisible or ambiguous background work**: expose lifecycle and needs-input state in polling and retain normal workspace visibility.
- **Cross-session control or stale publication**: bind every request to the authenticated route and generation, then reject unknown, stale, non-owned, or closing identities.
- **Resource exhaustion or duplicate starts**: bound batch size, request/text/frame sizes, concurrent calls, and duplicate request identities.
- **Concurrent edits conflict**: keep worktree isolation and role/profile policy in their dedicated follow-up issues rather than imply automatic safety here.

## Implementation Notes

- Keep MCP schemas and stdio protocol types in `src/agent/`; keep authenticated route management and controller-facing operations in `src/app/`; do not leak wire types or credentials into core, store, or UI.
- Preserve deterministic user MCP ordering and append one generated Kitten declaration.
- Add schema, route/ownership, and fake-transport integration coverage before expanding the operation set.
