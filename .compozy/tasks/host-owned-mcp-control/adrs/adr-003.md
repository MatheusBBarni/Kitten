# ADR-003: Extend the authenticated Kitten MCP bridge with atomic bounded agent control

## Status

Accepted

## Date

2026-07-15

## Context

The existing bundled MCP bridge starts one private, capability-bound child process for each Kitten session generation. It already keeps caller-selectable session identity out of the public tool contract, preserves user-configured MCP declaration order, validates framed local messages, and invalidates routes on replacement, close, and disposal.

The host-owned delegation registry already owns flat child identity, lifecycle, generation fencing, workspace visibility, and terminal state. The new child-control feature must expose only the approved MVP operations—bounded `start` and owner-scoped `poll`—without creating a second registry, a second local transport, or provider-specific behavior.

Technical clarification chose the existing Kitten MCP process and authenticated route, all-or-nothing preflight for invalid batches, a maximum of four children per start request, explicit poll IDs, and schema-to-fake-ACP integration verification.

## Decision

Generalize the existing generated Kitten MCP child mode so one authenticated, session-generation-bound route serves both `ask_user` and a new `agent_run` tool. The bridge uses a discriminated local frame union and resolves the parent session and generation only from the capability route.

`agent_run.start` accepts at most four validated `{ task, desired_outcome }` entries. It preflights the complete request before registering any child; an invalid route, duplicate entry, invalid item, or over-limit batch rejects the entire request with no child registration. After acceptance, child startup remains detached and parallel; an individual runtime failure becomes that child’s visible terminal failure rather than rolling back siblings.

`agent_run.poll` requires one or more explicit child IDs. It returns snapshots only when every requested ID belongs to the route-derived parent and generation; unknown, stale, fabricated, cross-parent, or duplicate IDs fail closed without leaking another child’s state.

The shared store registration primitive must not require the parent to be selected or move focus to it. That requirement belongs only to the UI adapter: route-authorized MCP calls can launch from a valid background parent while the developer’s current visible session remains unchanged. This narrow store adjustment preserves the existing protocol-free registry; it does not expose MCP types, capabilities, or controller runtime ownership to the store.

## Alternatives Considered

### Alternative 1: Separate agent-control bridge and generated MCP declaration

- **Description**: Start an additional local sidecar or publish a second generated MCP server for child control.
- **Pros**: Isolates the new tool implementation from the existing question bridge.
- **Cons**: Duplicates capability lifecycle, transport hardening, declaration ordering, and process management; creates competing session authority.
- **Why rejected**: The existing bridge is the proven authenticated route and one generated declaration is easier to reason about and test.

### Alternative 2: Provider-native subagents

- **Description**: Ask each ACP provider to create and manage its own children.
- **Pros**: Less host-side control protocol.
- **Cons**: Splits lifecycle, visibility, ownership, and safety semantics by provider and bypasses Kitten’s normal workspace model.
- **Why rejected**: It violates the provider-neutral, host-owned product and architectural boundary.

### Alternative 3: Partial batch acceptance and implicit poll-all

- **Description**: Start valid entries even when part of a batch is invalid, and let polling discover every child by default.
- **Pros**: Fewer retries for mixed submissions and less caller input for observation.
- **Cons**: Produces ambiguous partial effects, makes request recovery harder, expands result size, and increases information-disclosure surface.
- **Why rejected**: Atomic preflight plus explicit IDs gives the MVP deterministic, bounded, fail-closed behavior.

## Consequences

### Positive

- Reuses the established private-route lifecycle, strict validation, and single-declaration composition.
- Preserves the controller as the sole runtime owner and the registry as the sole delegation authority.
- Gives callers predictable batch behavior and bounded snapshots without exposing task content or unrelated child state.
- Makes real end-to-end verification possible through the existing stdio, local IPC, fake bridge, and fake ACP seams.

### Negative

- `ask_user` source and bridge naming/composition must be generalized to reflect a second bundled tool.
- Callers must retry an entire invalid batch and provide poll IDs explicitly.
- V1 omits wait, cancellation, steering, interaction response, role policy, worktree policy, and persistence.

### Risks

- **Route or generation confusion**: bind every operation to the capability route and reject stale or closing registrations.
- **Partial visibility after accepted start**: register each accepted child before asynchronous startup and publish subsequent runtime failure visibly.
- **Resource pressure**: enforce the four-child limit, bounded text/frame sizes, and per-route operation concurrency.
- **Regression of the existing question flow**: preserve the existing tool contract and extend same-binary integration tests to prove both tools coexist.
- **Unexpected focus mutation during a background launch**: keep the selected-parent guard in the UI adapter and add store/controller tests proving the shared registration primitive retains the current visible session.

## Implementation Notes

- Keep MCP wire schemas and child-mode code in `src/agent/`, authenticated local routing and controller-facing control in `src/app/`, and existing protocol-free lifecycle types in `src/core/`.
- Refactor the controller’s UI-oriented child launch into a UI adapter plus a route-authorized internal service; the UI retains its selected-parent guard while MCP uses the route-derived parent and generation.
- Remove the selected-parent requirement from the shared `AppStore.addDelegatedSession` registration primitive and preserve `selectedVisibleId`; the change is selection policy only, not a new ownership or lifecycle model.
- Add no new package, persistence schema, or results UI. Keep content-free telemetry to fixed operation/outcome enums, bounded batch-size buckets, and duration buckets.
