# ADR-006: Attention State Model and Jump-to-Next

## Status

Accepted

## Date

2026-07-09

## Context

The overview must tell the developer which session needs them and offer one action to jump to the next one that does (PRD).
Today the status model is `AgentStatus = "idle" | "working" | "awaiting_approval"` (`src/core/types.ts:17`), written by the reducer from a `status` domain event, and the ACP adapter emits `working` on prompt start and `idle` on finish, unconditionally (`src/agent/agentConnection.ts:184,192`).
The adapter exposes `PromptStopReason = "end_turn" | "max_tokens" | "max_turn_requests" | "refusal" | "cancelled"` but discards it.
There is no `finished` or `error` state, so the overview cannot distinguish "done, your move" from "still idle" or from "crashed."

## Decision

Extend the state model and derive a needs-you signal from it.

- `SessionStatus = "idle" | "working" | "awaiting_approval" | "finished" | "error"`.
- The adapter maps the stop reason instead of always emitting `idle`: `end_turn`, `max_tokens`, `max_turn_requests`, and `refusal` become `finished` (the turn ended and the developer's input is expected); `cancelled` becomes `idle` (the developer interrupted). A prompt call that throws, a failed or lost transport, or a subprocess exit becomes `error`.
- Needs-you is a pure derivation: `needsAttention(status)` is true for `awaiting_approval`, `error`, and `finished`; false for `working` and `idle`.
- New selectors: `selectSessionList` (each session with its status, in `order`) and `selectNextNeedy(afterSessionId)` (the next needs-you session after the focused one, wrapping around). The jump-to-next action sets focus to that session.
- Priority when several need attention: `awaiting_approval` before `error` before `finished`, then by `order`. This ordering is a defined default; a richer, configurable ranking is deferred (TechSpec Open Questions).

## Alternatives Considered

### Alternative 1: Add `finished` only, no `error`

- **Description**: Derive `finished` from `end_turn` and keep everything else as `idle`.
- **Pros**: Smallest change; no crash detection.
- **Cons**: Drops the PRD's error state and the "route me to the broken session" flow; a crashed adapter reads as merely idle.
- **Why rejected**: The overview would hide failures, which is when the developer most needs routing.

### Alternative 2: Store the raw stop reason on the session instead of a state enum

- **Description**: Keep `idle` and attach the last `PromptStopReason` for the UI to interpret.
- **Pros**: No new enum values.
- **Cons**: Pushes state interpretation into every consumer (overview, status strip, notifier), so they can disagree; the reducer stays the single writer only for turns, not for state meaning.
- **Why rejected**: One derived status keeps the strip, overview, and notifier consistent by construction.

## Consequences

### Positive

- The overview and status strip gain a truthful, shared state per session, and needs-you is one predicate every surface reads.
- The notifier (ADR-007) and the jump-to-next action both key off the same derivation.

### Negative

- Adds enum values that touch the reducer, the status labels, the theme's status tones, and the status strip.

### Risks

- Reliable `error` detection depends on observing transport/subprocess exit, which the adapter does not surface today. Mitigation: wire the transport close and prompt rejection to an `error` status event; treat detection gaps as an Open Question and default an unresponsive session to its last known state rather than a false `finished`.
- `finished` versus `idle` can flicker if the adapter reports intermediate stops. Mitigation: derive the state only from terminal stop reasons, not from streaming updates.

## Implementation Notes

Primary edit sites: `src/core/types.ts` (`SessionStatus`, the `status` domain event), `src/agent/agentConnection.ts` (map stop reason and failures to status), `src/store/selectors.ts` (`selectSessionList`, `selectNextNeedy`, `needsAttention`), `src/app/actions.ts` (jump-to-next action), `src/ui/StatusStrip.tsx` and `src/ui/theme.ts` (labels and tones for the new states).
