# ADR-003: Native OS-Level Attention Notifications in V1

## Status

Accepted

## Date

2026-07-09

## Context

The core pain this feature attacks is that an agent sits idle and blocked while the developer's attention is on a different terminal or a different application.
An in-app status strip and the Ctrl+S overview cannot reach a developer who is not looking at Kitten.
During PRD clarification the product owner chose a terminal bell plus in-app signaling, then extended the scope: notify the user at the operating-system level too, not only inside the terminal.

Kitten runs across macOS, Linux, and Windows terminals, and each platform exposes a different native notification channel.
Delivering a native notification therefore requires detecting the operating system and choosing the right channel, with a graceful fallback where none is available.

## Decision

V1 delivers attention notifications on three levels when a session newly enters a needs-you state (awaiting approval, finished, or errored) while Kitten is not the focused window:

1. In-app: the status strip always reflects fleet state and the Ctrl+S overview routes the developer to the session that needs them.
2. Terminal bell: a bell on the newly-needs-you transition, which most terminals surface as an alert or badge even when unfocused.
3. Native OS notification: a desktop notification naming the session and its working directory (for example, "Codex in ~/payments needs your approval"), using the platform-appropriate channel selected by OS detection, with the terminal bell as the fallback where a native channel is unavailable.

Notifications fire on the transition into a needs-you state, not repeatedly, and never carry prompt or transcript content.

## Alternatives Considered

### Alternative 1: In-app and terminal bell only

- **Description**: Signal attention through the status strip, the Ctrl+S overview, and a terminal bell, with no OS-level notification.
- **Pros**: Smallest scope; no cross-platform notification surface; nothing to detect or permission.
- **Cons**: A bell can be missed or silenced, and neither the strip nor the overview reaches a developer working in another application, which is the exact situation the feature exists to fix.
- **Why rejected**: It under-serves the core pain of an away developer, which the owner explicitly wanted addressed.

### Alternative 2: OS notifications only, no bell

- **Description**: Rely solely on native desktop notifications.
- **Pros**: One clear channel; strongest at reaching an away developer.
- **Cons**: Native notifications can be disabled or throttled by the OS, and a terminal-first user may prefer an in-terminal cue; a single channel has no fallback.
- **Why rejected**: Layering keeps a working cue on every platform even when native notifications are off.

## Consequences

### Positive

- Reaches the developer wherever their attention is, which is the difference between an agent waiting two minutes and waiting twenty.
- OS detection lets Kitten use the best channel each platform offers and degrade gracefully to the bell.

### Negative

- Adds a cross-platform notification surface and the OS-detection logic that selects the channel, widening the V1 build and the test matrix.

### Risks

- Notification behavior varies by platform and terminal, so a channel may be silently unavailable. Mitigation: always keep the terminal bell as a universal fallback, and detect the platform rather than assuming it.
- Notifications can become noise if they fire too often. Mitigation: fire only on the transition into a needs-you state, deduplicated per session, with no repeat nagging.
- Privacy: a desktop notification is visible on a shared or shared-screen machine. Mitigation: notifications carry only the session title, provider, working directory, and state, never prompt or transcript content.

## Implementation Notes

Out of scope for V1: user-configurable per-session notification rules, notification history, and Do-Not-Disturb scheduling. The channel set is macOS, Linux, and Windows native notifications with a terminal-bell fallback.
