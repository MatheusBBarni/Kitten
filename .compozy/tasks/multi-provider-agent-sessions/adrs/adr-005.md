# ADR-005: Fleet Configuration Model - Providers Plus a Sessions List

## Status

Accepted

## Date

2026-07-09

## Context

Today the config's `agents` field is a fixed two-key object (`claude-code`, `codex`), each a spawn override merged over `DEFAULT_AGENTS`, which pin the ACP adapter packages (`src/config/configLoader.ts:80-91,34-49`).
The controller opens one session per agent against a single shared working directory (`src/app/controller.ts:86,136`).
The pre-declared fleet (PRD) needs several sessions, each with its own working directory, an optional title, and an optional first task, and it must be able to run two sessions of the same provider.
The first-run gate assumes the launch directory is the project and requires it to be a git repository (`src/config/firstRun.ts`).

## Decision

Split the configuration into providers and sessions.

- `providers`: the spawn recipes, the existing per-provider `command`/`args`/`env` keyed by `ProviderKind`, still overridable and still pinned to known-good adapter versions. This is today's `agents` map, renamed to its real role.
- `sessions`: an ordered list of session descriptors, each `{ provider: ProviderKind; cwd: string; title?: string; task?: string }`. `cwd` is per session; `title` defaults to the directory basename; `task`, when present, is sent as the session's first prompt.
- Zero-config default: when the file declares no `sessions`, Kitten seeds one session per configured provider in the launch directory, which reproduces today's two-agent, single-repo behavior as the N=2 case.
- The loader resolves each session descriptor into the controller's per-session spawn-plus-cwd input, keeping validation in one zod schema that rejects unknown keys.

## Alternatives Considered

### Alternative 1: Self-contained sessions

- **Description**: Each session declares its own full spawn recipe (`command`/`args`/`env`) alongside `cwd` and `task`, with no provider concept.
- **Pros**: Flat; one place per session.
- **Cons**: Repeats the command, args, and env for every session of the same provider, and scatters the pinned-adapter version the base ADR-005 deliberately centralizes.
- **Why rejected**: It loses the single pinned-adapter definition and duplicates spawn detail.

### Alternative 2: Keep the agents map, add session references

- **Description**: Retain the `agents` map and add a parallel `sessions` array whose entries reference an agent by key plus a `cwd`/`task`.
- **Pros**: Smallest schema delta.
- **Cons**: Keeps conflating "agent" with "provider," and reads awkwardly when several sessions reference the same provider.
- **Why rejected**: The providers/sessions split names the two concepts correctly for the same effort.

## Consequences

### Positive

- Expresses an N-session, multi-directory, multi-provider fleet directly, including repeated providers.
- Preserves the zero-config experience and the pinned-adapter safety of today.
- Per-session `cwd` removes the single-shared-directory limitation the controller has now.

### Negative

- The config schema and its merge and default logic are rewritten, and the readiness and first-run checks must run per session rather than once.

### Risks

- A session may point at a directory that is not a git repository, which the current gate forbids globally. Mitigation: decide the per-session repo policy explicitly (see Open Questions in the TechSpec); at minimum, report a non-repo session as not-ready with its reason rather than blocking the whole fleet.
- Existing config files use the old `agents` object. Mitigation: accept the old `agents` key as an alias for `providers` during a deprecation window, or document a one-line migration.

## Implementation Notes

Primary edit sites: `src/config/configLoader.ts` (schema, defaults, merge, resolution to per-session inputs), `src/app/controller.ts` (start one runtime per session descriptor with its own `cwd`), `src/index.ts` and `src/config/firstRun.ts` (per-session readiness and repo checks).
