# ADR-004: Allocate verified worktrees in a repository-local managed container before child registration

## Status

Accepted

## Date

2026-07-15

## Context

`startDelegatedChild()` currently registers a child and then starts it with the parent `cwd`. Managed worktree creation must instead establish one collision-free workspace before any child session, ACP runtime, prompt, or success telemetry exists. The user selected a repository-local `.kitten/worktrees` root rather than a user-level or sibling directory.

The container must not pollute the parent repository's tracked state, and a failed allocation must not disturb pre-existing worktrees or branches.

## Decision

Create one app-owned `ManagedWorktreeService` in the controller layer. It resolves the parent attached branch and committed `HEAD`, reserves a bounded generated identifier, and creates the child branch and worktree under `<repo-root>/.kitten/worktrees/<binding-id>` before the controller registers the child. The service verifies canonical repository, path, branch, and base SHA against Git worktree state before returning the binding.

The service treats the container as private local workspace state. It ensures the managed root is ignored through local repository metadata without editing tracked project files, and fails closed if the target path is pre-existing, external, ambiguous, detached, submodule-based, or otherwise unverifiable. Pre-registration failures remove only clean artifacts created by that attempt and return no child session.

## Alternatives Considered

### Alternative 1: Use a user-level shared worktree root

- **Description**: Place every managed workspace under a Kitten directory outside repositories.
- **Pros**: Keeps project trees visually clean and centralizes cleanup.
- **Cons**: Separates workspaces from the repository they belong to and makes project-local discovery harder.
- **Why rejected**: The user selected a repository-local, self-contained workspace view.

### Alternative 2: Use a sibling directory beside the repository

- **Description**: Create worktrees next to the parent checkout.
- **Pros**: Avoids adding a directory inside the repository root.
- **Cons**: Naming and ownership are ambiguous when several repositories share a parent directory.
- **Why rejected**: It is less deterministic and less discoverable than the selected repository-local container.

### Alternative 3: Register the child before provisioning its worktree

- **Description**: Create the normal child session, then replace its workspace after Git succeeds.
- **Pros**: Reuses the current child-start order.
- **Cons**: Can expose the wrong checkout, leave ghost sessions, and emit misleading lifecycle state.
- **Why rejected**: The isolated workspace is a precondition of child execution.

## Consequences

### Positive

- Every managed child starts from a verified, isolated workspace.
- Branch/path reservation and rollback remain in one controller-owned boundary.
- Users can discover retained work next to the repository without tracked-file changes.

### Negative

- Repository-local ignore metadata requires careful idempotent handling.
- Provisioning adds a launch-time Git operation and new fail-closed outcomes.

### Risks

- **Concurrent collisions** — Reserve generated ids in-process and verify Git's authoritative worktree list before use.
- **Dirty parent expectation** — Use only committed `HEAD` and disclose the base; never copy dirty or ignored files.
- **Rollback deletes user work** — Roll back only paths and branches positively identified as created by the current attempt.

## Implementation Notes

Keep Git command execution behind injected service seams. The core and store receive immutable binding results only; ACP receives the final verified `cwd` only after provisioning succeeds.
