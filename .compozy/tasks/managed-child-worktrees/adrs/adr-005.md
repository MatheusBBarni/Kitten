# ADR-005: Restrict cleanup to terminal child review and verify Git lifecycle in two layers

## Status

Accepted

## Date

2026-07-15

## Context

Managed worktrees are retained review artifacts, not disposable child-session side effects. The PRD prohibits automatic deletion, and the user selected a dedicated controller action available only from terminal child review surfaces. The cleanup policy must reject dirty, unmerged, external, live-owned, and unverifiable worktrees while still being testable without relying only on host Git behavior.

## Decision

Expose `cleanupManagedWorktree` as a controller action that terminal child review surfaces may invoke after explicit confirmation. It is separate from `closeConversation`; closing a session does not remove a worktree. The action asks the managed-worktree service to verify provenance, terminal ownership, clean state, merge status, and normal removable state before performing non-force removal and safe branch deletion. It returns a bounded result code for user-visible explanation.

Verify the lifecycle in two layers: injected Git-service unit tests cover command order, state transitions, collision handling, and refusal codes; temporary real-Git-repository integration tests prove creation, verification, rollback, restoration, and clean-only removal against actual Git semantics.

## Alternatives Considered

### Alternative 1: Clean up when a child session closes

- **Description**: Couple worktree removal to the ordinary session-close path.
- **Pros**: Reduces retained artifacts automatically.
- **Cons**: Closing a transcript can discard unreviewed work and bypass the selected review loop.
- **Why rejected**: Session visibility and workspace disposal have different lifecycles.

### Alternative 2: Offer cleanup through a global slash command

- **Description**: Allow cleanup without selecting a terminal child review surface.
- **Pros**: Fast for experienced users.
- **Cons**: Weakens child-to-workspace provenance and makes target mistakes easier.
- **Why rejected**: V1 favors explicit contextual review over broad command surface area.

### Alternative 3: Use only mocked or only real-Git tests

- **Description**: Choose one test layer for all lifecycle coverage.
- **Pros**: Fewer test types to maintain.
- **Cons**: Mocks cannot prove Git semantics; real-only tests make command-order and failure matrices slow and opaque.
- **Why rejected**: The lifecycle needs both deterministic unit coverage and genuine Git validation.

## Consequences

### Positive

- Cleanup remains traceable to a reviewed terminal child workspace.
- Git safety checks are explicit, testable, and user-explainable.
- The test strategy catches both controller regressions and Git integration mismatches.

### Negative

- Users cannot bulk-clean or clean from ordinary session-close in V1.
- The feature adds temporary-repository integration fixtures to the test suite.

### Risks

- **Merge status is misread** — Test fast-forward, merged, and unmerged branch states in real repositories.
- **Unclear refusal copy** — Map all internal failures to a short bounded user-facing reason.
- **Cleanup races with a live child** — Recheck ownership and runtime liveness immediately before removal.

## Implementation Notes

The service performs non-force Git removal only after all checks pass. Telemetry records allowlisted lifecycle categories and refusal codes, never paths, branches, SHAs, task text, or command output.
