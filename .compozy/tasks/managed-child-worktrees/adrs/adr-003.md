# ADR-003: Persist managed bindings in versioned session records and reconcile on restore

## Status

Accepted

## Date

2026-07-15

## Context

Managed worktrees must remain reviewable after Kitten restarts, while the existing delegation projection intentionally does not restore live parent-child ownership. Current persisted conversation records retain a `cwd` but cannot identify whether it is a managed workspace, its branch, its committed base, or whether it is still safe to use.

The selected design must preserve review identity without reviving child execution, task text, runtime references, or any silent fallback to the parent checkout.

## Decision

Add an optional, strict managed-worktree binding to the next persisted conversation-record version. The binding stores only locally needed workspace identity: stable binding id, managed repository root, worktree path, child branch, parent base branch and SHA, owning child session id, and a bounded review/cleanup availability code.

On restore, Kitten restores the conversation as an ordinary session and clears delegation as it does today. The controller then reconciles each persisted managed binding against Git. A missing, external, dirty, live-owned, or unverifiable binding becomes explicit unavailable review state. It never becomes a parent `cwd`, and it never recreates a live parent-child relationship.

## Alternatives Considered

### Alternative 1: Keep worktree bindings in memory only

- **Description**: Discard all binding identity on process restart.
- **Pros**: No persistence schema change.
- **Cons**: Users cannot reliably review or resolve a completed child workspace after restart.
- **Why rejected**: It violates the retained-review requirement.

### Alternative 2: Reconstruct delegation from restored child sessions

- **Description**: Treat persisted child metadata as proof of a live parent-child graph.
- **Pros**: Preserves apparent lineage after restart.
- **Cons**: Can fabricate ownership, stale waiters, and live cleanup authority.
- **Why rejected**: Existing delegation safety deliberately forbids this behavior.

### Alternative 3: Store bindings in a separate manifest

- **Description**: Keep workspace lifecycle data outside the run record.
- **Pros**: Separates Git metadata from conversations.
- **Cons**: Introduces a second persistence lifecycle and reconciliation source of truth.
- **Why rejected**: A versioned optional session binding is smaller and preserves the session-to-workspace relationship.

## Consequences

### Positive

- Retained child workspaces remain attributable and reviewable across restarts.
- Strict schema evolution makes binding metadata explicit and testable.
- Restored ambiguity is visible rather than silently unsafe.

### Negative

- Run-record migration and restore reconciliation add lifecycle states.
- A restored child can be reviewable without retaining its original live delegation lineage.

### Risks

- **Sensitive lifecycle content leaks into telemetry** — Keep branch, path, SHA, and task data out of telemetry; emit only allowlisted status categories.
- **Stale path appears valid** — Reconcile provenance and Git identity before exposing an available state.
- **Older records fail to load** — Keep prior record versions readable and migrate them to no binding.

## Implementation Notes

The binding remains a protocol-free immutable value. Git inspection and reconciliation stay controller-owned; the store only holds controller-produced state transitions.
