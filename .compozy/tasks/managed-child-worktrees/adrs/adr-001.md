# ADR-001: Create managed worktrees only for spawned child sessions

## Status

Accepted

## Date

2026-07-15

## Context

Parallel children currently inherit their parent's `cwd`, so concurrent edits can collide in one checkout. Kitten's closed orchestration registry owns child lifecycle in the controller while keeping its core/store projection protocol-free and ephemeral. A worktree binding must therefore be resolved before ACP session creation, remain reviewable after a child ends, and never make cleanup or restore unsafe.

The user selected a child-only V1, based on the parent branch, with retained worktrees for explicit review and cleanup. Council review found that inherited, user-owned, and Kitten-managed worktrees have incompatible provenance and cleanup rights. It also established that a child must start at the parent branch's committed `HEAD`, never an implicit copy of dirty parent files.

## Decision

V1 will create one uniquely named, Kitten-managed Git worktree and branch only when the host spawns a child session. The app/controller layer will resolve the parent's branch and committed base SHA, allocate and verify the binding before ACP startup, then pass the verified worktree path as the child's sole `cwd`.

The binding will expose managed provenance, branch, path, base SHA, and availability for review. It may persist that content-free workspace metadata, but it must not reconstruct live parent-child ownership on restore. A stale or unavailable binding blocks dispatch and is visibly unavailable; it must never fall back to the parent checkout.

Successful and terminal child worktrees remain for explicit review. Cleanup is an explicit controller-owned action that refuses a dirty, unmerged, external, live-owned, or unverifiable worktree. V1 will not automatically merge, delete, force-remove, or bind arbitrary existing worktrees.

## Alternatives Considered

### Alternative 1: Keep shared parent working directories

- **Description**: Continue starting every child in the parent's `cwd`.
- **Pros**: No Git lifecycle surface or launch delay.
- **Cons**: Concurrent children can overwrite one another, and ownership of diffs is ambiguous.
- **Why rejected**: It fails the primary safety and trust problem the feature addresses.

### Alternative 2: Support inherit, arbitrary existing binding, and managed creation in V1

- **Description**: Let each child choose among the parent checkout, a pre-existing worktree, or a new managed worktree.
- **Pros**: Maximum flexibility for users with prepared local environments.
- **Cons**: Combines incompatible ownership, rollback, persistence, and cleanup contracts before the core path is validated.
- **Why rejected**: Kitten can fully guarantee provenance and cleanup only for a worktree it created and verified.

### Alternative 3: Automatically remove or merge child worktrees on completion

- **Description**: Treat a completed child workspace as disposable or promote it automatically.
- **Pros**: Reduces retained branch and directory clutter.
- **Cons**: Can discard unfinished work or hide unreviewed changes.
- **Why rejected**: Users need deliberate review and promotion; Git state must remain under explicit developer control.

## Consequences

### Positive

- Concurrent spawned children receive isolated checkouts before they can edit.
- Existing per-session `cwd` consumers can show the correct files and branch without ACP changes.
- Managed provenance gives users a clear review and cleanup contract.

### Negative

- Managed branches and paths remain until the user explicitly resolves them.
- Launch has Git provisioning latency and can fail before an ACP session starts.
- Existing-worktree binding and dirty-parent snapshot behavior are deferred.

### Risks

- **Path or branch collision** — Reserve and verify unique identifiers atomically before startup.
- **Partial startup failure** — Roll back only artifacts created by the failed attempt; preserve pre-existing Git state.
- **Stale restored metadata** — Reconcile against Git and mark unavailable rather than substituting a parent `cwd`.
- **Destructive cleanup** — Require controller ownership, no live child, verified provenance, clean state, and normal non-force Git removal.
- **Submodules** — Reject them in V1 with an actionable unsupported result until their lifecycle is designed.

## Implementation Notes

- Keep Git I/O in an app/controller-owned service; do not introduce it into ACP adapters, core, or store.
- Use the parent branch's committed `HEAD` as the creation base and display that identity with the child branch/path.
- Treat the integrated global shell as out of scope for V1 because it is not session-cwd-scoped.
