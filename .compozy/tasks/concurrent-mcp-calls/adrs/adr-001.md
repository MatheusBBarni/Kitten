# ADR-001: Keep concurrent MCP admission controller-owned and bounded

## Status

Accepted

## Date

2026-07-16

## Context

Kitten's bundled `ask_user` and `agent_run` MCP child opens a private local connection for each invocation. The authenticated bridge currently binds one socket to a session-generation route, so a second valid same-route invocation can fail at the socket boundary before the controller can apply its real limits or report a meaningful outcome. The failure is especially misleading for an agent author mid-task because the existing tool-call row reduces it to a generic failed MCP call.

The repair must preserve capability-derived authority, generation fences, per-session isolation, bounded frame and call limits, and opt-in content-free telemetry. It must not introduce a durable scheduler, an administration surface, or automatic retries that could duplicate `agent_run.start` side effects.

## Decision

Keep one private authenticated listener per session generation, but allow it to admit multiple independently authenticated, bounded calls instead of treating one socket as the route's exclusive owner. Route authority continues to come only from the capability and current generation. The controller-owned admission path applies legitimate capacity limits; each accepted or rejected call receives one bounded terminal outcome.

V1 distinguishes a legitimate, authoritative capacity condition from an unavailable route in the existing tool outcome surface. A busy state may offer manual recovery guidance only after the original request has an authoritative terminal settlement. V1 never automatically replays an ambiguous `agent_run.start` request.

## Alternatives Considered

### Alternative 1: Add a general per-route fairness queue and scheduler

- **Description**: Serialize or schedule calls through a configurable queue before controller admission.
- **Pros**: Makes fairness and overload handling explicit.
- **Cons**: Introduces new scheduling semantics, cancellation policy, priority policy, and operational surface.
- **Why rejected**: The existing defect is false socket exclusivity. Fixed bounded admission and controller-owned limits address it without turning a reliability repair into an orchestration platform.

### Alternative 2: Retain one bound socket and improve the generic error text

- **Description**: Preserve the socket-level rejection and only change presentation.
- **Pros**: Smallest code change.
- **Cons**: Valid parallel work still fails before the controller can distinguish a false contention from a configured limit.
- **Why rejected**: It does not satisfy the issue's continuity requirement or restore truthful failure ownership.

### Alternative 3: Automatically retry all busy or disconnected calls

- **Description**: Reconnect and resend requests without user action.
- **Pros**: Can hide transient transport failures.
- **Cons**: An ambiguous `agent_run.start` response can create duplicate child work and amplify load.
- **Why rejected**: V1 must fail closed around unknown execution state; idempotent replay requires a separately designed contract.

## Consequences

### Positive

- Valid same-session parallel MCP calls can reach controller-owned admission instead of failing because they use distinct sockets.
- Genuine configured limits remain bounded, content-free, and actionable.
- Existing session-generation authorization and cross-session isolation remain the authority boundary.
- The tool-call UI can communicate retryable capacity pressure without adding a second result surface.

### Negative

- Each call now needs independent lifecycle and exact-once settlement through disconnect, cancellation, and route replacement.
- Bounded admission needs saturation, stale-generation, and cross-session test coverage before release.
- No fairness guarantee beyond the bounded admission rules is added in V1.

### Risks

- **Duplicate `agent_run.start` after an ambiguous result** — do not auto-retry; retain call-ID deduplication and require authoritative settlement before manual retry guidance.
- **Stale or cross-session work** — validate capability-derived route and generation at admission and dispatch; cancel outstanding calls on route teardown.
- **Resource pressure or starvation** — keep explicit per-route/global bounds and prove deterministic overload failure in tests; revisit fair scheduling only with evidence.
- **Content leakage through diagnostics** — record only fixed failure/outcome categories and bounded dimensions, never route, capability, prompt, raw error, or identifier values.

## Implementation Notes

The implementation should reuse the existing generated MCP child, authenticated bridge, controller failure seam, closed telemetry vocabulary, and `ToolCallRow` rendering pipeline. It should not add persistence of bridge authority or a new dashboard.
