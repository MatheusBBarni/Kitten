# ADR-004: Fail Closed on Native Steering and Recover Unsent Text on Lifecycle Loss

## Status

Accepted

## Date

2026-07-15

## Context

The product promises a consistent steering outcome across providers, but the current standardized ACP client surface documents `prompt` and `cancel`, not a general steering operation. A provider-specific native path must therefore never be inferred from provider identity or an unverified configuration. The user also chose non-persistent recovery: when a live session closes, crashes, or is replaced, undispatched steering text must settle once and return to the current composer rather than resume automatically later.

## Decision

Model native steering as an explicit, verified adapter capability that defaults to unavailable. The controller uses a native path only when both the resolved capability and the adapter implementation prove it is supported; otherwise it uses the cancel-and-follow-up fallback. The initial implementation assumes no native capability until an adapter revision is deliberately certified.

On cancellation failure, terminal timeout, provider failure, close, or generation replacement, the controller terminalizes the relevant requests idempotently and exposes their exact ordered text as a one-time composer recovery draft. It never automatically retries or persists queued steering state. Once the composer acknowledges recovery, the raw queue is cleared.

## Alternatives Considered

### Alternative 1: Treat native steering as universally available

- **Description**: Add a generic provider-steering call and use it whenever a session is active.
- **Pros**: Shortest apparent flow for providers that happen to support it.
- **Cons**: Invents a cross-provider contract, risks unsupported transport calls, and creates inconsistent failure behavior.
- **Why rejected**: Current ACP documentation standardizes prompt and cancellation, not a generic steering method.

### Alternative 2: Use only the fallback and omit capability modeling

- **Description**: Permanently cancel and follow up for every provider.
- **Pros**: Minimal initial code path.
- **Cons**: Makes a later verified provider optimization invasive and hides capability evidence.
- **Why rejected**: A fail-closed capability seam costs little and preserves a safe extension path.

### Alternative 3: Persist queued steering for later automatic replay

- **Description**: Save queued message content in the run record and resend it after restoration.
- **Pros**: May reduce manual re-entry after a restart.
- **Cons**: Retains sensitive content, can replay stale intent, and violates the chosen recovery model.
- **Why rejected**: Restoring text visibly keeps the developer in control and maintains content-free run checkpoints.

## Consequences

### Positive

- Unsupported or ambiguous provider behavior fails safely into the same recovery experience.
- No queued prompt text enters run persistence or telemetry.
- Future native support can be introduced through an audited capability contract.

### Negative

- V1 uses the fallback even when a provider may have an undocumented convenience mechanism.
- Developers must explicitly resend a restored draft.

### Risks

- **A capability declaration drifts from an adapter revision** — bind certification to the resolved adapter identity and fail closed on mismatch.
- **Recovery is lost during application termination** — document that recovery applies while Kitten remains live; no replay occurs after process restart.
- **A late success follows a timeout** — request identity and generation fencing prevent it from replacing the visible recovery outcome.

## Implementation Notes

Keep the native-capability type protocol-free outside `src/agent/`. Do not route arbitrary provider extension calls until a certified adapter contract defines their semantics and terminal acknowledgement.
