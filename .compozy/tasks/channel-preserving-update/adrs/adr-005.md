# ADR-005: Prove Update Transactions with Isolated Local Tests

## Status

Accepted

## Date

2026-07-20

## Context

The update path mutates package installations and executable files. A unit-only suite can miss real launcher and replacement behavior, while a live GitHub Release exercise would make the normal gate dependent on publication timing and external network state. The project already favors injected seams for deterministic tests and local npm/package integration coverage.

## Decision

The update feature will use two layers of automated evidence. Unit tests will inject command, filesystem, hash, HTTP, clock, and output seams to prove every classification and failure-preservation branch. Deterministic local process integration tests will package the Node launcher and exercise a controlled standalone target/release fixture to prove global-vs-local routing, atomic replacement, and byte-identical failure preservation without a live GitHub Release.

The release workflow's existing published-package smoke remains a packaging check, not the primary behavioral proof for updates.

## Alternatives Considered

### Alternative 1: Unit tests only

- **Description**: Cover pure helpers and mocked command results without child processes or real temporary files.
- **Pros**: Fastest suite and fewest fixtures.
- **Cons**: Misses launcher path behavior and filesystem replacement semantics.
- **Why rejected**: The feature's most important promises cross process and filesystem boundaries.

### Alternative 2: Live GitHub Release update test

- **Description**: Exercise `--update` only after a real release is published.
- **Pros**: Uses production infrastructure.
- **Cons**: Slow, rate-limited, timing-sensitive, and unable to cover induced failure paths reliably.
- **Why rejected**: It cannot be the deterministic gate for fail-closed behavior.

## Consequences

### Positive

- The normal test suite proves no-mutation failures and launcher provenance without credentials or public-network dependence.
- Local integration fixtures validate behavior at the boundary users execute.

### Negative

- Test seams and fixture helpers add modest maintenance cost.
- Release metadata fixtures must be kept aligned with the public GitHub response shape.

### Risks

- Local fixtures can drift from GitHub's response contract. Strict schema parsing and a small representative fixture minimize that risk.

## Implementation Notes

- Run focused update suites plus the repository typecheck, test, self-check, and build gate before release.
- Assert unchanged target bytes and unchanged registry contents for every induced standalone failure.
