# ADR-004: Use Canonical-Path Records and Tagged Releases for Standalone Updates

## Status

Accepted

## Date

2026-07-20

## Context

The checksum installer proves a downloaded artifact before placing it on PATH, but it currently leaves no durable evidence that a later executable is that installed artifact. The direct `releases/latest/download` endpoint also does not provide an authoritative version before a download. Standalone updates need both a proven target and a stable release identity while preserving the old executable on failure.

## Decision

The installer will maintain a versioned XDG-state registry keyed by canonical installed path. Every record stores the canonical path, platform, installed version, and installed SHA-256. The installer creates or replaces a record only after checksum verification and successful target installation. The updater accepts only a canonical regular-file target whose record path, platform, embedded version, and current hash all agree.

The updater will request GitHub's latest stable release metadata, validate the Kitten version tag, and fetch that exact tag's matching artifact plus `SHA256SUMS`. It will require exactly one well-formed manifest row, verify the downloaded hash, then use a same-directory lock, private temporary file, adjacent backup, atomic replacement, and atomic registry write. Any inconclusive state or failed transaction returns nonzero without choosing another channel.

## Alternatives Considered

### Alternative 1: One record for the most recent standalone install

- **Description**: Store a single global installation record.
- **Pros**: Smallest state shape.
- **Cons**: A valid custom installation silently loses update eligibility when another location is installed.
- **Why rejected**: The installer already permits a custom destination; independently installed locations need independent ownership records.

### Alternative 2: Metadata sidecar beside the executable

- **Description**: Store provenance next to each installed binary.
- **Pros**: Record travels with the binary directory.
- **Cons**: Adds files to user-selected executable directories and makes directory moves or copies ambiguous.
- **Why rejected**: XDG state keeps generated ownership evidence separate and permits canonical-path validation.

### Alternative 3: Use the latest-download redirect as the version source

- **Description**: Download from `releases/latest/download` and infer the version after download.
- **Pros**: Reuses the existing installer URL.
- **Cons**: No structured version before download; redirect behavior is a weaker release contract.
- **Why rejected**: The GitHub Releases API provides an explicit stable tag for versioned output and exact artifact URLs.

## Consequences

### Positive

- Multiple installer-managed binaries can update independently without path guessing.
- The updater never executes a candidate binary before verifying its hash or replaces an unrecorded target.
- Result versions come from an explicit stable-release identity.

### Negative

- The installer and updater must keep a small versioned registry in sync.
- A stale, changed, moved, or malformed record intentionally produces a safe refusal.

### Risks

- GitHub API or release-asset availability can fail. The transaction leaves the current executable and record unchanged and prints recovery commands.
- Checksums do not defend against a compromised release publisher. That residual risk remains outside V1, as recorded in ADR-001.
- A registry-write failure after replacement requires rollback from the adjacent backup; automated tests must simulate that boundary.

## Implementation Notes

- Keep the registry under `$XDG_STATE_HOME/kitten`, falling back to `~/.local/state/kitten`; do not alter the unrelated first-run `state.json` schema.
- Release tags must represent a stable Kitten version. Draft, prerelease, malformed, duplicate-manifest, or platform-mismatched data is a refusal.
