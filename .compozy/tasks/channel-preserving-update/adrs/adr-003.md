# ADR-003: Keep Update Mutation at Its Provenance Boundary

## Status

Accepted

## Date

2026-07-20

## Context

`kitten --update` must preserve the ownership of a verified npm or standalone installation. The Node launcher can inspect the installed npm package and platform package paths before it launches the compiled executable. The compiled executable can inspect only its own canonical path and local state. Treating either view as universal would permit a local package, `npx` cache, copied executable, or user-provided environment marker to reach the wrong mutation path.

## Decision

The Node launcher owns the npm update transaction. It will handle `--update` before it spawns the compiled binary only after canonical comparison proves that both the launcher package and resolved platform binary belong to npm's global root. It will run `npm install --global @matheusbbarni/kitten@latest` and report the package version before and after the transaction.

The compiled executable owns only the standalone transaction. It will handle `--update` only after metadata-flag precedence, validate its installer-created standalone record, and otherwise refuse with the shared recovery guidance. It will never invoke npm. The launcher will refuse a local, `npx`, missing, or ambiguous npm invocation rather than delegate it to the binary.

## Alternatives Considered

### Alternative 1: Let the compiled binary choose either channel

- **Description**: Pass launch context through environment variables and centralize all update behavior in TypeScript.
- **Pros**: One implementation and one output formatter.
- **Cons**: The binary cannot independently prove the Node package ancestry; environment context can be stale or forged.
- **Why rejected**: The cross-process signal weakens the ownership proof that the feature promises.

### Alternative 2: Infer npm ownership from the executable name or PATH

- **Description**: Treat a command named `kitten` or found in a common prefix as global npm.
- **Pros**: Small implementation surface.
- **Cons**: Local installs, `npx`, copied binaries, and shims can look identical.
- **Why rejected**: Filename and PATH position are not provenance.

### Alternative 3: Provide only manual npm recovery

- **Description**: Automate standalone updates and print npm instructions for every package install.
- **Pros**: Avoids launcher changes.
- **Cons**: Does not meet the co-primary supported-channel goal.
- **Why rejected**: Global npm ownership can be proven safely at the launcher boundary.

## Consequences

### Positive

- Each mutating operation runs where its installation provenance is observable.
- Local package and `npx` contexts fail closed before either npm or standalone mutation begins.
- The compiled binary remains independent of Node/npm for standalone users.

### Negative

- User-facing result formatting is implemented across the Node launcher and compiled binary boundaries.
- Launcher tests need injected command, path, and package-version seams.

### Risks

- npm layouts vary by prefix and platform. Canonical path comparison and local process fixtures mitigate false classification.
- npm can fail after beginning its own transaction. Kitten reports the nonzero npm result and does not attempt a channel fallback.

## Implementation Notes

- Preserve `--version` before `--help` precedence by forwarding those invocations through the existing compiled-binary path before considering `--update`.
- Use explicit command argument arrays; never construct a shell command from paths or versions.
