# ADR-003: Move Cockpit as one self-contained Bun workspace app

## Status

Accepted

## Date

2026-07-16

## Context

Cockpit is currently one root-owned public package. Its source, tests, launcher, build script, and TypeScript configuration depend on sibling paths and, in several integration tests, the Cockpit working directory. Phase 1 requires a workspace conversion without changing published Cockpit behavior or extracting shared capabilities.

Current Bun workspace guidance supports a private root with workspace globs, one root lockfile, and filtered package-script execution. The user selected a whole-app move and app-local test invocation.

## Decision

Create a private Bun workspace root with `workspaces: ["apps/*"]`. Move the entire Cockpit package to `apps/cockpit/`: its public package manifest, `src/`, `test/`, `bin/`, `scripts/`, and TypeScript configuration move as one unit. Keep its public package name and binary declaration unchanged.

The root owns only workspace coordination, shared install policy, and future top-level commands. It delegates Cockpit commands with Bun package filters; Cockpit scripts and tests execute with `apps/cockpit` as their working directory. Phase 1 creates no `packages/` workspace, shared runtime, or compatibility wrapper for root-relative Cockpit source paths.

## Alternatives Considered

### Alternative 1: Keep Cockpit source at the workspace root

- **Description**: Add a workspace root but leave Cockpit source and package ownership in place.
- **Pros**: Minimizes immediate path changes.
- **Cons**: Leaves the application without an independent workspace boundary and makes a later desktop app structurally ambiguous.
- **Why rejected**: It does not satisfy the two-application monorepo boundary required by ADR-001.

### Alternative 2: Extract shared packages during the move

- **Description**: Split domain, transport, or UI-adjacent code into internal packages before moving Cockpit.
- **Pros**: Appears to accelerate later reuse.
- **Cons**: Multiplies moving parts, changes runtime ownership, and makes parity failures harder to attribute.
- **Why rejected**: ADR-001 excludes shared-capability extraction from Phase 1.

## Consequences

### Positive

- Preserves existing source-to-test and CWD relationships inside one app.
- Creates an unambiguous app boundary without prematurely designing a shared capability layer.
- Lets root commands target Cockpit explicitly through current Bun filtering.

### Negative

- Every root-relative workflow, release path, and test fixture must be updated deliberately.
- A temporary single-app workspace has less immediate structural payoff than a broad package split.

### Risks

- **Hidden root-path assumption** — move source, tests, package metadata, and scripts as a single commit and preserve contract coverage.
- **Wrong working directory** — require root delegators to execute app scripts through the workspace package rather than invoke source paths directly.

## Implementation Notes

Use one root `bun.lock` and retain the existing root `bunfig.toml` install policy. Cockpit owns its public package metadata and local development scripts.
