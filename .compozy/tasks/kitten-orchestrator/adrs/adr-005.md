# ADR-005: Preserve Cockpit local state without migration in Phase 1

## Status

Accepted

## Date

2026-07-16

## Context

Cockpit configuration is already user-scoped and has strict path precedence, validation, and session semantics. Phase 1 changes repository ownership only; it introduces no Orchestrator data model, shared profile registry, or cross-app user experience. The user selected unchanged configuration and persisted local state.

## Decision

Preserve every existing Cockpit configuration location, schema, and local-state behavior exactly. Do not add a migration, copy, new registry, workspace-level configuration file, or dual-read compatibility path in Phase 1.

The moved Cockpit app continues to resolve configuration through its existing precedence and preserves malformed-configuration failures. Future shared profiles and predecessor-data import remain separately specified work.

## Alternatives Considered

### Alternative 1: Copy existing state into a compatibility location

- **Description**: Duplicate Cockpit data into a new location while retaining the original as rollback material.
- **Pros**: Starts a future data-migration path early.
- **Cons**: Adds migration correctness, idempotency, and rollback obligations to a release that promises no user-visible change.
- **Why rejected**: Phase 1 needs no new state model and must leave user state unchanged.

### Alternative 2: Introduce the shared profile registry now

- **Description**: Move Cockpit configuration into the future cross-app agent profile registry.
- **Pros**: Reduces a later configuration move.
- **Cons**: Alters user state and crosses the Phase-1 boundary before the second application exists.
- **Why rejected**: It violates ADR-001 and ADR-002 scope discipline.

## Consequences

### Positive

- Existing users retain configuration compatibility without migration risk.
- Malformed configuration stays a visible error rather than a silent fallback.
- Later state changes can be designed with their own import and rollback evidence.

### Negative

- Cockpit configuration remains app-specific until later shared-profile work.

### Risks

- **Accidental path behavior change** — retain exact configuration tests and launch-directory semantics in the app-local suite.
- **Premature data coupling** — reject any Phase-1 change that reads or writes future Orchestrator state.

## Implementation Notes

No persisted data or configuration-file migration runs in Phase 1.
