# ADR-006: Relocate Cockpit contract tests and delegate them from the workspace root

## Status

Accepted

## Date

2026-07-16

## Context

Cockpit's test suite contains colocated and integration coverage for package metadata, launcher resolution, compiled self-checks, configuration behavior, installer integrity, release workflows, and CWD-sensitive process launches. Replacing or duplicating it during the workspace move would weaken the only formal Phase-1 advancement evidence.

The user selected relocation and preservation of the existing suite, with root commands delegating to Cockpit-local scripts.

## Decision

Move the established Cockpit test suite with `apps/cockpit` and preserve its intent. Update assertions only where the approved workspace boundary changes a path or command. Run Cockpit's typecheck, tests, coverage, self-check, and build from the Cockpit package working directory.

Root aggregate commands delegate to the named Cockpit workspace package and do not emulate the old root CWD or duplicate the suite. Add narrowly scoped root workflow tests for delegation and app-path release orchestration.

## Alternatives Considered

### Alternative 1: Replace the suite with workspace-specific tests

- **Description**: Retire current contract tests and write a smaller suite for the new layout.
- **Pros**: Reduces legacy path assertions.
- **Cons**: Loses parity evidence and can normalize missing public behavior.
- **Why rejected**: The Phase-1 gate is the existing contract matrix.

### Alternative 2: Run Cockpit tests from the workspace root

- **Description**: Rewrite CWD-sensitive tests and scripts around root invocation.
- **Pros**: Centralizes commands at the root.
- **Cons**: Changes the app's execution semantics and obscures application ownership.
- **Why rejected**: App-local CWD preserves existing source and fixture relationships with less behavioral change.

## Consequences

### Positive

- The parity gate remains meaningful after the directory move.
- Test failures identify whether an intentional path change lacks a matching contract update.
- Root coordination remains thin and explicit.

### Negative

- Some assertions must be changed even though user behavior does not change.
- Root and app command definitions require ongoing consistency checks.

### Risks

- **Over-broad assertion edits** — review each changed path assertion against the public contract it protects.
- **Root commands mask app failure** — ensure delegation propagates failures unchanged.

## Implementation Notes

Keep the separate showcase package verification independent from Cockpit-local commands.
