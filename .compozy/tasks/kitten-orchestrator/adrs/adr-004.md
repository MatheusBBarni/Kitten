# ADR-004: Keep root release orchestration while building Cockpit locally

## Status

Accepted

## Date

2026-07-16

## Context

Cockpit's published installer URL, README guidance, release workflow, npm shim, and four native platform packages are public contracts. The current release workflow and its tests use root-relative paths, while the workspace decision moves the Cockpit implementation and public package to `apps/cockpit`.

The user selected root-owned public release orchestration rather than moving the installer, public documentation, and release entry points in Phase 1.

## Decision

Keep the public installer, README install guidance, release configuration, and GitHub workflow orchestration at the repository root. Update those root-owned release paths to invoke Cockpit's app-local build, package, and verification scripts, and consume the app-local staged artifacts when publishing the same public npm shim and four platform packages.

The published command, package names, release asset names, checksums, and Bun-free smoke behavior remain unchanged. Phase 1 does not introduce an independent Orchestrator release train.

## Alternatives Considered

### Alternative 1: Move installer and release ownership into Cockpit

- **Description**: Relocate public release documentation, installer, and workflow ownership beside the app package.
- **Pros**: Co-locates all Cockpit assets.
- **Cons**: Breaks a stable public root URL and expands the migration surface.
- **Why rejected**: Root orchestration preserves the existing public contract while app-local ownership changes internally.

### Alternative 2: Introduce a new multi-app release train

- **Description**: Replace the current Cockpit release process with final two-app release architecture during Phase 1.
- **Pros**: Avoids a later release evolution.
- **Cons**: Couples an absent desktop app to a Cockpit parity release and adds untestable future behavior.
- **Why rejected**: Independent application releases are a later phase, not a Phase-1 requirement.

## Consequences

### Positive

- Public install and release references remain stable.
- Existing release-contract tests continue to describe the actual public path.
- The app package owns implementation build inputs without changing user artifact names or installation behavior.

### Negative

- Root workflow paths must bridge into the app package during the transition.
- Release ownership is temporarily split between root orchestration and app build inputs.

### Risks

- **Artifact-path drift** — update workflow, build, installer, and contract tests atomically around the app-local staging location.
- **Version mismatch** — use Cockpit's public package version as the source for platform package and shim validation.

## Implementation Notes

The root installer continues to download release assets by their existing public names. The workflow remains responsible for native-runner assembly and published-package smoke checks.
