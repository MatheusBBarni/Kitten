# ADR-001: Gate the two-app migration on Cockpit workspace parity

## Status

Accepted

## Date

2026-07-16

## Context

Issue #25 transforms Kitten into a two-application Bun monorepo: the existing
Cockpit for live developer-led sessions and a future Orchestrator for governed
unattended work. Cockpit is currently a root-owned public CLI and release
surface. Its package shim, config semantics, self-check, native builds,
installer, and release workflow have contract coverage that must survive any
workspace move.

The primary user is an individual developer. The user chose preservation of
existing workflows as the first outcome and selected the complete automated
Cockpit contract matrix as the formal unlock for the desktop migration. Market
research confirms demand for asynchronous coding agents, but also shows that
trust, review, and bounded authority—not early agent parallelism—are the
product's credible differentiation.

## Decision

Make the first delivery a narrow, behavior-preserving Cockpit workspace
conversion. The repository root becomes a private Bun workspace while Cockpit
keeps its public package, CLI, configuration behavior, test/self-check,
four-platform binary, installer, and release contracts. The complete existing
Cockpit contract matrix is the mandatory Phase-1 engineering gate before
Orchestrator import begins.

Keep Orchestrator import, shared-capability extraction, shared UI, shared
application lifecycle, new governance behavior, and visual redesign out of
this delivery. Record one representative live Cockpit workflow as supplementary
learning evidence, but do not weaken or replace the automated parity gate with
it. Treat later cross-app workflow learning as a gate for broader investment,
not as a reason to blur Phase-1 ownership.

## Alternatives Considered

### Alternative 1: Prototype cross-app orchestration before workspace conversion

- **Description**: Build a disposable handoff or orchestration experiment while
  the applications remain separately structured.
- **Pros**: Tests user demand and cross-app trust assumptions early.
- **Cons**: Creates a temporary integration path and does not de-risk the
  existing Cockpit packaging, release, or path-coupling migration.
- **Why rejected**: The selected milestone must first preserve Cockpit's public
  contract; a later, explicit learning slice can test orchestration value
  without making a throwaway path the production foundation.

### Alternative 2: Import Orchestrator or extract shared packages during Phase 1

- **Description**: Combine the workspace move with the desktop import or early
  common-capability work.
- **Pros**: Appears to reach the two-app end state faster.
- **Cons**: Multiplies regressions, obscures parity loss, couples app ownership,
  and makes failure attribution difficult.
- **Why rejected**: It violates the parity-first contract and turns an
  independently testable migration into a multi-product rewrite.

## Consequences

### Positive

- Preserves the individual developer's current Cockpit experience before adding
  a second application lifecycle.
- Makes packaging, configuration, install, binary, and release regressions
  visible through existing automated evidence.
- Keeps future shared capabilities below application ownership and preserves
  independent release and data boundaries.

### Negative

- The first delivery is primarily foundational and creates little direct new
  end-user value.
- Root-path assumptions across source, tests, build, installer, and release
  tooling require deliberate migration rather than a mechanical move.

### Risks

- **False confidence from tests alone** — retain one representative live
  workflow as supplementary evidence and reserve cross-app learning for the
  next investment gate.
- **Release or installer drift** — preserve the published CLI, platform-package
  order, compiled self-check, and release-contract tests as mandatory evidence.
- **Scope creep** — enforce explicit Phase-1 exclusions and create separately
  gated child issues for import, capability extraction, and cross-app flows.

## Implementation Notes

The workspace conversion must keep Cockpit's public `@matheusbbarni/kitten`
distribution and standalone binary behavior unchanged while introducing
workspace-local commands that never hide Cockpit-local failures. Later phases
must cite the governing Kitten ADRs for app separation, parity-first desktop
migration, independent releases, separate product data, and reviewed handoffs.
