# ADR-001: Two-Layer Whole-Cockpit Resume - Reliable Relationship, Best-Effort Liveness

## Status

Accepted

## Date

2026-07-09

## Context

Kitten runs two ACP agents (Claude Code + Codex) side by side and hands a live task between them.
Its trust story to date is an *egress* guarantee: nothing leaves unredacted, nothing is sent to the other agent until the user confirms.

The proposed "Resume previous sessions" feature (`Ctrl+R`) introduces a durable session concept.
The user's chosen direction is: resume means *continue the conversation live* (not read-only), the unit is *the whole cockpit run* (both agents' threads + focus + hand-off state) restored atomically, and the primary driver is *picking up work across days*.

Two forces constrain the design:

1. **Uneven adapter fidelity.** ACP defines `session/load` (full history replay via notifications) gated by an advertised `loadSession` capability. Claude's adapter supports it (verified in source); Codex's adapter advertises it but has reported replay gaps; Cursor-class adapters shipped `session/load` returning nothing as late as mid-2026. Live resume across days also depends on the agent adapter durably persisting its own session and the old session id still being valid days later - none of which Kitten can verify statically. A naive "resume BOTH agents live, atomically" promise is stacked on two unverified third-party reload paths, either of which can silently half-fail.
2. **A category change in disk posture.** Kitten today writes only opt-in, content-free telemetry to `~/.local/state/kitten/`. Resume writes actual conversation *content* - both transcripts, tool output, pending diffs, and the curated hand-off bundle - to disk for the first time. The hand-off bundle in particular is denser in sensitivity-per-byte than any raw transcript, and keeping both agents behind one serialized store is a juicier single target than the agents' two separate stores.

A five-advisor council (pragmatic-engineer, architect-advisor, security-advocate, product-mind, devils-advocate) debated the scope. Four favored shipping the whole-cockpit unit with degradable liveness; the devils-advocate dissented toward a read-only-history floor. The key reframing that carried the room: the *reliable* part of resume (the hand-off relationship, focus, pending diffs) is Kitten's own state, and the *unreliable* part (live conversation replay) lives in adapters Kitten does not control - the inverse of a naive risk read.

## Decision

Ship **whole-cockpit resume in V1, defined as two layers**:

- **Relationship layer (Kitten-owned, always restored):** `focusedAgentId`, the hand-off ledger / last curated bundle, per-agent pending diffs and touched files, and run metadata. Persisted as a **versioned, slot-keyed run record** - a stable `runId` plus per-agent slot ids independent of ACP session ids - so hand-off references survive the session-id remap that adapters perform on load. This layer restores atomically and reliably because no adapter can break it.
- **Live-conversation layer (agent-owned, best-effort per side):** probe each adapter's `loadSession` capability at `initialize`; restore live via `session/load` where supported and the id is still valid; otherwise the pane loads as clearly-labeled **read-only history**. Restoration is **explicit per side** (`restored` / `history-only` / `unavailable`), never silent, never an all-or-nothing block.

**Hand-back after resume reuses the existing `Ctrl+T` redacted-preview-confirm flow** - no new egress path. A restored-live focused pane plus the restored bundle is sufficient for "continue and hand back."

**A data-at-rest posture is a V1 gate:** atomic `0600` file / `0700` dir permissions; first-run disclosure of what is stored, where, and how to purge; per-session delete from the `Ctrl+R` picker plus a global delete-all; a retention cap; and reuse of the existing secret redactor to strip already-detected secrets from anything persisted. Prefer an index + redacted-bundle as the durable source of truth and rehydrate transcripts via `session/load`; persist a redacted, bounded transcript snapshot only as the labeled fallback for degraded adapters.

**A runtime capability + durability probe against the pinned adapters (`claude-code-acp@0.57.0`, `codex-acp@1.1.0`) is a required spike at the front of the work**, before the "resume live" promise is committed.

## Alternatives Considered

### Alternative 1: Read-only history floor + single-agent live (devils-advocate)

- **Description**: V1 restores best-effort read-only history for both panes plus live-resume of only the single focused agent; hand-off-edge restore and dual-live deferred to V2; resume possibly ranked below the model/provider selector.
- **Pros**: Only promises what the adapters have provably earned; smallest trust exposure; sidesteps the most fragile piece (hand-off re-anchoring) on its first outing.
- **Cons**: Spends the entire resume budget on the undifferentiated, commodity half; defers the one thing no competitor can copy; still writes content to disk, so it pays the at-rest cost without the differentiated payoff.
- **Why rejected**: The reliable part of resume *is* the differentiated part (Kitten's own relationship state), so scoping it out inverts the risk - it defers the cheap, safe, valuable layer while shipping the commodity layer that competitors already have. Its core caution (do not promise live where unearned) is fully absorbed by the two-layer model's explicit per-side degradation.

### Alternative 2: Crash-safe autosave + instant last-run resume (no picker)

- **Description**: Continuously persist the current run and offer "resume last run in this project?" on startup, mirroring `claude --continue`. No `Ctrl+R` picker, no history browsing.
- **Pros**: Smallest surface; high feasibility; delivers crash recovery and same-project continuity immediately.
- **Cons**: No cross-run navigation (the `Ctrl+R` job the user asked for); matches `--continue` without differentiating.
- **Why rejected**: Not rejected as a mechanism - **folded in**. Continuous autosave is *how* a run becomes resumable, and "resume last run" becomes a fast-path alongside the `Ctrl+R` picker rather than a replacement for it.

### Alternative 3: All-or-nothing dual-live (block resume unless both agents reload)

- **Description**: Treat the cockpit as truly atomic - if either adapter cannot reload live, the whole resume fails.
- **Pros**: Simple mental model; never shows a half-restored workspace.
- **Cons**: Codex's known replay gaps would hold every release hostage to the weakest adapter; discards the reliable relationship layer whenever one side is degraded; brittle against a stale days-old id.
- **Why rejected**: Atomicity is a fiction the moment one adapter reloads and the other returns nothing; a clean model over an unreliable substrate produces confident, wrong state, which is worse than an honest partial.

## Consequences

### Positive

- Ships the differentiated hand-off-relationship restore in V1 on the layer Kitten fully controls, while liveness degrades gracefully on the layer it does not.
- Honest per-side status protects the trust brand: the product never claims "live" where the adapter cannot deliver.
- The slot-keyed, versioned run record turns V2 fork / list / multi-run / share into "copy a run record," not a schema rewrite and a live migration.
- The at-rest gate keeps the "nothing leaves unredacted" promise coherent even as Kitten writes content to disk for the first time.

### Negative

- More runtime branching than a naive one-blob serialize: per-adapter capability probing, per-side restoration status, and a store rehydrate path (`startSession` currently empties the slice; a bind-without-empty variant is needed so replayed history is not dropped).
- Two-sided live fidelity is capped by the weaker adapter, so the "both live" experience will be inconsistent until V2 closes the gaps.

### Risks

- **Days-old ids may not reload.** If the pinned adapters do not persist their own sessions across restarts or reject a days-old id, "live across days" degrades to read-only history for that adapter. *Mitigation*: run the capability + durability probe first and set the honest default before building the picker.
- **Re-inject is the redaction danger zone.** Re-anchoring the bundle to freshly minted ids is where "nothing leaves unredacted" could silently regress into an unredacted re-send. *Mitigation*: hand-back only through the existing preview-confirm gate in V1; defer any automatic re-anchor-and-send to V2 behind an explicit redaction gate + preview + tests.
- **At-rest exposure.** *Mitigation*: `0600`/`0700` perms, redact already-detected secrets on the persist path, retention cap, delete story, first-run disclosure; OS-keychain encryption-at-rest named as the V1.1 target.

## Implementation Notes

- Capture `InitializeResponse.agentCapabilities.loadSession` in `connect()` (currently discarded) and thread it to the resume path.
- Persist source-of-truth `turns` (+ `status`, `plan`, `sessionId`, `focusedAgentId`); recompute `referencedFiles` (a `Map`, not JSON-serializable) and `pendingDiffs` via the existing `withDerived`.
- Reuse `createJsonlFileSink` + XDG-state path resolution from `src/telemetry/recorder.ts` for the run store; reuse `src/core/secretRedactor.ts` on the persist path.
- Add a store rehydrate/bind action so the slice is subscribed before `session/load` notifications arrive.
- The `Ctrl+R` picker overlay clones the `ApprovalPrompt`/`HandoffPreview` modal pattern; add a `sessionPicker` slot to `OverlayState` and include it in `selectHasOpenOverlay`.
