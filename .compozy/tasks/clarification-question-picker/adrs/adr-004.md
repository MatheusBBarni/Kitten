# ADR-004: Coordinate agent interactions in the controller with clarification priority

## Status

Accepted

## Date

2026-07-12

## Context

The controller currently owns a FIFO queue for permission requests, while the store holds a visible approval overlay. Clarification requests add a second agent-originated interaction that must remain semantically distinct from permissions, preserve session attribution, and immediately preempt an already open modal according to ADR-002.

Independent queues or UI-owned arbitration would make response ordering, stale-response guards, disposal, and resumption inconsistent. At the same time, a generic interaction abstraction must not erase distinct permission and clarification payloads or outcomes.

## Decision

Replace the permission-only pending queue with a controller-owned interaction coordinator. It owns stable request IDs, session ownership, pending order, terminal settlement, and the active interaction. Its entries are a discriminated union with separate permission and clarification payloads and outcome mappings.

A newly received clarification becomes the active interaction immediately. The coordinator suspends the previous active agent interaction without settling it, then resumes it after the clarification reaches a terminal outcome. The UI receives only the coordinator's active interaction and renders clarification with top overlay priority; existing non-agent overlays retain their state but do not receive input while clarification is active.

## Alternatives Considered

### Alternative 1: Separate controller queues

- **Description**: Add an independent clarification queue beside the current permission queue.
- **Pros**: Smaller change to the existing permission code.
- **Cons**: Requires duplicated lifecycle guards and ad hoc precedence/resumption rules.
- **Why rejected**: Preemption cannot be made deterministic without a single arbitration owner.

### Alternative 2: UI-owned modal stack

- **Description**: Let overlay components decide visibility, priority, and resumption.
- **Pros**: Keeps controller code smaller.
- **Cons**: Moves agent callback lifecycle into the UI and risks response routing, hidden pending promises, and inconsistent cleanup.
- **Why rejected**: The controller is the correct owner of blocked agent callbacks.

### Alternative 3: Queue clarification behind the current modal

- **Description**: Preserve first-in-first-out display order.
- **Pros**: Simplest ordering model.
- **Cons**: Contradicts the immediate clarification behavior selected in ADR-002.
- **Why rejected**: The product requires clarification preemption.

## Consequences

### Positive

- Provides one deterministic owner for request identity, preemption, resumption, and terminal settlement.
- Reuses proven permission lifecycle safeguards without conflating permission and clarification semantics.
- Keeps ACP callbacks blocked only until the matching user interaction resolves.

### Negative

- Expands controller test surface and requires migration of the existing permission queue.
- Preemption policy must be maintained deliberately as new modal types are added.

### Risks

- **Stale or duplicate submission**: compare the active request ID before settlement.
- **Lost suspended interaction**: preserve it in the coordinator and resume only after the active clarification settles.
- **Keyboard leakage from paused overlays**: gate modal input on the active overlay priority.

## Implementation Notes

Keep the coordinator in `src/app/controller.ts` with protocol-free entry types. Store and UI state remain projections of the active interaction; they do not own resolver promises. Add a dedicated clarification overlay and selector rather than reusing approval terminology.
