# ADR-002: Present supported clarification requests as immediate session-attributed dialogs

## Status

Accepted

## Date

2026-07-12

## Context

Kitten users can work across multiple live agent sessions. A supported agent may need a user decision before it can continue. The product must resolve that decision quickly and clearly, without letting a waiting agent appear stalled or making the user compose an ambiguous reply.

Research shows that developer tools increasingly ask clarifying questions before consequential work and that keyboard-first, structured choices are an expected interaction. The user chose immediate handling for every supported structured request, broad eligibility within that boundary, and a return to work immediately after submission. Fast unblocking is the governing success outcome.

## Decision

For every structured clarification request from a verified supported provider, Kitten will immediately present a session-attributed dialog that requires an explicit answer or a terminal cancellation outcome before normal cockpit input resumes. The dialog will offer the agent-provided options, support one or many compatible selections, and provide a text response path when predefined choices do not fit.

After an answer is submitted or the user explicitly declines through terminal cancellation, Kitten will close the dialog, return the user to their prior work, and provide the waiting agent with a clear outcome. V1 does not create a persistent decision-history experience.

## Alternatives Considered

### Alternative 1: Show a passive waiting indicator for background sessions

- **Description**: Leave the user’s current work uninterrupted and mark the requesting session as needing attention.
- **Pros**: Minimizes interruption and lets the user control timing.
- **Cons**: Delays an agent that cannot proceed and risks the request remaining unnoticed.
- **Why rejected**: The user prioritized fast unblocking over deferred attention.

### Alternative 2: Send a notification and let the user open the question later

- **Description**: Notify the user about the request without opening a dialog.
- **Pros**: Reduces focus disruption and works well for long-running background tasks.
- **Cons**: Adds a separate attention-management step and makes response timing less predictable.
- **Why rejected**: It does not meet the chosen immediate-interruption behavior.

### Alternative 3: Restrict pickers to selected classes of questions

- **Description**: Only show a picker for high-impact or user-configured question categories.
- **Pros**: Limits interruption volume.
- **Cons**: Makes behavior harder to predict and can leave valid structured requests in plain chat.
- **Why rejected**: The user chose consistent handling for all supported structured requests.

## Consequences

### Positive

- Gives users an unambiguous, fast path to unblock agents.
- Makes the requesting agent and decision context visible at the moment of need.
- Creates predictable behavior for every supported structured request.
- Preserves user agency through selection, text input, and explicit cancellation.

### Negative

- Immediate dialogs can interrupt a user who is focused on another session.
- The V1 experience does not retain a browsable decision history after submission.
- Provider support determines where the feature is available.

### Risks

- **Excessive interruption**: monitor cancellation and response latency, and encourage agents to use structured requests only for genuine decisions.
- **Poor option fit**: retain a text path and measure option-fit satisfaction in user research.
- **Unsupported provider expectations**: clearly state capability availability before the feature is used.
- **Privacy concerns**: keep measurement opt-in, local, and content-free.

## Implementation Notes

The user experience must make the requesting session, question context, available choices, text path, and explicit cancellation clear. The PRD must preserve the immediate-display rule without prescribing the underlying implementation.
