# ADR-005: Terminally cancel pending clarification on session loss

## Status

Accepted

## Date

2026-07-12

## Context

A clarification callback blocks a specific live agent session until it receives an explicit answer or terminal cancellation. Session disconnect, restart, controller disposal, or restoration breaks that callback's identity. Replaying a saved response into a new session could send a decision to the wrong agent context.

The PRD requires a clear terminal cancellation outcome and does not include a persistent pending-question or decision-history experience in V1.

## Decision

When the requesting connection or session is lost, restarted, disposed, or replaced, the interaction coordinator resolves every matching pending clarification with terminal cancellation, removes its visible overlay, and does not restore it. A restarted agent must issue a new structured clarification request if it still needs input.

The same terminal cancellation policy applies when the user presses Escape. The adapter maps cancellation to the ACP response only while the original callback remains live.

## Alternatives Considered

### Alternative 1: Persist and replay after session restoration

- **Description**: Store pending clarifications locally and reconnect them to a restored session.
- **Pros**: Avoids asking the user again.
- **Cons**: Cannot prove the restored session is waiting on the same request or accepts the same response.
- **Why rejected**: Risks misrouting a user decision into a different agent context.

### Alternative 2: Show an expired request without resolving it

- **Description**: Retain a read-only dialog after the session is lost.
- **Pros**: Preserves visible history.
- **Cons**: Leaves the agent callback unresolved and conflicts with V1's no-history boundary.
- **Why rejected**: The user chose a terminal outcome, not a deferred workflow.

## Consequences

### Positive

- Prevents stale answers and orphaned callbacks.
- Gives users and agents one clear outcome on session loss.
- Keeps V1 free of persistence and recovery complexity.

### Negative

- Agents may need to ask the same question again after recovery.
- A user can lose an in-progress text answer when the requesting session disconnects.

### Risks

- **Unexpected cancellation during transient failure**: make the terminal result visible at the moment it occurs and rely on the agent to re-request if necessary.
- **Disposal regression**: cover active and suspended entries in controller cleanup tests.

## Implementation Notes

Bind each interaction to Kitten's live session ID and original connection generation. Never resolve a new session with an outcome captured for an old generation.
