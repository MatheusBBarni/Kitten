# ADR-003: Extend the existing statusline contract with a selector-validated CONTEXT field

## Status

Accepted

## Date

2026-07-20

## Context

The Statusline Context Headroom Field PRD requires a focused-session **CONTEXT** item that omits itself for invalid or unavailable values, persists only its identifier, previews against the dialog's captured session, and does not create new telemetry or storage.

Kitten already has a pure statusline contract, one selector that derives remaining headroom from in-memory session usage, an identifier-only proposal flow, and shared preview/footer rendering. The current headroom selector handles missing usage and non-positive capacity, but it can expose malformed numeric inputs as an invalid percentage. V1 needs one honest validity boundary rather than separate behavior between the legacy footer and custom layouts.

## Decision

Extend the existing statusline field allowlist and renderer with **CONTEXT**. Add an optional **contextHeadroom: number | null** value to the existing statusline context read model. The pure renderer formats only a valid value as **ctx <remaining>%** and otherwise omits the field under its normal omission and trailing-field budget rules.

Harden the existing focused-session headroom selector to return **null** unless usage values are finite, capacity is positive, and the rounded remaining percentage is within 0 through 100 inclusive. The custom footer and statusline preview consume that selector for their respective existing session ownership models: current focus for the footer and captured target for the preview.

Persist only the literal **CONTEXT** identifier in the existing layout line. Do not add new store state, configuration keys, usage events, telemetry, ACP translation, or resolved-value persistence.

## Alternatives Considered

### Alternative 1: Validate only inside the CONTEXT formatter

- **Description**: Leave the selector permissive and make the custom renderer reject malformed values.
- **Pros**: Limits changed behavior to the new field.
- **Cons**: Creates two definitions of valid headroom and leaves legacy consumers able to render malformed data.
- **Why rejected**: A shared selector establishes one honest read model for all present and future consumers.

### Alternative 2: Persist the latest resolved percentage

- **Description**: Store a last-known context value in the saved statusline configuration.
- **Pros**: Keeps a number visible while live usage is unavailable.
- **Cons**: Makes stale data look current and violates the PRD's omission-first and identifier-only boundaries.
- **Why rejected**: Current session state is the only authoritative display source.

### Alternative 3: Build a dedicated context-status subsystem

- **Description**: Add a new store slice, formatter, and configuration model for context management.
- **Pros**: Could support future alerts and coordination features.
- **Cons**: Duplicates existing statusline and session-usage seams, broadens scope, and encourages policy features outside V1.
- **Why rejected**: The existing closed statusline contract is sufficient for the requested field.

## Consequences

### Positive

- One validated session-headroom projection serves both legacy and custom status displays.
- The proposal, preview, runtime rendering, and persistence paths remain schema-native and aligned.
- The diff stays small and avoids new state, wire, telemetry, or configuration surfaces.

### Negative

- Malformed legacy usage values now become unknown rather than producing an out-of-range display.
- The preview intentionally reflects its captured session and may differ from a globally refocused footer while the dialog remains open.

### Risks

- **A caller bypasses the selector and supplies an invalid value** — keep pure renderer validation defensive and cover invalid values in direct tests.
- **Preview and saved footer drift** — derive both contexts from the same field formatting and canonical renderer, with their documented ownership difference.
- **Scope expands into freshness or handoff policy** — preserve identifier-only persistence and do not add threshold, action, or telemetry seams.

## Implementation Notes

- Modify existing files only: the pure statusline contract, headroom selector, proposal instruction, custom footer, preview context, and their colocated tests.
- Treat valid 0% and 100% as renderable values; treat missing, non-finite, non-positive-capacity, and out-of-range results as null.
- Keep the existing layout-null branch unchanged so the legacy footer remains intact when no custom layout is saved.
