# ADR-001: Keep CONTEXT as a local, optional, field-only headroom indicator

## Status

Accepted

## Date

2026-07-20

## Context

Kitten issue #29 proposes `CONTEXT` as an additional selectable `/statusline` field. The value should help operators of long-running coding-agent sessions time voluntary handoffs by rendering the selected session's existing remaining-context headroom as `ctx <remaining>%`.

The product already has a strict declarative statusline layout, a canonical renderer with trailing-field truncation, and a content-free usage path: `usage_update` flows into `SessionState.usage`, then `selectSessionHeadroom`. The field must not create a new usage source, persist resolved values, change the legacy footer, or expand into new session or handoff behavior.

The council agreed that this is a low-cost, reversible visibility improvement, but identified a material trust risk: an apparently precise percentage can be delayed, provider-dependent, or semantically misunderstood. The correct response for V1 is to prove the existing focused-session signal is accurate enough for presentation, rather than adding warnings, provenance chrome, or automation before the field's value is validated.

## Decision

V1 adds only `CONTEXT` to the existing closed statusline field allowlist. It renders `ctx <remaining>%` from the selected session's existing content-free headroom when that value is valid and available; otherwise it omits the field and its separator. The same strict validator, proposal parser, renderer, and grapheme-budget trailing-field behavior used by every other statusline item remain authoritative.

The agent-facing `/statusline` instruction may name `CONTEXT` as a supported identifier but must never include a resolved session value. A saved custom layout can opt into the field; no saved custom layout preserves the legacy footer exactly.

V1 excludes freshness/provenance UI, thresholds, warnings, handoff recommendations or automation, new telemetry, persistence of resolved usage, new ACP usage collection, and all session-isolation changes. Correct selected-session ownership, invalid/unavailable omission, and current headroom semantics are release prerequisites.

## Alternatives Considered

### Alternative 1: Add freshness labels, thresholds, or low-context guidance

- **Description**: Pair the percentage with an age/provenance indicator, warnings, or passive guidance when context becomes low.
- **Pros**: Could reduce the chance that a user treats an estimate as a precise countdown.
- **Cons**: Adds UI semantics and policy before demand for the base signal is established; diverges from the user-approved field-only V1.
- **Why rejected**: Preserve the smallest credible learning probe. Revisit only if user feedback demonstrates confusing or unreliable headroom behavior.

### Alternative 2: Keep context available only on demand

- **Description**: Document the existing footer or add an occasional `/context`-style readout instead of a selectable layout field.
- **Pros**: Avoids persistent statusline chrome and reduces apparent precision.
- **Cons**: Adds a discoverability and interaction cost at the moment a user needs to assess an active long-running session.
- **Why rejected**: An optional field is a smaller reuse of the existing statusline seam and can be removed from a layout without adding a second interaction path.

### Alternative 3: Automate context-aware handoffs

- **Description**: Introduce thresholds, recommendations, or automatic handoff behavior based on remaining headroom.
- **Pros**: Could turn a passive cue into a more forceful workflow intervention.
- **Cons**: Requires provider-normalized semantics, reliability guarantees, policy design, lifecycle integration, and substantially broader user control.
- **Why rejected**: It makes the product depend on an approximate signal before that signal has proved useful for voluntary decisions.

## Consequences

### Positive

- Long-running-session users can place an at-a-glance handoff cue in a layout they already control.
- The feature reuses a content-free, selected-session projection and avoids new telemetry or storage.
- The canonical statusline renderer keeps preview and runtime behavior aligned, including narrow-terminal truncation.

### Negative

- The field cannot communicate freshness or provider-specific semantics in V1.
- Its value depends on the quality and cadence of existing usage updates.
- It remains an informational hint, not a reliable capacity or handoff guarantee.

### Risks

- **Stale or misowned session data**: prove selected-session ownership and conservative unavailable/invalid omission in focused tests.
- **Over-trust of a percentage**: use neutral `ctx <remaining>%` copy and omit urgency cues, thresholds, and automation.
- **Narrow terminal layouts**: prove that the existing trailing-field grapheme budget drops `CONTEXT` deterministically when required.
- **Scope creep**: retain the closed field allowlist and exclude new usage collection, telemetry, persistence, and workflow behavior.

## Implementation Notes

- `CONTEXT` is an identifier in the declarative layout, not a persisted runtime value.
- Resolve headroom only for the focused session through the existing `selectSessionHeadroom` projection.
- Test parser/validator acceptance, agent-proposal identifier-only contract, preview and saved-layout rendering, unavailable/invalid omission, selected-session changes, narrow-width truncation, and unchanged legacy-footer behavior.
