# ADR-002: Center V1 on a Verified Two-Agent Handoff

## Status

Accepted

## Date

2026-07-11

## Context

The Kitten showcase site must turn evaluation into a verified installation attempt while the project becomes public and open source. The current codebase supports a concrete, reviewable Claude Code-to-Codex handoff: Kitten assembles bounded recent context, touched files, and pending diffs; the developer can review, edit, trim, confirm, or cancel before anything is sent.

The working branch contains broader capabilities that are not reflected in the public two-agent story. The repository is still private and lacks a public license, tested public release route, and authentic handoff recording. Promising more than the released experience, or publishing the page before a real installation route exists, would undermine the product's control-and-trust positioning.

## Decision

V1 will market only the released, reviewable Claude Code ↔ Codex handoff.

The primary visitor action is a verified installation attempt. The site may become public only when all of the following are true:

1. The repository is public and has an explicit open-source license.
2. The promoted installation route is tested and accurate.
3. A real recording proves the prepare → review/trim → confirm → continue handoff flow.
4. Product claims match released behavior and do not imply complete context transfer, automatic sending, or guaranteed secret removal.

The page will make the review, trimming, explicit confirmation, and redaction boundary central to the proof. It will use a secondary GitHub control with the accurate public star count. It will collect only aggregate, privacy-preserving install and proof-engagement signals, supplemented by GitHub and release metrics.

## Alternatives Considered

### Alternative 1: Market the Broader Multi-Session Cockpit

- **Description**: Lead with working-branch multi-session, target-selection, and related broader capabilities.
- **Pros**: Sounds more expansive and may appeal to power users.
- **Cons**: Creates a mismatch with the public product story, complicates the message, and risks promising unreleased behavior.
- **Why rejected**: The verified two-agent handoff is differentiated, understandable, and appropriate for an initial public launch.

### Alternative 2: Publish the Page Before Launch Readiness

- **Description**: Use the site to collect interest before the repository, license, installation route, and recording are ready.
- **Pros**: Starts public awareness sooner.
- **Cons**: Sends users to an incomplete experience and prevents meaningful measurement of installation intent.
- **Why rejected**: The primary business goal is verified installation, so the site and launch prerequisites must be available together.

### Alternative 3: Make GitHub Exploration or the Recording the Primary CTA

- **Description**: Ask visitors to inspect the repository or watch the recording before offering installation.
- **Pros**: Reduces early commitment and may suit skeptical developers.
- **Cons**: Adds friction to the stated conversion goal and treats proof as a destination rather than support for the install decision.
- **Why rejected**: The recording and repository remain important evidence, but a verified installation route is the highest-value primary action.

## Consequences

### Positive

- The public message matches one demonstrated, differentiated workflow.
- The handoff recording explains the trust boundary better than generic feature claims.
- Visitors receive a direct, truthful route from evaluation to installation.
- Measurement can validate the conversion hypothesis without surveillance-style tracking.

### Negative

- The page cannot launch until public-release prerequisites are complete.
- Broader working-branch capabilities receive no launch-page promotion.
- A low initial star count remains a possible negative social-proof signal.

### Risks

- **Unclear installation**: Verify every displayed path immediately before launch and remove unavailable alternatives.
- **Overclaiming safety or completeness**: Review all copy against released behavior and use the recording as evidence.
- **Low public trust**: Keep the star count secondary and lead with transparent product proof.
- **Measurement overreach**: Restrict signals to documented aggregate events and avoid user-level or third-party tracking.
- **Weak demonstration**: Use a stable, real task scenario that visibly includes review, trimming, confirmation, and continuation.

## Implementation Notes

The detailed site design, publishing configuration, data retrieval behavior, and measurement mechanism belong in the TechSpec. This decision constrains only the user-facing promise, required launch conditions, and measurement boundary.
