# ADR-001: Build a Focused Proof-Led Astro Showcase

## Status

Accepted

## Date

2026-07-10

## Context

Kitten needs a public website that explains why an individual Claude Code or Codex user should try the product. The requested V1 is a focused Astro landing page on GitHub Pages with installation as the primary action and a GitHub star count as secondary community proof.

The product's verified differentiator is not generic multi-agent orchestration. Kitten keeps agent sessions live and lets the developer hand a bounded bundle of recent transcript, touched files, and pending diffs to another agent. The developer previews, edits, trims, confirms, or cancels the bundle; nothing auto-sends, and secrets are redacted before preview.

The repository is private before launch and lacks an explicit license, tags, GitHub Releases, complete public metadata, and a finished public installer. The current truthful path runs from source with Bun. The launch also lacks a real product screenshot or recording. Public repository access, accurate GitHub data, a truthful advertised install path, and a real proof asset are dependencies of the requested experience, but they are not separate website features.

Market research shows high adoption of AI development tools alongside persistent trust concerns. Competitors already market multi-agent execution, so the landing page must prove Kitten's reviewable cross-agent handoff rather than rely on category-level claims.

## Decision

Build the original focused Astro landing page for GitHub Pages. Do not expand V1 into an interactive handoff lab, reduce it to a single launch card, or turn it into a distribution-first project.

V1 will include:

1. A precise hero promise centered on moving live work between Claude Code and Codex without blind copy-paste.
2. One authentic 20-30 second recording annotated around the three essential beats: prepare, review, and confirm.
3. A concise explanation of the bounded handoff, explicit confirmation, secret redaction, and local opt-in telemetry.
4. One verified installation route with a copyable command as the primary CTA. Only working installation methods may be advertised.
5. A secondary GitHub CTA with an accurate star count sourced from GitHub. The count will appear from launch because it is an explicit product requirement, remain visually subordinate to installation, and degrade without a fabricated value if GitHub is unavailable.
6. Minimal prerequisites and troubleshooting guidance sufficient to prevent avoidable setup confusion.
7. Privacy-preserving, first-party aggregate events for install-command copies, install CTA activations, and proof engagement. V1 will not use cookies, fingerprints, user identifiers, cross-site tracking, or third-party analytics.

The experience depends on repository visibility, an explicit open-source license, accurate public metadata, correct repository ownership in every published command, and at least one verified install route. The page must not advertise broken methods or claim unreleased branch functionality. Broader packaging and distribution improvements remain outside the website scope.

## Alternatives Considered

### Alternative 1: Interactive Handoff Lab

- **Description**: Let visitors manipulate a simulated bundle, inspect redaction, remove files, and confirm a browser-based handoff.
- **Pros**: Makes the trust boundary tangible and creates a differentiated, shareable demonstration.
- **Cons**: Adds state, accessibility, testing, and maintenance work that threatens the focused weekend scope.
- **Why rejected**: The user selected the original landing-page direction; an annotated real recording is sufficient proof for V1.

### Alternative 2: Single Launch Card

- **Description**: Reduce V1 to a hero, one annotated recording, one install command, and the GitHub star control.
- **Pros**: Ships fastest and isolates the core message-to-install hypothesis.
- **Cons**: Provides too little room for capability explanation, trust facts, prerequisites, and common evaluation questions.
- **Why rejected**: The user selected the original focused page rather than the stripped-down alternative.

### Alternative 3: Distribution-First Hybrid

- **Description**: Prioritize a scoped package or signed standalone installer and treat the page as a smaller companion to that distribution effort.
- **Pros**: Removes installation friction and improves conversion after the CTA.
- **Cons**: Expands the idea into release engineering and can delay the requested showcase.
- **Why rejected**: The user selected the original website idea. The page will advertise only a verified route, while packaging improvements remain separate follow-up work.

## Consequences

### Positive

- The page tells one differentiated, verifiable product story.
- Every primary CTA leads to a working path rather than a promise.
- The annotated recording makes the handoff's control boundary visible without interactive-demo scope.
- Public-release hygiene reinforces the product's trust positioning.
- Minimal aggregate measurement supports learning without introducing surveillance-style analytics.
- The GitHub count remains authentic and grows with the community.

### Negative

- Website launch depends on release work outside the page itself.
- A source-run installation has more friction than a package or standalone installer.
- A low launch star count may weaken rather than strengthen social proof.
- First-party measurement adds a small privacy, disclosure, and maintenance obligation.
- Producing a credible recording requires a stable demo scenario and careful capture.

### Risks

- **Broken installation**: Verify every displayed command immediately before launch and remove unavailable methods.
- **Unsupported claims**: Source copy from released behavior and avoid unreleased branch features.
- **Low-star negative signal**: Keep the count secondary, frame it as community participation, and compare conversion after traction grows.
- **GitHub availability or rate limits**: Preserve a recently verified GitHub-sourced value or omit only the number on failure; never display zero as an error fallback.
- **Privacy drift**: Limit measurement to documented aggregate events with no identifiers or third-party scripts.
- **Weak proof asset**: Use a real end-to-end handoff and label prepare, review, and confirm rather than staging abstract terminal activity.

## Implementation Notes

- V1 excludes a documentation portal, blog, roadmap, community system, competitor comparison, testimonials, interactive simulator, third-party analytics, and unsupported npm or curl installation claims.
- A scoped package or signed, versioned standalone installer is the preferred conversion improvement after launch.
- An accessible interactive handoff simulator or annotated step-through replay is a V2 proof experiment if the recording does not meet the comprehension target.
- The initial 30-day targets are 12% qualified install intent, 25 net-new GitHub stars, 8 of 10 target users accurately explaining the handoff after 30 seconds, 40% proof engagement, and no more than 20% of launch feedback concerning unclear setup.
