# ADR-004: Defer site telemetry collection until post-launch

## Status

Accepted

## Date

2026-07-12

## Context

The PRD requires privacy-conscious signals and a launch that does not rely on intrusive tracking. The current team preference is explicit, low-friction privacy handling and the first release should prioritize trust and install reliability over telemetry complexity.

## Decision

V1 of the showcase site will ship with no automatic aggregate analytics endpoint and no client beacon emission. The site will include:

- explicit install and requirements copy,
- explicit GitHub link and star control,
- no cookies, no identifiers, and no third-party behavioral scripts.

Install-intent and proof engagement will be assessed initially through:
- launch feedback,
- GitHub/release observable signals,
- and future post-launch instrumentation behind a separate change.

## Alternatives Considered

### Alternative 1: Add endpoint-backed aggregate telemetry V1

- **Description**: Emit privacy-preserving non-identifying events (install copy/CTA, proof section visits) to a hosted endpoint.
- **Pros**: Faster product analytics in the first 30 days.
- **Why rejected**: Operational overhead is disproportionate for V1 and creates a retention/privacy boundary that is not yet justified in this launch.

### Alternative 2: Use third-party analytics (e.g., GA/ Plausible / Simple Analytics)

- **Description**: Add a managed analytics script for conversion and engagement counters.
- **Pros**: Minimal implementation effort.
- **Why rejected**: Conflict with the trust-first positioning and no-cross-site-tracking requirement.

## Consequences

### Positive

- Zero new backend surface and no dependency on an observability service.
- Lower compliance risk and easier explainable privacy posture.
- Conversion hypotheses can still be validated via external signals.

### Negative

- Slower feedback loop for install-intent measurement in V1.
- No direct KPI signal source for site engagement without additional tooling.

### Risks

- Insufficient launch data for optimization.
- Mitigation: treat post-launch milestone as explicit follow-up, gated by privacy review and install-route confidence.

## Implementation Notes

Do not include scripts that transmit behavioral events. Keep event-specific KPIs in scope as explicit post-launch follow-up requirements in task planning.
