# ADR-002: V1 product scope - reliable release pipeline plus a self-describing install

## Status

Accepted

## Date

2026-07-09

## Context

ADR-001 fixed the technical scope of the automated release train (release-please cut, npm-native binary via `optionalDependencies`, atomic token-less publish with OIDC provenance, PR-title enforcement, slug fix).
This ADR fixes the *product* scope: how complete the user-facing release experience should be in V1.

Codebase research through a user-experience lens surfaced gaps the pipeline alone does not close.
The README documents only the from-source flow (`bun install` / `bun start`) and never mentions the shipped channels the feature is built around.
kitten has no `--version` and no `--help`; the only flag is `--self-check`, and `package.json` is `0.0.0` - even the version kitten reports to agents over ACP is a hardcoded `0.0.0`.
So a user cannot learn which kitten they are running, which is incoherent under a feature whose whole point is versioned releases.

Market research on install/release *experience* established the conventions a developer CLI is judged against: a one-line install in the README hero, feedback that never goes silent (progress plus an explicit version-stamped success line), `--version`/`--help` that always work, changelogs written for humans (auto-drafted then curated, grouped Breaking/Features/Fixes, SemVer stated), and - increasingly - a proactive update notifier and an in-tool "what changed" view (the Claude Code benchmark).

The product owner, through clarifying questions, chose: add `--version`/`--help` with the real version wired through; do not add a proactive update notifier (knowing your version plus upgrade docs is enough for V1); and set a curated-before-merge quality bar for release notes.
Three approaches were considered against those answers.

## Decision

Adopt Approach A: ship the full council-scoped pipeline plus the user-facing completeness that makes the release feature coherent from the user's point of view, and no more.

In scope for V1, beyond the ADR-001 pipeline:

1. **A self-describing binary.**
   Add `--version` and `--help`, and wire the real release version through them and through the `clientInfo` kitten sends over ACP, replacing the hardcoded `0.0.0`.
2. **Honest, channel-accurate docs.**
   Rewrite the README to lead with the shipped install channels - `npx kitten` as the hero, `npm i -g kitten` for daily use, the checksummed curl one-liner for the no-Node case - over the from-source flow, and document the real requirements including the git-repo launch gate.
3. **A trustworthy install moment.**
   Each install path ends with an explicit, version-stamped success confirmation rather than silence, with a size expectation set for the heavier `npx` download.
4. **Human-readable release notes.**
   release-please auto-drafts the changelog into the release PR grouped into Breaking/Features/Fixes; the maintainer curates it before merge, and it publishes verbatim to the GitHub Release.

Explicitly excluded from V1 (deferred): a proactive update notifier, an in-app version surface inside the running TUI, and an in-tool release-notes/changelog view.

## Alternatives Considered

### Alternative 1: Approach B - pipeline-first, polish as a fast-follow

- **Description**: Ship the pipeline plus the slug fix now; defer `--version`/`--help`, the README rewrite, and notes curation to a follow-up.
- **Pros**: Fastest path to "it ships"; smallest first increment.
- **Cons**: Leaves a window where kitten publishes versioned releases users cannot introspect and the docs still point at `bun install`; a weaker first impression at exactly the moment new users arrive.
- **Why rejected**: It walks back the owner's explicit choices (version surface, curated notes) and ships a versioning feature whose users cannot see a version.

### Alternative 2: Approach C - experience-maximal

- **Description**: Approach A plus a proactive update notifier, an in-app version surface, and an in-tool `/release-notes` view.
- **Pros**: Closest to the gold standard (Claude Code); fully closes the upgrade-awareness loop.
- **Cons**: The notifier and in-app surfaces are net-new subsystems (background version check, TTY/opt-out handling, TUI work) that the owner deferred; over-scoped for a pre-1.0, zero-user project.
- **Why rejected**: It re-adds precisely the pieces set aside for V2 and inflates V1 past the point of coherence.

## Consequences

### Positive

- The release feature is complete from the user's point of view: they can install on any Node machine, see a version-stamped success line, ask `kitten --version`, and read human-readable notes.
- Docs match reality, so the first command a new user runs works.
- Scope stays minimal: no background update checks, no new in-TUI surfaces.

### Negative

- More than raw plumbing: V1 also touches the README and adds CLI flags and version wiring.
- Users still are not proactively told when they are outdated; they must check or read docs.

### Risks

- **Version drift** between `package.json`, the compiled binary's reported version, and the ACP `clientInfo` - mitigated by wiring all three to the single release-please version at build time.
- **Docs regressing to stale install commands** - mitigated by a CI check that the README's literal install commands resolve.
- **Curation skipped under time pressure**, degrading notes to raw commit logs - mitigated by the release PR being the existing merge gate where curation naturally happens.

## Implementation Notes

- `--help` should lead with examples and the install/upgrade commands matched to how the user installed.
- The install success line and `--version` must report the same string the release published.
- Keep the deferred items (notifier, in-app version, in-tool changelog) named in Non-Goals so they are a conscious V2 slice, not an oversight.
