# ADR-004: Use strict config, canonical commands, and bounded evidence for the experiment

## Status

Accepted

## Date

2026-07-15

## Context

The PRD requires a default-off experiment, keyboard-accessible history actions, and content-free evidence before default adoption. Kitten already has a strict user-config schema with default merging, a single keymap/command registry shared by slash commands and help, and an opt-in local telemetry recorder with an allowlisted record shape. Settings currently owns only theme preference, so a runtime Settings toggle would expand a separate UI contract.

## Decision

Add `transcriptWindowingEnabled` as an optional strict user-config boolean that resolves to `false` by default. When enabled, expose history reveal and return-to-live through the focusable history marker and registered slash commands; do not add a global chord or Settings tab in V1. Prove behavior with deterministic core, store, and real-renderer fixtures, and add only fixed, content-free telemetry buckets for projection dimensions and duration under the existing telemetry opt-in.

## Alternatives Considered

### Alternative 1: Add a runtime Settings toggle

- **Description**: Add a new settings control for the experiment.
- **Pros**: More visible discovery and immediate switching.
- **Cons**: Expands the single-tab Settings contract and creates persistence/UI work unrelated to the projection hypothesis.
- **Why rejected**: A strict config boolean is the lowest-scope default-off experiment surface.

### Alternative 2: Use an environment-only flag

- **Description**: Gate the feature only through process environment.
- **Pros**: Minimal product configuration code.
- **Cons**: Less discoverable, less durable, and inconsistent with user-facing Kitten preferences.
- **Why rejected**: The experiment needs an intentional, validated user configuration path.

### Alternative 3: Use marker-only controls and manual testing

- **Description**: Omit slash commands and rely on ad hoc human evaluation.
- **Pros**: Fewer command and test changes.
- **Cons**: Weak keyboard discoverability and no repeatable evidence for default adoption.
- **Why rejected**: The PRD requires command/help discovery and measurable safety/performance evidence.

## Consequences

### Positive

- Keeps opt-in behavior explicit, validated, and disabled by default.
- Makes all history commands discoverable through the existing source of truth.
- Preserves the recorder's content-free privacy boundary while producing comparable evidence.

### Negative

- Early adopters edit configuration rather than toggling an in-app preference.
- Adds command, config-fixture, telemetry-allowlist, and test maintenance.

### Risks

- **Undiscoverable experiment**: document the config field and list slash commands in help.
- **Telemetry scope creep**: use only closed enums and coarse count/duration buckets; reject free-form fields.
- **Misleading metrics**: require deterministic 1,000-turn fixtures in addition to opt-in runtime measurements.

## Implementation Notes

Extend existing config default/merge/write tests and the central `CockpitCommand` registry. Add no environment-only override, Settings tab, global key chord, transcript text field, raw ID, path, or free-form telemetry attribute.
