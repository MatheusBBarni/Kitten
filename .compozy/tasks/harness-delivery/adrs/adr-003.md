# ADR-003: Own delivery state by controller generation and persist only a content-free checkpoint

## Status

Accepted

## Date

2026-07-14

## Context

The PRD requires a fresh or replacement conversation to receive the versioned harness exactly once, a successfully loaded conversation to receive none, and an uncertain first delivery to stop rather than silently continue. The existing controller already owns the live ACP session identity and increments a generation when it creates or replaces a connection. The visible `SessionId` can span multiple ACP sessions, so it cannot be the delivery identity on its own.

The controller must also survive restart safely. Live-only state would lose an ambiguous first-delivery outcome and could allow a restored conversation to look healthy despite an unknown prior submission. Persisting the harness or user request to solve this would violate the PRD's content boundary.

## Decision

Create a controller-private, protocol-free delivery machine keyed by `{ SessionId, generation, acpSessionId }`. It owns `not_required`, `pending`, `in_flight`, `delivered`, and terminal `failed` states. A successful `session/load` starts `not_required`; every fresh creation path starts `pending`; a replacement always creates a new state instance.

Persist only a V3 `harnessDelivery` checkpoint containing the harness version, generation-relative state, and fixed failure category. It contains no rendered harness, user blocks, ACP session ID, path, raw error, or reversible digest. On startup, a successful provider load remains `not_required`; an unresolved or failed checkpoint produces the content-free recovery state rather than reinjecting into an existing provider conversation.

Mark `delivered` only after the installed ACP SDK's terminal `prompt()` request resolves. Before invoking that request, a known failure can remain `pending` for a controlled retry. Once invocation may have started, any throw, timeout, cancellation, close, or disposal becomes terminal indeterminate failure for that generation and cannot automatically resend the first user turn.

New harness versions apply only to future fresh generations. Loaded conversations retain their existing context and never receive a mid-conversation version update.

## Alternatives Considered

### Alternative 1: Track delivery by visible Kitten `SessionId`

- **Description**: Store one `delivered` flag per visible session.
- **Pros**: Minimal state and simple lookup.
- **Cons**: Replacement and fallback sessions inherit stale truth from a different ACP connection.
- **Why rejected**: The controller generation is the real lifecycle boundary.

### Alternative 2: Keep delivery state only in memory

- **Description**: Derive state from each new process's creation or load operations.
- **Pros**: No persistence migration.
- **Cons**: Loses unresolved-delivery safety after restart and cannot distinguish a safe loaded continuation from an indeterminate prior first turn.
- **Why rejected**: It risks silent continuation after the failure mode the PRD explicitly forbids.

### Alternative 3: Persist request and harness payloads for replay

- **Description**: Save the exact first-turn material and raw errors to enable replay.
- **Pros**: More forensic detail and possible idempotent replay.
- **Cons**: Violates content-free storage and expands the privacy surface.
- **Why rejected**: A fixed checkpoint plus fail-closed recovery contains ambiguity without retaining content.

## Consequences

### Positive

- Replacement cannot inherit a delivered result from an old connection.
- Restart behavior remains conservative without storing prompt content.
- The truth claim matches the installed SDK: terminal request resolution, not inferred provider consumption.

### Negative

- The persistence schema requires a versioned migration.
- An indeterminate first turn requires user recovery rather than automatic convenience.

### Risks

- **Stale asynchronous completion** — generation-check state transitions after every await.
- **Old saved runs** — treat a successful provider load as an existing continuation and surface recovery only when the checkpoint says unresolved or failed.
- **Overbroad checkpoint fields** — validate fixed enums and reject arbitrary strings or content.

## Implementation Notes

Keep the transition helper beside the controller in `src/app/`; it must remain protocol-free and independently unit-testable. The UI receives only a derived content-free notice; it does not become the source of delivery truth. Use a new persisted record version rather than overloading the existing V2 schema.
