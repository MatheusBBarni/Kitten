# ADR-001: Scope harness delivery by live ACP session generation

## Status

Accepted

## Date

2026-07-14

## Context

Kitten will receive a versioned, protocol-free base harness from issue #18. Issue #19 must deliver that harness for genuinely fresh ACP sessions without putting it in the visible transcript, persisted user turns, handoff bundles, composer history, or content-bearing diagnostics.

The visible Kitten `SessionId` is insufficient to identify a delivery opportunity: restore can load an existing provider session, failed load can create a fresh replacement, and connection replacement advances a controller generation. ACP provides neither a portable hidden host-instruction field nor an early prompt-acceptance acknowledgement. A network or provider failure after submission is therefore ambiguous and cannot be retried safely as if nothing happened.

The user selected a complete lifecycle V1 for daily Kitten users and an actionable fail-closed outcome when harness attachment cannot be established.

## Decision

Define harness delivery as a controller-owned, content-free state machine scoped to the pair of live ACP session identity and monotonic controller generation. The controller, rather than the core, store, UI, renderer, or ACP adapter, decides whether the current generation is `not_required`, `pending`, `in_flight`, `delivered`, or `failed` with a fixed failure category.

Every path that creates a genuinely fresh generation starts `pending`: normal `session/new`, failed-load fallback, fresh-run replacement, handoff-first target, and recovery after a closed or crashed connection. A successful `session/load` is `not_required` and never receives another harness. The controller atomically enters `in_flight` before dispatching the first real user prompt, passes the original user blocks unchanged to transcript/persistence pathways, and supplies a separate protocol-free delivery envelope to the adapter. The adapter alone applies an explicitly verified runtime-profile encoding.

For V1, `delivered` means exactly one harness-bearing prompt submission resolved successfully through the selected transport for that generation; it does not claim that the provider consumed or followed the harness. A conclusively pre-dispatch failure may retry without creating a transport submission. Any outcome after dispatch may have started, including timeout, partial output, cancel, close, crash, or provider error, becomes a fixed `failed`/indeterminate state: Kitten must not automatically submit the first user turn again and must show an actionable, content-free degraded status. Replacing the generation creates a new, independent delivery opportunity.

Persist only the minimum content-free control facts necessary to prevent silent unsafe continuation after restoration: harness version, generation-relative delivery state, and fixed failure category. Never persist the rendered harness, user blocks, paths, adapter command lines, raw errors, or a reversible prompt-derived digest.

## Alternatives Considered

### Alternative 1: Inject only for the normal fresh-session path

- **Description**: Attach the harness on the initial `session/new` path and defer restore fallback, replacement, handoff-first, cancellation, and crash behavior.
- **Pros**: Small initial change; faster validation of a happy path.
- **Cons**: Leaves lifecycle transitions inconsistent and risks later retrofitting stale or duplicate delivery behavior.
- **Why rejected**: The controller already has explicit creation and replacement seams. A partial policy would establish the wrong ownership boundary for the selected full-lifecycle V1.

### Alternative 2: Let the ACP adapter track and prepend the harness

- **Description**: Have each adapter remember whether it has attached a harness and prepend it directly to prompt blocks.
- **Pros**: Localizes provider-specific encoding.
- **Cons**: The adapter cannot distinguish successful load from fresh fallback reliably, races with replacement, or protect Kitten transcript/persistence boundaries by itself.
- **Why rejected**: It conflates controller lifecycle truth with protocol conversion and would leak policy into provider-specific code.

### Alternative 3: Retry every failed first turn or send it without the harness

- **Description**: Treat all prompt failures as retryable, or continue with the original user prompt when harness delivery fails.
- **Pros**: Preserves apparent availability.
- **Cons**: Can duplicate a user turn after partial provider processing or silently create an unharnessed session.
- **Why rejected**: It violates the selected fail-closed behavior and makes the exactly-once claim misleading.

### Alternative 4: Persist full prompt and harness material to make retries idempotent

- **Description**: Save rendered content, request payloads, and raw errors for exact restart-time replay.
- **Pros**: More evidence for reconciliation across a host-process crash.
- **Cons**: Enlarges the privacy surface, adds a second source of truth, and conflicts with the content-free persistence contract.
- **Why rejected**: V1 can safely contain ambiguity by failing closed; full payload persistence is disproportionate and unsafe.

## Consequences

### Positive

- One lifecycle authority prevents stale `delivered` state from crossing replacements.
- Visible conversation artifacts retain only intentional user-visible content.
- Adapter profiles remain replaceable while the domain and store stay ACP-free.
- The contract is honest about observable transport submission versus provider behavior.

### Negative

- Some otherwise usable sessions are stopped when attachment is ambiguous or unsupported.
- Full lifecycle coverage requires a substantial matrix of deterministic tests.
- A future ACP acknowledgement feature may justify refining the `delivered` definition.

### Risks

- **Adapter incorrectly claims a hidden encoding is available** — require profile-level contract tests before enabling it.
- **Race between first prompt, close, and replacement** — guard each transition with the current generation and settle promises exactly once.
- **Old restored records lack delivery facts** — treat successful provider load as `not_required`; surface a fixed degraded status for indeterminate restored states rather than injecting mid-conversation.
- **Content leaks through observability** — allow only fixed fields such as version, state, lifecycle path, profile identifier, and failure category.

## Implementation Notes

`src/app/controller.ts` owns freshness and generation. `src/app/actions.ts` must continue recording original prompt blocks only; a controller dispatch seam should prepare a separate protocol-free envelope. `src/agent/` owns conversion to verified provider-specific ACP payloads. The core, store, UI, handoff assembler, persistence content, and telemetry must not receive harness text.

Required contract tests cover fresh new, successful load, failed-load fallback, follow-up, generation replacement, handoff-first, definite pre-dispatch retry, post-dispatch ambiguity, partial output, close, cancellation, crash, and sibling-session isolation.
