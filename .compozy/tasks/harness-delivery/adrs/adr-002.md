# ADR-002: Keep baseline guidance silent by default and recovery-oriented on failure

## Status

Accepted

## Date

2026-07-14

## Context

Daily Kitten users expect a new agent conversation to start with reliable baseline guidance, while a resumed conversation should remain a continuation of its existing context. Comparable coding-agent products establish durable instructions at the start of a session, but users need clear boundaries between fresh and continued work.

The approved idea and ADR-001 establish that Kitten must never silently continue a fresh conversation when it cannot safely establish the baseline. The PRD clarification round resolved the user-facing behavior: successful initialization should not add routine status noise; recovery should prioritize a safe fresh conversation; and only new or replaced conversations need an understandable guidance state.

## Decision

Make successful baseline-guidance initialization silent. A new or replaced conversation proceeds normally when Kitten establishes its baseline, and an existing restored conversation continues without a new baseline or a routine status notice.

When Kitten cannot establish the baseline for a new or replaced conversation, it must stop that initial task, keep the user's intent recoverable, and show a concise, actionable explanation with a safe fresh-conversation path. The notice must say what the user can do next without revealing the hidden guidance or creating the impression that the task was sent under unknown conditions.

The V1 user experience will not show a per-turn or per-session success badge. It will expose only recovery-relevant state and will keep visible transcripts, history, handoff material, saved runs, and telemetry limited to intentional user-visible content.

## Alternatives Considered

### Alternative 1: Show a guidance-ready indicator for every new conversation

- **Description**: Display a visible success status whenever a conversation starts with baseline guidance.
- **Pros**: Makes the hidden state explicit and may reassure some users.
- **Cons**: Adds repetitive visual noise to the normal workflow and invites users to infer details from a status badge.
- **Why rejected**: The user selected a silent normal path; the product value is predictable work, not ongoing ceremony.

### Alternative 2: Continue the initial task with a warning when guidance is unavailable

- **Description**: Allow the user task to proceed without baseline guidance while showing a warning.
- **Pros**: Minimizes immediate interruption.
- **Cons**: Violates the reliability promise and leaves users uncertain which conditions governed the agent's work.
- **Why rejected**: The user selected a safe fresh-conversation recovery path rather than degraded continuation.

### Alternative 3: Require users to confirm guidance status before every fresh task

- **Description**: Ask for an explicit confirmation before sending the first task of each new conversation.
- **Pros**: Maximizes visible control.
- **Cons**: Slows the common path and duplicates the existing human-control moments that matter, such as handoff review.
- **Why rejected**: Baseline guidance is a host responsibility; users should be interrupted only when that responsibility cannot be met.

## Consequences

### Positive

- Daily users receive a calm normal workflow and an honest recovery path when needed.
- Resumed conversations retain a clear continuity promise.
- Failure notices focus on the next safe action rather than hidden implementation detail.

### Negative

- Users do not receive a routine visual proof that baseline guidance is active.
- A failed initialization interrupts the first task until the user starts a safe replacement conversation.

### Risks

- **Users misinterpret silence** — document the fresh-versus-resumed promise and make failures unambiguous.
- **Recovery costs users work** — retain the initial task for the recovery route wherever the product can do so safely.
- **Provider coverage varies** — expand eligibility only when the same user promise can be upheld.

## Implementation Notes

This ADR defines product behavior only. ADR-001 contains the lifecycle-safety decision that enables it. Failure copy, recovery affordance, and accessibility treatment must remain concise, actionable, and content-free.
