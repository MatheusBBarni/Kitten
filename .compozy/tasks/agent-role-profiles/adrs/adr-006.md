# ADR-006: Verify the Explore Contract Through Layered Tests

## Status

Accepted

## Date

2026-07-15

## Context

`explore` crosses pure policy, configuration eligibility, asynchronous controller lifecycle, internal MCP exposure, terminal UI, and content-free telemetry. A single test layer cannot prove both deterministic denials and the operator-visible contract. Existing Kitten tests already favor pure reducer tests, injectable controller seams, integration tests, and selector-driven UI checks.

## Decision

Use layered verification: pure tests for policy resolution, immutable snapshots, and atomic reservations; controller and integration tests for attestation denial, fresh restricted launch, MCP filtering, cleanup, and stale lifecycle fencing; UI tests for textual availability, denial, and accessibility; telemetry tests for allowlisted content-free fields. Run the repository typecheck, full tests, self-check, and build before declaring implementation complete.

## Alternatives Considered

### Alternative 1: Controller integration tests only

- **Description**: Exercise the feature solely through controller fakes.
- **Pros**: Covers a broad path quickly.
- **Cons**: Makes policy and race failures hard to isolate and leaves UI contract gaps.
- **Why rejected**: The safety contract needs deterministic unit evidence and operator-visible evidence.

### Alternative 2: End-to-end terminal tests only

- **Description**: Depend on a full live terminal flow for all evidence.
- **Pros**: High realism.
- **Cons**: Slow, flaky, and poorly suited to exhaustive denial and concurrency matrices.
- **Why rejected**: It cannot replace focused pure and injected-seam tests.

## Consequences

### Positive

- Each invariant has an appropriate, reproducible proof layer.
- Fail-closed behavior and content-free telemetry are tested independently of live providers.
- UI regressions cannot silently turn a safe denial into a generic failure.

### Negative

- The implementation adds tests across several existing suites.
- Browser-backed visual smoke remains supplementary to deterministic UI tests.

### Risks

- **Over-mocking provider behavior**: keep verifier fixtures versioned and include negative unknown/stale cases.
- **Incomplete direct-handler coverage**: use the same controller policy service from UI and agent-control tests.

## Implementation Notes

Place tests beside their owning layers and retain existing injectable seams. Do not add a new end-to-end framework for this feature.
