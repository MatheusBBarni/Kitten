# ADR-002: Validation-First Thin Slice for V1 (Human-Curated Hand-off, LLM Curation Behind a Gate)

## Status

Accepted

## Date

2026-07-07

## Context

ADR-001 committed Kitten's V1 to a cross-agent hand-off wedge rather than a generic multi-agent switcher, and flagged the central risk: the hand-off *feature* is unvalidated (only the *pain* is documented), and building an LLM-powered curation engine is the most model-risky, expensive part of the product.
The council was explicit that raising build risk before proving demand is the failure mode to avoid.

During PRD clarification the user made four decisions that shape the build: V1 targets Claude Code + Codex; the hand-off shows a **preview-and-edit** step before sending; V1 supports a single **continue** intent; and the two-agent experience is a **focused pane with quick switch**.

The preview-and-edit decision is pivotal.
Because a human curates the bundle in the preview, the MVP can deliver real value without an automated curation engine.
That opens a phasing choice: build the smart curation now (strongest first impression, highest risk) or defer it behind a validation gate (cheapest honest test of demand).

## Decision

V1 ships as a **validation-first thin slice**:

- The MVP is a focused two-agent cockpit (Claude Code + Codex over ACP) with a **continue** hand-off whose bundle is assembled **deterministically** from ACP's structured session data: a transcript excerpt, the referenced file set (derived from tool-call `locations`), and pending diffs.
- The **human curates** the bundle in the preview-and-edit step. The MVP does **not** include an LLM-powered curation engine (no automatic dead-end dropping, no source-agent-generated summary).
- The MVP includes opt-in instrumentation for the honest metrics (hand-off adoption, 7-day repeat use, re-explanation eliminated, curated-brief acceptance).
- **Smart curation** (source-agent summary, automatic dead-end dropping, auto file selection) and **hunk-level diff review** are deferred to Phase 2 and built only after the MVP hits the pre-registered kill/scale thresholds.
- The **second-opinion intent**, additional agents, and the parallel-agent conductor are deferred to Phase 3 and beyond.

## Alternatives Considered

### Alternative 1: Smart-curation MVP

- **Description**: Include the LLM-powered curation engine (source-agent summary, dead-end dropping, auto file selection) and hunk-level diff review in the MVP.
- **Pros**: Strongest first impression; directly tests whether a curated brief beats a human summary; the differentiator is fully present at launch.
- **Cons**: Builds the expensive, model-risky component before demand is proven; larger surface delays a flicker-free, stable first release; contradicts ADR-001's validation discipline.
- **Why rejected**: The preview-and-edit step already lets a human curate, so the engine is not required to deliver value; building it first inverts the risk order the council warned against.

### Alternative 2: Cockpit-complete MVP

- **Description**: Match toad's polish (two agents + hand-off + `@`-file picker + hunk diffs + referenced-file viewer + persistent shell) at launch.
- **Pros**: Competitive on breadth immediately; feels complete.
- **Cons**: Largest scope; competes with toad on its established strengths; slowest to reach the validation signal; dilutes the wedge.
- **Why rejected**: Scope discipline from ADR-001; the goal of V1 is to reach the kill/scale decision cheaply, not to out-feature toad.

## Consequences

### Positive

- Cheapest honest path to the demand signal that decides whether to invest in the curation engine.
- The human-in-the-loop preview doubles as a safeguard against the poisoned-transcript failure mode without any model logic.
- Small surface makes a flicker-free, stable first release realistic.
- Every deferred capability (smart curation, hunk review, second opinion, more agents, conductor) is strictly additive, so Phase 2+ requires no rewrite.

### Negative

- The auto-assembled bundle will be rougher than a curated one, so users edit more in the preview.
- A rough bundle risks under-selling the concept if the deterministic assembly is poor.
- The headline "curated brief beats a human summary" hypothesis is only tested in Phase 2, not at launch.

### Risks

- **Rough bundle undersells the wedge** - mitigate by making the deterministic assembly genuinely useful (accurate referenced-file set, clean pending diffs) and the preview fast to edit.
- **Users edit so much that the hand-off is not worth it** - this is itself a signal; instrument edit volume and treat heavy editing as evidence that Phase 2 curation is needed.
- **Threshold gaming** - pre-register the kill/scale thresholds and the metric definitions before launch so results are not rationalized after the fact.

## Implementation Notes

- Assemble the referenced file set from ACP tool-call `locations` and pending diffs from `edit`-kind tool calls, not from screen scraping.
- Keep the deterministic assembly logic isolated behind the same boundary the LLM curation engine will later occupy, so Phase 2 swaps the strategy without touching the UI.
