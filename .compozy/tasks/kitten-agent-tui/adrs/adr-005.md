# ADR-005: BYO Agents via Config-Driven ACP Subprocess Spawn

## Status

Accepted

## Date

2026-07-07

## Context

ACP local agents run as subprocesses over stdio.
The two V1 targets differ in packaging: Claude Code needs an ACP adapter wrapper to speak ACP, while other agents may speak it more directly.
Kitten needs a uniform way to launch and talk to both without owning third-party binaries or their auth.

## Decision

Kitten spawns each agent as a stdio subprocess from a **config entry** (`command`, `args`, `env`) using `Bun.spawn`, wraps the process in an ACP `ClientSideConnection`, and runs a **readiness check** at startup (spawn, `initialize` handshake, capability negotiation).
Kitten ships default config for the two known agents (Claude Code through its ACP wrapper, and Codex), while the user installs the agent CLIs and provides their own auth.
Each agent shows a clear ready / not-ready state, and a single agent failing to start degrades gracefully instead of taking down the app.

## Alternatives Considered

### Alternative 1: Kitten bundles or auto-installs the adapters

- **Description**: Vendor or auto-install the ACP adapter shims for both agents.
- **Pros**: Smoother first run.
- **Cons**: Kitten owns updating and debugging third-party adapters; couples release cycles; bloats the binary.
- **Why rejected**: Ownership and maintenance burden outweigh the onboarding gain for a validation-stage MVP.

### Alternative 2: Hybrid (BYO agents, bundled shims)

- **Description**: User provides agent CLIs and auth; Kitten bundles only the thin ACP shims.
- **Pros**: Some first-run smoothing.
- **Cons**: Still owns shim updates and version drift against the user's agent version.
- **Why rejected**: Keeps a maintenance liability without removing the BYO requirement.

## Consequences

### Positive

- Kitten stays thin and respects the user's existing installs and auth.
- One uniform spawn-and-connect path handles both native-ACP and wrapper-adapter agents.
- New agents in later phases are added by config, not code.

### Negative

- The user bears install and auth setup, which is real first-run friction.
- Kitten must detect and clearly report per-agent failures (missing binary, bad auth, wrong version).

### Risks

- An agent's ACP adapter breaks or changes flags - mitigate with the startup readiness check and a legible per-agent error, and by pinning recommended adapter versions in the default config.

## Implementation Notes

- Validate readiness by completing the `initialize` handshake, not just by spawning the process.
- Isolate all subprocess and JSON-RPC handling in the Agent Adapter Layer (ADR-003); surface only normalized readiness/state to the store.
