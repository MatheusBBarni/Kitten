# ADR-004: React Binding for the OpenTUI UI Layer

## Status

Accepted

## Date

2026-07-07

## Context

OpenTUI ships three ways to build the UI: `@opentui/core` (imperative), `@opentui/react`, and `@opentui/solid`.
The agent conversation streams tokens at high frequency, which stresses the rendering path.
Solid's fine-grained reactivity (the pairing OpenCode ships in production) is technically better suited to streaming, but the team prioritized contributor familiarity and ecosystem breadth.

## Decision

Kitten uses `@opentui/react` for the UI layer, with explicit measures to keep streaming cheap:

- An **external store with targeted subscriptions** (a scoped store such as Zustand, or a narrowly-scoped reducer) so only the streaming message view re-renders on a token chunk, not the whole tree.
- **Coalesced token updates**: buffer incoming `agent_message_chunk` updates and flush at most once per frame, so the reconciler runs on batched state, not per token.
- **Overlays via conditional, absolutely-positioned boxes** (the hand-off preview, approval prompts), since the React binding does not expose Portal.

## Alternatives Considered

### Alternative 1: Solid (@opentui/solid)

- **Description**: Build the UI with Solid's fine-grained reactivity.
- **Pros**: Best streaming performance; Portal for overlays; richest hook set; production-proven with OpenTUI in OpenCode.
- **Cons**: Smaller contributor pool than React; less familiar patterns.
- **Why rejected**: The team chose React for familiarity and ecosystem; the streaming gap is closable with a store plus update coalescing.

### Alternative 2: Core imperative (@opentui/core)

- **Description**: Manage the renderable tree and state by hand.
- **Pros**: Maximal control, no framework overhead.
- **Cons**: Slow to build a cockpit with overlays, approval prompts, and reactive status; error-prone.
- **Why rejected**: Poor fit for the interaction complexity; slows delivery.

## Consequences

### Positive

- Broad contributor base and familiar patterns.
- Large ecosystem and React DevTools support during development.

### Negative

- The team must actively manage render scope; naive `useState` on the transcript would re-render the whole conversation per token.
- Overlays need manual absolute positioning rather than Portal.

### Risks

- Uncoalesced streaming could reintroduce flicker, the exact failure mode Kitten must avoid - mitigate by making per-frame update coalescing a non-optional part of the streaming path and covering it with a snapshot test.

## Implementation Notes

- Put the streaming buffer in the adapter layer so the store receives batched updates.
- Use `useKeyboard`, `useOnResize`, and `usePaste` from `@opentui/react`; render diffs with the built-in `<diff>` component and messages with `<markdown>`.
