# ADR-001: V1 Scope - Cross-Agent Hand-off Wedge, Not a Generic Multi-Agent Switcher

## Status

Accepted

## Date

2026-07-07

## Context

Kitten was conceived as a unified terminal UI (TUI) to run multiple AI coding agents (Claude Code, OpenAI Codex, Cursor) from one keyboard-driven interface, built on the Agent Client Protocol (ACP) with Bun, TypeScript, and OpenTUI.
The original feature set was broad: text editor, file explorer, shell, and multi-agent chat.

Market research surfaced a decisive constraint: **toad** (by Will McGugan, author of Rich/Textual) already ships the exact "unified multi-agent ACP terminal frontend" concept - 12+ agents, integrated shell, `@`-file context, and an OpenHands partnership.
The single-agent multi-model tools (OpenCode, Crush, aider) are a different category, and Zed/JetBrains are multi-agent but GUI, not terminal.
"First unified ACP TUI" is therefore already taken.

A five-advisor council (Architect, Pragmatic Engineer, Product Mind, Devil's Advocate, Thinker) debated the scope.
Four of five independently converged: a generic switcher is a losing race against toad's head start, and the defensible, unsolved wedge is **cross-agent shared context / hand-off** - eliminating the top documented pain of re-explaining your project to every separate agent.
The Pragmatic Engineer resolved the "switcher vs. wedge" framing as a false split: the two-agent client is the unavoidable substrate (you cannot hand off between agents you cannot talk to), so the switcher is the floor and the hand-off is the product.
The Devil's Advocate held the essential dissent: the hand-off *feature* is unvalidated (only the *pain* is documented), a client-side layer can only reach agents "through the front door" (prompt + files), and carrying a failed agent's transcript verbatim can poison the receiving agent - so curation, not raw forwarding, is the real product.

## Decision

Kitten V1 is scoped as a **hand-off cockpit**, not a generic all-in-one workspace:

1. **One ACP client, exactly two agents** (config-driven; e.g. Claude Code + Codex).
2. **One conversation view per agent session** with ACP-native streaming and approval/diff plumbing.
3. **A one-keystroke "hand off to `<agent>`" action** that forwards a **curated, normalized context bundle**: a transcript summary (produced by the source agent, with failed/dead-end turns dropped), the referenced file set, and pending diffs carried as structured ACP objects (not pasted text).
4. **Persistent, resumable sessions on both sides** so a task can be handed off *and handed back*.
5. **Opt-in telemetry** for two honest metrics: *re-explanation eliminated* and *hand-off frequency / repeat use*.

Before over-investing in the curation engine, validate demand with a **fake-door + Wizard-of-Oz test** and **pre-registered kill thresholds** (hand-off invoked in > 40% of multi-agent sessions; > 25% of first-time users repeat within 7 days).

The differentiation bet is **permissive license + TypeScript/Bun stack** (vs. toad's AGPL-3.0 + Python/Textual) and **owning the cross-agent memory layer that ACP deliberately leaves undefined** - orthogonal to toad's agent-count breadth.

## Alternatives Considered

### Alternative 1: Generic multi-agent switcher / all-in-one cockpit

- **Description**: Ship the original vision - multi-agent chat + file explorer (+ later editor and shell) as a clean unified TUI across three agents.
- **Pros**: Crisp definition of done; demoable fast; matches the literal IDEA.md.
- **Cons**: Directly competes with toad on breadth, where toad has a structural head start; maximum surface area against two pre-1.0 dependencies with zero cohesion to defend; does not touch the top documented pain.
- **Why rejected**: A commodity ACP client is a race to the bottom of the stack that a better-resourced, more-mature competitor is already winning.

### Alternative 2: Concurrent multi-agent orchestration ("conductor")

- **Description**: Run N agents on the same task in parallel, then diff, compare, and merge their outputs.
- **Pros**: Genuinely novel; high ceiling; compounds on a shared-context spine.
- **Cons**: Much heavier; only works *after* the shared-context boundary exists; unproven demand at even higher build risk.
- **Why rejected (for V1)**: Sequenced as a V2 stretch that builds on the V1 context spine; premature as a first ship.

### Alternative 3: Strip to the curated hand-off brief only (no persistent sessions)

- **Description**: A minimal command that forwards transcript + files into the next agent's opening prompt, without live resumable sessions.
- **Pros**: Smallest possible build; fastest validation.
- **Cons**: Degrades to a shell script; cannot hand *back*; loses the structured-diff and session-state advantages that make it a product.
- **Why rejected**: Persistent resumable sessions + structured ACP diffs are exactly the parts a 40-line script cannot cheaply fake; cutting them removes the moat.

## Consequences

### Positive

- Attacks an unsolved, documented pain instead of re-fighting a battle toad has already won.
- The owned cross-agent context boundary doubles as an anti-corruption layer that insulates Kitten from ACP SDK churn and OpenTUI instability.
- Permissive license + TS/Bun stack opens contribution and embedding to the JS/TS ecosystem and removes the corporate-adoption blocker AGPL creates for toad.
- Small, reversible V1 that sequences cleanly into orchestration (V2) with no rewrite.

### Negative

- Narrower literal feature set than the original "all in one" promise (no editor/shell/file explorer in V1).
- The hand-off *feature* demand is unproven; the product could fail validation.
- Requires a curation capability (summarize + drop poisoned turns) that carries model-quality risk.

### Risks

- **Unvalidated demand** - mitigate with fake-door + Wizard-of-Oz and pre-registered kill thresholds before building the curation engine.
- **Poisoned-transcript failure mode** (the highest-value switch is often "agent A got stuck") - mitigate by forwarding a curated brief that drops failed turns and by proving the brief beats both the raw transcript and a human's two-sentence summary.
- **Feasibility ceiling** - a client-side layer can only reach agents through prompt + files, not their internal session state; design the bundle accordingly and set expectations.
- **Two pre-1.0 dependencies on the critical path** (ACP SDK at v1.2.0 with frequent breaking changes; OpenTUI at v0.4.x, 176 open issues, Zig in the build) - mitigate with a versioned anti-corruption layer around ACP and a hard render boundary around OpenTUI, isolating capability negotiation to a single seam.
- **Competitive / platform risk** - toad can relicense (McGugan owns the copyright) and Zed/JetBrains own the protocol and distribution - mitigate by owning the undefined cross-agent-memory layer (orthogonal to breadth) and shipping permissively.
- **Small, unsized TAM** (~31% of developers use agents at all; multi-agent users are a subset) - accept as a strategic bet, de-risked by cheap validation before scale.

## Implementation Notes

- Build the anti-corruption layer for the two concrete agents only; generalize when a real third agent reveals where the abstraction is wrong. Premature generality is the failure mode that turns a 3-week V1 into a 3-month one.
- Prefer the source agent's own summarization over a bespoke summarization pipeline in V1.
- Keep session state in-memory-and-resumable within a run for V1; defer durable/cross-machine persistence.
