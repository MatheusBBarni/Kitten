# ADR-003: Discover Repository Files Through an Injected Controller-Owned Git Source

## Status

Accepted

## Date

2026-07-12

## Context

The `@` selector needs a repository-relative candidate list for the focused session. Each session has its own `cwd`, while `PromptEditor` owns the transient textarea buffer and must not perform process I/O. Kitten's layering rules require UI capabilities to pass through `ControllerActions`; `src/config/gitBranch.ts` already provides the project pattern for an injected, fail-soft Bun subprocess seam.

The PRD requires ignored and outside-repository paths to stay out of the candidate list, prompt composition to remain responsive, and no persistent cross-session catalog.

## Decision

Add a repository-file source in `src/app/fileDiscovery.ts`. It receives a session working directory, resolves the Git worktree root, and lists repository-relative candidates with Git metadata. It returns a typed ready or unavailable result; expected Git, stream, and filesystem failures never throw.

Expose `ControllerActions.listRepositoryFiles(sessionId: SessionId)`. The caller captures the focused `SessionId` before starting work. The action synchronously resolves that session's `cwd` from the store, then catches every source rejection and converts it to `{ kind: "unavailable", reason: ... }`. An unknown session is unavailable. A not-ready session may still discover its configured repository; the existing readiness gate still prevents prompt submission.

Keep only the eligible path list in `PromptEditor` refs for the currently focused session. Clear it and dismiss file completion when focus changes. Do not add a watcher, store slice, disk cache, or cross-session cache. The source owns any discovery-internal optimization and never exposes binary verdicts to the UI.

## Alternatives Considered

### Alternative 1: UI-owned Git helper

- **Description**: Call Bun or a filesystem helper from `PromptEditor`.
- **Pros**: Fewer layers and direct access to the active token.
- **Cons**: Crosses the UI/I/O boundary and makes render-path tests depend on process behavior.
- **Why rejected**: It breaks the established controller action boundary and bypasses the injectable subprocess convention.

### Alternative 2: Persistent store-backed catalog

- **Description**: Keep file lists in `AppStore`, keyed by workspace.
- **Pros**: Reuses lists across focus changes and components.
- **Cons**: Adds mutable global state, invalidation policy, and cross-session persistence beyond the MVP.
- **Why rejected**: The list exists solely to serve one active composition interaction.

### Alternative 3: Filesystem walk

- **Description**: Recursively scan the session directory and maintain ignore logic in Kitten.
- **Pros**: Can include untracked files without Git.
- **Cons**: Reimplements repository policy and makes ignored-file correctness difficult.
- **Why rejected**: Git is already the repository authority and exposes the required tracked/unignored boundary.

## Consequences

### Positive

- Keeps process I/O testable and out of React.
- Scopes every result to an explicitly captured session and its own repository.
- Reuses Git's ignore semantics and repository-relative path model.
- Avoids a persistent indexing subsystem.

### Negative

- The first invocation can wait for local Git and filesystem work.
- A focus switch discards a warm list and can require another discovery call.

### Risks

- **Large repository latency**: cache only for the active focus and record content-free discovery and warm-query timing when telemetry is enabled.
- **Late async result after focus changes**: bind each request to a session/cwd generation and discard a non-current result.
- **Git unavailable or non-repository cwd**: return an unavailable result with a concise user-facing explanation; never reject into React.

## Implementation Notes

- Use `git rev-parse --show-toplevel` before listing so every returned path is relative to the repository root even when a session starts in a subdirectory.
- Use `git ls-files --cached --others --exclude-standard -z`, then subtract every path matched by `git check-ignore --no-index -z --stdin`. This defines “ignored” as any path currently matched by repository exclude rules, including tracked paths.
- Model the source after `readGitBranch`: injectable spawn, fixed command options, swallowed expected failures, and colocated tests.
