# ADR-005: Use Conservative Attributes and Bounded Binary Detection

## Status

Accepted

## Date

2026-07-12

## Context

Git's tracked/unignored listing excludes ignored and outside-repository paths, but it may include generated artifacts and binary files. The PRD defines the V1 candidate set as normal repository files and explicitly excludes generated and binary content. Heuristic directory and extension lists would be incomplete, repository-specific, and difficult to maintain. Repository-controlled filenames can also contain terminal control characters or whitespace, which must not corrupt the terminal selector or create ambiguous prompt text.

## Decision

Resolve the Git root, list candidate paths with `git ls-files --cached --others --exclude-standard -z`, and subtract every path that `git check-ignore --no-index -z --stdin` reports. This treats any path currently matched by repository ignore rules as excluded, whether or not it is tracked.

Reject a path before display when it has a C0 or C1 control character (U+0000–U+001F or U+007F–U+009F), resolves outside the worktree root, or is not a regular file. Pass remaining NUL-delimited paths to `git check-attr -z --stdin linguist-generated text`. Parse stdout as repeated `path\0attribute\0value\0` triples. Exclude a path when:
- `linguist-generated` is `set` or any explicit string value other than `false`;
- `text` is `unset` or the explicit string `false`.

Treat `unspecified`, `text=set`, `text=auto`, and `linguist-generated=unset` as not excluded by that attribute check. A malformed triple or command failure makes discovery unavailable; it never silently broadens the candidate set.

For every remaining file, read at most a fixed 4 KiB prefix and reject it if the bytes contain a NUL. Process checks through a named bounded worker-pool concurrency constant. Convert the bytes directly to a boolean and discard them; retain neither source bytes nor binary verdicts outside the single discovery call.

Do not introduce build-directory, filename, or extension denylists in V1. An unmarked generated text file remains eligible; that is the accepted trade-off for an explicit, repository-controlled generated-file policy.

## Alternatives Considered

### Alternative 1: Heuristic directory and extension filtering

- **Description**: Exclude conventional build directories and known binary suffixes.
- **Pros**: Cheap and familiar.
- **Cons**: Misses custom layouts, incorrectly excludes legitimate files, and requires ongoing policy maintenance.
- **Why rejected**: Kitten cannot reliably infer every repository's generated-file conventions.

### Alternative 2: Attributes only for binary exclusion

- **Description**: Trust `.gitattributes` to mark every binary file.
- **Pros**: Avoids reading file bytes.
- **Cons**: Unmarked binary files can appear in the selector.
- **Why rejected**: It does not meet the V1 binary-file boundary on ordinary repositories.

### Alternative 3: Full-content classification

- **Description**: Read every candidate completely and classify its content.
- **Pros**: Potentially richer detection.
- **Cons**: Adds avoidable latency and reads more source content than necessary.
- **Why rejected**: A bounded prefix check is sufficient for the V1 candidate gate.

## Consequences

### Positive

- Enforces ignored, outside-repository, generated, binary, and terminal-safe-path boundaries without a dependency.
- Keeps source-content handling bounded and content-free outside the detection operation.
- Lets a repository declare generated policy using its existing Git metadata.

### Negative

- Prefix detection can misclassify unusual binary or text formats.
- Unmarked generated text files remain visible by design.
- Control-character pathnames cannot be selected in V1.

### Risks

- **Large result set scan**: process candidates through a bounded worker pool, show loading while discovery runs, and retain the filtered list only for the current focus.
- **Attribute command failure**: return the discovery result as unavailable rather than broadening the candidate set silently.
- **Special filename handling**: NUL-delimited Git transport and explicit control-character rejection preserve terminal safety.

## Implementation Notes

- Use NUL-delimited Git input/output internally for every path-bearing command.
- Define the 4 KiB prefix and worker-pool concurrency as named constants with tests; do not read full files or cap the searchable candidate set.
- Display only repository-relative safe paths; never reveal an absolute session or worktree path.
