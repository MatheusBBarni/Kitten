# ADR-002: Validation-gated honest MVP for the agent usage gauge

## Status

Accepted

## Date

2026-07-09

## Context

ADR-001 set the shape of the feature: an always-on per-agent context-headroom signal in the status strip and the `Ctrl+T` handoff preview, context-only and absence-aware.
This ADR decides the *delivery strategy* for the MVP - how to sequence the work and how opinionated the first release should be - given two open forces:

- **Unverified emission.** No code path has confirmed that the Claude Code or Codex ACP adapters actually emit `usage_update`. Building a full gauge on a signal that never arrives would ship a permanent "unknown / unknown" strip.
- **Untrustworthy denominator.** The context window size reported over ACP may not equal the effective limit under internal auto-compaction and reserved output headroom, so a directive "hand off now" verdict could mislead before thresholds are calibrated against real sessions.

The clarifying round with the product owner resolved three forks: ship both surfaces (status strip + handoff preview) in the MVP; keep the MVP gauge neutral (no color verdict); and measure the MVP by data availability/honesty rather than behavior-change telemetry, deferring correlation to Phase 2.

## Decision

Deliver the MVP as a **validation-gated, honest, neutral** first release:

1. **Instrument-first gate.** Before UI work, confirm through logging that at least one adapter emits `usage_update`. If neither does, the gauge still renders honest "unknown" for both agents, and the finding reshapes the plan (raise with adapter owners) rather than shipping a false signal.
2. **Both surfaces in the MVP.** The always-on status-strip headroom and the target-agent headroom in the handoff preview ship together, since both read the same derived value and the handoff preview is where the decision is made.
3. **Neutral presentation.** The MVP shows headroom as a percent + bar (and "unknown" when absent) with no "consider/hand off now" color verdict. Color threshold bands are deferred to Phase 2, tuned against observed sessions.
4. **Availability-and-honesty metrics.** The MVP is judged by whether usage arrives and is rendered honestly (never a fabricated percent) and by render isolation, not by behavior-change. Behavior-change telemetry (earlier hand-offs, correct direction) is added in Phase 2.

## Alternatives Considered

### Alternative 1: Conservative data-path spike first

- **Description**: A standalone Phase 0 that logs adapter emission and ships nothing user-facing, deferring all UI until emission is confirmed.
- **Pros**: Maximally de-risks the unverified-emission assumption before any UI investment.
- **Cons**: Delays all user value; risks the feature stalling in an open-ended "gate"; the honest "unknown" state already lets the UI ship safely without a blocking spike.
- **Why rejected**: The instrument-first check can run alongside the first UI slice rather than blocking it, because absent data degrades to an honest "unknown" rather than a broken screen.

### Alternative 2: Directive, richer MVP

- **Description**: Ship both surfaces plus amber/red handoff color bands plus behavior-change telemetry from day one.
- **Pros**: More glanceable and directive; earlier read on whether the feature changes behavior.
- **Cons**: Asserts "hand off now" precision before thresholds are calibrated against real compaction behavior; the untrustworthy denominator makes an early verdict a trust hazard; larger first slice.
- **Why rejected**: A misleading verdict on an always-on surface erodes trust faster than a neutral gauge earns it; thresholds should follow data, not precede it.

## Consequences

### Positive

- Ships user value quickly without betting the release on unverified emission or uncalibrated thresholds.
- The neutral, honest MVP protects trust - the feature never asserts precision it cannot back.
- Phasing is additive: thresholds and behavior telemetry (Phase 2) and the overlay/breakdown/custody nudge (Phase 3) layer onto the same derived value.

### Negative

- The MVP is less directive: users read a number and a bar and make their own call, rather than being told "hand off now."
- Behavior-change impact is not measured until Phase 2, so early success is judged on availability and honesty rather than outcomes.

### Risks

- **Neither adapter emits usage**: the strip shows "unknown / unknown." Mitigation: the instrument-first gate surfaces this before launch messaging promises a working comparison; escalate to adapter owners.
- **Users want a verdict sooner**: neutral framing may feel under-baked. Mitigation: Phase 2 thresholds are explicitly planned and calibrated, not cut.

## Implementation Notes

- Deferred to the TechSpec. The gate is an observability step (log `usage_update` arrivals per adapter), not a user-facing phase.
