# ADR-003: Keep explorer I/O behind separate controller-owned capabilities

## Status

Accepted

## Date

2026-07-16

## Context

Kitten's existing `fileDiscovery.ts` supports prompt `@` completion through Git-backed, safe-text-only repository discovery. The File Explorer must instead show hidden and ignored workspace entries, omit only `.git`, traverse eligible contained links, and open regular files in an external editor. Reusing the existing source would conflate two incompatible policies.

The application already places filesystem and process work behind injected controller dependencies and `ControllerActions`, while the external store owns cross-view session state. The explorer needs the same separation to preserve session isolation, avoid React-owned mutable state, and make containment and launch failures deterministic under test.

## Decision

Add separate injected app-layer `WorkspaceExplorerSource` and `ExternalEditorLauncher` capabilities. Thread them through `SessionControllerOptions` into `createControllerActions`, where all explorer I/O is explicitly addressed by `SessionId`. The source resolves and revalidates the captured session workspace boundary at every directory list, refresh, traversal, and file-open operation; the launcher receives only an already revalidated regular file target and a validated editor preference.

Keep explorer visibility, focus, notices, lazy directory snapshots, selection, expansion, and scroll position in the external `AppStore`, keyed by `SessionId` for current-run state. The UI consumes narrow selectors and invokes `ControllerActions`; it performs no filesystem or process work. Explorer operations commit only to their captured session and generation, so focus changes, session removal, or stale results cannot overwrite another session's tree.

## Alternatives Considered

### Alternative 1: Reuse `fileDiscovery.ts`

- **Description**: Adapt the Git-backed prompt file source into the explorer's tree source.
- **Pros**: Fewer new files and existing filesystem safety tests.
- **Cons**: Its Git-rooted, safe-text-only, ignored-entry filtering policy conflicts with the explorer contract and would couple prompt completion to workspace navigation.
- **Why rejected**: The two features have intentionally distinct visibility, lifecycle, and security semantics.

### Alternative 2: Perform tree reads and launching in React

- **Description**: Let the explorer component read directories and spawn the external editor directly.
- **Pros**: Shorter initial wiring path.
- **Cons**: Violates Kitten's layer boundary, duplicates mutable state, complicates test injection, and lets UI focus timing influence security-sensitive I/O.
- **Why rejected**: Controller-owned capabilities and store-owned state are established repository invariants.

### Alternative 3: Maintain one global explorer cache

- **Description**: Keep one tree and selection that updates whenever focus changes.
- **Pros**: Fewer state objects.
- **Cons**: Mixes workspaces across sessions and cannot restore a session's current-run navigation position.
- **Why rejected**: Session identity is the feature's defining boundary.

## Consequences

### Positive

- Preserves separate product semantics for prompt completion and workspace exploration.
- Gives containment, launch, and fallback behavior one injected, deterministic enforcement boundary.
- Makes concurrent operations and session switches safe to reason about through captured session and generation state.

### Negative

- Adds focused app, store, selector, controller, and UI contracts rather than one component-local implementation.
- Requires explicit lifecycle handling when sessions are created, removed, or their workspace becomes unavailable.

### Risks

- **A missed I/O call site bypasses revalidation**: expose no direct UI filesystem or process capability and centralize every explorer operation in the new actions.
- **Stale async results mutate a wrong tree**: fence commits by captured `SessionId`, workspace identity, and operation generation.
- **Tree state grows with large workspaces**: lazy-load only expanded directories and clear current-run state with the session lifecycle.

## Implementation Notes

Use existing `RepositoryFileSource`, `ControllerActions`, `SessionControllerOptions`, `AppStore`, and selector patterns as shape references only. New explorer contracts belong in `src/app/`; persistent preferences remain config data, and tree snapshots remain current-run store data.
