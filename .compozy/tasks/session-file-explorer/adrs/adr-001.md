# ADR-001: Keep a safety-complete session explorer as the V1 boundary

## Status

Accepted

## Date

2026-07-16

## Context

Issue #23 proposes a File Explorer that follows Kitten's focused session workspace and opens selected regular files in an external editor. The value is workflow continuity: an active agent-session developer can inspect or open workspace files without abandoning the cockpit or confusing one session's workspace with another.

The feature crosses filesystem traversal, symlink handling, process launch, strict configuration persistence, terminal layout, and local telemetry. It must not reuse the Git-backed, safe-text-only `@` file-discovery source, and it must preserve Kitten's controller-owned I/O, external-store ownership, failure isolation, and content-free telemetry contracts. The principal scope tension is whether to remove contained symlink traversal and custom editor fallback from V1 to reduce risk, despite their explicit issue requirements.

## Decision

Ship a complete but narrow V1: a session-scoped, lazy File Explorer with the specified docked/narrow layouts; explicit toggle and navigation commands; contained Workspace Entry traversal; regular-file opening through the system default or a validated saved custom editor command; deterministic fallback; and opt-in, content-free outcome telemetry.

All directory reads, containment decisions, revalidation, and launches must pass through a separate injected app/I/O capability reached through `ControllerActions`. Every traversal and open re-resolves the current target and proves canonical containment in the captured session workspace at use time; all failures, broken links, escapes, or changed entries fail closed. Custom editors are an executable plus discrete arguments with exactly one `{file}` placeholder, never a shell string. A failed custom launch attempts the system-default path exactly once and reports the final coarse outcome without exposing command, path, filename, or error text.

## Alternatives Considered

### Alternative 1: System-default tree only

- **Description**: Deliver browsing and system-default regular-file opening, but defer custom editor configuration, automatic fallback, and contained link traversal.
- **Pros**: Smaller initial process-launch and filesystem-risk surface; fewer persistence and platform cases.
- **Cons**: Fails the explicit product and acceptance requirements; cannot support developers whose established editor is not the OS default; makes the explorer less useful for real agent-session workflows.
- **Why rejected**: It is a different, incomplete product boundary. The risks are better managed through a single revalidating gateway, strict argv validation, deterministic fallback, and focused tests.

### Alternative 2: Workspace hub platform

- **Description**: Treat the explorer as a foundation for previews, search, mutations, Git decorations, watchers, multiple roots, and richer editor integration.
- **Pros**: More breadth and future extensibility.
- **Cons**: Turns Kitten into a general IDE, dilutes the active-session continuity hypothesis, and compounds TUI, I/O, state, and security complexity.
- **Why rejected**: The product needs a measurable, reversible learning loop, not an IDE platform.

### Alternative 3: Display contained links but never traverse or open them

- **Description**: Show symlinks only as inert rows to remove traversal and open race exposure.
- **Pros**: Lowest symlink risk and simplest enforcement.
- **Cons**: Violates the issue's contained-link contract and produces misleading visible entries that cannot serve the central navigation task.
- **Why rejected**: Contained links can participate only through central, use-time canonical containment checks. Any code path unable to provide that proof must reject the link rather than weaken the rule.

## Consequences

### Positive

- Preserves active-session workflow continuity while retaining Kitten's focused multi-agent differentiation.
- Gives every security-sensitive filesystem and process action one controller-owned, injectable enforcement seam.
- Supports the issue's editor-preference workflow without shell interpretation or broad editor-platform scope.
- Produces privacy-preserving beta evidence using only allow-listed, coarse local outcomes.

### Negative

- Requires broad deterministic testing across tree listing, link swaps, session switching, launch failures, config persistence, layout modes, and telemetry schemas.
- macOS and Linux launch behavior needs explicit supported-platform handling and may still only prove process-start success.
- Revalidation and lazy loading may surface local operation failures that need clear, non-disruptive notices.

### Risks

- **Containment bypass or TOCTOU race**: centralize use-time resolution and containment checks; fail closed for every changed, broken, looping, or escaping target; add race and chained-link regression tests.
- **Command injection or unsafe launcher behavior**: store executable and arguments separately; require exactly one placeholder; spawn direct argv only; never shell-parse input.
- **Session leakage from asynchronous work**: capture and recheck `SessionId` and workspace root for every result; retain explorer position only in current-run session state.
- **Telemetry privacy drift**: enforce an allow-listed outcome schema that excludes paths, names, commands, arguments, errors, tree shape, and stable identifiers.

## Implementation Notes

The later TechSpec must define the source and launcher contracts, central enforcement point, strict config shape, atomic writer and watcher updates, exact launch semantics for supported platforms, width threshold behavior, and the complete security regression matrix. This ADR deliberately does not prescribe internal implementation details.
