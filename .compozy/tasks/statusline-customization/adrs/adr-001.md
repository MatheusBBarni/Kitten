# ADR-001: Constrain V1 to declarative conversational statusline configuration

## Status

Accepted

## Date

2026-07-13

## Context

Kitten's status strip is a fixed presentation of the focused session. Developers can already see useful workspace and agent state, but cannot choose its order, density, or branch/path representation. The requested experience is a `/statusline` command where an individual developer describes the line they want in natural language, inspects the result, and chooses whether to persist it.

Comparable agent CLIs establish both the opportunity and the boundary: natural-language statusline customization is familiar, while command-backed status lines create an execution, portability, and support burden. Kitten also has a strict user configuration contract and an explicit-preview product philosophy. The feature must preserve those properties even when an LLM interprets the developer's request.

## Decision

V1 will use a dedicated conversational request-and-review flow. The LLM is an interpretation boundary only: it may propose one schema-validated, non-executable `statusline` configuration, never a script, template, terminal control sequence, or command.

The persisted layout has a printable separator and an ordered line assembled only from `FOLDER`, `FULL_PATH`, `BRANCH`, `ELLIPSIS_BRANCH` (with a bounded `maxChars` option), `PROVIDER`, `MODEL`, `EFFORT`, and `HELP_TEXT`. Unavailable values omit cleanly. The exact same deterministic renderer produces both the review preview and the runtime line. Saving requires an explicit confirmation, shows the exact config diff and destination, and tells the user if the change takes effect on the next launch.

Natural-language creation remains the primary V1 path. Three fixed, schema-native presets are an opt-out or recovery fallback, not a separate visual editor. V1 persists only the validated layout; it does not retain the request, raw model response, or resolved runtime values. It also ships without statusline-specific telemetry.

## Alternatives Considered

### Alternative 1: Preset or manual editor only

- **Description**: Offer a few layouts or a field picker without natural-language interpretation.
- **Pros**: Smallest implementation and no LLM dependency.
- **Cons**: Does not test the requested conversational experience; forces users to translate intent into configuration mechanics.
- **Why rejected**: Retained as the three-preset fallback, but insufficient as the primary product hypothesis.

### Alternative 2: Arbitrary scripts, templates, ANSI, or dynamic commands

- **Description**: Let the LLM generate a powerful statusline program, similar to command-backed statusline systems.
- **Pros**: Maximum apparent flexibility and access to any local data.
- **Cons**: Creates code execution, terminal escape, portability, timeout, refresh, support, and privacy risks.
- **Why rejected**: It violates Kitten's bounded, inspectable configuration model and is disproportionate to the V1 learning goal.

### Alternative 3: Conversational layout with no fallback

- **Description**: Require every customization to depend on the LLM flow.
- **Pros**: The cleanest measurement of conversational creation.
- **Cons**: Model unavailability, latency, or misunderstandings would be indistinguishable from a poor product hypothesis.
- **Why rejected**: Three schema-native presets provide a reliable baseline without adding a second editing system.

## Consequences

### Positive

- Developers get the requested intent-first customization flow without learning a formatting language.
- The layout remains portable, deterministic, previewable, and safe to persist.
- The request/review/commit pattern can later inform other bounded personal settings if users prove its value.

### Negative

- V1 cannot satisfy requests outside the allowlisted fields and formatting options.
- The initial implementation may require a restart rather than generalized live configuration reload.
- Presets add a small recovery surface that must share the same schema, renderer, preview, and save path.

### Risks

- **Prompt injection or malformed LLM output** — Validate against the strict schema; reject unknown fields and never execute generated text.
- **Preview/save divergence** — Use one canonical normalized representation and renderer for both surfaces; revalidate immediately before writing.
- **Privacy or unsafe persistence** — Disclose any off-device model processing before submission; store only the validated layout; use restrictive, symlink-safe atomic writes; do not collect statusline telemetry in V1.
- **Narrow-terminal or Unicode overflow** — Apply deterministic field omission and truncation, with the specified branch ellipsis bounded by grapheme-aware limits.

## Implementation Notes

- The product-owned LLM instruction may request only the declarative layout schema. It is not a user-configurable execution policy, and neither the user request nor the raw response belongs in `config.json`.
- `/statusline` needs a dedicated request/review state because the existing slash-command dispatcher intentionally invokes commands without passing trailing prose.
- A future TechSpec must define the exact JSON shape, separator restrictions, field precedence, fallback behavior, config migration, and live-reload decision.
