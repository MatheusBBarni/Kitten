# ADR-004: Use the focused agent transcript with a strict fenced proposal contract

## Status

Accepted

## Date

2026-07-13

## Context

The developer selected the focused agent's ordinary transcript as the LLM path for `/statusline`. Kitten's existing prompt action records every user turn and returns only a stop reason; it has no separate structured-response API. Agent messages are already normalized into ordered transcript turns before prompt completion resolves.

The product still requires an inspectable layout proposal, no automatic save, and rejection of malformed output. Parsing arbitrary prose would make the acceptance boundary ambiguous.

## Decision

The statusline flow will submit a product-owned instruction plus the developer's request through the focused session's existing prompt path. The request and response intentionally appear in that agent's normal transcript. The instruction requires the response to consist solely of one fenced `json` block containing the statusline configuration shape. The flow captures the agent turns produced after its request completes, requires exactly that response form, parses it with the same strict layout schema used for user config, and opens the preview only after validation.

Malformed, multiple, or prose-wrapped responses are proposal failures, never partial layouts. The overlay keeps the failure legible and offers the recovery presets. Raw request and response text remain in the chosen normal transcript but are never copied into user configuration, telemetry, or statusline persistence.

## Alternatives Considered

### Alternative 1: Dedicated hidden proposal service

- **Description**: Add an isolated request/response channel outside the coding conversation.
- **Pros**: No transcript noise and direct structured result handling.
- **Cons**: Adds a new agent integration boundary not present in Kitten today.
- **Why rejected**: The developer explicitly selected the normal focused-agent transcript for V1.

### Alternative 2: Parse JSON anywhere in a natural-language response

- **Description**: Search an agent explanation for a likely JSON object.
- **Pros**: More forgiving agent output.
- **Cons**: Ambiguous extraction, accidental acceptance, and difficult error reporting.
- **Why rejected**: A sole fenced block creates an unambiguous, testable contract.

### Alternative 3: Remove the LLM path and offer presets only

- **Description**: Never send a request to an agent.
- **Pros**: No parse or transcript integration.
- **Cons**: Fails the requested conversational feature.
- **Why rejected**: Presets remain recovery-only by product decision.

## Consequences

### Positive

- Reuses the existing focused-agent authentication and prompt path.
- Makes the request and reply visible to the developer.
- Keeps proposal acceptance strict and schema-backed.

### Negative

- Statusline requests and replies become part of the normal coding transcript.
- The focused agent must follow a narrow response contract.
- A malformed response requires recovery rather than heuristic repair.

### Risks

- **Agent emits prose or multiple blocks** — reject deterministically and offer presets.
- **Prompt completion races transcript extraction** — collect turns only after the normal prompt path reaches its terminal status, which flushes buffered agent messages first.
- **Request content includes sensitive context** — disclose the transcript path before first send and do not add resolved path/branch values to the product-owned instruction.

## Implementation Notes

- The prompt template is product-owned source, not a user-configurable prompt or persisted configuration value.
- The response parser must accept only a complete fenced JSON response and must share the same schema as the config loader.
