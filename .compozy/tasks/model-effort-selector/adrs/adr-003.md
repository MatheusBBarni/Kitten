# ADR-003: Generic config-option channel in the domain core

## Status

Accepted

## Date

2026-07-09

## Context

ADR-001 committed V1 to "generic data, narrow render": model the whole ACP config-option surface in the domain core, but expose only model and effort in the UI.
This ADR fixes the concrete type shape.

ACP delivers configuration as a `SessionConfigOption[]`, each with an opaque `id`, a `category` (`"model" | "thought_level" | "model_config" | "mode" | string`), and either a select (`currentValue` plus `options[]`) or a boolean value.
The `session/new` response carries the initial `configOptions`, `session/set_config_option` returns the full refreshed set, and a `config_option_update` notification carries agent-initiated changes.
Kitten's domain core (`src/core/types.ts`) is protocol-free by design (ADR-003 of the base cockpit): the ACP adapter translates wire types into Kitten's own domain types, and nothing above the adapter imports the SDK.

## Decision

Add a normalized, Kitten-owned config-option list to `SessionState`, keyed by opaque category id:

```typescript
interface ConfigSelectOption { value: string; name: string }
interface ConfigOption {
  id: string           // opaque ACP config id, passed back verbatim on change
  category: string     // "model" | "thought_level" | ... , kept opaque
  label: string
  currentValue: string
  options: ConfigSelectOption[]
}
```

`SessionState` gains `configOptions: ConfigOption[]`, defaulting to `[]`.
A new domain event `{ kind: "config_options"; options: ConfigOption[] }` carries every source (session start, a switch response, a `config_option_update`), and the reducer applies it as a wholesale replace, because the agent always returns the complete option set.
Selectors filter to the visible categories (`model`, `thought_level`); the category strings stay opaque to the store and are never hardcoded as named fields.
V1 only handles select options; boolean options (Fast mode) are not represented.

## Alternatives Considered

### Alternative 1: Typed model/effort fields

- **Description**: Add `currentModelId`, `availableModels`, `currentEffortId`, `availableEfforts` directly to `SessionState`.
- **Pros**: Simplest to read in the UI; no category mapping.
- **Cons**: Hardcodes the two categories; V2 (provider swap, more categories) is a rewrite; needs projection code to squash the generic wire shape into named fields.
- **Why rejected**: Contradicts ADR-001 and is more code today plus a rewrite tomorrow.

### Alternative 2: Separate keyed ConfigState map

- **Description**: Hold config outside `SessionState` in a parallel `Map<categoryId, {currentValue, options}>`.
- **Pros**: Generic like the chosen option.
- **Cons**: A second per-agent structure the reducer and selectors must track and keep in sync with the session.
- **Why rejected**: `SessionState` is already the per-agent slice the store and selectors are built around; a parallel map adds coordination for no gain.

## Consequences

### Positive

- V2 provider swap and additional categories become a render/policy change, not new plumbing.
- The wholesale-replace reducer is trivial and immune to partial-update drift, matching how the agent reports state.
- The domain type is Kitten-owned, so the anti-corruption boundary holds.

### Negative

- The UI maps opaque category strings to sections, one indirection more than typed fields.

### Risks

- ACP `category` is an open union (`| string`); a strict UI allowlist (ADR-004) protects against surfacing an unexpected category.

## Implementation Notes

Default `configOptions: []` in `createSessionState` (`src/core/sessionReducer.ts`), add the `config_options` case, and keep the `assertNever` exhaustiveness guard compiling.
