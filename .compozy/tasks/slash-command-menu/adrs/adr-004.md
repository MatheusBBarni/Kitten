# ADR-004: Non-modal editor-local menu with a shared cockpit-command dispatcher

## Status

Accepted

## Date

2026-07-10

## Context

The slash menu must detect the `/` token as the user types, which is only possible inside `PromptEditor` because it owns the OpenTUI textarea buffer (`plainText`, `cursorOffset`, `insertText`, and the `onContentChange`/`onKeyDown` seams).
But the Kitten actions the menu invokes are dispatched inside `CockpitFrame`'s `onKey` switch across four different owners: `controller.actions.switchFocus()`, `controller.store.openSessions()`, the `handoff` flow's `begin()`, and shell-local `setHelpOpen()`.
`handoff` and `helpOpen` are not in scope inside `PromptEditor`.

Every existing overlay (approval, hand-off preview, sessions) is modal: it lives in a store overlay slot, `selectHasOpenOverlay` blurs the editor, and the shell early-returns on `overlayOpen`.
The slash menu is the opposite - the user keeps typing to filter, so the textarea must stay focused. Putting the menu in the store overlay slot would blur the editor and break filtering.

## Decision

Build the menu as **non-modal, editor-local React state** in `PromptEditor`, dispatching cockpit actions through a **shared `runCockpitCommand` callback** passed down from `CockpitFrame`.

- **State ownership**: menu open/filter/highlight index are `useState` inside `PromptEditor`. The menu is not a store overlay; `SessionState` never holds menu state; `overlayOpen` stays false so the textarea keeps focus.
- **Dispatcher extraction**: refactor `CockpitFrame`'s `onKey` switch into a `runCockpitCommand(command: CockpitCommand): void` function that both `onKey` and the menu call. Pass it to `PromptEditor` as an `onRunCommand` prop. This keeps the single dispatch path and reuses the `handoff`/`helpOpen`/`controller` closures already in `CockpitFrame`.
- **Trigger**: on `onContentChange`, read `plainText` + `cursorOffset` and arm the menu only when `/` begins the token under the caret (start of input or after whitespace). Disarm on no-match, a second `/` in the token, or the caret leaving the token.
- **Key capture**: while armed, `onKeyDown` calls `key.preventDefault()` for Up/Down/Enter/Esc/Tab and drives the menu; when disarmed, keys fall through (Enter submits normally).
- **Invocation**: a cockpit row calls `onRunCommand(command)`; an agent-command row calls `textarea.insertText` to replace the token with `"/name "` and closes the menu (never routes through `ControllerActions`, never auto-sends).
- **Rows**: cockpit rows are derived from `COCKPIT_KEYMAP` (reusing each binding's `keys` and `description`); agent rows come from `selectSessionCommands(focusedSessionId)`. A new `MENU_KEYMAP` + `matchMenuCommand` defines the menu's own navigation keys.
- **Rendering**: an absolutely-positioned `<box>` anchored above the textarea within `PromptEditor`'s tree (no Portal exists; existing overlays use the same absolute-box technique).

## Alternatives Considered

### Alternative 1: Lift the menu into `CockpitFrame`

- **Description**: Render the menu at the frame level so it reaches `handoff`/`helpOpen` directly.
- **Pros**: No dispatcher prop.
- **Cons**: The editor still owns the buffer, so `PromptEditor` must publish token/filter/keystroke events upward, adding cross-component plumbing and splitting the interaction across two components.
- **Why rejected**: More coupling than passing one callback down.

### Alternative 2: Modal menu in the store overlay slot

- **Description**: Reuse the tested overlay machinery; open the menu as a modal that blurs the editor and filters internally.
- **Pros**: Reuses key-capture/focus/dismissal infrastructure.
- **Cons**: Blurs the editor, so "type `/` then keep typing to filter" is impossible; contradicts ADR-001's non-modal decision.
- **Why rejected**: Held only as the fallback from ADR-001 if the non-modal overlay proves janky.

## Consequences

### Positive

- Buffer ownership stays in one place; the menu reads the buffer it lives next to.
- One dispatch path (`runCockpitCommand`) serves both chords and the menu, so behavior cannot drift.
- The transcript never re-renders on menu keystrokes because menu state is local and never touches the store.

### Negative

- Introduces the codebase's first non-modal overlay concept and overloads editor keys (Enter/Esc) with menu-armed meaning.
- Requires a small refactor of `CockpitFrame.onKey` into an extracted dispatcher.

### Risks

- **Key-capture ordering** (Enter = accept vs. submit). Mitigation: the render is the signal - keys are only intercepted while the menu is visibly armed; covered by interaction tests.
- **Trigger edge cases** on paths/URLs. Mitigation: token-begin guard + no-match disarm, covered by edge-case tests.

## Implementation Notes

- Menu component: a single new `src/ui/SlashMenu.tsx` plus state and handlers added to `PromptEditor.tsx`; no new package or directory.
- Keep `focused={!overlayOpen}` as-is; the menu deliberately does not participate in `overlayOpen`.
