# ADR-005: Record a closed live-config capability result in the native contract

## Status

Accepted

## Date

2026-07-16

## Context

Kitten correctly derives model and reasoning controls from the active ACP session and applies changes through the session configuration operation. Existing mocks cover the generic update path, but the native Cursor contract currently stops after authentication, session creation, and a completed prompt. It therefore cannot establish whether Cursor offers a live switchable option or accepts an update.

The PRD requires an explicit ready-but-no-switchable-options state and prohibits direct CLI model discovery as a substitute. Contract evidence must remain content-free and must not record raw configuration IDs, values, prompts, credentials, or provider output.

## Decision

Extend the opt-in native Cursor contract with one closed live-config capability result. After session creation, the contract records either `not_advertised`, `accepted`, or `rejected`. If a safe visible option is advertised, it submits that option's currently active value through the existing ACP operation, proving the request path without intentionally changing user-visible behavior. The contract records only the closed outcome and continues to cleanly dispose the session.

The product renders controls only from live session options and applies changes only when ACP confirms the updated option set. A `not_advertised` or `rejected` result preserves the explicit unavailable state; it never triggers direct CLI probing or inferred controls.

## Alternatives Considered

### Alternative 1: Observe advertised options only

- **Description**: Record whether options appear but never call the live configuration operation.
- **Pros**: Avoids an additional native request.
- **Cons**: Does not prove that the advertised configuration path works.
- **Why rejected**: The issue explicitly requires experimental confirmation of live configuration support.

### Alternative 2: Defer all Cursor config evidence

- **Description**: Keep configuration outside the native contract for V1.
- **Pros**: Minimizes the contract scope.
- **Cons**: Leaves the model-control support claim unverified.
- **Why rejected**: It would make the ready-state capability behavior speculative.

### Alternative 3: Discover models through Cursor CLI flags

- **Description**: Use direct Cursor CLI model listing or selection as a fallback source.
- **Pros**: May expose options even when ACP does not.
- **Cons**: Creates a second source of truth and does not prove a live ACP session can use the value.
- **Why rejected**: It violates the session-authoritative PRD boundary.

## Consequences

### Positive

- Converts the live-config question into a repeatable native evidence outcome.
- Preserves no-options as an explicit supported state rather than a generic error.
- Avoids recording sensitive or provider-specific free-form data.

### Negative

- Native verification now depends on one additional ACP request.
- The contract must carefully choose only a safe currently active advertised value.

### Risks

- **Provider rejects the request**: record `rejected` and retain unavailable controls rather than guessing.
- **No option is advertised**: record `not_advertised` and render the ready no-options state.
- **Evidence leaks option data**: restrict persisted evidence to the three closed outcomes.

## Implementation Notes

- Keep the request in `src/agent/agentConnection.ts` and the opt-in evidence in `test/cursorAcp.contract.test.ts`.
- Reuse the existing atomic confirmed-config-options update path; no optimistic store mutation.
- Keep normal unit tests mocked and the native contract opt-in.
