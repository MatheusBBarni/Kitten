# ADR-001: Keep Cursor support evidence-gated and fail closed

## Status

Accepted

## Date

2026-07-16

## Context

Kitten already has a built-in Cursor `agent acp` recipe and a sealed runtime-profile seam, but no reviewed Cursor profile is currently certified. The product therefore rejects Cursor before ACP initialization rather than claim that an unverified local CLI is usable. The existing model selector then misstates this readiness failure as an absence of model or reasoning options.

The intended audience is local macOS Cursor subscribers. Cursor's CLI changes rapidly and no durable public ACP compatibility contract was found, so a version string or direct-CLI model list alone cannot establish support. The product must restore a useful integration without weakening the existing per-agent resilience, native authentication, ACP-only configuration, or fail-closed boundaries.

## Decision

Define V1 as an evidence-gated, local Cursor ACP path for exactly one reviewed configuration. A profile may be described as certified only after content-free native evidence confirms the exact executable and runtime context, unmodified `agent acp` recipe, native authentication, ACP initialization, session creation, and a completed prompt. Any version, invocation, environment, or capability outside that reviewed snapshot remains unsupported and fails closed.

Ship the truthful Cursor readiness and recovery state independently when it is accurate, but do not treat that UI improvement as certification. A ready Cursor session exposes model and reasoning controls only when that ACP session advertises supported options; Kitten never derives live controls from direct Cursor CLI flags or model lists. The reviewed profile is owned and revocable: runtime changes require renewed evidence before support is claimed again.

## Alternatives Considered

### Alternative 1: Broadly enable any detected Cursor CLI

- **Description**: Allow any installed Cursor CLI to start through the built-in recipe and infer compatibility from local discovery or a version range.
- **Pros**: Reaches more potential users quickly and reduces recertification work.
- **Cons**: Treats a fast-changing, unverified runtime as trusted and makes failures harder to diagnose.
- **Why rejected**: It violates the existing fail-closed contract and would turn a narrow local integration into an unsupported compatibility promise.

### Alternative 2: Ship only the recovery-message correction

- **Description**: Replace the generic selector notice but leave Cursor entirely uncertified.
- **Pros**: Low risk, immediately truthful, and independently valuable.
- **Cons**: Does not restore a working Cursor path or collect end-to-end evidence.
- **Why rejected**: It is an acceptable incremental improvement, but insufficient as the complete V1 outcome requested by the issue.

### Alternative 3: Add direct-CLI model discovery and controls

- **Description**: Use `agent --list-models` or `--model` as an alternate control plane for an ACP session.
- **Pros**: Could show more choices when ACP advertises none.
- **Cons**: Creates two sources of truth and assumes a direct CLI flag can configure an existing ACP session.
- **Why rejected**: It would misrepresent unsupported capability and bypass the session's negotiated ACP contract.

## Consequences

### Positive

- Restores a concrete local Cursor path without weakening authentication, version, recipe, or capability boundaries.
- Separates honest recovery guidance from an unsupported claim of readiness.
- Produces a small, reviewable evidence snapshot that can be renewed or revoked as Cursor changes.

### Negative

- Initial reach is deliberately limited to one reviewed local configuration.
- Profile renewal creates recurring maintenance whenever Cursor or its runtime contract changes.
- Some users with an installed Cursor CLI will remain unsupported until their configuration is reviewed.

### Risks

- **Runtime drift or binary replacement**: bind evidence to the reviewed local runtime context and revoke support on a mismatch.
- **Expired or failed authentication**: surface the normalized recovery cause and keep native credentials outside Kitten.
- **ACP capability drift**: render only session-advertised options and make unavailable controls explicit.
- **Misleading release claims**: distinguish recovery UX from certified lifecycle support in onboarding and release notes.

## Implementation Notes

- Preserve Kitten's existing per-agent resilience: a failed or unsupported Cursor session must not affect Claude Code or Codex.
- Keep contract evidence content-free and local, consistent with opt-in telemetry boundaries.
- Treat successful support as the full native lifecycle rather than binary discovery, `--list-models`, or a version check alone.
