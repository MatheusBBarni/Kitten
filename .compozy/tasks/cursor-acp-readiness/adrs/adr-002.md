# ADR-002: Define support by a completed first Cursor task after reviewed proof

## Status

Accepted

## Date

2026-07-16

## Context

The approved idea establishes a fail-closed, exact-profile Cursor ACP boundary. PRD research shows that local coding-agent users expect readiness to distinguish discovery, authentication, connection, and capability availability, with a specific recovery action for each blocked state. The current generic model-options message hides that distinction.

The primary MVP outcome is a supported local macOS Cursor subscriber completing a first coding task in Kitten. Users whose configuration is unsupported still need a clear, safe explanation and a next action; hiding Cursor or presenting a generic empty state would not meet that need. The support claim must remain narrow because a detected Cursor CLI is not necessarily a verified, authenticated, capability-compatible local agent.

## Decision

Define V1 as a proof-backed first-task journey. Kitten calls one local Cursor profile supported only after a reviewed, complete native lifecycle for that exact profile has succeeded. Once supported, the customer outcome is a ready Cursor session that can complete a first task in Kitten; model and reasoning choices appear only when that ready session makes them available.

For every blocked state, Kitten provides specific recovery guidance that distinguishes user-remediable local problems from a profile that is not yet verified by Kitten maintainers. The experience remains visible and honest rather than hiding Cursor. Real-world reliability is measured after support is granted, but a pilot is not a prerequisite for the first narrow support claim.

## Alternatives Considered

### Alternative 1: Require a pilot before calling any profile supported

- **Description**: Delay the first support claim until reviewed proof and a minimum set of voluntary real-world attempts are complete.
- **Pros**: Produces more reliability evidence before users see a supported label.
- **Cons**: Delays a narrowly verified customer outcome and creates a circular adoption requirement.
- **Why rejected**: A complete reviewed lifecycle is the appropriate threshold for the first exact-profile claim; post-release reliability metrics can validate the claim without blocking it.

### Alternative 2: Keep Cursor permanently preview-only

- **Description**: Show the integration as experimental even after the exact profile passes review.
- **Pros**: Limits expectations while the provider evolves.
- **Cons**: Undercuts a successful verification and gives users no clear contract for when the path is usable.
- **Why rejected**: It weakens the product promise without adding safety beyond the exact-profile and revocation boundaries already required.

### Alternative 3: Hide unsupported Cursor sessions

- **Description**: Remove Cursor from user-facing choices until every prerequisite is satisfied.
- **Pros**: Avoids exposing a blocked state.
- **Cons**: Conceals a configured provider, prevents diagnosis, and fails to distinguish recovery from lack of certification.
- **Why rejected**: Specific recovery guidance is more useful and preserves trust without overstating support.

## Consequences

### Positive

- Gives the first supported user a concrete end-to-end outcome instead of an abstract certification state.
- Makes failure states actionable and intelligible for both supported and unsupported local configurations.
- Keeps the support promise small, reviewable, and compatible with recurring runtime changes.

### Negative

- Some installed Cursor users will see a blocked but visible provider until their configuration is reviewed.
- Support communications and onboarding must carefully distinguish user remediation from maintainer-owned certification.
- Real-world reliability evidence begins after the initial support decision rather than before it.

### Risks

- **Users interpret visible Cursor as supported**: use explicit readiness labels and recovery copy that state the profile's actual status.
- **A reviewed profile becomes stale**: retain the revocation and renewal expectation from ADR-001.
- **The first-task journey fails after readiness**: measure content-free lifecycle outcomes and promptly narrow or revoke the support claim if needed.

## Implementation Notes

- Frame all product copy around the user’s next safe action and avoid asking for credentials or secrets.
- Preserve the distinction between a ready session with no optional model controls and an unready session that cannot advertise them.
- Treat the first support claim and subsequent reliability monitoring as separate milestones.
