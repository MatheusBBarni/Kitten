# ADR-003: Shared `<Markdown>` Renderer - One Owned Leaf Mirroring ToolCallDiff, With `markup.*` Theme Registration

## Status

Accepted

## Date

2026-07-09

## Context

Kitten renders Markdown in only one place today: `MessageView`'s single `<markdown>` call site in the transcript.
`theme.ts` registers only code-syntax scopes, so Markdown structure (headings, emphasis, lists, quotes, links) renders flat while fenced code is styled.
The TechSpec must render Markdown on a second surface, the hand-off summary, consistently with the transcript, and must preserve two hard invariants: the OpenTUI 0.4.3 streaming pin (a `MarkdownRenderable` must keep `streaming` permanently true or it blanks multi-block content) and live theme reactivity (a dark/light flip must recolor).
The codebase already solves the "same renderer on multiple surfaces" problem for diffs: `ToolCallDiffBody` is a shared leaf that owns `useSyntaxStyle()`, wrapped by thin per-surface components (`ToolCallDiffView` for overlays, a connector for the transcript).

## Decision

Introduce a single shared leaf component `<Markdown>` (`src/ui/Markdown.tsx`) that owns `useSyntaxStyle()` and `usePalette()`, hard-pins `streaming` to `MARKDOWN_STREAMING` (not a caller-toggleable prop), and sets `conceal`.
Thin per-surface wrappers consume it: the transcript message in `MessageView`, and a new read-mode summary view in `HandoffPreview`.
This mirrors the existing `ToolCallDiffBody` / `ToolCallDiffView` pattern exactly.
Extend `theme.ts` `syntaxThemeFor` to register the `markup.*` scopes the grammar emits - `markup.heading.1`-`.6` and bare `markup.heading`, `markup.strong`, `markup.italic`, `markup.strikethrough`, `markup.raw` and `markup.raw.block`, `markup.list`/`.checked`/`.unchecked`, `markup.quote`, and `markup.link`/`.label`/`.url` - theme-aware in dark and light.

## Alternatives Considered

### Alternative 1: A single flexible `<Markdown>` used directly everywhere

- **Description**: One component at every call site, with props for per-surface variation.
- **Pros**: Fewer files.
- **Cons**: No wrapper layer to enforce per-surface styling; per-surface concerns leak into call sites; a caller can omit or wrongly toggle `streaming` and reintroduce the blank-content bug.
- **Why rejected**: The wrapper discipline is what keeps the streaming pin and theme reactivity from leaking; the ToolCallDiff precedent already proves the shape.

### Alternative 2: Theme-only, with a bespoke summary renderer

- **Description**: Register `markup.*` scopes (lights up `MessageView` for free) and hand-roll a separate renderer for the summary.
- **Pros**: Smallest change to the transcript.
- **Cons**: Two divergent Markdown paths that drift; the streaming and theme invariants are duplicated and can fall out of sync.
- **Why rejected**: Drift risk on the two surfaces the product most cares about looking identical.

## Consequences

### Positive

- One place OpenTUI's Markdown API is named; the streaming pin and theme reactivity are enforced centrally.
- The transcript lights up for free from the theme change, with no call-site edit.
- The hand-off summary and transcript render through the same leaf, so they cannot drift.

### Negative

- A new component plus wrappers to maintain.
- The leaf must call `useSyntaxStyle()`/`usePalette()` in its own body (not accept frozen props) to preserve live dark/light repaint.

### Risks

- `SyntaxStyle.fromTheme` supports `foreground`, `bold`, `italic`, `underline`, and `dim`, but has no strikethrough attribute, so `markup.strikethrough` can only be colored or dimmed, not struck. Any element needing an attribute beyond that set is not expressible through `fromTheme`.

## Implementation Notes

Preserve `getSelectedText()` copy-cleanliness in the leaf: no borders or box glyphs that would enter the clipboard.
Add a `theme.ts` unit test asserting the `markup.*` scopes are registered in both modes (mirror the existing `syntaxStyleFor` tests).
