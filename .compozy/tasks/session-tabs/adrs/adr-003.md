# ADR-003: Use a Mutable Registry with One Dedicated Runtime per Conversation

## Status

Accepted

## Date

2026-07-12

## Context

Kitten resolves a fixed session plan from configuration at boot, constructs one store slice and one agent runtime per seed, and restores only those configured sessions. Session Tabs requires users to create fresh conversations during a run, keep them background-running, close them independently, and restore dynamic conversations later.

The PRD requires a fresh tab to inherit the focused conversation's provider and working directory while receiving an independent ACP conversation and empty transcript. The existing controller already isolates one connection, one ACP session, one event subscription, and one readiness state per configured session.

## Decision

Replace the controller's immutable configuration-derived lifetime plan with a mutable controller-owned conversation registry.

- Configuration remains the read-only source of provider recipes and initial workspace seeds.
- Each conversation has a stable Kitten-assigned SessionId and a dedicated AgentConnection, ACP session, event subscription, permission ownership, and readiness state.
- Creating a fresh tab inherits the focused visible conversation's provider recipe and working directory, allocates a new SessionId, then opens a new ACP session. It never reuses an ACP session or transcript.
- If no visible conversation exists, the new-conversation action uses the first available configured provider and the launch directory. If no provider is available, it returns `null` and commits a workspace-level `no-provider-available` notice; it does not create a providerless conversation placeholder.
- Backgrounding changes workspace metadata only. Cancelling and closing are separate controller operations: cancellation targets one ACP session; closing acquires one per-conversation teardown promise, settles owned permissions, and disposes only that conversation's runtime after lifecycle policy is resolved.
- A descriptor-backed creation or restore failure retains that conversation as unavailable. A cancellation or disposal failure retains its current Visible or Background lifecycle as unavailable and retryable. Each failure remains isolated and never tears down sibling conversations.

## Alternatives Considered

### Alternative 1: Share one provider connection among several conversations

- **Description**: Run multiple ACP sessions over a single provider connection.
- **Pros**: Fewer child processes and lower idle resource use.
- **Cons**: Requires a new multiplexing/concurrency contract, couples failures, and departs from Kitten's existing per-session isolation.
- **Why rejected**: It adds significant concurrency risk before the tab lifecycle is proven.

### Alternative 2: Reuse a fixed configuration-seeded runtime pool

- **Description**: Limit tabs to a pre-created set of configured sessions.
- **Pros**: Preserves the existing controller plan and minimizes runtime changes.
- **Cons**: Cannot support dynamic fresh conversations or restoration of user-created tabs.
- **Why rejected**: It contradicts the PRD's fresh-tab and full-workspace requirements.

## Consequences

### Positive

- Reuses Kitten's established one-conversation-one-runtime resilience model.
- Keeps agent failures, permissions, cancellation, and event streams attributable to a stable conversation ID.
- Lets dynamic tabs inherit familiar provider and working-directory behavior without mutating user configuration.

### Negative

- More open tabs consume more agent processes and connections.
- The controller gains per-conversation startup, teardown, and registry lifecycle responsibilities.
- The empty-workspace new-tab default must degrade gracefully when no provider is available.

### Risks

- **Leaked subscriptions or child processes:** use one idempotent close promise, dispose each runtime at most once, and ignore late events for a closing or removed conversation.
- **Misattributed permissions:** remove or settle only the pending requests owned by the affected conversation.
- **Creation failure:** commit a `starting` identity only when a descriptor exists, retain it as unavailable after a runtime failure, and return `null` with a workspace notice when the empty workspace has no configured provider.

## Implementation Notes

The controller remains the only owner of connections and ACP effects. Core and store code remain protocol-free; UI components invoke the new behavior exclusively through ControllerActions.
