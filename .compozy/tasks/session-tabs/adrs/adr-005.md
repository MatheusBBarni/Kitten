# ADR-005: Gate Requested Tab Chords on Kitty Keyboard Events and Retain Sessions Fallback

## Status

Accepted

## Date

2026-07-12

## Context

The PRD requests Ctrl+H and Ctrl+L for previous and next tab navigation, but legacy terminal input encodes Ctrl+H as the same control byte commonly used for Backspace and Ctrl+L as a redraw or clear control byte. Kitten's integrated shell forwards control bytes to its PTY whenever the shell owns focus. Existing global bindings are intentionally sparse, and modal overlays already suppress global input.

OpenTUI supports enabling the Kitty keyboard protocol for disambiguated modifier reporting. Current tests expose a Kitty event source, but the current renderer bootstrap does not enable the protocol and the app has no tab-key capability policy. The existing keyboard-accessible Sessions overlay is the reliable fallback navigation surface.

## Decision

Use the requested Ctrl+H and Ctrl+L tab chords only for Kitty-protocol key events, and retain the Sessions overlay as the universal fallback.

- Configure the cockpit renderer to request Kitty keyboard disambiguation and alternate-key reporting.
- Initialize ephemeral keyboard capability as `unknown`. Promote it to `kittyConfirmed` only after the renderer boundary observes a valid Kitty-source event; do not infer support from terminal names or environment variables.
- Extend the keymap capability boundary so the previous/next tab commands match only when capability is `kittyConfirmed` and the current event is explicitly reported through the Kitty protocol.
- Never intercept these chords while a modal overlay or the integrated shell owns input.
- Until capability is confirmed, do not advertise or claim the requested chords; expose the existing keyboard-accessible Sessions overlay and next-attention path as the fallback.
- Keep no user-configurable shortcut system and no second universal global chord in V1.
- Test both qualifying Kitty events and legacy/non-qualifying key events.

## Alternatives Considered

### Alternative 1: Bind Ctrl+H and Ctrl+L unconditionally

- **Description**: Treat every Ctrl+H/L event as tab navigation when the agent pane is focused.
- **Pros**: Meets the requested key labels on more terminals.
- **Cons**: Risks consuming Backspace or redraw input and makes prompt editing unreliable.
- **Why rejected**: It violates the PRD's safe-fallback requirement.

### Alternative 2: Add a second universal global chord

- **Description**: Introduce a new fallback chord for all terminals.
- **Pros**: Direct adjacent tab navigation without opening the Sessions overlay.
- **Cons**: Adds another global collision surface and a discovery burden before V1 demand is validated.
- **Why rejected**: The existing overlay already provides a keyboard path and attention routing.

### Alternative 3: Add user-configurable shortcuts in V1

- **Description**: Build a shortcut preference system alongside tabs.
- **Pros**: Gives power users control.
- **Cons**: Broadens settings, persistence, help, conflict detection, and validation scope.
- **Why rejected**: It is explicitly deferred by the PRD.

## Consequences

### Positive

- Protects prompt editing and the integrated shell from ambiguous control bytes.
- Keeps global key handling consistent with modal and shell ownership rules.
- Makes the requested chords available where the input protocol can actually distinguish them.

### Negative

- Some terminals will use the Sessions overlay rather than direct adjacent chords.
- Renderer bootstrap, ephemeral AppStore capability state, and keymap tests must model keyboard capability explicitly.
- Help and hints must communicate conditional availability accurately.

### Risks

- **Incorrect capability signal:** require both persistent Kitty confirmation and the current Kitty-source event, never terminal-name heuristics, and cover unknown, confirmed, and legacy paths in tests.
- **Shortcut leakage:** preserve overlay and shell focus guards before tab command dispatch.
- **Discoverability gap:** render a clear fallback hint whenever direct tab chords are unavailable.

## Implementation Notes

The keymap remains the single source of truth for matching, help text, and command dispatch. Renderer input promotion is the only writer of the ephemeral capability state; UI components must not install ad hoc keyboard listeners for global tab navigation.
