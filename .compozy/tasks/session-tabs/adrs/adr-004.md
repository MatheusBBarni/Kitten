# ADR-004: Separate Workspace Metadata from Session State and Persist a Versioned Workspace

## Status

Accepted

## Date

2026-07-12

## Context

SessionState is the protocol-free execution record written only by the session reducer. It stores an ACP-backed conversation's transcript, status, working directory, provider identity, and agent-advertised capabilities. It does not model user-owned lifecycle, display order, background state, attention acknowledgement, or a valid empty workspace.

The current app store assumes at least one configuration-seeded session and the version-1 saved-run record persists only focused session identity plus light per-session ACP metadata. That cannot restore dynamic conversations, renamed tabs, background-running work, or an empty visible workspace.

## Decision

Create a separate pure workspace model in the core and persist it as a version-2 saved-run contract.

- Keep ACP-derived execution state in SessionState and its existing pure session reducer.
- Add a protocol-free workspace reducer and metadata keyed by SessionId. It owns lifecycle (visible, background-running, closed), user display name, creation/display order, nullable selected visible conversation, and attention acknowledgement.
- Store session execution records and workspace metadata separately, and update them atomically through AppStore methods.
- Derive the attention queue from current SessionStatus plus workspace lifecycle and acknowledgement. Selecting a conversation records acknowledgement without clearing its underlying execution status.
- Persist a version-2 run record with top-level dynamic conversation descriptors and ACP resume metadata only. Persist lifecycle, display names, order, attention acknowledgement, and nullable selected conversation only inside `workspace`, making it the single user-metadata source of truth.
- Validate both version 1 and version 2 records. Migrate version-1 records at restore time by deriving visible workspace order from matching resolved configuration descriptors and treating the legacy `PersistedAgent.sessionId` as the ACP resume pointer; preserve the legacy focused ID only when it maps to a migrated visible conversation.
- Never write user-owned tab metadata back into the configuration file.
- Exclude closed conversations from snapshots and restore. Preserve background-running conversations and their attention eligibility.

## Alternatives Considered

### Alternative 1: Add all tab fields to SessionState

- **Description**: Store lifecycle, display name, order, and acknowledgement directly with ACP-derived session state.
- **Pros**: Fewer top-level structures and direct lookup from a session record.
- **Cons**: Mixes user workspace policy with agent-event state, complicates the single-writer reducer, and makes lifecycle changes look like ACP events.
- **Why rejected**: It weakens the core boundary between agent execution and user-owned workspace state.

### Alternative 2: Keep tab metadata in React state only

- **Description**: Let the tab strip own lifecycle and names locally.
- **Pros**: Small initial implementation.
- **Cons**: Breaks persistence, background attention routing, controller actions, and narrow store subscriptions.
- **Why rejected**: It cannot satisfy the PRD's restore and lifecycle requirements.

### Alternative 3: Reject version-1 saved runs

- **Description**: Require users to start fresh after the new workspace model lands.
- **Pros**: Simplifies the new decoder.
- **Cons**: Breaks established resume behavior and discards usable ACP pointers.
- **Why rejected**: The user chose migration into visible tabs.

## Consequences

### Positive

- Preserves core purity and keeps tab lifecycle independently testable.
- Makes an empty workspace a valid state rather than a malformed session collection.
- Restores user-created visible and background-running conversations without config mutation.
- Maintains compatibility for existing saved runs.

### Negative

- Adds a versioned persistence schema and migration coverage.
- Requires selectors and writer code to handle nullable visible focus.
- Requires explicit synchronization between execution status transitions and workspace attention metadata.

### Risks

- **Migration mismatch:** migrate only version-1 entries with a matching resolved configuration descriptor; safely omit unmatched legacy pointers while preserving independent siblings and an empty selection when necessary.
- **Late events after close:** use an idempotent teardown state, remove runtime subscriptions before deleting execution state, and ignore events for closing or unknown IDs.
- **State drift:** keep all workspace transitions in one pure reducer, make `workspace` the only persisted user-metadata source, and atomically commit session/workspace updates.

## Implementation Notes

The persistence layer validates record shape and the controller supplies resolved configuration descriptors only for version-1 migration. Version-2 restoration is record-driven: a persisted descriptor without an available provider remains an unavailable conversation without blocking its siblings.
