# ADR-002: Make Prompt Recall Visible and Collapse Adjacent Duplicates

## Status

Accepted

## Date

2026-07-12

## Context

Prompt recall must be understandable at a glance. A user who is browsing earlier prompts needs to know that the composer is showing a recalled entry rather than their ordinary draft. Repeatedly sending the same unchanged prompt should not make the short recall list noisy. The primary product outcome is faster recall, editing, and resubmission.

## Decision

Show a compact position indicator while the user is browsing recalled prompts, such as `History 2/5`. Collapse consecutive identical submitted prompts into one history entry. Measure V1 primarily by whether users recall, edit, and resend prompts faster.

## Alternatives Considered

### Alternative 1: Silent recall

- **Description**: Replace the composer text with recalled prompts without any visible state.
- **Pros**: Keeps the interface visually minimal.
- **Cons**: Makes it easy to mistake recalled text for an ordinary draft and obscures the clear-on-Down boundary.
- **Why rejected**: The narrow indicator gives users confidence without adding a separate browsing surface.

### Alternative 2: Preserve every repeated submission

- **Description**: Add a new history entry for every send, including adjacent identical text.
- **Pros**: Represents an exact activity log.
- **Cons**: Adds noise without helping the immediate reuse workflow.
- **Why rejected**: V1 is a recall convenience, not an audit trail.

## Consequences

### Positive

- Users can orient themselves while moving through history.
- The recall list remains concise and useful during rapid retries.
- Measurement aligns with the user value hypothesis rather than raw feature exposure.

### Negative

- The indicator adds a small amount of visual information to the composer.
- Collapsing duplicates means history does not represent every individual send.

### Risks

- **Indicator is overlooked**: use concise, stable wording near the composer.
- **Users expect audit history**: document that V1 retains a short recall list, not a complete record.
- **Raw usage is mistaken for value**: evaluate edit-and-resend outcomes, not only recall activation.

## Implementation Notes

- This decision specifies user-visible behavior and measurement only; implementation belongs in the TechSpec.
