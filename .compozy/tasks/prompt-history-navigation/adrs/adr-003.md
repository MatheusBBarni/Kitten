# ADR-003: Store Bounded Prompt History in Each Session Slice

## Status

Accepted

## Date

2026-07-12

## Context

Prompt history must survive focus changes between live agent sessions, disappear when a run is replaced, and remain observable through the same immutable state model that owns each session. The feature also needs deterministic navigation, adjacent-duplicate collapsing, and a fixed capacity of 50 prompts.

## Decision

Add a `PromptHistoryState` to every `SessionState`. It holds up to 50 chronological prompt entries and a nullable recall cursor. Drive it through a pure prompt-history reducer called from `sessionReducer`. The composer records a nonblank prompt when Kitten accepts the local submission, before the asynchronous agent call settles. Consecutive identical entries collapse into one.

## Alternatives Considered

### Alternative 1: Keep a map in `PromptEditor`

- **Description**: Store history and cursor state in a component ref keyed by session id.
- **Pros**: Smaller initial diff and no core event changes.
- **Cons**: Session lifecycle, focus restoration, and test ownership become implicit UI behavior.
- **Why rejected**: The history is session domain state, not an incidental view cache; it needs the store’s explicit lifecycle and pure tests.

### Alternative 2: Derive recall directly from transcript turns

- **Description**: Scan session turns every time the user presses an arrow.
- **Pros**: Avoids a new state field.
- **Cons**: Includes non-composer turns, has no bounded retention policy, and mixes transcript display with recall semantics.
- **Why rejected**: V1 needs a separate, bounded composer-only history with duplicate and cursor rules.

## Consequences

### Positive

- Per-session ownership and new-run cleanup follow existing store behavior.
- Ring capacity, duplicate handling, and navigation are deterministic and unit-testable.
- UI subscribers can derive the history indicator from one immutable session slice.

### Negative

- `SessionState`, domain events, selectors, controller actions, and test doubles gain a small new contract.

### Risks

- **Transcript leakage into history**: record only composer-originated submissions.
- **Unbounded growth**: trim at the fixed 50-entry cap on every accepted record.
- **Mutation regressions**: keep all transitions in a pure reducer with immutable outputs.

## Implementation Notes

- Entries remain in memory only and are never serialized to config, saved runs, or telemetry.
