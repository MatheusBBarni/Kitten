# ADR-004: Keep Composition Metadata Ephemeral and Telemetry-Only

## Status

Accepted

## Date

2026-07-16

## Context

The composition result is useful for bounded diagnosis, but the existing persisted delivery checkpoint intentionally stores only lifecycle state. Persisting fragment identifiers or capability facts would expand saved-run records, create migration work, and increase the chance of coupling hidden guidance to restored conversations.

The product requires content-free observability: no prompt text, task content, repository content, paths, credentials, bridge endpoints or tokens, ACP identifiers, provider recipes, environment values, or raw errors may be recorded.

## Decision

Keep the full composition result only in the live controller runtime for the current fresh generation. Emit optional, content-free telemetry only after final composition using the existing closed telemetry seam. The record may contain contract version, selected static fragment IDs, selected count, and whether the result is base-only.

Do not add composition metadata to persisted run records in V1. A process restart or replacement recomputes a new fresh-generation result from current authoritative facts; a successful loaded session remains harness-free.

## Alternatives Considered

### Alternative 1: Persist selected fragment identifiers in the delivery checkpoint

- **Description**: Extend saved-run records with the selected static IDs for post-restart inspection.
- **Pros**: Gives diagnostic continuity across restarts.
- **Cons**: Requires a record-schema migration, widens the persistence surface, and risks confusing historical selection with current capability truth.
- **Why rejected**: V1 needs only current-generation debugging; fresh replacement already recomputes the authoritative result.

### Alternative 2: Keep no composition metadata

- **Description**: Render and send the harness without any diagnostic record.
- **Pros**: Smallest privacy surface.
- **Cons**: Makes safe rollout and diagnosis of base-only fallback difficult.
- **Why rejected**: Bounded static identifiers and counts provide useful evidence without retaining content.

## Consequences

### Positive

- Preserves the strict content-free persistence contract and avoids a schema migration.
- Keeps diagnostic data tied to current runtime truth rather than stale historical state.
- Supports rollout metrics without exposing hidden guidance or sensitive session details.

### Negative

- Detailed composition diagnostics disappear when the process exits.
- Maintainers cannot use saved runs to reconstruct a historical optional-fragment selection.

### Risks

- **Telemetry field creep** — Use an explicit closed record shape and negative assertions for prohibited content.
- **Misinterpreted base-only result** — Include a fixed base-only outcome flag and document it as a valid state, not an error.

## Implementation Notes

Reuse the existing opt-in telemetry recorder. Do not create a new storage mechanism or change persisted run-record versions for V1.
